## 1. Number Formatting ‚Äî Decimal Separators & Digit Grouping

Locale-dependent number formatting is governed by Unicode CLDR via `Intl.NumberFormat` in JavaScript.

| Locale | Decimal Separator | Grouping Separator | Example (123456.789) |
|--------|-------------------|-------------------|----------------------|
| `en-US` | `.` | `,` | `123,456.789` |
| `de-DE` | `,` | `.` | `123.456,789` |
| `pt-BR` | `,` | `.` | `123.456,789` |
| `en-IN` | `.` | `,` (lakh/crore) | `1,23,456.789` |
| `ar-EG` | `Ÿ´` (Arabic) | `Ÿ¨` | `Ÿ°Ÿ¢Ÿ£Ÿ¨Ÿ§Ÿ•Ÿ¶Ÿ´ŸßŸ®Ÿ©` |
| `ja-JP` | `.` | `,` | `123,456.789` |

**Key patterns from CLDR (`<symbols>`)**:
- `decimal` ‚Äî separates integer/fraction parts
- `group` ‚Äî separates digit clusters (primary grouping size varies: 3 for most, 4 for some CJK locales, irregular for `en-IN`)
- `currencyDecimal` / `currencyGroup` ‚Äî optional overrides for currency context (allows different separators for currency vs plain numbers)
- India uses irregular grouping: `1,23,456` ‚Äî first group is 3 digits, subsequent groups are 2 digits (lakh/crore system)

**Implementation**: Use `Intl.NumberFormat` with `useGrouping: true` (default). For financial precision, specify `minimumFractionDigits` and `maximumFractionDigits` explicitly rather than relying on defaults.

## 2. Currency Formatting ‚Äî Symbol Placement & Negative Display

### Symbol placement varies by locale and currency

From CLDR currency format patterns:
```
en-US:  $1,234.57     (symbol before, no space)
de-DE:  1.234,57 ‚Ç¨    (symbol after, no space)
de-AT:  ‚Ç¨ 1.234,57    (symbol before, with space)
ja-JP:  ¬•123,457      (symbol before, no space, no minor units for JPY)
pt-BR:  R$ 1.234,57   (symbol before, with space)
```

### CLDR currency pattern types

From Unicode LDML Part 3:
```xml
<currencyFormat type="standard">
  <pattern>¬§#,##0.00</pattern>           <!-- standard: -$1,000.00 -->
  <pattern alt="alphaNextToNumber">¬§ #,##0.00</pattern>  <!-- when symbol ends with letter -->
  <pattern alt="noCurrency">#,##0.00</pattern>  <!-- column of same-currency values -->
</currencyFormat>
<currencyFormat type="accounting">
  <pattern>¬§#,##0.00;(¬§#,##0.00)</pattern>  <!-- accounting: ($1,000.00) -->
</currencyFormat>
```

The `alt="alphaNextToNumber"` variant is critical: when the narrow currency symbol ends with a letter (e.g., `CA$`, `kr`), a non-breaking space is inserted before the number. This prevents `CA$1,000` from looking like part of the number.

### Negative currency display modes (from ICU Number Skeletons)

| Skeleton | Display | Use case |
|----------|---------|----------|
| `sign-auto` | `-1,000` (only negatives show sign) | Default for most numbers |
| `sign-always` / `+!` | `+1,000` / `-1,000` | When sign position matters in columns |
| `sign-accounting` / `()` | `(1,000)` for negatives | **Accounting format** ‚Äî standard in Western finance |
| `sign-accounting-always` / `()!` | `+1,000` / `(1,000)` | Always show sign in accounting |
| `sign-except-zero` / `+?` | `+1,000` / `0` | Show sign for non-zero only |
| `sign-accounting-except-zero` / `()?` | `+1,000` / `(1,000)` for neg / `0` | Accounting without zero sign |

### Currency codes vs symbols

CLDR defines 5 widths for currency symbols:
- `¬§` (standard), `¬§¬§` (ISO code like `USD`), `¬§¬§¬§¬§¬§` (narrow symbol like `$`)
- Use `¬§¬§` (ISO code) when the narrow symbol is ambiguous (e.g., `$` could be USD, CAD, AUD)
- `currencyPatternAppendISO`: `{0} ¬§¬§` produces `$1,432.00 USD`

### Financial date conventions

## 4. RTL Support ‚Äî Arabic & Hebrew Financial Layouts

From next-intl documentation and CLDR:

### Requirements for proper RTL financial UI

1. **`dir` attribute**: Set `dir="rtl"` on container elements for Arabic, Hebrew, Persian
2. **CSS logical properties**: Use `margin-inline-start` instead of `margin-left`, `padding-inline-end` instead of `padding-right`
3. **Number rendering**: Numbers in RTL text are still LTR (bidirectional algorithm handles this), but currency symbols may need repositioning
4. **Table alignment**: Financial tables in RTL should align numeric columns to the right (start in RTL context), text columns to the left (end in RTL context)
5. **Charts**: Bar charts should mirror horizontally; line charts may not need mirroring (values still read left-to-right)

### Arabic number system considerations

CLDR supports multiple numbering systems per locale:
```xml
<!-- Arabic with Western digits -->
<html lang="ar" dir="rtl">  <!-- Western digits in financial context -->
<!-- Arabic with native digits -->
<html lang="ar-u-nu-arab">  <!-- Arabic-Indic digits: ŸÝŸ°Ÿ¢Ÿ£Ÿ§Ÿ•Ÿ¶ŸßŸ®Ÿ© -->
```

Financial platforms typically use Western Arabic digits (0-9) even in RTL contexts for clarity, but the UI chrome (labels, headings) uses RTL text.

### Bidi text in financial data

CLDR provides `approximatelySign` (`~`), `range` patterns (`{0}‚Äì{1}`), and currency spacing rules that are built to work visually in bidi contexts. The range collapsing algorithm explicitly accounts for visual reordering in bidi.

## 5. Multi-Language Architecture ‚Äî ICU MessageFormat

### ICU MessageFormat for financial plurals

From next-intl/FormatJS (both built on ICU Message syntax):

```json
{
  "transaction_count": "You have {count, plural, =0 {no transactions} =1 {one transaction} other {# transactions}}.",
  "balance_negative": "Your balance is {amount, number, ::sign-accounting currency/USD}.",
  "overdraft_fee": "Overdraft fee: {fee, number, currency} per {days, plural, =1 {day} other {# days}}."
}
```

### Number skeletons in ICU messages

From ICU documentation, number skeletons use `::` prefix in messages:
```
{0, number, ::currency/USD}                    ‚Üí $1,234.57
{0, number, ::percent}                         ‚Üí 25%
{0, number, ::sign-accounting currency/EUR}    ‚Üí (‚Ç¨1,234.57)
{0, number, ::.##}                             ‚Üí 1,234.5 (at most 2 fraction digits)
{0, number, ::@@@}                             ‚Üí 123 (exactly 3 significant digits)
```

### ICU skeleton key financial formats

| Skeleton | Purpose | Example |
|----------|---------|---------|
| `currency/USD` | Format as USD | `$1,000.00` |
| `sign-accounting` | Parentheses for negatives | `(1,000.00)` |
| `compact-short` | Abbreviated | `$5K` |
| `precision-currency-standard` | Currency-standard precision | `$10.00` |
| `precision-currency-cash` | Cash rounding (e.g., CHF 0.05) | `$10.05` |
| `group-min2` | Group only if 4+ digits | `5000` vs `15,000` |
| `scale/100` | Multiply before format | For percentage conversion |

### String externalization patterns

Best practice architecture:
1. **Separate translation files per locale**: `en.json`, `pt-BR.json`, `ar.json`, `ja.json`
2. **Nested key structure**: Group by domain (`transactions.balance`, `invoices.tax`)
3. **ICU format strings in values**: Keep format logic in message strings, not in code
4. **Named parameters**: `{amount}` not `{0}` for maintainability
5. **Locale-specific format overrides**: Use `formats` config for reusable number/date patterns

## 6. Timezone Handling

### UTC storage vs local display

**Rule**: Store all timestamps as UTC (or with explicit timezone offset). Format for display using user's local timezone or the transaction's originating timezone.

From MDN `Intl.DateTimeFormat`:
```js
new Intl.DateTimeFormat('en-US', {
  timeZone: 'America/New_York',
  dateStyle: 'full',
  timeStyle: 'long'
}).format(date);
// "Sunday, December 20, 2020 at 10:23:16 AM EST"
```

### Fiscal period boundaries across timezones

- A transaction at `2026-01-01T00:30:00+01:00` (Paris) is `2025-12-31T23:30:00Z` (UTC) ‚Äî different fiscal day in UTC vs local
- **Decision**: Fiscal periods should be determined by the **local time of the transaction origin**, not UTC
- Server must store the original timezone alongside the UTC timestamp
- Fiscal year boundaries per jurisdiction must be configurable (not hardcoded)

### Timezone configuration for servers

- Application servers should run in UTC
- Display formatting always uses `Intl.DateTimeFormat` with explicit `timeZone` option
- Never use `new Date().toLocaleString()` without specifying timezone ‚Äî the server's timezone is meaningless for global users
- Use IANA timezone identifiers (`America/Sao_Paulo`, not `BRT`)

## 7. Locale-Specific Accounting Formats

### Brazilian Real (R$) formatting

```js
new Intl.NumberFormat('pt-BR', {
  style: 'currency',
  currency: 'BRL'
}).format(1234.56);
// "R$ 1.234,56"
```
- Symbol: `R$` (with space before number)
- Decimal: `,` | Group: `.`
- Negative: `-R$ 1.234,56` (standard) or `(R$ 1.234,56)` (accounting)

### European accounting notation

Parentheses for negatives are standard across Western accounting:
```js
// CLDR accounting pattern for en-US:
// ¬§#,##0.00;(¬§#,##0.00)
new Intl.NumberFormat('en-US', {
  style: 'currency',
  currency: 'USD',
  currencySign: 'accounting'
}).format(-1234.56);
// "($1,234.56)"
```

### Japanese Yen (¬•) formatting

```js
new Intl.NumberFormat('ja-JP', {
  style: 'currency',
  currency: 'JPY'
}).format(123456);
// "Ôø•123,457"  (no minor units ‚Äî JPY has 0 decimal places)
```
- JPY and KRW have 0 minor units in ISO 4217
- CLDR `precision-currency-cash` can handle cash rounding for currencies like CHF (0.05 increment)

### Chinese financial numerals

CLDR defines a `finance` numbering system for Chinese to prevent fraud (traditional characters harder to alter):
```xml
<numberingSystem id="hant" type="algorithmic" rules="zh_Hant/SpelloutRules/spellout-cardinal"/>
```
Use `zh-u-nu-finance` locale for financial Chinese display.

### Indian Rupee grouping

```js
new Intl.NumberFormat('en-IN', {
  style: 'currency',
  currency: 'INR'
}).format(1234567.89);
// "‚Çπ12,34,567.89"  (lakh/crore grouping: 3, then 2s)
```

## 8. Tax Localization

### VAT display variations

| Region | Convention | Display |
|--------|-----------|---------|
| EU (most) | Tax-inclusive pricing (consumer) | `‚Ç¨100.00` (VAT included) |
| EU (B2B) | Tax-exclusive + VAT line | `‚Ç¨82.64 + ‚Ç¨17.36 VAT` |
| US | Tax-exclusive (added at checkout) | `$100.00 + $8.00 tax` |
| Japan | Tax-inclusive (since 2021 reform for B2C) | `¬•1,100 (Á®éËæº)` or `¬•1,000 (Á®éÊäú)` |
| Brazil | Tax-inclusive (ICMS/ISS/PIS/COFINS embedded) | `R$ 100,00` (taxes embedded in price) |

### Brazilian tax complexity

Brazil has multiple overlapping taxes:
- **ICMS** (state): 7-25% depending on state and product
- **ISS** (municipal): 2-5% for services
- **PIS/COFINS** (federal): ~9.25% combined
- Display must show the total with a note that taxes are embedded, or break down by tax type for B2B invoices

### Tax display format patterns

Financial platforms need:
1. **Toggle between inclusive/exclusive** display (user preference or jurisdiction default)
2. **Tax breakdown line items** for invoices (varying per jurisdiction)
3. **Rounding rules**: Tax amounts may have jurisdiction-specific rounding (e.g., Germany rounds tax to nearest cent, some jurisdictions round to nearest whole unit)

## 9. i18n Framework Comparison for Financial Formatting

### FormatJS (react-intl)

- **Built on**: ICU MessageFormat + ECMAScript Internationalization API (ECMA-402)
- **Financial formatting**: Direct access to `Intl.NumberFormat` options; ICU skeletons in messages
- **Strengths**: Industry standard (50M+ npm downloads/month), used by Yahoo, Coinbase, Ethereum, Mozilla; polyfills for older browsers; native Rust implementation for server-side performance
- **Currency support**: Full CLDR currency data; `formatNumber` with `style: 'currency'`; `currencySign: 'accounting'` for parenthesized negatives
- **Date support**: Full `Intl.DateTimeFormat` access with timezone control
- **RTL**: Delegates to CSS/HTML `dir` attribute; no built-in RTL layout mirroring

### next-intl

- **Built on**: ICU MessageFormat + `Intl` APIs, Next.js App Router native
- **Financial formatting**: `useFormatter().number()` with named global formats; ICU number skeletons in messages (`{price, number, ::currency/EUR}`)
- **Strengths**: First-class Next.js integration; server component support; TypeScript augmentation; named reusable formats
- **Currency support**: Same underlying `Intl.NumberFormat` options; global format presets reduce boilerplate
- **RTL**: Explicit guidance ‚Äî set `dir` attribute, use CSS logical properties, mirror icons
- **Plural rules**: Uses `Intl.PluralRules` ‚Äî supports Arabic's 6 plural forms (zero, one, two, few, many, other)

### Recommendation for cashflow platform

**Use `formatjs` (react-intl) as the core formatting library** for these reasons:
1. **Rust native implementation** aligns with ATLAS's Rust-first direction (D-022)
2. **Framework-agnostic** ‚Äî works with Next.js, React, Vue, Ember, or plain Node.js
3. **Polyfill support** for older environments (still relevant for financial enterprise clients)
4. **ICU MessageFormat** is the same standard both libraries use ‚Äî message portability
5. **Larger ecosystem** ‚Äî more community examples, tooling, and enterprise adoption

**Use `next-intl` only if** the platform is exclusively Next.js App Router and you want the tightest framework integration with named format presets.

**In both cases**: The actual number/date formatting comes from `Intl.NumberFormat` / `Intl.DateTimeFormat` (ECMA-402). The libraries differ in how they expose ICU message syntax and integrate with React frameworks ‚Äî the formatting engine is the same CLDR data.

## Summary of Key Architecture Decisions

1. **Store**: Amounts as integers (cents/smallest unit) + currency code (ISO 4217) + UTC timestamp + originating timezone
2. **Format**: Via `Intl.NumberFormat` with locale-specific options from CLDR; never hand-craft format strings
3. **Messages**: ICU MessageFormat with number skeletons for inline formatting
4. **Accounting mode**: Use `currencySign: 'accounting'` / `sign-accounting` skeleton for parenthesized negatives
5. **RTL**: CSS logical properties + `dir` attribute; numbers stay LTR within bidi context
6. **Fiscal periods**: Configurable per jurisdiction; determined by local time of transaction, not UTC
7. **Tax display**: Configurable inclusive/exclusive toggle; jurisdiction-specific breakdown templates
8. **Locale data**: Let CLDR drive all formatting decisions; never hardcode `.` or `,` as decimal separator
