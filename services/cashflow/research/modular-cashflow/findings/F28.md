# F28: Real-Time Collaboration Patterns for Financial Platforms

**Research date**: 2026-07-09
**Angle**: Concurrent editing, record locking, multi-user period closing, conflict resolution, presence indicators, collaborative review, offline collaboration, permission-based collaboration

---

## 1. OT vs CRDTs for Financial Data

**Finding**: CRDTs are the right model for financial platforms; OT requires a central server and is unsuitable for offline-capable systems.

**OT (Operational Transformation)**:
- Requires a central server to serialize operations and resolve conflicts.
- Works for free-text editing (Google Docs, Figma) where character-level merging is natural.
- Poor fit for structured financial records: a journal entry has debit/credit constraints that cannot be resolved by transforming character positions.
- No offline support without a server.

**CRDTs (Conflict-free Replicated Data Types)**:
- Automatically merge concurrent edits without coordination.
- Two dominant libraries: **Yjs** (fastest, largest ecosystem, `Y.Map`/`Y.Array` for structured data) and **Automerge** (local-first, version-controlled, stronger consistency guarantees for structured documents).
- For structured financial records (journal entries, ledger rows), use **CRDT Maps** (`Y.Map` in Yjs, Automerge objects) where each field (account, amount, date) is a separate key. Concurrent edits to different fields merge cleanly.
- For free-text fields (descriptions, notes, comments), use **CRDT Text** (`Y.Text`) which supports character-level collaboration.
- **Key insight**: Financial platforms need a **hybrid approach** — CRDT Maps for structured record fields, CRDT Text for free-text fields, with server-side validation to enforce accounting invariants (debits = credits, account existence).

**Libraries**:
- **Yjs**: `yjs` (core), `y-websocket` (provider), `y-indexeddb` (offline persistence), `y-leveldb` (server persistence). Y.Map, Y.Array, Y.Text shared types. WebSocket or WebRTC providers.
- **Automerge**: `automerge` (core), automerge-repo (sync layer), automerge-repo-react-hooks. Works with React out of the box. Compressed columnar storage.

**Recommendation**: Use **Yjs** for the collaboration layer due to its performance benchmarks (2-10x faster than Automerge) and mature ecosystem. Reserve Automerge for scenarios requiring full document versioning/history.

## 2. Record Locking Strategies for PostgreSQL

**Finding**: A tiered locking strategy combining optimistic locking at the application layer with PostgreSQL advisory locks for fiscal periods.

### 2a. Optimistic Locking (Version Check at Save)

- Add a `version INTEGER` column to every editable table.
- On read: `SELECT ... FOR UPDATE` or just read the version.
- On update: `UPDATE ... SET version = version + 1 WHERE id = $1 AND version = $2`. If affected rows = 0, the record was modified by another user — reject with a conflict.
- **Best for**: Individual journal entries, line items, account records where conflicts are rare and short-lived.
- **Tradeoff**: Users get no warning until they try to save. Good UX requires polling for remote changes.

### 2b. Pessimistic Locking (Lock Row Before Edit)

- `SELECT ... FOR UPDATE` acquires a row-level lock. Other transactions attempting `UPDATE/DELETE/SELECT FOR UPDATE` on the same row block until the lock is released.
- PostgreSQL row lock modes: `FOR UPDATE`, `FOR NO KEY UPDATE`, `FOR SHARE`, `FOR KEY SHARE`. `FOR UPDATE` blocks all writers; `FOR SHARE` allows concurrent readers.
- **Best for**: Critical operations where conflict resolution UX is unacceptable (e.g., period close, reconciliations).
- **Risk**: Holding locks too long (e.g., while user is thinking) degrades performance. PostgreSQL docs explicitly warn: "it is a bad idea for applications to hold transactions open for long periods of time (e.g., while waiting for user input)."

### 2c. Advisory Locks for PostgreSQL

- `pg_advisory_lock(key)` / `pg_advisory_unlock(key)` — application-level locks without touching table rows.
- Two variants: **session-level** (held until explicit release or session end, survives transaction rollback) and **transaction-level** (auto-released at transaction end).
- Faster than table locks, avoid table bloat, automatically cleaned up on session end.
- Upper limit: tens to hundreds of thousands of advisory locks depending on `max_locks_per_transaction` and `max_connections`.
- **Best for**: Fiscal period locks (`pg_advisory_lock(period_id)`), module locks, preventing concurrent close operations.

### 2d. Per-Period Locking Pattern

- Use advisory locks keyed by `period_id` to prevent concurrent period-close operations.
- Use optimistic locking (version column) for individual record edits within an open period.
- Use `SERIALIZABLE` isolation for consistency-critical operations (balance checks, reconciliations).

**Recommendation**: Implement a **3-tier locking strategy**:
1. **Optimistic locking** (version columns) for routine record editing.
2. **Advisory locks** for fiscal period operations (open/close/reopen).
3. **Serializable transactions** for cross-record consistency checks (balance validation, period-end reports).

## 3. Multi-User Period Closing

**Finding**: Period closing requires a sequential workflow with distributed ownership, not a single-owner model.

### Period Closing Architecture

**Fiscal period states**: `draft` → `open` → `closing` → `closed` → `locked`

**Multi-user closing workflow**:
1. **Pre-close checklist**: Each module (AP, AR, GL, payroll) has its own checklist. Accountants work on their assigned module in parallel.
2. **Module completion**: When an accountant marks their module "ready to close," it transitions to a review state. A supervisor reviews and approves.
3. **Final close**: Once all modules are approved, a designated user (typically the controller) executes the final period close. This acquires an advisory lock on the period.
4. **Post-close lock**: After closing, set period status to `locked`. All editing of closed-period records is blocked at the application layer.

**Conflict prevention during closing**:
- Advisory lock on `period_id` during the close operation itself (brief, seconds).
- Application-level status check on every write: `if period.status == 'closed' then reject`.
- Real-time broadcast when a module marks "ready to close" so other users know the period is transitioning.

**Approval workflows**:
- Each module close has an approval chain (accountant → supervisor → controller).
- Approval state is stored per-module-per-period: `{ period_id, module, status, approved_by, approved_at }`.
- Period cannot be fully closed until all modules have `approved` status.

**Libraries/patterns**:
- **State machine** (XState, Rust `statig`, or custom): Model period lifecycle with explicit transitions and guards.
- **Event sourcing**: Record every close-related action as an immutable event for audit trail.

## 4. Conflict Resolution Strategies

**Finding**: Financial data requires user-resolved conflicts with a complete audit trail — never last-write-wins.

### Why Last-Write-Wins is Dangerous for Financial Data

- LWW silently drops one user's changes, which is unacceptable for accounting data where every modification must be traceable.
- Financial regulations require a complete audit trail of who changed what, when, and why.

### Recommended Conflict Resolution

**Tier 1: CRDT-based automatic merge (for independent fields)**
- When two users edit different fields of the same record (e.g., one edits the description, another edits an amount), CRDTs merge automatically without conflict.
- No user intervention needed.

**Tier 2: Version-based conflict detection (for same-field edits)**
- When two users edit the same field, the second user receives a "conflict detected" notification.
- Present a merge UI: "Your change: X | Their change: Y | Choose one or merge manually."
- Store the conflict resolution in an audit log: `{ record_id, field, your_value, their_value, chosen_value, resolved_by, resolved_at }`.

**Tier 3: Server-side invariant enforcement**
- After merge, run server-side validation: debits = credits, account exists, period is open.
- If invariants are violated, reject the merged result and surface both changes for manual resolution.

**Audit trail pattern**:
- Every record has an associated `audit_log` table: `{ record_id, field_name, old_value, new_value, user_id, action, timestamp, conflict_id }`.
- `conflict_id` links entries that were part of the same conflict resolution.

## 5. Presence Indicators

**Finding**: Use the Yjs Awareness CRDT (state-based, ephemeral, not persisted) for real-time presence.

### Yjs Awareness Protocol

- **Not persisted**: Awareness state is ephemeral — when a user goes offline, their state is automatically removed and all peers are notified.
- **JSON-encodable fields**: Set any key-value pairs: user name, color, cursor position, current page, active record.
- **Propagated to all users**: Changes are broadcast to all connected peers in real-time.

**Recommended presence fields for financial platforms**:
```json
{
  "user": {
    "id": "user-123",
    "name": "Maria Silva",
    "color": "#ffb61e",
    "avatar": "/avatars/maria.jpg"
  },
  "cursor": {
    "recordId": "je-456",
    "field": "debit_amount",
    "page": "journal_entries"
  },
  "status": "editing" | "viewing" | "idle"
}
```

**UI patterns**:
- **Record-level indicators**: Show colored dots/borders on records being viewed/edited by other users.
- **Field-level cursors**: Show cursor position within specific fields (amount, description).
- **Active users sidebar**: List all users currently in the same period/module with their status.
- **Idle timeout**: After 30-60 seconds of no activity, transition user to "idle" status.

**Implementation**: Use Yjs Awareness CRDT with `provider.awareness` to share presence state. Editor bindings (QuillBinding, ProsemirrorBinding) automatically render remote cursors.

## 6. Collaborative Review Patterns

**Finding**: Implement threaded commenting and approval workflows at the record level.

### Commenting on Journal Entries

- Attach comments to specific records (journal entries, line items) or specific fields.
- Comments should support:
  - Text content (use CRDT Text for real-time collaborative editing of a comment thread)
  - Mentions (@user) to notify specific accountants
  - Resolved/unresolved status
  - Linked audit evidence (file attachments)

### Approval Threads

- Each journal entry can have an approval workflow: `draft` → `submitted` → `reviewed` → `approved` / `rejected`.
- Approval actions are part of the audit trail.
- Rejection includes a reason and links back to the reviewer's comments.

### Audit Notes

- Separate from approval workflow — these are general observations about the financial data.
- Support threading (reply to an audit note).
- Link to specific accounts, periods, or records.

**Pattern**: Model comments as a separate collection (not inside the CRDT document) to avoid CRDT merge complexity for thread structures. Use WebSocket events for real-time comment delivery, with PostgreSQL as the source of truth for comments.

## 7. Offline Collaboration

**Finding**: Yjs with IndexedDB persistence provides robust offline support; sync is automatic when reconnected.

### Yjs Offline Architecture

- **y-indexeddb**: Persists Yjs document state to IndexedDB (browser local storage).
- **Automatic sync**: When the network provider reconnects, only the delta (new changes) is synced — not the entire document.
- **Meshable providers**: Can use WebSocket + WebRTC + IndexedDB simultaneously for maximum reliability.
- **Conflict-free**: CRDTs guarantee convergence regardless of sync order.

### Offline Collaboration Flow

1. User goes offline. Local changes are persisted to IndexedDB.
2. Other users continue editing (syncing via server or WebRTC).
3. User reconnects. y-indexeddb replays local changes, network provider syncs remote changes.
4. CRDTs automatically merge all changes. No conflicts.

### Financial-Specific Offline Considerations

- **Read-only access offline**: For critical operations (period close), restrict to online-only to ensure real-time visibility.
- **Sync validation**: After offline sync, run server-side invariant checks (balance validation). If invariants are violated, flag the record for manual review.
- **Audit trail**: Offline changes must be attributed to the correct user and timestamp. Store the original client timestamp alongside the sync timestamp.

**Limitations**:
- Advisory locks (period close) cannot work offline — these require server coordination.
- Optimistic locking (version checks) will detect conflicts after sync, but the user may be offline when the conflict is detected. Resolution happens on next sync.

## 8. Permission-Based Collaboration

**Finding**: Implement row-level security (RLS) at the database layer combined with application-level permission checks.

### PostgreSQL Row-Level Security

- Enable RLS on tables: `ALTER TABLE journal_entries ENABLE ROW LEVEL SECURITY;`
- Create policies based on user role and period assignment:
  ```sql
  CREATE POLICY period_access ON journal_entries
    FOR ALL
    USING (
      period_id IN (
        SELECT period_id FROM user_period_access
        WHERE user_id = current_setting('app.current_user_id')::uuid
      )
    );
  ```
- Users can only see/modify records for periods they have access to.

### Permission Model

**Roles**: `admin`, `controller`, `accountant`, `viewer`

**Granular permissions**:
- **Period-level**: User A can edit January, User B can edit February. Both can view each other's periods.
- **Module-level**: User A works AP, User B works AR. Both operate in the same period but different modules.
- **Record-level**: For shared records (e.g., inter-company entries), both users can edit but with conflict resolution.

### Real-Time Permission Enforcement

- WebSocket server validates permissions on every incoming change before applying to the CRDT.
- If a user's permissions are revoked mid-session, the server broadcasts a "permission revoked" event and the client switches to read-only for affected records.

### Concurrent Module Editing

- Two accountants working on different modules in the same period: no conflicts (different records).
- Two accountants working on the same module: CRDTs handle field-level merges, version checks catch same-field conflicts.
- Permission checks on every write: `if user has write_access to module AND period is open`.

## Architecture Decision: Hybrid CRDT + Server Validation

The recommended architecture for a modular cashflow platform:

1. **Client-side**: Yjs CRDTs for real-time collaboration on structured records (journal entries as Y.Map, line items as Y.Array, descriptions as Y.Text).
2. **Server-side**: WebSocket server (y-websocket or custom) that:
   - Relays CRDT updates between clients.
   - Validates accounting invariants on every update (debits = credits, account exists).
   - Enforces permissions (RLS policies checked at application layer).
   - Manages advisory locks for period operations.
3. **Database**: PostgreSQL with RLS, version columns for optimistic locking, advisory locks for periods.
4. **Audit trail**: Immutable audit_log table recording every change, conflict, and approval action.
5. **Offline**: y-indexeddb for local persistence, automatic sync on reconnect, server-side invariant validation post-sync.

This hybrid approach gives the real-time responsiveness of CRDTs with the data integrity guarantees required for financial systems.
