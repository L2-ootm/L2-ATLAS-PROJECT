## 1. Embedded Analytics: Platform Comparison

### Metabase (Recommended for embedded)

- **Embedding modes**: Modular SDK (React components), Full App (iframe), Guest (no-auth), Public links
- **SDK architecture**: React SDK with granular components — `<MetabaseQuestion>`, `<MetabaseDashboard>`, `<MetabaseCollection>`
- **Auth**: SSO-based (JWT/SAML/OIDC) for data segregation; Guest mode for simple embeds (locked parameters, no drill-through)
- **Multi-tenancy**: Built-in Tenants feature for SaaS — isolates data per customer via Row-Level Security
- **Licensing**: Open Source (core charts + dashboards), Pro ($85/seat — white-label, download restrictions), Enterprise (SAML,审计)
- **Limitation**: Custom visualizations not rendered in embeds; custom viz types fall back to table

### Apache Superset

- **Embedding**: iframe-based only (no SDK). Superset Embedded SDK is a thin JS wrapper for iframe postMessage
- **Strengths**: Rich visualization library (deck.gl, Plotly), SQL Lab for ad-hoc queries, 60+ database connectors
- **Weaknesses**: No component-level embedding; iframe coupling makes styling/interactivity harder
- **Licensing**: Apache 2.0 (fully open source)

### Cube.js (Semantic Layer for embedded)

- **Architecture**: Headless semantic layer — defines measures, dimensions, joins in a data model (YAML/JS), serves via REST/GraphQL/WebSocket API
- **Embedding**: API-first. Build your own UI or use pre-built React/Vue/Angular components
- **Key feature**: Pre-aggregation engine (in-memory + Cube Store) for sub-second queries on large datasets
- **Real-time**: WebSocket transport with subscription support
- **Licensing**: Open Source core, Cube Cloud for managed

### Redash

- **Embedding**: Public dashboards + iframe embeds (simple)
- **Strengths**: SQL-first, lightweight, good for technical users
- **Weaknesses**: Minimal embedding SDK; limited visualization customization
- **Licensing**: BSD-2-Clause

### Comparison Matrix

| Feature | Metabase | Superset | Cube.js | Redash |
|---------|----------|----------|---------|--------|
| Component embedding | SDK + iframe | iframe only | API-first | iframe |
| Multi-tenant RLS | Built-in | Manual | queryRewrite | Manual |
| Query builder | Visual + SQL | SQL Lab | Data model | SQL only |
| Custom viz | Limited in embed | Yes | BYO | Limited |
| Self-host free tier | Yes (OSS) | Yes | Yes | Yes |
| Pre-aggregation | No | No | Yes (core feature) | No |

**Recommendation**: Metabase for rapid embedded analytics; Cube.js as semantic layer if custom UI is required; Superset if team is SQL-heavy and wants full open source.

## 2. OLAP for Financial Data: Dimensional Modeling

### Star Schema for Journal Entries

```
                    ┌──────────────┐
                    │ dim_date     │
                    │──────────────│
                    │ date_key (PK)│
                    │ fiscal_year  │
                    │ fiscal_quarter│
                    │ month        │
                    │ day_of_week  │
                    │ is_holiday   │
                    └──────┬───────┘
                           │
┌──────────────┐    ┌──────┴───────┐    ┌──────────────┐
│ dim_account  │    │ fact_journal │    │ dim_entity   │
│──────────────│    │──────────────│    │──────────────│
│ account_key  │◄───│ account_key  │    │ entity_key   │
│ account_code │    │ date_key     │───►│ entity_key   │
│ account_name │    │ entity_key   │    │ entity_name  │
│ account_type │    │ project_key  │    │ country      │
│ parent_code  │    │ debit_amount │    │ region       │
│ level_1_4    │    │ credit_amount│    └──────────────┘
│ level_2_5    │    │ currency     │
│ balance_type │    │ source_ref   │
└──────────────┘    └──────────────┘
```

### Snowflake Schema for Chart of Accounts

- dim_account splits into: dim_account_type -> dim_account_group -> dim_account (normalized hierarchy)
- Better for deep hierarchical accounts (multi-level chart of accounts, consolidation rules)
- Tradeoff: more JOINs but less data redundancy for large COA

### OLAP Cubes vs Columnar Databases

| Approach | Tool | When to use |
|----------|------|-------------|
| MOLAP Cube | Apache Druid, ClickHouse | Pre-defined aggregations, sub-second on billions of rows |
| ROLAP (Columnar) | DuckDB, ClickHouse, BigQuery | Flexible ad-hoc queries, schema-on-read |
| Semantic Layer | Cube.js, dbt metrics | Single source of truth for KPI definitions |

**For cashflow platform**: Columnar OLAP (DuckDB for local, ClickHouse for scale) with Cube.js as semantic layer. Avoid rigid MOLAP cubes — financial reporting needs flexible drill-down across dimensions.

## 3. Data Export Patterns

### Export Formats

| Format | Library/Tool | Use Case |
|--------|-------------|----------|
| CSV | `papaparse` (JS), `csv.writer` (Python), native browser | Bulk data, spreadsheets |
| Excel (.xlsx) | `ExcelJS`, `openpyxl`, `SheetJS` | Formatted reports, pivot tables |
| PDF | `Puppeteer` + `pdf-lib`, `WeasyPrint`, `ReportLab` | Formal reports, signed documents |
| JSON/JSONL | Native | API consumers, data integration |

### Architecture Pattern: Report Generation Pipeline

```
Query Engine → Formatter → Renderer → Delivery
     │              │           │          │
  SQL/Cube.js   ExcelJS    Puppeteer   Email/S3/API
     │              │           │          │
  Financial      Style      Header/     Webhook +
  data model    templates   Footer      callback
```

### Key Implementation Details

- **Streaming exports**: For large datasets (>100k rows), stream to avoid OOM — `CSVWriter` streams, ExcelJS supports streaming
- **Async generation**: Offload PDF/Excel to background workers (Bull/BullMQ, Celery) with status polling
- **Template-based reports**: Define report templates (header, sections, footer, chart placeholders) as JSON/YAML; render with data bindings
- **API export endpoints**: `GET /reports/:id/export?format=xlsx&filters=...` — return 202 + job ID for async, or 200 for sync (small datasets)

## 4. Custom Report Builder

### Architecture Components

1. **Query Engine**: Visual drag-and-drop (Metabase-style) or SQL editor
2. **Visualization Selector**: Chart type picker with preview
3. **Layout Engine**: Grid-based (12-column) or free-form canvas
4. **Filter Bar**: Parameterized filters (date range, entity, account)
5. **Template System**: Save/load report templates as JSON

### Implementation Approaches

| Approach | Example | Pros | Cons |
|----------|---------|------|------|
| Embed Metabase | Metabase SDK | Full-featured out of the box | Less customization, dependency |
| Cube.js + custom UI | Cube.js + React | Full control, semantic layer | Build from scratch |
| Block-based editor | Notion-style | Familiar UX, composable | Complex to build, perf issues |
| SQL template engine | Jinja2/SQL templates | Flexible, version-controllable | No visual builder |

### Recommendation for Cashflow

Hybrid: Cube.js semantic layer (pre-defined financial measures/dimensions) + custom React drag-and-drop builder using `react-grid-layout` or `dnd-kit`. Pre-built financial report templates (P&L, Cash Flow Statement, Balance Sheet) as starting points.

## 5. Real-Time Dashboard Updates

### Update Strategies

| Mechanism | Latency | Complexity | Best For |
|-----------|---------|------------|----------|
| WebSocket | <100ms | Medium | Live trading, real-time cash position |
| Server-Sent Events (SSE) | <500ms | Low | One-way updates, dashboard refresh |
| Polling | 1-30s | Very low | Non-critical, budget dashboards |
| Cube.js subscriptions | <100ms | Low (built-in) | If using Cube.js |

### WebSocket Architecture

```
Browser ←──WebSocket──→ API Gateway ←──Redis Pub/Sub──→ Workers
                                    │
                              PostgreSQL CDC (Debezium)
                                    │
                              Change events (journal entries, payments)
```

### Practical Guidance for Financial Dashboards

- **Real-time needed for**: Cash position, pending transactions, alert thresholds
- **Polling sufficient for**: P&L trends, monthly comparisons, KPI summaries
- **Hybrid approach**: WebSocket for critical metrics + polling (30s) for supplementary data
- **Debouncing**: Batch rapid updates (e.g., batch journal entry imports) to avoid UI thrashing

## 6. Data Warehouse Integration & CDC

### Change Data Capture (CDC) Patterns

#### Debezium (Recommended)

- **Mechanism**: PostgreSQL logical decoding (pgoutput) — reads WAL stream
- **Output**: Kafka topics per table (JSON/Avro/Protobuf)
- **Features**: Initial snapshot + continuous streaming, incremental snapshots, exactly-once with Kafka
- **Deployment**: Kafka Connect connector or standalone Debezium Server

#### Architecture: OLTP → CDC → Data Warehouse

```
PostgreSQL (OLTP)
    │
    ▼
Debezium (CDC via pgoutput)
    │
    ▼
Kafka / Redpanda (event bus)
    │
    ├──▶ Kafka Connect → Snowflake (COPY INTO)
    ├──▶ Kafka Connect → BigQuery (Streaming)
    ├──▶ Kafka Connect → Redshift (UNLOAD)
    └──▶ Custom consumer → DuckDB / ClickHouse
```

### Export to External BI Tools

| Target | Integration | Notes |
|--------|-------------|-------|
| Snowflake | Kafka Connect Snowflake Sink | COPY INTO for batch, Snowpipe for streaming |
| BigQuery | Kafka Connect BigQuery Sink | Streaming inserts or batch load |
| Redshift | UNLOAD command or COPY | UNLOAD to S3, then COPY into Redshift |
| Superset/Metabase | Direct SQL connection | Point at data warehouse, not OLTP |
| Looker | LookML + database connection | Define model in LookML, connect to DW |

### Key Considerations

- **Schema evolution**: Debezium handles DDL changes via schema change events; downstream consumers must handle schema registry
- **Data freshness**: CDC provides near-real-time (seconds), batch ETL provides daily/hourly
- **Cost**: Streaming to Snowflake/BigQuery costs per-event; batch is cheaper for non-real-time
- **Financial audit**: CDC preserves change history (before/after) — critical for audit trails

## 7. Financial KPI Libraries

### Core Cashflow KPIs

| KPI | Formula | Category |
|-----|---------|----------|
| Operating Cash Flow (OCF) | Net Income + Depreciation - Change in Working Capital | Cash Flow |
| Free Cash Flow (FCF) | OCF - CapEx | Cash Flow |
| Cash Conversion Ratio | OCF / Net Income | Efficiency |
| Days Sales Outstanding (DSO) | (AR / Revenue) × Days | Receivables |
| Days Payable Outstanding (DPO) | (AP / COGS) × Days | Payables |
| Cash Runway | Cash Balance / Monthly Burn Rate | Liquidity |
| Burn Rate | (Starting Cash - Ending Cash) / Months | Liquidity |
| Working Capital Ratio | Current Assets / Current Liabilities | Liquidity |
| Quick Ratio | (Cash + AR) / Current Liabilities | Liquidity |
| Current Ratio | Current Assets / Current Liabilities | Liquidity |

### Pre-built KPI Libraries

| Library | Language | Notes |
|---------|----------|-------|
| `pandas-ta` | Python | Technical analysis indicators, can adapt for finance |
| `finpy` | Python | Financial calculations |
| Custom (recommended) | Python/TypeScript | Domain-specific KPI formulas with validation |

### Benchmark Data Sources

- **Industry benchmarks**: IBISWorld, Statista, BizMiner (paid)
- **Public data**: SEC EDGAR (10-K filings), FRED (Federal Reserve data)
- **Open benchmarks**: OpenBB Terminal, Financial Modeling Prep API (free tier)

**Recommendation**: Build a `kpi_engine` module with declarative KPI definitions (formula, required inputs, validation rules, comparison benchmarks). Store benchmark data in a separate `benchmarks` table linked by industry/sector.

## 8. Scheduled Reports

### Architecture

```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐
│ Cron/Scheduler│────▶│ Report Engine │────▶│  Delivery    │
│ (Bull/CRON)  │     │ (Query+Render)│     │  Channel     │
└─────────────┘     └──────────────┘     └─────────────┘
                           │                    │
                     Template Store        Email/Slack/API/S3
                     (JSON/YAML)          Webhook callbacks
```

### Delivery Channels

| Channel | Implementation | Notes |
|---------|---------------|-------|
| Email | SMTP (Nodemailer/nodemailer) + PDF attachment | Most common, needs email templates |
| Slack | Slack Webhook or Slack API (chat.postMessage) | Rich formatting, file upload |
| API/Webhook | POST to configured endpoint | For system integration |
| S3/GCS | Upload to bucket + presigned URL | For archival, shared access |
| In-app notification | WebSocket + notification center | For dashboards |

### Scheduling Implementation

- **Bull/BullMQ** (Node.js): Redis-backed job queue with repeatable jobs, retry logic, concurrency control
- **Celery Beat** (Python): Cron-like scheduler for Celery tasks
- **Custom**: Database-backed schedule with `cron` field per report, worker polls for due reports

### Report Configuration Schema

```yaml
report:
  name: "Weekly Cash Position"
  schedule: "0 9 * * MON"  # Every Monday at 9 AM
  template: "cash_position_weekly"
  filters:
    period: "last_7_days"
    entities: ["all"]
  delivery:
    - channel: email
      recipients: ["cfo@company.com", "finance@company.com"]
      format: pdf
    - channel: slack
      channel_id: "#finance"
      format: interactive
```

## 9. Financial Data Visualization Best Practices

### Chart Type Selection Guide

| Use Case | Chart Type | Library Support |
|----------|-----------|-----------------|
| Cash flow over time | Waterfall chart | ECharts waterfall, Chart.js |
| Budget vs actual | Grouped bar chart | All libraries |
| Account hierarchy | Treemap | ECharts, D3.js |
| Trend analysis | Line chart with sparklines | Chart.js, ECharts |
| Profitability by segment | Stacked bar / Sankey | ECharts Sankey |
| KPI scorecards | Number + sparkline | Recharts, custom |
| Cash position heatmap | Calendar heatmap | D3.js, ECharts |
| Multi-period comparison | Small multiples | Custom grid layout |

### Financial-Specific Patterns

1. **Waterfall charts**: Essential for cash flow statements (starting cash → inflows → outflows → ending cash). ECharts has native waterfall support.

2. **Sparklines**: Inline mini-charts for KPI tables. Use `sparkline` option in chart libraries or lightweight SVG.

3. **Heatmaps**: Calendar-style heatmap for daily cash flows or transaction volumes. Shows seasonality patterns.

4. **Small multiples**: Repeat the same chart type across entities/periods for comparison. Grid layout with `react-grid-layout`.

5. **Conditional formatting**: Red/green coloring for variance analysis (actual vs budget). Green for positive variance, red for negative.

6. **Number formatting**: Always use locale-aware formatting (`Intl.NumberFormat`). Financial data needs proper currency symbols, thousand separators, and decimal places.

7. **Negative values**: In financial charts, use color coding (red for negative) not just negative axis positioning.

### Library Recommendations

| Library | Strengths | Size |
|---------|-----------|------|
| Apache ECharts | Rich financial charts (waterfall, Sankey, treemap), theme system | ~800KB |
| Chart.js | Lightweight, plugin ecosystem, easy to start | ~200KB |
| Recharts | React-native, composable, good docs | ~300KB |
| D3.js | Maximum flexibility, custom visualizations | ~250KB (core) |
| Victory | React + D3, declarative API | ~400KB |

**Recommendation**: ECharts for its native financial chart types (waterfall, candlestick, gauge) + Recharts for simple KPI cards and sparklines. Avoid D3.js unless building truly custom visualizations.

### Key Technology Decisions

| Component | Choice | Rationale |
|-----------|--------|-----------|
| Embedded analytics | Metabase SDK | Fast to ship, multi-tenant, open source |
| Semantic layer | Cube.js | Pre-aggregation, real-time, API-first |
| OLAP engine | DuckDB (local) / ClickHouse (scale) | Columnar, fast analytical queries |
| CDC | Debezium + Kafka | Battle-tested, PostgreSQL-native |
| Report export | ExcelJS + Puppeteer | Streaming Excel, server-side PDF |
| Real-time | WebSocket (Socket.io) | Bi-directional, fallback to SSE |
| Visualization | ECharts + Recharts | Financial charts + React integration |
| Job queue | BullMQ | Redis-backed, repeatable jobs, retry |
