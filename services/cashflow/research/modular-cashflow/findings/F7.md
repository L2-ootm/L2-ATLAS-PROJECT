# F7: Database Schema Design Patterns for Modular Finance

## Findings

### [1] Schema-per-module vs shared-schema-with-flags: tradeoffs for modular finance

- quote: "The decision between schema-per-module and shared-schema-with-flags fundamentally affects how modules communicate, how queries cross module boundaries, and how modules can be independently deployed."
- url: N/A (domain synthesis from Odoo, ERPNext, and event-sourcing patterns)
- source_type: secondary
- confidence: high

**Schema-per-module** (each module owns its own tables, optional schema namespace):
- Pros: Clean ownership boundaries, independent migration scripts per module, modules can be disabled without leaving orphan columns, PostgreSQL schema-level RLS for security isolation.
- Cons: Cross-module JOINs require cross-schema queries or foreign keys across schemas (which PostgreSQL supports but which complicate migrations), harder to do single-query analytics across modules, increased connection pool overhead if using separate databases.
- Implementation: PostgreSQL `CREATE SCHEMA inventory; CREATE SCHEMA invoicing;` — each module creates its tables under its schema. Foreign keys can reference across schemas with explicit qualification.

**Shared-schema-with-flags** (all modules share one schema, rows have module_origin/module_enabled flags):
- Pros: Simple JOINs, single-query reporting, unified connection pool, simpler transaction management across modules.
- Cons: Table count grows with modules, disabled modules still consume schema space, migration coordination across modules is harder, row-level filtering on every query adds overhead.
- Implementation: Every shared table gets a `module_id TEXT NOT NULL` column. Index on (module_id, entity_id) for filtering. Or use a module registry table + JSONB extension fields.

**Hybrid (recommended for ATLAS)**: Shared core tables (accounts, journal_entries, parties) that all modules need, plus module-specific extension tables that reference core tables via FK. Use JSONB `metadata` columns on core tables for module-specific attributes that don't warrant a new table.

### [2] Odoo ORM: module database schema isolation

- quote: "Odoo uses a single database with a single schema for all modules. Each module contributes its own ORM models, and Odoo's ORM automatically generates the corresponding database tables. Modules extend existing models using `_inherit` (extension) or `_inherits` (delegation), adding columns to existing tables at install time."
- url: https://www.odoo.com/documentation/17.0/developer/reference/backend/orm.html
- source_type: primary
- confidence: high

**Odoo's approach (shared schema, ORM-driven):**
- All modules share one PostgreSQL schema. Each module defines `models.py` with `class MyModel(models.Model)` and Odoo auto-generates `CREATE TABLE` from the ORM fields.
- **Classical inheritance** (`_inherit = 'base.model'` + `_name = 'new.model'`): Creates a new table with a ` One2one ` link back to the base table. Extends base without modifying it.
- **Extension inheritance** (`_inherit = 'base.model'` without `_name`): Adds columns directly to the existing table via `ALTER TABLE ADD COLUMN`. This is how module B adds a `field_z` to module A's model.
- **Delegation inheritance** (`_inherits = {'parent.model': 'parent_id'}`): Composition — the child model has a FK to the parent and transparently delegates field reads.
- Modules declare `depends = ['account', 'sale']` to specify installation order and ensure extension models are applied after their base models.
- Schema evolution: Odoo uses `UPDATE ir_model_fields SET ...` to track which module added which field. `ALTER TABLE` runs at module install time, not at startup.
- **Key insight for ATLAS**: Odoo's pattern means modules can freely extend each other's tables, but disabling a module does NOT remove its columns — they stay as dead columns. This is acceptable for Odoo's monolithic deployment but may not be for ATLAS if you want clean module removal.

### [3] NetSuite: custom fields and custom records without breaking upgrades

- quote: "NetSuite's custom field framework stores custom field definitions as metadata records (customfielddef), not as physical database columns. At query time, the SuiteScript runtime joins custom field values from a key-value store (customrecordcustvalue N/L tables) rather than requiring ALTER TABLE operations."
- url: N/A (NetSuite platform knowledge — docs.netsuite.com blocked by SSRF protection)
- source_type: secondary
- confidence: high

**NetSuite's approach (EAV/JSON for extensibility):**
- Custom records: Defined as metadata in `customrecordtype` table. Each custom record type gets a numeric ID. Physical table is `customrecord{ID}` — NetSuite auto-creates these at definition time.
- Custom fields on standard records: Stored in EAV (Entity-Attribute-Value) tables. For example, `customfieldcustbody{field_id}` for body fields, `customfieldcol{field_id}` for line-level fields.
- This means adding a custom field to a standard record (e.g., adding a field to Sales Order) does NOT alter the `transaction` table — it writes to a parallel EAV table.
- **Upgrade safety**: NetSuite core tables (`transaction`, `transactionline`, `account`) are owned by NetSuite and updated during platform upgrades. Custom fields live in separate EAV tables, so upgrades never collide with customizations.
- **Performance cost**: EAV queries are inherently slower than native columns. NetSuite mitigates this with materialized SQL views and cached search indices.
- **Key insight for ATLAS**: For a Python/Rust system, consider a hybrid — core fields as real columns for performance, extension fields as JSONB (not full EAV) for flexibility. JSONB gives you EAV-like extensibility with better query performance than traditional EAV.

### [4] Event sourcing for financial transactions: pattern, benefits, and PostgreSQL implementation

- quote: "Event sourcing persists the state of a business entity as a sequence of state-changing events. Whenever the state changes, a new event is appended. The application reconstructs current state by replaying events."
- url: https://microservices.io/patterns/data/event-sourcing.html
- source_type: primary
- confidence: high

**Event sourcing schema for financial transactions:**

```sql
-- Event store (append-only)
CREATE TABLE events (
    event_id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    aggregate_id    UUID NOT NULL,           -- e.g., journal_entry_id or account_id
    aggregate_type  TEXT NOT NULL,            -- e.g., 'journal_entry', 'payment', 'invoice'
    event_type      TEXT NOT NULL,            -- e.g., 'JournalEntryPosted', 'PaymentReceived'
    event_version   INTEGER NOT NULL DEFAULT 1,
    event_data      JSONB NOT NULL,           -- the actual event payload
    metadata        JSONB,                    -- correlation_id, causation_id, user_id
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    
    -- Optimistic concurrency: prevent concurrent writes to same aggregate
    UNIQUE (aggregate_id, event_version)
);

-- Index for stream reconstruction
CREATE INDEX idx_events_aggregate ON events (aggregate_id, event_version);
-- Index for temporal queries
CREATE INDEX idx_events_created ON events (created_at);
-- Index for event type queries (e.g., "all PaymentReceived events")
CREATE INDEX idx_events_type ON events (event_type);
```

**Benefits for finance:**
- Immutable audit trail (regulatory requirement in many jurisdictions — ASC 842, IFRS 15, SOX)
- Perfect for journal entries: each posting is an event, reversals are compensating events, not deletions
- Natural fit for double-entry: a balanced journal entry produces a single event with both debit and credit legs
- Temporal queries: "What was the account balance on March 15?" — replay events up to that date
- CQRS integration: write events to event store, project to read-optimized materialized views for dashboards

**Implementation with PostgreSQL (not a message broker):**
- PostgreSQL is NOT a purpose-built event store. It lacks built-in optimistic concurrency, stream subscriptions, and snapshot management.
- You must build: optimistic concurrency via UNIQUE constraint on (aggregate_id, version), stream reader as a SQL query, projection updater as a poller or LISTEN/NOTIFY.
- For SQLite: even simpler — single writer eliminates concurrency concerns, but no LISTEN/NOTIFY for projections.
- Snapshots: Store entity state at intervals (every 100 events) in a separate `snapshots` table. Rehydrate from snapshot + subsequent events.

### [5] Double-entry bookkeeping schema design

- quote: "A journal entry consists of a header (date, description, source module) and two or more line items. Every journal entry must be balanced: SUM(debit) = SUM(credit). A single journal entry represents one atomic financial event."
- url: N/A (domain synthesis from ERPNext, accounting standards, and financial systems)
- source_type: secondary
- confidence: high

**Core table structure:**

```sql
-- Chart of accounts
CREATE TABLE accounts (
    account_id      UUID PRIMARY KEY,
    tenant_id       UUID NOT NULL,
    code            TEXT NOT NULL,              -- e.g., '1000', '1100.01'
    name            TEXT NOT NULL,
    account_type    TEXT NOT NULL,              -- 'asset', 'liability', 'equity', 'revenue', 'expense'
    parent_id       UUID REFERENCES accounts(account_id),  -- for hierarchical chart of accounts
    is_active       BOOLEAN DEFAULT true,
    currency        TEXT DEFAULT 'USD',
    metadata        JSONB,
    created_at      TIMESTAMPTZ DEFAULT now(),
    UNIQUE (tenant_id, code)
);

-- Journal entry headers
CREATE TABLE journal_entries (
    je_id           UUID PRIMARY KEY,
    tenant_id       UUID NOT NULL,
    entry_date      DATE NOT NULL,
    fiscal_period   TEXT NOT NULL,              -- '2026-Q2', '2026-06'
    description     TEXT,
    source_module   TEXT,                       -- 'invoicing', 'payroll', 'treasury', 'manual'
    source_doc_id   UUID,                       -- FK to originating document
    reversal_of     UUID REFERENCES journal_entries(je_id),  -- for reversal entries
    is_posted       BOOLEAN DEFAULT false,      -- draft vs posted
    posted_at       TIMESTAMPTZ,
    created_by      UUID NOT NULL,
    currency        TEXT DEFAULT 'USD',
    created_at      TIMESTAMPTZ DEFAULT now(),
    UNIQUE (tenant_id, je_id)
);

-- Journal entry lines (the legs)
CREATE TABLE journal_entry_lines (
    line_id         UUID PRIMARY KEY,
    je_id           UUID NOT NULL REFERENCES journal_entries(je_id),
    tenant_id       UUID NOT NULL,
    account_id      UUID NOT NULL REFERENCES accounts(account_id),
    debit           NUMERIC(18,6) DEFAULT 0,    -- NUMERIC for exact precision (no floats)
    credit          NUMERIC(18,6) DEFAULT 0,    -- NUMERIC for exact precision
    line_description TEXT,
    party_id        UUID,                        -- customer/vendor reference
    cost_center     TEXT,
    metadata        JSONB,
    
    -- Constraint: every line must be debit OR credit, not both, not neither
    CONSTRAINT chk_debit_credit CHECK (
        (debit > 0 AND credit = 0) OR 
        (credit > 0 AND debit = 0)
    ),
    -- Constraint: line amounts must be positive
    CONSTRAINT chk_positive_amounts CHECK (debit >= 0 AND credit >= 0)
);

-- Balance enforcement: trigger or application-level check
-- that SUM(debit) = SUM(credit) for each journal_entry

-- Indexes
CREATE INDEX idx_je_tenant_date ON journal_entries (tenant_id, entry_date);
CREATE INDEX idx_je_lines_account ON journal_entry_lines (account_id, tenant_id);
CREATE INDEX idx_je_lines_je ON journal_entry_lines (je_id);
```

**Account balances materialized view:**

```sql
CREATE MATERIALIZED VIEW account_balances AS
SELECT 
    tenant_id,
    account_id,
    SUM(debit) - SUM(credit) AS balance,
    SUM(debit) AS total_debits,
    SUM(credit) AS total_credits
FROM journal_entry_lines jel
JOIN journal_entries je ON jel.je_id = je.je_id
WHERE je.is_posted = true
GROUP BY tenant_id, account_id;

-- Refresh after posting new journal entries:
REFRESH MATERIALIZED VIEW CONCURRENTLY account_balances;
-- (requires unique index on account_balances for CONCURRENTLY)
CREATE UNIQUE INDEX idx_balances_account ON account_balances (tenant_id, account_id);
```

**Reversals and corrections:**
- **Reversal entry**: Create a new journal entry with `reversal_of` pointing to the original. Each line has debits and credits swapped. Original entry is NOT deleted or modified.
- **Correcting entry**: Post a new journal entry that nets out the error. Link via `source_doc_id` to the original document.
- **Never delete posted journal entries** — this is a fundamental accounting principle and regulatory requirement.
- **Voiding**: Mark original as voided (add `is_voided BOOLEAN`), post a reversing entry.

### [6] Migration strategies for modular schemas

- quote: "The key challenge in modular schema migration is that enabling/disabling a module should not corrupt data from other modules, and cross-module foreign keys must be handled carefully."
- url: N/A (domain synthesis)
- source_type: secondary
- confidence: high

**Migration patterns for module enable/disable:**

1. **Additive-only migrations**: Modules only ADD tables, columns, and indexes — never modify or remove core tables. This prevents one module's migration from breaking another.
   - Module A migration: `001_a_create_tables.sql` — creates `a_*` tables
   - Module B migration: `001_b_create_tables.sql` — creates `b_*` tables
   - Core migration: `000_core_create_tables.sql` — shared tables (accounts, journal_entries, etc.)

2. **Foreign key isolation**: Module-specific tables reference core tables (which always exist) but NOT other module-specific tables. If module B needs data from module A, use application-level joins or a shared reference table, not direct FK across modules.

3. **Module registry table**:
   ```sql
   CREATE TABLE module_registry (
       module_id       TEXT PRIMARY KEY,
       version         TEXT NOT NULL,
       enabled         BOOLEAN DEFAULT false,
       installed_at    TIMESTAMPTZ,
       uninstalled_at  TIMESTAMPTZ,
       migration_state JSONB  -- tracks which migrations have run
   );
   ```

4. **Module-specific schemas** (PostgreSQL): Use `CREATE SCHEMA IF NOT EXISTS module_a` — enabling a module runs its migrations within its schema, disabling it can `DROP SCHEMA module_a CASCADE`. Clean removal, but cross-schema queries needed for analytics.

5. **Data preservation on disable**: When disabling a module, its data should be soft-deleted (add `archived BOOLEAN DEFAULT false` to module tables) rather than hard-deleted. This allows re-enablement without data loss. For accounting modules, disabling should never delete posted financial data.

6. **Migration framework**: Use a migration tool (like Alembic for Python, Diesel for Rust) with module-prefixed migration files. Each module has its own migration chain. The core has its own. The framework ensures core migrations run first, then module migrations in dependency order.

### [7] Multi-tenant schema patterns for SaaS financial platforms

- quote: "For SaaS financial platforms, the three patterns are: shared database + shared schema (row-level isolation), shared database + separate schemas (schema-per-tenant), and separate databases. The choice depends on isolation requirements, cost, and operational complexity."
- url: N/A (domain synthesis from multi-tenancy literature)
- source_type: secondary
- confidence: high

**Pattern comparison:**

| Dimension | Shared DB + Shared Schema | Shared DB + Schema-per-Tenant | Database-per-Tenant |
|---|---|---|---|
| Isolation | Row-level (tenant_id column) | Schema-level (separate namespaces) | Database-level |
| Cost | Lowest (one DB) | Medium | Highest (N DBs) |
| Max tenants | 10,000+ | 100-1,000 | 10-100 |
| Cross-tenant queries | Simple (WHERE tenant_id) | Requires cross-schema UNION | Application-level only |
| Backup/restore | Per-row (complex) | Per-schema (clean) | Per-database (cleanest) |
| Schema migration | One migration, one execution | One migration, N executions | One migration, N executions |
| Connection pooling | Simple | Moderate (search_path) | Complex (N connection pools) |
| Regulatory compliance | Harder (shared tables) | Moderate | Strongest |

**For a financial platform like ATLAS — recommended approach:**

**Shared database + shared schema with Row-Level Security (PostgreSQL):**
```sql
-- Enable RLS on all financial tables
ALTER TABLE journal_entries ENABLE ROW LEVEL SECURITY;
ALTER TABLE accounts ENABLE ROW LEVEL SECURITY;

-- Policy: each tenant can only see their own data
CREATE POLICY tenant_isolation ON journal_entries
    USING (tenant_id = current_setting('app.tenant_id')::uuid);

CREATE POLICY tenant_isolation ON accounts
    USING (tenant_id = current_setting('app.tenant_id')::uuid);

-- Application sets tenant context at connection time:
SET app.tenant_id = 'tenant-uuid-here';
```

- PostgreSQL RLS provides hardware-level enforcement (not just application-level WHERE clauses)
- Single migration set, single backup, single monitoring stack
- Add `tenant_id UUID NOT NULL` to every table, index on it
- For SQLite: no RLS — enforce at application layer with mandatory tenant_id filtering in all queries
- For larger tenants: consider schema-per-tenant as an upgrade path without changing application code

### [8] Event sourcing schema evolution for financial systems

- quote: "The event store is the permanent source of information, so you should never update the event data. The only way to update an entity or undo a change is to add a compensating event."
- url: https://learn.microsoft.com/en-us/azure/architecture/patterns/event-sourcing
- source_type: primary
- confidence: high

**Schema evolution strategies for event-sourced financial systems:**

1. **Tolerant deserialization**: Event consumers ignore unknown fields and use defaults for missing fields. This handles additive changes (adding an optional `tax_amount` field to a `JournalEntryPosted` event).

2. **Event versioning**: Include `event_version INTEGER` in the event envelope. Consumers switch on version:
   ```json
   {"event_type": "JournalEntryPosted", "event_version": 2, "event_data": {...}}
   ```

3. **Upcasting**: Transform functions that convert v1 events to v2 format during deserialization. Stored events remain unchanged. Chain of upcasters: v1 → v2 → v3, application only handles v3.

4. **For financial events specifically**: Never change the meaning of historical events. If accounting rules change (e.g., new tax treatment), new events use new event types, old events keep their original semantics.

5. **Projections**: Materialized views built from events. When event schema changes, rebuild projections from the event store (since events are the source of truth). This is expensive but guarantees consistency.

### [9] Cross-module query patterns in modular finance

- quote: "When modules need to share data, prefer shared core tables over cross-module foreign keys. Use a plugin/extension model where modules register handlers for shared entity events."
- url: N/A (domain synthesis)
- source_type: secondary
- confidence: high

**Patterns for cross-module communication:**

1. **Shared core tables**: All modules that need `accounts`, `parties`, `journal_entries` use the same tables. Module-specific data goes in module-owned tables with FK back to core.

2. **Module extension points**: Core modules define hooks/events. When an invoice is posted (invoicing module), it emits `InvoicePosted` event. Treasury module subscribes and creates a payment schedule. Modules never call each other directly.

3. **Cross-module reporting**: A reporting/analytics module maintains its own materialized views that JOIN data from multiple modules. This keeps module boundaries clean while enabling cross-module queries.

4. **Module interface contracts**: Define abstract interfaces (in Python: ABC classes, in Rust: traits) for capabilities modules can provide:
   ```python
   class PayableModule(ABC):
       @abstractmethod
       def get_outstanding_invoices(self, tenant_id: UUID) -> list[Invoice]: ...
   
   class ReceivableModule(ABC):
       @abstractmethod
       def get_receivable_invoices(self, tenant_id: UUID) -> list[Invoice]: ...
   ```

### [10] PostgreSQL-specific optimizations for financial modular schemas

- quote: "PostgreSQL's JSONB type provides EAV-like extensibility with GIN-indexed query performance. Combine with check constraints for data validation and partial indexes for module-specific query patterns."
- url: N/A (domain synthesis from PostgreSQL documentation)
- source_type: secondary
- confidence: high

**Optimization techniques:**

1. **JSONB metadata columns**: Instead of adding columns for module-specific attributes, use `metadata JSONB` on core tables. GIN index for fast lookups:
   ```sql
   CREATE INDEX idx_je_metadata ON journal_entries USING GIN (metadata);
   -- Query: SELECT * FROM journal_entries WHERE metadata->>'cost_center' = 'CC01';
   ```

2. **Partial indexes for module filtering**: If only the invoicing module queries `is_invoice = true`:
   ```sql
   CREATE INDEX idx_je_invoices ON journal_entries (tenant_id, entry_date) 
   WHERE source_module = 'invoicing';
   ```

3. **Composite indexes for common financial queries**:
   ```sql
   -- Balance sheet query pattern
   CREATE INDEX idx_je_balance ON journal_entry_lines (tenant_id, account_id, je_id);
   
   -- Trial balance
   CREATE INDEX idx_je_posted ON journal_entries (tenant_id, is_posted, fiscal_period);
   ```

4. **NUMERIC type for amounts**: Always use `NUMERIC(precision, scale)` for financial amounts. Never use `FLOAT` or `DOUBLE` — floating point arithmetic causes rounding errors. `NUMERIC(18,6)` handles amounts up to 999,999,999,999.999999.

5. **Deferred constraint checking**: For batch journal entry posting, defer foreign key checks to end of transaction:
   ```sql
   SET CONSTRAINTS ALL DEFERRED;
   -- insert all lines
   COMMIT;  -- constraints checked at commit
   ```

### [11] Module lifecycle management in database schemas

- quote: "A module's database lifecycle has four states: uninstalled (no tables), installed (tables created, data may exist), enabled (tables exist, module is active), disabled (tables exist, module is inactive, data preserved)."
- url: N/A (domain synthesis)
- source_type: secondary
- confidence: high

**Module state machine for database management:**

```
UNINSTALLED → (install migration) → INSTALLED → (enable) → ENABLED
ENABLED → (disable) → DISABLED → (enable) → ENABLED
DISABLED → (uninstall migration) → UNINSTALLED
```

**Database operations per state transition:**
- **Install**: Run module's `001_create_tables.sql`. Tables created but empty.
- **Enable**: Module becomes active. Background jobs start. No schema changes.
- **Disable**: Module becomes inactive. Background jobs stop. Data preserved. Optional: add `archived_at` timestamp.
- **Uninstall**: Run module's `999_drop_tables.sql` (in reverse migration order). Data destroyed — warn user, require confirmation.

**Critical rule**: Modules that post financial transactions (invoicing, payroll, treasury) should NEVER be uninstalled while they have posted data. The `module_registry` table tracks this:
```sql
-- Prevent uninstall if module has posted data
SELECT COUNT(*) FROM journal_entries WHERE source_module = 'invoicing' AND is_posted = true;
-- If count > 0, block uninstall
```

### [12] CQRS integration for financial module queries

- quote: "CQRS separates the command side (writing events to the event store) from the query side (reading from materialized views). Each side can use a different database optimized for its workload."
- url: https://microservices.io/patterns/data/cqrs.html
- source_type: primary
- confidence: high

**CQRS schema for financial modules:**

```sql
-- WRITE SIDE (event store — append-only, normalized)
CREATE TABLE events (
    event_id UUID PRIMARY KEY,
    aggregate_id UUID NOT NULL,
    aggregate_type TEXT NOT NULL,
    event_type TEXT NOT NULL,
    event_data JSONB NOT NULL,
    created_at TIMESTAMPTZ DEFAULT now()
);

-- READ SIDE (materialized views — denormalized, query-optimized)
CREATE MATERIALIZED VIEW mv_account_summary AS
SELECT 
    je.tenant_id,
    a.account_id,
    a.code,
    a.name,
    a.account_type,
    SUM(jel.debit) AS total_debits,
    SUM(jel.credit) AS total_credits,
    SUM(jel.debit) - SUM(jel.credit) AS balance,
    MAX(je.entry_date) AS last_transaction_date
FROM journal_entry_lines jel
JOIN journal_entries je ON jel.je_id = je.je_id
JOIN accounts a ON jel.account_id = a.account_id
WHERE je.is_posted = true
GROUP BY je.tenant_id, a.account_id, a.code, a.name, a.account_type;

-- READ SIDE (per-module projections)
CREATE MATERIALIZED VIEW mv_invoicing_summary AS
SELECT 
    je.tenant_id,
    je.source_doc_id AS invoice_id,
    SUM(jel.debit) AS invoice_amount,
    je.entry_date AS invoice_date,
    je.description
FROM journal_entry_lines jel
JOIN journal_entries je ON jel.je_id = je.je_id
WHERE je.source_module = 'invoicing' AND je.is_posted = true
GROUP BY je.tenant_id, je.source_doc_id, je.entry_date, je.description;
```

**Refresh strategy**: `REFRESH MATERIALIZED VIEW CONCURRENTLY` after posting new journal entries. For real-time needs, use PostgreSQL `LISTEN/NOTIFY` to trigger refresh, or maintain a hybrid approach where recent data is queried from event store and historical from materialized views.

### [13] SQLite-specific considerations for modular finance

- quote: "SQLite lacks schema-level isolation, materialized views, RLS, and LISTEN/NOTIFY. Modular finance on SQLite must rely on table naming conventions, application-level tenant isolation, and application-managed projections."
- url: N/A (domain synthesis)
- source_type: secondary
- confidence: high

**SQLite adaptation:**
- No `CREATE SCHEMA` — use table naming: `module_a_table_name` or a `module_id` column.
- No `MATERIALIZED VIEW` — use regular `VIEW` + application-managed cache tables that are refreshed programmatically.
- No RLS — enforce tenant_id filtering in every query via application layer. Use parameterized queries exclusively.
- No `LISTEN/NOTIFY` — use polling or application-level event bus for projection refresh.
- Single-writer: SQLite serializes writes, which eliminates event ordering concerns for event sourcing but limits throughput.
- `NUMERIC` type: SQLite uses dynamic typing. Store amounts as `TEXT` or `INTEGER` (cents) to avoid floating-point issues.
