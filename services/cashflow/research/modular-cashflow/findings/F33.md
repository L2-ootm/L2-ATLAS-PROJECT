## 1. Budget Creation Methodologies

### 1.1 Top-Down vs Bottom-Up Budgeting

**Top-down budgeting**: Senior management sets overall targets and allocates downward to departments. Fast, consistent with strategic goals, but may miss operational realities. Best for organizations with clear strategic direction.

**Bottom-up budgeting**: Each department/manager builds their own budget from operational needs, which is aggregated upward. More accurate at the department level, increases ownership, but time-consuming and prone to sandbagging (over-asking). Requires validation against financial targets.

**Hybrid approach** (recommended): Management sets strategic targets and a total envelope; departments propose within that envelope. Finance reconciles proposals against constraints. This combines strategic alignment with operational accuracy.

### 1.2 Zero-Based Budgeting (ZBB)

- **Definition**: All expenses must be justified from scratch each budget period; no automatic carryover of prior allocations.
- **Origin**: Peter Pyhrr, Texas Instruments, 1970s; adopted by Jimmy Carter for Georgia state budget in 1973.
- **How it works**:
  1. Identify "decision units" (departments, programs, cost centers).
  2. Create "decision packages" at multiple funding levels (e.g., minimum viable, current level, enhanced).
  3. Rank packages by priority; management allocates funds from highest to lowest rank until budget is exhausted.
  4. Performance measures are mandatory — packages without metrics cannot be ranked.
- **Key variants**: Modified ZBB — only a subset of departments undergo full zero-base review each year (rotating cycle), reducing administrative burden.
- **Advantages**: Eliminates redundant spending; forces justification of every dollar; can achieve 15-20% cost savings.
- **Disadvantages**: Extremely time-consuming; requires extensive documentation; risk of "pet project" gaming; departments may struggle with justification under market uncertainty.
- **Notable users**: AB InBev, 3G Capital (Kraft Heinz, Burger King), Unilever — reported significant savings (e.g., 18% savings, 20% share price increase at one consumer goods company).

### 1.3 Rolling Forecasts vs Annual Budgets with Quarterly Reforecasts

**Annual budget with quarterly reforecast**:
- Fixed annual budget set once per year.
- Quarterly reforecast adjusts the remaining quarters based on year-to-date actuals and updated assumptions.
- Balances planning stability with responsiveness.
- Most common in mid-size organizations.

**Rolling forecast**:
- Continuously updated 12-month (or 18-month) forecast that always extends to the same future horizon.
- Updated monthly or quarterly.
- Eliminates the annual "budget season" — planning becomes a continuous process.
- Advantages: More agile, reduces budget gaming, better resource allocation.
- Requires organizational culture shift and robust data systems.

### 1.4 Driver-Based Forecasting

- Forecasts are built from key business drivers (e.g., revenue per employee, cost per unit, headcount growth) rather than line-item detail.
- A small number of drivers (typically 5-15) can explain 80-90% of financial performance.
- Simplifies the forecasting process and makes it more transparent.
- Enables rapid "what-if" analysis by changing driver assumptions.

### 2.2 Budget Versioning Strategy

- **Original budget**: Created during annual planning, locked after approval.
- **Reforecast**: New version created from approved parent, with adjusted amounts for remaining periods. Original budget preserved for comparison.
- **What-if / Scenario**: Non-binding versions for modeling; never approved, but can be "promoted" to official reforecast.
- **Forecast**: Mid-year estimate of full-year results; typically not a separate budget version but derived from actuals + remaining forecast.
- **Lineage tracking**: Every version records `parent_version_id` for audit trail.

### 2.3 Multi-Dimensional Budget Lines

Budget lines support multiple dimensions for flexible analysis:

| Dimension | Purpose | Example |

| Location | Geographic | US, EU, APAC |
| Vendor | Supplier-specific | AWS, Salesforce |

The combination of dimensions creates a "budget cube" that can be sliced/diced in any direction for reporting.

## 3. Variance Analysis

### 3.1 Variance Calculation Methods

**Absolute Variance**:
```
variance = actual_amount - budget_amount
```

**Percentage Variance**:
```
variance_pct = (actual_amount - budget_amount) / |budget_amount| * 100
```
When budget is zero or near-zero, use absolute variance or a threshold flag.

**Favorability Logic** (account-type dependent):
- **Revenue accounts**: Favorable if actual > budget (earned more than planned).
- **Expense accounts**: Favorable if actual < budget (spent less than planned).
- **Balance sheet accounts**: Context-dependent (e.g., lower receivables may be favorable for cash, unfavorable for sales).

### 3.2 Variance Types (CFI Framework)

From Corporate Finance Institute's variance analysis methodology:

| Variance Type | Formula | Interpretation |
|---------------|---------|----------------|
| **Price Variance** | (Actual Price - Standard Price) × Actual Quantity | Cost difference per unit |
| **Quantity/Efficiency Variance** | (Actual Qty - Standard Qty) × Standard Price | Usage difference at standard cost |
| **Volume Variance** | (Actual Volume - Budgeted Volume) × Standard Cost | Activity level difference |
| **Budget/Spending Variance** | Actual Fixed Cost - Budgeted Fixed Cost | Fixed cost overrun |
| **Mix Variance** | Difference in product/service mix from budget | Portfolio shift impact |
| **Total Variance** | Sum of all sub-variances | Overall deviation |

### 3.3 Materiality Thresholds

Materiality determines when a variance is "significant enough" to investigate:

**Common approaches**:
- **Fixed threshold**: Flag any variance > $X (e.g., $5,000) or > Y% (e.g., 10%).
- **Tiered thresholds**: Different limits by account type or department size.
  - Revenue accounts: 5% materiality
  - Operating expenses: 10% materiality
  - Capital expenses: 15% materiality
  - Total company: 3% materiality
- **Dynamic/materiality percentage**: `materiality = max(fixed_floor, budget_amount * pct_threshold)`
- **Statistical approach**: Flag > 2 standard deviations from historical variance pattern.

```

### 4.2 Three-Scenario Framework

| Scenario | Revenue Growth | Cost Inflation | Headcount | Cash Position |
|----------|---------------|----------------|-----------|---------------|
| **Best Case (Optimistic)** | +15% | -3% (efficiency gains) | +20% | Strong |
| **Base Case (Most Likely)** | +8% | +4% (normal inflation) | +10% | Stable |
| **Worst Case (Pessimistic)** | -5% | +8% (supply chain) | 0% (freeze) | Tight |

### 4.3 Sensitivity Analysis

Identifies which assumptions have the greatest impact on financial outcomes:

**Tornado diagram approach**: For each key variable, calculate the impact of ±10% (or ±1 standard deviation) change on the target metric (e.g., net income, cash balance). Rank variables by impact magnitude.

**Key sensitivity variables for cash flow**:
1. Revenue timing (payment terms, collection speed)
2. COGS percentage
3. Headcount and salary growth
4. CapEx timing
5. Tax rate changes
6. Working capital cycle (DSO, DPO, DIO)
7. Interest rate on debt
8. FX rates (for multi-currency)

### 4.4 Monte Carlo Simulation for Cash Flow

Monte Carlo simulation uses repeated random sampling to model uncertainty:

**Algorithm**:
1. **Define distributions** for each uncertain variable (e.g., revenue growth ~ Normal(8%, 3%), COGS ~ Uniform(45%, 55%)).
2. **Run N simulations** (typically 1,000-10,000): For each simulation, randomly sample values from each distribution and compute the resulting cash flow.
3. **Aggregate results**: Compute mean, median, standard deviation, percentiles (P10, P50, P90) of the simulated cash flows.
4. **Output**: Probability distribution of cash position, likelihood of cash shortfall, value-at-risk.

**Monte Carlo pattern (from Wikipedia)**:
```
s = 0;
for i = 1 to n:
    run simulation with random inputs -> result_i
    s = s + result_i
mean_estimate = s / n
```

**Application to cash flow**:
- Define probability distributions for: revenue collection, expense timing, CapEx, debt repayments.
- Run 5,000 simulations of monthly cash flows over 12 months.
- Output: "There is a 10% chance cash will fall below $X in month Y."
- Use for: cash reserve sizing, credit line planning, investment timing.

**Limitations**: Computationally expensive; requires accurate distribution assumptions; correlation between variables must be modeled.

  id: UUID
  forecast_id: UUID FK
  account_id: UUID FK
  department_id: UUID? FK
  fiscal_month: int
  amount: Decimal
  method: enum                     // driver_based, trend, manual, statistical
  driver_id: UUID? FK             // link to business driver if driver-based
  confidence: enum                 // high (±5%), medium (±15%), low (±30%)
}
```

### 5.2 Continuous Planning Cycle

Monthly update cycle:
1. **Week 1**: Import actuals from prior month into the model.
2. **Week 1-2**: Department managers review YTD actuals vs forecast; adjust remaining months.
3. **Week 2-3**: Finance consolidates, validates assumptions, identifies cross-department impacts.
4. **Week 3-4**: Present updated rolling forecast to leadership; lock the period.
5. **Day 1 of next month**: New forecast window opens (drop oldest month, add new month at end).

### 5.3 Forecast Accuracy Tracking

Track how well forecasts predict actuals:

| Metric | Formula | Target |
|--------|---------|--------|
| **MAPE** (Mean Absolute Percentage Error) | (1/n) * Σ|actual - forecast| / |actual| * 100 | < 10% |
| **Bias** | (1/n) * Σ(actual - forecast) | Near 0 (no systematic over/under) |
| **Weighted MAPE** | Σ|actual - forecast| / Σ|actual| * 100 | < 8% |
| **Hit Rate** | % of forecasts within ±X% of actual | > 80% within ±10% |

Forecast accuracy by lead time:
- Month-ahead: ~95% accurate
- Quarter-ahead: ~85% accurate
- Annual forecast: ~70-75% accurate

## 7. Capital Expenditure (CapEx) Budgeting

### 7.1 CapEx vs OpEx

| Characteristic | CapEx | OpEx |
|----------------|-------|------|
| **Definition** | Investments in long-term assets | Day-to-day operating costs |
| **Time horizon** | > 1 year benefit | Current period |
| **Accounting treatment** | Capitalized on balance sheet | Expensed on income statement |
| **Cash flow impact** | Large initial outflow, depreciation spread | Regular periodic outflows |
| **Examples** | Buildings, equipment, software licenses (3yr+), R&D projects | Salaries, rent, utilities, maintenance |
| **Depreciation** | Yes (straight-line, declining balance, units of production) | No |
| **Tax impact** | Depreciation deductions; potential bonus depreciation | Fully deductible in year |

### 7.2 Depreciation Methods

**Straight-line**:
```
annual_depreciation = (cost - salvage_value) / useful_life
```

**Declining balance** (double-declining):
```
annual_depreciation = book_value * (2 / useful_life)
```

**Units of production**:
```
depreciation_per_unit = (cost - salvage_value) / total_expected_units
annual_depreciation = depreciation_per_unit * units_produced_this_year
```

### 7.3 ROI Calculation for CapEx

```
ROI = (Net Benefits - Cost) / Cost * 100

NPV = Σ [Cash_Flow_t / (1 + discount_rate)^t] - Initial_Investment
IRR = discount_rate where NPV = 0
Payback Period = time to recover initial investment
Profitability Index = PV of future cash flows / Initial investment
```

**Capital budgeting techniques** (from Wikipedia):
- **NPV**: Gold standard — accept all positive NPV projects.
- **IRR**: Rate of return; accept if IRR > cost of capital.
- **Payback period**: Time to recover investment; simpler but ignores time value.
- **Equivalent Annual Cost (EAC)**: Annualized cost for comparing projects of different lifespans.
- **Real options**: Values managerial flexibility to expand, abandon, or delay.

## 9. Integration with Accounting

### 9.1 Budget-to-Actual Connection Flow

```
Accounting System (GL)
  ↓
Journal Entries (actual transactions)
  ↓
Budget System pulls actuals via:
  - API integration (real-time or scheduled batch)
  - Direct GL query
  - File import (CSV/JSON from accounting export)
  ↓
Variance Engine computes:
  - Period-by-period comparison (budget line vs actual by same dimensions)
  - Cumulative YTD comparison
  - Rolling 12-month comparison
  ↓
Variance Alerts triggered based on materiality rules
  ↓
Dashboard updated with current period + YTD variances
```

### 9.2 Dimensional Mapping

Budget lines and actual entries must share the same dimensional taxonomy:

```
// Chart of Accounts mapping
BudgetAccount mapping -> GL Account
BudgetDepartment mapping -> GL Department/Class
BudgetCostCenter mapping -> GL Cost Center
BudgetProject mapping -> GL Project Code

// Validation rules
- Every journal entry must map to a valid BudgetAccount
- Unmapped entries flagged for review (e.g., new GL accounts without budget counterpart)
- Cross-period matching: budget for month M compared against actuals posted in month M
  (not the posting date but the accounting period)
```

### 9.3 Automatic Variance Calculation

Real-time variance computation approach:

**Event-driven**: When a journal entry is posted:
1. Extract account, department, period, amount.
2. Look up corresponding budget line(s).
3. Compute variance = actual - budget.
4. Update materialized variance record.
5. Check against alert rules; trigger notifications if exceeded.

**Batch computation**: Nightly or hourly:
1. Pull all new/modified journal entries since last run.
2. Aggregate by budget dimensions.
3. Recompute all variances for affected periods.
4. Update dashboard materialized views.

**Recommended**: Hybrid — event-driven for real-time dashboards, batch for comprehensive reporting.

### 9.4 Period Close Integration

```
Monthly Close Process:
1. Accounting closes the period in GL (posts final adjusting entries)
2. Budget system pulls final actuals for the closed period
3. Variance engine recomputes all variances for closed period
4. Forecast system updates: actuals replace forecast for closed period
5. Rolling forecast extends by one month
6. Variance report generated and distributed
7. Budget version updated (if reforecast cycle)
8. Dashboard refreshed with final period data
```
