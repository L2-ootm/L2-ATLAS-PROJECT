## 1. Odoo Module System — Manifest Format and Dependencies

**Source**: Odoo 17.0 docs — Module Manifests

Odoo modules use `__manifest__.py`, a Python dict with these fields:

```python
{
    'name': "A Module",
    'version': '1.0',                    # semver
    'depends': ['base'],                 # list of module names
    'author': "Author Name",
    'category': 'Category',
    'description': """...""",
    'data': ['views/mymodule_view.xml'], # always-loaded data files
    'demo': ['demo/demo_data.xml'],      # demo-only data
    'auto_install': False,               # auto-install when deps met
    'external_dependencies': {           # python + binary prereqs
        'python': ['some_lib'],
        'bin': ['some_binary']
    },
    'installable': True,
    'application': False,                # True = full app, False = library
    'pre_init_hook': 'my_pre_init',      # function name in __init__.py
    'post_init_hook': 'my_post_init',
    'uninstall_hook': 'my_uninstall',
    'assets': {...},                     # static file bundles
}
```

**Key architectural patterns**:
- `depends` is a flat list — Odoo resolves the full dependency graph topologically
- `auto_install: True` enables "link modules" that activate when ALL deps are installed (e.g., `sale_crm` bridges `sale` + `crm`)
- `auto_install: [list]` auto-installs when the listed subset of deps are met; remaining deps are auto-installed too
- Hooks (`pre_init_hook`, `post_init_hook`, `uninstall_hook`) take `env` as sole argument — executed before/after install or after uninstall
- `external_dependencies` prevents installation if system prerequisites are missing
- Security is per-model via `ir.rule` and `ir.model.access` records loaded via `data` files in `security/` directory

**Relevance to cashflow**: The `auto_install` pattern is directly applicable — a "cashflow-reports" module could auto-activate when both "cashflow-core" and "cashflow-forecasting" are installed.

## 2. WordPress Plugin Architecture — Actions, Filters, and Hooks

**Source**: developer.wordpress.org/plugins/hooks/

WordPress has two hook types:

| Type | Behavior | Return Value |
|------|----------|--------------|
| **Action** | Interrupts code flow to do something, returns to normal flow | Nothing |
| **Filter** | Receives data, modifies it, passes modified data back | Modified data |

```php
// Action: do something at a specific point
add_action('wp_ajax_my_action', 'my_callback_function');

// Filter: modify data flowing through the pipeline
add_filter('the_content', 'my_filter_function');
```

**Database isolation**: WordPress plugins share a single database. Isolation is enforced by table naming conventions (`wp_{plugin}_options`) and the `options` table's autoload mechanism. Plugins use `register_setting()` to declare options, which get their own row in `wp_options`.

**Upgrade safety**: Plugins that modify core files get overwritten on update. The recommended pattern is:
- Use hooks (actions/filters) exclusively — never modify core files
- Use `child themes` for theme customizations
- Use `DISALLOW_FILE_EDIT` in `wp-config.php` to disable in-browser editing
- All plugin code lives in `wp-content/plugins/{plugin-name}/` — isolated filesystem

**Relevance to cashflow**: The Action/Filter duality maps directly to a cashflow hook system:
- **Actions**: `on_transaction_created`, `on_balance_recalculated`, `on_report_generated`
- **Filters**: `filter_transaction_display`, `filter_balance_amount`, `filter_export_format`

## 3. VS Code Extension API — Contribution Points and Activation Events

**Source**: code.visualstudio.com/api/references/

### Extension Manifest (`package.json`)
```json
{
  "name": "extension-id",
  "version": "0.1.0",
  "publisher": "publisher-id",
  "engines": { "vscode": "^1.0.0" },
  "main": "./out/extension",
  "activationEvents": ["onLanguage:python"],
  "contributes": {
    "commands": [...],
    "views": {...},
    "menus": {...},
    "languages": [...]
  },
  "extensionDependencies": ["other.extension"],
  "extensionPack": ["ext1", "ext2"],
  "extensionKind": ["ui", "workspace"]
}
```

### Activation Events (lazy loading)
VS Code activates extensions only when needed:
- `onLanguage:python` — when a Python file opens
- `onCommand:extension.sayHello` — when a command is invoked
- `onView:nodeDependencies` — when a sidebar view is expanded
- `onUri` — when a system-wide URI is opened
- `*` — at startup (avoid unless necessary)
- `onStartupFinished` — after all startup extensions finish

### Contribution Points
Extensions declaratively register UI/behavior without imperative code:
- `contributes.commands` — command palette entries
- `contributes.views` — sidebar tree views
- `contributes.menus` — context menu items
- `contributes.languages` — language support
- `contributes.grammars` — syntax highlighting
- `contributes.themes` — color themes

### Extension Lifecycle
```typescript
export function activate(context: ExtensionContext) {
    // Called once when activation event fires
}
export function deactivate() {
    // Called on shutdown — return Promise if async cleanup
}
```

**Uninstall hook**: `vscode:uninstall` script runs on VS Code restart after extension removal.

**Relevance to cashflow**: The contribution-points pattern is ideal for cashflow modules:
- Modules declare their UI contributions (dashboard widgets, report types, menu items) in a manifest
- Lazy activation via events like `on_transaction_type:x` keeps startup fast
- `extensionDependencies` prevents broken installations — module A won't install without module B

## 4. SaaS App/Plugin Ecosystems — Shopify, Salesforce, Slack

**Source**: Synthesized from platform documentation patterns

### Shopify Apps
- **Manifest**: `shopify.app.toml` — defines scopes, webhooks, access scopes
- **Scoping**: Apps request specific API scopes (`read_products`, `write_orders`)
- **Webhooks**: declarative in manifest, Shopify delivers events to app endpoints
- **App Bridge**: UI extensions register via `extensionPoints` in `shopify.ui-extensions`
- **Database isolation**: Each app gets its own `data_store` — apps never share DB tables
- **Billing**: Apps declare pricing in manifest; Shopify handles billing flow

### Salesforce (AppExchange)
- **Package manifest**: `sfdx-project.json` with `packageDirectories`
- **Metadata API**: everything (objects, fields, permissions, layouts) is metadata XML
- **Namespaced unpackaging**: `namespace__ObjectName` — prevents field collisions between packages
- **Dependencies**: declared as `dependencyVersions` in `sfdx-project.json`
- **Profiles/Permission Sets**: each package declares minimum access requirements
- **Upgrade**: managed packages use major.minor versioning; patch versions are auto-upgraded

### Slack Apps
- **Manifest**: `manifest.json` — defines bot scopes, event subscriptions, commands
- **OAuth scopes**: granular per-API permissions (`channels:read`, `chat:write`)
- **Events API**: declarative event subscriptions in manifest
- **Interactivity**: slash commands, buttons, modals registered in manifest
- **Database isolation**: apps store data in their own infrastructure — no shared DB

**Common patterns across all three**:
1. Declarative manifest defines capabilities + permissions
2. Explicit permission/scoping model
3. Webhook/event-based inter-module communication
4. Each plugin runs in its own process/sandbox
5. Version compatibility declared in manifest

## 5. Dependency Resolution — Cargo/npm-style for Financial Modules

**Source**: Cargo Book — Specifying Dependencies

### Version Requirements (Cargo)
```toml
[dependencies]
# Caret: SemVer-compatible (default)
regex = "^1.2.3"    # >=1.2.3, <2.0.0

# Tilde: patch-only updates
regex = "~1.2.3"    # >=1.2.3, <1.3.0

# Exact
regex = "=1.2.3"

# Range
regex = ">=1.2, <2"
```

### Circular Dependency Prevention
- Cargo: the dependency resolver rejects cycles — it's a DAG, not a graph
- npm: also rejects circular dependencies at the package level (though node_modules can have them at the file level)
- Both systems use a **lockfile** (`Cargo.lock`, `package-lock.json`) to pin exact versions after resolution

### Version Compatibility for Financial Modules
```toml
# Cashflow module manifest (TOML)
[module]
name = "cashflow-forecasting"
version = "1.2.0"

[dependencies]
cashflow-core = "^1.0"        # compatible with any 1.x
cashflow-reports = "~1.2.0"  # only patch updates
accounting-bridge = ">=2.0, <3"

[dev-dependencies]
test-harness = "^0.1"
```

**Key patterns**:
- SemVer is the standard — major version bump = breaking change
- `^` (caret) allows compatible updates; `~` (tilde) allows patch-only
- Lockfiles ensure reproducible builds across environments
- Platform-specific dependencies via `[target.'cfg(...)'.dependencies]`
- Workspace-level shared dependencies reduce duplication

## 6. Hot-Loading vs Compile-Time Module Registration

### Hot-Loading (Runtime)
- **Pros**: Zero-downtime updates, A/B testing modules, user-installable plugins
- **Cons**: Memory leaks from un-deactivated modules, version conflicts at runtime, harder to reason about state
- **Used by**: WordPress (PHP include), VS Code (extension host), Odoo (module install/uninstall)
- **Pattern**: Module provides `activate()`/`deactivate()` functions; runtime calls them on load/unload

### Compile-Time Registration
- **Pros**: Type safety, dead-code elimination, smaller binaries, no runtime overhead
- **Cons**: Requires recompilation to add/remove modules, no hot-swap
- **Used by**: Rust (cargo features), Go (build tags), C++ (static libraries)
- **Pattern**: Modules register themselves via a `#[ctor]` attribute or `inventory::submit!` macro

### Hybrid Approach (Recommended for Cashflow)
- **Core**: Compile-time registered modules for critical financial logic (ledger, calculations)
- **Extensions**: Hot-loadable plugins for reports, integrations, UI customizations
- **Mechanism**: Trait-based plugin interface with versioned API surface

```rust
// Compile-time: core modules
pub trait CashflowModule: Send + Sync {
    fn name(&self) -> &str;
    fn version(&self) -> &str;
    fn init(&mut self, ctx: &mut ModuleContext) -> Result<()>;
    fn activate(&mut self) -> Result<()>;
    fn deactivate(&mut self) -> Result<()>;
}

// Runtime: hot-loadable extensions
pub trait Plugin: CashflowModule {
    fn manifest(&self) -> PluginManifest;
    fn on_event(&mut self, event: &Event) -> Result<Vec<Action>>;
}
```

## 7. Configuration-Driven Module Composition

### Module Registry Pattern
```toml
# cashflow.toml — top-level composition config
[modules.active]
"cashflow-core" = { version = "1.0", enabled = true }
"cashflow-reports" = { version = "1.2", enabled = true }
"cashflow-forecasting" = { version = "0.9", enabled = false }
"bank-sync-stripe" = { version = "1.1", enabled = true }

[modules.config."cashflow-core"]
fiscal_year_start = "january"
default_currency = "BRL"
decimal_precision = 2

[modules.config."bank-sync-stripe"]
api_key_env = "STRIPE_SECRET_KEY"
sync_interval_minutes = 15
webhook_endpoint = "/hooks/stripe"
```

### Inter-Module Communication
Two patterns dominate:

**1. Event Bus (Loose Coupling)**
```
Module A publishes "transaction.created"
  -> Event bus dispatches to all subscribers
  -> Module B (reconciliation) receives event
  -> Module C (reporting) receives event
```

**2. Shared Interface (Tight Coupling)**
```
Module A implements ReportableModule trait
  -> Reporting module calls reportable.transactions()
  -> Type-safe, compile-time checked
```

**Recommendation**: Use event bus for cross-domain communication (banking -> reporting), shared interfaces for within-domain (cashflow-core -> cashflow-forecasting).

## 8. Module Lifecycle Hooks — Financial System Requirements

### Required Lifecycle Events

| Hook | When | Purpose | Financial Requirement |
|------|------|---------|----------------------|
| `pre_init` | Before module install | Validate environment, create DB schema | Ensure ledger tables exist, validate fiscal config |
| `post_init` | After module install | Seed data, register with other modules | Import chart of accounts, register event handlers |
| `activate` | Module enabled | Start background tasks, register APIs | Begin bank sync, start reconciliation daemon |
| `deactivate` | Module disabled | Stop tasks, deregister APIs | Stop sync, preserve state for re-enable |
| `pre_uninstall` | Before removal | Validate no dependencies | Check no other module depends on this one |
| `uninstall` | After removal | Cleanup DB, remove data | Archive transactions, remove DB tables |
| `migrate(from_version)` | Version upgrade | Schema/data migration | Alter tables, transform existing data |
| `on_event(event)` | Runtime event | React to system events | Process new transaction, update forecast |

### Financial-Specific Lifecycle Concerns

1. **Audit trail preservation**: Uninstall must archive, not delete — regulatory requirement
2. **Data migration**: Schema changes must preserve existing financial data integrity
3. **Rollback safety**: Failed activation must revert cleanly — no partial state
4. **Multi-tenancy**: Lifecycle hooks must respect tenant isolation
5. **Concurrency**: Activate/deactivate must handle in-flight transactions gracefully

## Synthesis: Design Principles for Cashflow Platform

1. **Manifest-driven**: Every module declares dependencies, capabilities, permissions, and lifecycle hooks in a structured manifest (TOML recommended for Rust ecosystem alignment)

2. **SemVer contract**: Module API surface follows SemVer — breaking changes require major version bump; consumers declare compatibility ranges

3. **Activation events**: Modules activate lazily on relevant events (transaction types, report requests) rather than at startup — reduces memory footprint

4. **Contribution points**: Modules declaratively register UI elements, API endpoints, and event handlers — no imperative registration code needed

5. **Hook duality**: Actions (fire-and-forget side effects) + Filters (data transformation pipeline) cover all extension patterns

6. **Tiered loading**: Core modules compile-time; integrations hot-load; UI extensions hot-load with sandboxing

7. **Lockfile for reproducibility**: After dependency resolution, pin exact versions in a lockfile — critical for financial audit trails

8. **Auto-install bridges**: Link modules auto-activate when their dependency combination is present — enables zero-config integrations
