## 1. Service Worker Caching Strategies for Financial Data

**Pattern**: Use a **hybrid caching strategy** — cache-first for static assets, network-first for financial API data.

**Why this matters for cashflow**: Financial dashboards need fresh data but must degrade gracefully offline. The service worker acts as a request router with per-URL strategies.

| Resource Type | Strategy | Rationale |
|---|---|---|
| Static assets (JS, CSS, fonts) | CacheFirst | Versioned, never change between deploys |
| Dashboard data (KPIs, balances) | NetworkFirst with timeout | Fresh data preferred, cache fallback for offline |
| Transaction history | StaleWhileRevalidate | Show cached immediately, background refresh |
| Payment submission endpoints | NetworkOnly | Never cache sensitive POST operations |
| Invoice PDFs / receipts | CacheFirst with expiration | Large files, cache after first fetch |
| User preferences | CacheOnly | Local-first, sync to server separately |

**Code — Workbox strategy registration**:
```js
import { registerRoute } from 'workbox-routing';
import { CacheFirst, NetworkFirst, StaleWhileRevalidate, NetworkOnly } from 'workbox-strategies';
import { ExpirationPlugin } from 'workbox-expiration';
import { BackgroundSyncPlugin } from 'workbox-background-sync';

// Static assets — cache forever, bust on deploy
registerRoute(
  ({ request }) => ['style', 'script', 'font', 'image'].includes(request.destination),
  new CacheFirst({ cacheName: 'static-assets', plugins: [new ExpirationPlugin({ maxAgeSeconds: 30 * 24 * 60 * 60 })] })
);

// Financial dashboard data — network-first with 3s timeout
registerRoute(
  ({ url }) => url.pathname.startsWith('/api/dashboard'),
  new NetworkFirst({ cacheName: 'dashboard-data', networkTimeoutSeconds: 3 })
);

// Transaction history — stale-while-revalidate
registerRoute(
  ({ url }) => url.pathname.startsWith('/api/transactions'),
  new StaleWhileRevalidate({ cacheName: 'transactions' })
);

// Payment submissions — network only, queue on failure
const bgSyncPlugin = new BackgroundSyncPlugin('payment-queue', { maxRetentionTime: 24 * 60 });
registerRoute(
  ({ url }) => url.pathname.startsWith('/api/payments'),
  new NetworkOnly({ plugins: [bgSyncPlugin] })
);
```

## 3. IndexedDB for Offline Financial Data Storage

**Libraries** (ranked by fit for financial data):

| Library | Type | Best For | Size |
|---|---|---|---|
| **Dexie.js** | Wrapper | Most use cases, great query API | ~30KB |
| **idb** (by Jake Archibald) | Minimal wrapper | When you want raw IndexedDB with Promises | ~1.2KB |
| **RxDB** | Full database | Complex sync, CRDT support, multi-device | ~120KB |
| **PouchDB** | CouchDB-compatible | CouchDB/CouchDB sync backend | ~120KB |
| **LocalForage** | Simple key/value | Simple caching, not relational | ~30KB |

**Recommended**: **Dexie.js** for the cashflow platform — excellent TypeScript support, compound indexes, and sync-friendly design.

**Schema for offline financial data**:
```ts
import Dexie, { Table } from 'dexie';

interface Transaction {
  id: string;           // UUID, client-generated
  invoiceId?: string;
  type: 'income' | 'expense' | 'transfer';
  amount: number;
  currency: string;
  category: string;
  description: string;
  date: string;         // ISO 8601
  status: 'pending' | 'completed' | 'failed';
  syncStatus: 'synced' | 'pending' | 'conflict';
  syncVersion: number;  // Vector clock or version counter
  createdAt: string;
  updatedAt: string;
}

interface Invoice {
  id: string;
  clientId: string;
  items: InvoiceItem[];
  total: number;
  currency: string;
  status: 'draft' | 'sent' | 'paid' | 'overdue' | 'cancelled';
  dueDate: string;
  syncStatus: 'synced' | 'pending' | 'conflict';
  syncVersion: number;
  createdAt: string;
  updatedAt: string;
}

const db = new Dexie('CashflowDB');
db.version(1).stores({
  transactions: 'id, invoiceId, type, category, date, syncStatus, [syncStatus+syncVersion]',
  invoices: 'id, clientId, status, dueDate, syncStatus, [syncStatus+syncVersion]',
  accounts: 'id, name, type',
  contacts: 'id, name, email',
  syncQueue: '++queueId, tableName, recordId, operation, timestamp',
});

export type { Transaction, Invoice };
export { db };
```

**Why `syncVersion` matters**: Enables conflict detection on sync — if server version > client version, a conflict exists. The `syncQueue` table stores operations performed offline for replay when connectivity returns.

## 4. Offline-to-Online Sync Strategy

**Pattern**: **Outbox with retry queue** — write locally first, queue sync operations, replay on reconnect.

**Architecture**:
```
User Action → IndexedDB Write → Sync Queue Entry → Background Sync fires → 
  POST to server → Server ACKs → Queue entry removed → UI updates syncStatus
```

**Implementation**:
```ts
// lib/sync-manager.ts
import { db } from './db';

class SyncManager {
  private isOnline: boolean = navigator.onLine;
  private syncInProgress: boolean = false;

  constructor() {
    window.addEventListener('online', () => { this.isOnline = true; this.flushQueue(); });
    window.addEventListener('offline', () => { this.isOnline = false; });
  }

  async enqueueOperation(table: string, recordId: string, operation: 'create' | 'update' | 'delete', data: any) {
    await db.syncQueue.add({
      tableName: table,
      recordId,
      operation,
      data,
      timestamp: new Date().toISOString(),
    });
    
    // Register background sync
    const reg = await navigator.serviceWorker.ready;
    await reg.sync.register('cashflow-sync');
  }

  async flushQueue() {
    if (this.syncInProgress || !this.isOnline) return;
    this.syncInProgress = true;
    
    try {
      const pending = await db.syncQueue.orderBy('queueId').toArray();
      for (const entry of pending) {
        try {
          const response = await fetch(`/api/sync/${entry.tableName}`, {
            method: 'POST',
            headers: { 'Content-Type': 'application/json' },
            body: JSON.stringify({ operation: entry.operation, id: entry.recordId, data: entry.data }),
          });
          
          if (response.ok) {
            await db.syncQueue.delete(entry.queueId);
            await this.markSynced(entry.tableName, entry.recordId);
          }
        } catch (err) {
          // Server unreachable — will retry on next online event
          break;
        }
      }
    } finally {
      this.syncInProgress = false;
    }
  }

  private async markSynced(table: string, recordId: string) {
    if (table === 'transactions') {
      await db.transactions.update(recordId, { syncStatus: 'synced' });
    } else if (table === 'invoices') {
      await db.invoices.update(recordId, { syncStatus: 'synced' });
    }
  }
}

export const syncManager = new SyncManager();
```

**Conflict resolution strategy for financial data**:
- **Last-Write-Wins (LWW)** with **server as source of truth** for non-critical fields (descriptions, categories)
- **Client-wins** for locally-initiated edits the user explicitly made offline
- **Server-wins** for balance calculations and aggregate values (always compute server-side)
- **Manual resolution** for conflicts on payment status or invoice amounts — show the user both versions

## 5. Background Sync and Push Notifications

**Background Sync** (for payment queue replay):
```js
// service-worker.js
self.addEventListener('sync', (event) => {
  if (event.tag === 'cashflow-sync') {
    event.waitUntil(flushSyncQueue());
  }
});

async function flushSyncQueue() {
  // Import Dexie in SW context via IndexedDB directly or share logic
  const db = await openDB('CashflowDB', 1);
  const pending = await db.getAll('syncQueue');
  
  for (const entry of pending) {
    const response = await fetch(`/api/sync/${entry.tableName}`, {
      method: 'POST',
      body: JSON.stringify(entry),
    });
    if (response.ok) await db.delete('syncQueue', entry.queueId);
  }
}
```

**Push Notifications** for financial alerts:
```js
// Subscribe to push
const reg = await navigator.serviceWorker.ready;
const subscription = await reg.pushManager.subscribe({
  userVisibleOnly: true,
  applicationServerKey: urlBase64ToUint8Array(VAPID_PUBLIC_KEY),
});

// Send to server
await fetch('/api/push/subscribe', { method: 'POST', body: JSON.stringify(subscription) });

// Service worker — handle push
self.addEventListener('push', (event) => {
  const { title, body, url } = event.data.json();
  event.waitUntil(
    self.registration.showNotification(title, {
      body,
      icon: '/icons/notification.png',
      badge: '/icons/badge.png',
      data: { url },
      actions: [
        { action: 'view', title: 'View' },
        { action: 'dismiss', title: 'Dismiss' },
      ],
    })
  );
});

self.addEventListener('notificationclick', (event) => {
  event.notification.close();
  if (event.action === 'view') {
    event.waitUntil(clients.openWindow(event.notification.data.url));
  }
});
```

**Notification use cases for cashflow**:
- Payment overdue alerts (cron on server checks due dates)
- Budget threshold warnings (e.g., "You've spent 80% of your marketing budget")
- New invoice received from a client
- Payment received confirmation
- Recurring payment reminder (day before)

**Library**: `web-push` (npm) for server-side push message sending with VAPID signing.

## 6. Responsive Financial Dashboard Design

**Breakpoint strategy** for financial UIs:

| Breakpoint | Width | Layout | Key Adaptations |
|---|---|---|---|
| Mobile S | < 375px | Single column | Stack KPIs, hide secondary metrics |
| Mobile L | 375-767px | Single column | Card-based KPIs, collapsible tables |
| Tablet | 768-1023px | 2 columns | Side-by-side charts, condensed tables |
| Desktop | 1024-1439px | 3 columns | Full dashboard grid |
| Wide | ≥ 1440px | 4 columns | Expanded charts, full data tables |

**Mobile financial table patterns**:
- **Card layout** instead of wide tables on mobile — each transaction as a card with amount, date, and status badge
- **Horizontal scroll** with sticky first column for detailed data tables (client name + amounts)
- **Pull-to-refresh** for dashboard data refresh
- **Bottom sheet** for transaction detail / filter panels (instead of modals)

**KPI cards on mobile**:
```css
.kpi-grid {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(150px, 1fr));
  gap: 0.75rem;
  padding: 1rem;
}

@media (min-width: 768px) {
  .kpi-grid {
    grid-template-columns: repeat(4, 1fr);
  }
}
```

**Chart sizing**:
- Mobile: full-width single-column charts, reduced data points (monthly aggregation)
- Tablet: 2 charts per row, weekly aggregation
- Desktop: 3-4 charts per row, daily granularity
- Use `react-responsive` or CSS container queries to detect breakpoints

## 7. Mobile Touch UX for Financial Actions

**Swipe gestures for invoice status changes**:
- Swipe right → Mark as Paid (green background reveal)
- Swipe left → Mark as Overdue (red background reveal)
- Implement with `react-swipeable` or native touch events

**Touch-optimized financial forms**:
- Large touch targets (minimum 44x44px for interactive elements)
- Numeric keyboard for amount inputs: `<input type="text" inputmode="decimal" pattern="[0-9]*\.?[0-9]*">`
- Date picker: native `<input type="date">` on mobile for better UX than custom pickers
- Category selection: bottom sheet with search, not dropdown
- Amount input: large font, auto-format with currency symbol, right-aligned

**Biometric authentication** (Web Authentication API):

## 8. Data Sync Patterns: CRDTs and Operational Transformation

**For financial data, the recommended approach is NOT full CRDTs** — financial data has natural hierarchies and balance constraints that make pure CRDT merge semantics dangerous. Instead:

**Recommended: Event Sourcing + Version Vectors**
- Each mutation is an event (e.g., `TransactionCreated`, `InvoiceStatusChanged`)
- Events are append-only and idempotent (client-generated UUIDs)
- Version vectors track causality between client and server
- Server replays events in order to compute current state
- Conflicts are detected by comparing version vectors

**When CRDTs DO make sense**:
- **Collaborative document editing** (invoice line items being edited by multiple people)
- **Todo/task lists** associated with financial items
- **User preferences and settings**

**Libraries for CRDT-based sync**:
| Library | Use Case | Complexity |
|---|---|---|
| **Yjs** | Collaborative editing, real-time | Medium |
| **Automerge** | JSON document merge | Medium |
| **Replicache** | Full-stack sync with mutation spec | High |
| **ElectricSQL** | Postgres sync to client | High (infra) |

**Practical recommendation**: Use **event sourcing with a sync queue** (Finding #4) rather than CRDTs for the core cashflow platform. Reserve CRDTs only for collaborative features like shared invoice editing.

```tsx
// Intersection Observer for chart lazy loading
const LazyChart = ({ children }) => {
  const ref = useRef(null);
  const [visible, setVisible] = useState(false);

  useEffect(() => {
    const observer = new IntersectionObserver(
```

## 10. Security on Mobile PWA

**Secure storage**:
- **Web Crypto API** for encryption keys — never store raw keys in IndexedDB
- **WebAuthn** for biometric-gated access (see Finding #7)
- IndexedDB stores data in the browser's isolated storage — not accessible to other origins
- For sensitive data (access tokens, refresh tokens): store in `sessionStorage` and re-authenticate on app resume

**Content Security Policy** for financial PWA:
```
Content-Security-Policy:
  default-src 'self';
  script-src 'self' 'wasm-unsafe-eval';
  style-src 'self' 'unsafe-inline';
  connect-src 'self' https://api.l2systems.com.br;
  img-src 'self' data: https://cdn.l2systems.com.br;
  worker-src 'self';
  frame-ancestors 'none';
  base-uri 'self';
  form-action 'self';
  upgrade-insecure-requests;
```

**Other security measures**:
- **TLS everywhere** — enforce HTTPS, HSTS header with long max-age
- **Service worker scope restriction** — serve SW from root to prevent scope escape
- **No sensitive data in service worker cache** — mark API responses with `Cache-Control: no-store` for auth endpoints
- **Session timeout** — auto-lock after 5 minutes of inactivity, require re-authentication
- **Jailbreak/root detection** is NOT possible in PWA — this is a limitation vs native. Mitigate with server-side risk scoring and step-up authentication for high-value operations.

## 12. Offline-First UX Patterns for Financial Apps

**Key UX principles from web.dev** (applicable to cashflow):

1. **Inform users of state immediately** — show a persistent banner: "Offline — changes will sync when connected"
2. **Don't block content** — let users create/edit invoices offline, queue for sync
3. **Skeleton loading** — show wireframe placeholders while data loads
4. **Show sync status per record** — visual indicator (dot, icon) on each transaction/invoice showing sync state
5. **Auto-update on reconnect** — silently sync, show toast: "3 transactions synced"
6. **Provide offline by default** — cache dashboard data, transaction history, and contact list on first visit
7. **Educate about data cost** — show download sizes, offer low-bandwidth mode

**Connection state management**:
```tsx
// hooks/useConnectionStatus.ts
export function useConnectionStatus() {
  const [status, setStatus] = useState<'online' | 'offline' | 'slow'>('online');

  useEffect(() => {
    const handleOnline = () => {
      setStatus('online');
      // Trigger sync
      syncManager.flushQueue();
    };
    const handleOffline = () => setStatus('offline');

    // Detect slow connections
    if ('connection' in navigator) {
      const conn = navigator.connection;
      const updateSlow = () => setStatus(conn.effectiveType === '2g' || conn.saveData ? 'slow' : 'online');
      conn.addEventListener('change', updateSlow);
    }

    window.addEventListener('online', handleOnline);
    window.addEventListener('offline', handleOffline);
    return () => {
      window.removeEventListener('online', handleOnline);
      window.removeEventListener('offline', handleOffline);
    };
  }, []);

  return status;
}
```

## 13. PWA Update Mechanism

**Service worker update flow**:
1. New SW file detected (byte-different from current)
2. Browser installs new SW in waiting state
3. All old tabs close → new SW activates
4. `controllerchange` event fires in pages → trigger UI refresh

**User-facing update notification**:
```tsx
// hooks/useSWUpdate.ts
export function useSWUpdate() {
  const [updateAvailable, setUpdateAvailable] = useState(false);

  useEffect(() => {
    if ('serviceWorker' in navigator) {
      navigator.serviceWorker.addEventListener('controllerchange', () => {
        window.location.reload(); // Or show "Update available" banner
      });

      navigator.serviceWorker.ready.then((reg) => {
        reg.addEventListener('updatefound', () => {
          const newWorker = reg.installing;
          newWorker.addEventListener('statechange', () => {
            if (newWorker.state === 'installed' && navigator.serviceWorker.controller) {
              setUpdateAvailable(true);
            }
          });
        });
      });
    }
  }, []);

  const applyUpdate = () => {
    navigator.serviceWorker.controller?.postMessage({ type: 'SKIP_WAITING' });
  };

  return { updateAvailable, applyUpdate };
}
```

## 15. Key Risks and Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| Data conflicts on reconnect | Financial amounts could be inconsistent | Server-authoritative for balances; client-wins for edits; manual resolution UI for conflicts |
| Service worker cache staleness | User sees old financial data | Network-first for dashboard data; background sync for updates; explicit "refresh" action |
| IndexedDB storage limits | Data loss on low-storage devices | Use `navigator.storage.persist()`; implement storage quota monitoring; offer data export |
| Browser SW lifecycle | SW killed during long sync operations | Break sync into small chunks; use Background Sync API for retry; implement idempotent operations |
| No native security features | Can't detect jailbreak/root | Server-side risk scoring; step-up auth for high-value ops; session timeout; HTTPS enforcement |
| Offline payment queuing | Duplicate payments on reconnect | Idempotency keys per transaction; server deduplication; show pending payments prominently |
