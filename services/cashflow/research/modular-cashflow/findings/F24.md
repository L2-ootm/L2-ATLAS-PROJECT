## 1. API Documentation

### OpenAPI/Swagger Spec Generation
- **Primary source of truth**: All production-grade financial APIs (Stripe, Mercury, Plaid) use OpenAPI 3.x specs.
- **Stripe**: Maintains a versioned OpenAPI spec per release. Each major version (`2026-06-24.dahlia`) has its own spec. Specs are machine-readable and used to generate their SDK types.
- **Mercury**: Exposes OpenAPI spec at `https://docs.mercury.com/llms.txt` (Markdown-indexed for AI agents) and a full API description page. Docs powered by ReadMe (ReadMe.io).

### Interactive Documentation Platforms
| Platform | Strength | Used By |
|----------|----------|---------|
| **Redocly/Redoc** | Beautiful 3-panel layout, OpenAPI-native, SSR support | Many fintech APIs |
| **Scalar** | Modern, lightweight, Try-It-Out built-in | Growing adoption |
| **Stoplight Elements** | Bundles design + docs + mock server | Stoplight users |
| **ReadMe.io** | Full developer portal + docs + guides | Mercury, many startups |
| **Swagger UI** | Standard, widely recognized | OpenAPI Generator default |

### Recommendation for Cashflow Platform
- Use **OpenAPI 3.1** as the canonical spec format.
- Serve docs via **Redocly** (self-hosted or cloud) for the reference docs.
- Use **ReadMe.io** or a custom portal for guides, recipes, and onboarding (Mercury's model).

## 2. SDK Generation

### Tools Compared

| Tool | Languages | Strengths | Trade-offs |
|------|-----------|-----------|------------|
| **Speakeasy** (⭐ recommended) | TS, Python, Go, Java, Ruby, C#, PHP, Unity | Polished idiomatic SDKs, auto-generated docs snippets, contract tests, Terraform providers, MCP servers, npm/pub/pip publishing | Commercial (free tier available) |
| **OpenAPI Generator** | 50+ languages | Massive community (26.5k stars), server stubs + clients, Apache 2.0 | Quality varies per language, templates need customization |
| **Stencil (SmartBear)** | TS, Java, Python | Official Swagger tooling, tightly coupled to Swagger ecosystem | Less community traction |

### Speakeasy Specifics (from GitHub repo)
- Generates "SDK code that looks like you wrote it" — optimized for debuggability and modern idiomatics.
- Supports: TypeScript, Python, Go, Java, C#, PHP, Ruby, Unity.
- Built-in features: retry logic, pagination helpers, auth handling, error types, versioning.
- Generates clean code samples for syncing with API docs.
- CLI workflow: `speakeasy init` → `speakeasy generate` → publish.
- AI agent skills available: `/speakeasy:start-new-sdk-project`, `/speakeasy:validate-openapi-spec`, `/speakeasy:regenerate-sdk`.

### OpenAPI Generator Specifics
- 26.5k GitHub stars, 7.6k forks, Apache 2.0 license.
- Generates: client SDKs, server stubs, documentation, config, schema (Avro, GraphQL, Protobuf, MySQL, PostgreSQL, Postman Collection).
- Active technical committee per language.
- Community-maintained templates with quality varying per generator.

### Recommendation
- **Speakeasy** for TypeScript and Python SDKs (primary dev languages for cashflow consumers).
- **OpenAPI Generator** for Go and Ruby (broader coverage, open source).
- Automate SDK generation in CI: spec change → generate → publish to npm/PyPI.

## 3. Sandbox Environment

### Mercury Sandbox (Best Practice Model)
1. **Separate signup**: `sandbox.mercury.com/signup` (no real onboarding).
2. **Pre-loaded dummy data**: Organizations, accounts, transactions, balances.
3. **Separate API tokens**: Tokens created in sandbox don't work in production and vice versa.
4. **Dedicated base URL**: `https://api-sandbox.mercury.com/api/v1/` (production: `api.mercury.com`).
5. **Visual confirmation**: All sandbox API calls appear in the sandbox UI.
6. **OAuth2 sandbox**: Separate OAuth base URL (`oauth2-sandbox.mercury.com`).

### Stripe Test Mode
- Test mode uses the same API endpoints with test API keys (`sk_test_...`).
- Sandbox data: fake cards, test bank accounts, simulated payment flows.
- Stripe CLI provides local forwarding for webhook testing.
- Per-account API version control in test and live.

### Rate Limit Differences (Typical Pattern)
| Tier | Rate Limit | Burst |
|------|-----------|-------|
| Sandbox | 100 req/min (liberal) | High burst allowed |
| Free/Dev | 100 req/min | 10 burst |
| Pro | 1000 req/min | 100 burst |
| Enterprise | Custom | Custom |

### Recommendation
- Mirror Mercury's model: separate sandbox signup, dedicated base URL, pre-seeded data.
- Sandbox tokens are completely isolated from production.
- Provide a "seed data" API endpoint so developers can generate test scenarios programmatically.

- Tiered pricing: free tier → pro → enterprise.

  1. App registration flow (OAuth2 or API key).
  2. API key management with role-based scoping.
  3. Usage dashboard (requests, errors, latency).
  4. Webhook endpoint management with delivery logs.
  5. Billing integration for API usage metering.

## 5. Webhook Testing

### Stripe Webhook Testing (Gold Standard)
1. **CLI-based local testing**: `stripe listen --forward-to localhost:4242/webhook`
2. **Event triggering**: `stripe trigger payment_intent.succeeded`
3. **Signature verification**: HMAC-SHA256 with timestamp, 5-minute tolerance.
4. **Manual resend**: Dashboard or CLI (`stripe events resend <event_id>`).
5. **Event delivery logs**: HTTP status codes, retry status, pending/failed/delivered.
6. **Automatic retries**: Exponential backoff for up to 3 days (live), 3 retries over hours (sandbox).

### Webhook Payload Patterns
- **Snapshot events**: Full object at time of event (`data.object`).
- **Thin events**: Minimal payload + `fetchRelatedObject()` to retrieve current state.
- Both models allow handling: snapshot for simplicity, thin for reduced payload size.

### Signature Verification Pattern
```
Stripe-Signature: t=1492774577, v1=5257a869e7...
```
- Timestamp prevents replay attacks.
- Multiple v1 signatures during secret rotation (24h window).
- HMAC-SHA256 with endpoint secret as key.

### Recommendation
- Provide a CLI tool for local webhook testing (`cashflow listen --forward-to localhost:xxxx/webhook`).
- Support both snapshot and thin event models.
- Implement HMAC-SHA256 signature verification with timestamp tolerance.
- Build a webhook dashboard with delivery logs, retry status, and manual resend.

## 7. Versioning and Deprecation

### Stripe Versioning (Gold Standard)
- **Date-based versions**: `2026-06-24.dahlia` — each major release is a named date.
- **Monthly minor releases**: Backward-compatible, same major version name.
- **Per-account default**: Controlled in Workbench/Dashboard.
- **Per-request override**: `Stripe-Version` header.
- **SDK version pinning**: SDKs (v6+) pin to the API version at release time.
- **Webhook versioning**: Events use the API version at endpoint creation time.
- **Migration guides**: Each version has upgrade notes (`/upgrades.md`).
- **Backward-compatible changes**: Never break existing endpoints.

### Deprecation Best Practices
- **Sunset headers**: `Sunset: Sat, 01 Jan 2028 00:00:00 GMT`
- **Deprecation headers**: `Deprecation: true`
- **Migration guides**: Step-by-step for each breaking change.
- **Compatibility matrices**: Show which SDK version works with which API version.
- **Long runway**: 12-18 months minimum before removing deprecated endpoints.

### Recommendation
- Adopt Stripe's date-based versioning: `v{YYYY-MM-DD}`.
- Pin SDK versions to API versions at release time.
- Publish migration guides for every breaking change.
- Maintain a compatibility matrix (SDK version × API version).

## 8. Rate Limiting and Quotas

### Standard Rate Limit Headers
```
X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 742
X-RateLimit-Reset: 1625097600
```

### Burst Handling
- Allow short bursts (10-50 requests above limit) with `Retry-After` header on 429.
- Exponential backoff recommended for clients.
- Token bucket algorithm for smooth rate limiting.

### Per-Key Limits
- Different limits for test vs production keys.
- Role-based limits (read-only keys get higher limits).
- Per-endpoint rate limiting for expensive operations (report generation).

### Recommendation
- Implement standard `X-RateLimit-*` headers on all responses.
- Use `429 Too Many Requests` with `Retry-After` header.
- Per-key limits: sandbox (100/min), free (100/min), pro (1000/min), enterprise (custom).
- Burst allowance: 2x limit for 10-second windows.

## 9. Error Handling

### Stripe Error Response Structure (Gold Standard)
```json
{
  "error": {
    "type": "invalid_request_error",
    "code": "resource_missing",
    "message": "No such customer: 'cus_xxx'",
    "param": "customer",
    "doc_url": "https://docs.stripe.com/error-codes/resource-missing",
    "request_log_url": "https://dashboard.stripe.com/test/logs/req_xxx"
  }
}
```

### Error Types
| Type | Description | HTTP Status |
|------|-------------|-------------|
| `invalid_request_error` | Bad parameters | 400 |
| `authentication_error` | Auth failed | 401 |
| `authorization_error` | Insufficient permissions | 403 |
| `not_found_error` | Resource doesn't exist | 404 |
| `rate_limit_error` | Too many requests | 429 |
| `api_error` | Server-side issue | 500 |

### Key Error Fields
- `type`: Programmatic error category.
- `code`: Specific error code for handling (e.g., `resource_missing`, `card_declined`).
- `message`: Human-readable (safe to show to users for card errors).
- `param`: Which parameter caused the error (for form field highlighting).
- `doc_url`: Link to error documentation.
- `request_log_url`: Link to request log in dashboard.

### Recommendation
- Return structured JSON errors on all endpoints.
- Include `doc_url` linking to the error code page.
- Include `request_log_url` for debugging in the developer portal.
- Catalog all error codes with descriptions and troubleshooting steps.

## Cross-Task Transferable Findings

1. **Speakeasy is the strongest SDK generator** for TypeScript and Python — produces idiomatic, debuggable, publishable SDKs with auto-generated docs snippets.
2. **Stripe's date-based versioning** (`2026-06-24.dahlia`) is the gold standard — adopt this pattern for any financial API.
3. **Mercury's sandbox model** (separate signup, pre-seeded data, dedicated base URL) is simpler to implement than Stripe's same-endpoint test/live model.
4. **Webhook signature verification** (HMAC-SHA256 + timestamp) is non-negotiable for security — implement from day one.
5. **Structured error responses** with `doc_url` and `request_log_url` dramatically reduce developer support burden.
6. **`X-RateLimit-*` headers** on every response prevent 429 surprises and enable client-side backoff.
