# F15: Deployment and DevOps Patterns for Modular Financial Platforms

## 1. Vercel Deployment for Next.js Financial Apps

### Edge Functions and Serverless Functions
- **Serverless Functions**: Vercel handles zero server management with fluid compute (reduced cold starts, lower latency). Functions run in a single region by default (`iad1`), configurable per project.
- **Regional control**: Pin functions to a region close to your data source (e.g., Supabase region). Critical for financial data locality compliance.
- **Fluid compute**: Concurrent execution within the same instance reduces cold starts and lowers cost — important for financial dashboards with bursty traffic patterns.
- **Limits**: Function duration, payload size, and memory limits apply. For heavy financial calculations, consider chunking work or using background tasks.

### ISR for Financial Dashboards
- ISR serves cached static pages while regenerating content in the background (stale-while-revalidate).
- **Recommended for**: Dashboard pages that update on a schedule (minutes to hours), not real-time transaction views.
- **Implementation** (Next.js App Router):
  ```typescript
  // app/dashboard/page.tsx
  export const revalidate = 300; // 5 minutes
  ```
- **On-demand revalidation**: Trigger via API when data changes (e.g., after a transaction import).
- **Failure behavior**: If revalidation fails, Vercel keeps serving stale content with a 30-second TTL retry.
- **ISR cache lives 31 days** or until revalidation — scoped per deployment.

### Deployment Environments
- **Three default environments**: Local, Preview (pre-production), Production.
- **Custom environments** (Pro/Enterprise): Create `staging`, `QA`, etc. with unique env vars.
- **Preview deployments**: Auto-created on non-production branch pushes and PRs. Each gets a unique URL.
- **Production**: Push to `main` auto-deploys. Manual promotion available via `vercel --prod`.
- **Branch tracking**: Custom environments can track specific branches for auto-deployment.

**Recommendation**: Use `preview` for PR testing, create a `staging` custom environment for pre-production validation. Each environment gets its own `DATABASE_URL` pointing to separate Supabase projects or schemas.

## 2. Database Migrations in Production

### Prisma Migrate Production Workflow
- **Never use `migrate dev` in production** — use `migrate deploy` instead.
- `migrate deploy` applies pending migrations without shadow database, drift detection, or reset.
- **Advisory locking**: Prisma uses advisory locks with a 10-second timeout to prevent concurrent migration runs.
- Disable with `PRISMA_SCHEMA_DISABLE_ADVISORY_LOCK` env var if needed.

### Zero-Downtime Migration Strategies
1. **Expand-contract pattern**: Add new columns/tables first (expand), migrate data, then remove old columns (contract).
2. **Backfill strategies**: Run data migrations as background jobs, not inline with schema migrations.
3. **Migration locking**: Use advisory locks or distributed locks (e.g., `pg_advisory_lock`) to prevent concurrent migrations.
4. **Blue-green schema changes**: Deploy new code that tolerates both old and new schema, then migrate schema, then clean up.

### Supabase-Specific Considerations
- Supabase runs isolated Postgres instances per project.
- Compute upgrades incur <2 minutes downtime typically.
- Use `pg_cron` for scheduled backfill jobs within Supabase.
- **Disk IOPS**: Monitor `Disk IO % consumed` — if >1%, consider upgrading compute for migration-heavy periods.

### Backfill for Large Tables
- Use batched updates: `UPDATE ... WHERE id IN (SELECT id FROM ... WHERE not_migrated LIMIT 1000)`.
- Run during low-traffic windows.
- Consider `pg_repack` for table reorganization after large migrations without heavy locks.
- Use `CREATE INDEX CONCURRENTLY` to avoid blocking reads.

## 3. Feature Flags for Financial Modules

### Unleash (Recommended for Multi-Tenant Module Toggle)
- **Open-source**, self-hostable, with enterprise tier.
- Feature flags exist within projects with per-environment activation strategies.
- **Key types for financial modules**:
  - `Release` — manage rollout of new modules (40-day expected lifetime)
  - `Kill switch` — gracefully degrade functionality (permanent)
  - `Permission` — control feature access based on user roles/entitlements (permanent)

### Tenant-Based Module Toggling Without Redeployment
- **Custom context fields**: Define `tenantId`, `plan`, `region` as context properties.
- **Activation strategies with constraints**:
  ```json
  {
    "strategy": "gradualRollout",
    "constraints": [
      {
        "contextField": "plan",
        "operator": "IN",
        "values": ["premium", "enterprise"]
      }
    ]
  }
  ```
- **Segments**: Reusable constraint sets (e.g., "all enterprise tenants in Brazil").
- **Strategy variants**: A/B testing for module layouts or calculation algorithms.

### Custom Feature Flag Approach (Lightweight Alternative)
For simpler setups, use a database-driven approach:
```sql
CREATE TABLE tenant_feature_flags (
  tenant_id UUID REFERENCES tenants(id),
  feature_key TEXT NOT NULL,
  enabled BOOLEAN DEFAULT false,
  config JSONB DEFAULT '{}',
  PRIMARY KEY (tenant_id, feature_key)
);
```
Cache in Redis with 60-second TTL. Evaluate at middleware level.

### LaunchDarkly vs Unleash vs Custom
| Criteria | LaunchDarkly | Unleash | Custom |
|----------|-------------|---------|--------|
| Cost | $$$$ (per-seat) | Free/OSS + enterprise | Dev time |
| Multi-tenant | Yes (via contexts) | Yes (via custom fields) | Full control |
| Audit log | Yes | Yes (Enterprise) | Build yourself |
| Self-hosted | No | Yes | Yes |
| SDK ecosystem | Excellent | Good | Build yourself |

**Recommendation**: Start with Unleash OSS for module toggling. Migrate to LaunchDarkly only if you need advanced experimentation at scale.

## 4. CI/CD Pipelines for Financial Logic

### Testing Financial Logic
- **Property-based testing**: Use `fast-check` (JS/TS) or `hypothesis` (Python) to generate random inputs for calculation functions. Verify invariants (e.g., sum of debits = sum of credits).
- **Golden master testing**: Snapshot expected outputs for known inputs. When calculations change, review diffs explicitly.
  ```typescript
  // Example: golden master for cashflow calculation
  const result = calculateCashflow(testInput);
  expect(result).toMatchSnapshot();
  ```
- **Regression tests**: Maintain a suite of real-world scenarios with expected outcomes.

### Automated Compliance Checks
- **Dependency vulnerability scanning**: `npm audit`, `snyk`, or `socket.dev` in CI pipeline.
- **License compliance**: `license-checker` to enforce approved licenses only.
- **Secret scanning**: `gitleaks` or `trufflehog` to prevent credential commits.
- **Schema validation**: Run `prisma validate` and `prisma generate` in CI.

### CI Pipeline Structure
```yaml
# .github/workflows/ci.yml
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run lint
      - run: npm run typecheck
  
  test:
    needs: lint
    steps:
      - run: npm run test:unit
      - run: npm run test:integration
      - run: npm run test:property  # fast-check tests
  
  security:
    steps:
      - run: npx gitleaks detect
      - run: npm audit --audit-level=high
  
  migrate-check:
    steps:
      - run: npx prisma validate
      - run: npx prisma migrate diff --from-schema-datasource prisma/schema.prisma --to-schema-datasource prisma/schema.prisma
```

## 5. Blue-Green / Canary Deployments for Financial Systems

### Rollback Strategies
- **Vercel instant rollback**: Cached content from previous deployments is preserved. Roll back by promoting a previous deployment.
- **Database rollback**: Use migration down scripts (`prisma migrate down`). For critical financial data, prefer forward-only migrations with compensating transactions.
- **Feature flag rollback**: Kill switches instantly disable features without deployment.

### Data Migration During Deploys
- **Dual-write pattern**: Write to both old and new schemas during transition.
- **CQRS**: Separate read and write models during migration periods.
- **Event sourcing**: Replay events to rebuild state after schema changes.

### Canary Deployment Pattern
1. Deploy to 5% of traffic (Vercel supports this via configuration).
2. Monitor error rates, latency, and financial calculation accuracy.
3. Gradually increase to 100% or rollback.

**Recommendation for financial systems**: Prefer blue-green with feature flags over canary. Financial calculations must be atomic and consistent — partial rollouts can create data inconsistency.

## 6. Environment Management

### Dev/Staging/Production Parity
- **Database parity**: Use Supabase branches (Enterprise) or separate projects per environment.
- **Seed data**: Use `prisma db seed` with realistic but anonymized financial data.
- **Synthetic data generation**: Use `faker.js` for generating test transactions, accounts, and cashflow entries.

### Seed Data Strategy
```typescript
// prisma/seed.ts
import { PrismaClient } from '@prisma/client';
import { faker } from '@faker-js/faker';

const prisma = new PrismaClient();

async function main() {
  // Create tenant
  const tenant = await prisma.tenant.create({
    data: { name: 'Test Company', plan: 'premium' }
  });
  
  // Create 12 months of cashflow data
  for (let month = 0; month < 12; month++) {
    await prisma.cashflowEntry.createMany({
      data: Array.from({ length: 50 }, () => ({
        tenantId: tenant.id,
        amount: parseFloat(faker.finance.amount({ min: -10000, max: 50000 })),
        category: faker.helpers.arrayElement(['revenue', 'expense', 'investment']),
        date: faker.date.recent({ days: 30 }),
      }))
    });
  }
}
```

### Environment Variables
| Variable | Dev | Staging | Production |
|----------|-----|---------|------------|
| `DATABASE_URL` | Local SQLite | Supabase staging | Supabase production |
| `NEXTAUTH_SECRET` | Dev secret | Staging secret | Production secret |
| `UNLEASH_URL` | localhost:4242 | staging-unleash | production-unleash |
| `FEATURE_FLAGS` | All enabled | Subset | Production subset |

## 7. Container Orchestration

### Docker for Self-Hosted Option
```dockerfile
# Dockerfile
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npx prisma generate
RUN npm run build

FROM node:20-alpine AS runner
WORKDIR /app
COPY --from=builder /app/.next/standalone ./
COPY --from=builder /app/.next/static ./.next/static
COPY --from=builder /app/prisma ./prisma
EXPOSE 3000
CMD ["node", "server.js"]
```

### Docker Compose for Development
```yaml
# docker-compose.yml
services:
  app:
    build: .
    ports:
      - "3000:3000"
    environment:
      - DATABASE_URL=postgresql://postgres:postgres@db:5432/cashflow
    depends_on:
      - db
      - redis
  
  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: cashflow
      POSTGRES_PASSWORD: postgres
    volumes:
      - pgdata:/var/lib/postgresql/data
  
  redis:
    image: redis:7-alpine
  
  unleash:
    image: unleashorg/unleash-server:latest
    ports:
      - "4242:4242"
    environment:
      DATABASE_URL: postgresql://postgres:postgres@db:5432/unleash

volumes:
  pgdata:
```

### Kubernetes for Enterprise
- Use Helm charts for Supabase (self-hosted) or connect to Supabase Cloud.
- Deploy Next.js as a Deployment with HPA (Horizontal Pod Autoscaler).
- Use ConfigMaps/Secrets for environment variables.
- Consider ArgoCD for GitOps-based deployments.

## 9. Logging and Observability

### Structured Logging
```typescript
// lib/logger.ts
import pino from 'pino';

export const logger = pino({
  level: process.env.LOG_LEVEL || 'info',
  formatters: {
    level: (label) => ({ level: label }),
  },
  base: {
    service: 'cashflow-api',
    version: process.env.APP_VERSION,
  },
});

// Usage in route handlers
logger.info({ tenantId, amount, category }, 'Cashflow entry created');
logger.error({ error, transactionId }, 'Payment processing failed');
```

### OpenTelemetry Integration
```typescript
// lib/tracing.ts
import { NodeSDK } from '@opentelemetry/sdk-node';
import { getNodeAutoInstrumentations } from '@opentelemetry/auto-instrumentations-node';
import { OTLPTraceExporter } from '@opentelemetry/exporter-trace-otlp-http';

const sdk = new NodeSDK({
  traceExporter: new OTLPTraceExporter({
    url: process.env.OTEL_EXPORTER_OTLP_ENDPOINT,
  }),
  instrumentations: [getNodeAutoInstrumentations()],
});

sdk.start();
```

### Log Aggregation for Audit Compliance
- **Vercel**: Built-in function logs with search and filtering.
- **External**: Ship logs to Datadog, Grafana Loki, or CloudWatch for long-term retention.
- **Audit requirements**: Financial systems typically require 7-year log retention. Use immutable storage (S3 with Object Lock).
- **Key events to log**: All CRUD operations on financial data, user authentication, feature flag changes, calculation results.

## 10. Backup and Disaster Recovery

### PostgreSQL Point-in-Time Recovery (PITR)
- **WAL archiving**: Enable `wal_level = replica`, `archive_mode = on`.
- **Archive command**: Copy completed WAL segments to secondary storage.
- **Base backups**: Use `pg_basebackup` regularly (daily recommended).
- **Recovery**: Restore base backup + replay WAL to desired point in time.

### Supabase-Specific
- **Automated backups**: Supabase Pro plans include daily backups with 7-day retention.
- **Point-in-time recovery**: Available on paid plans (hourly granularity).
- **Manual backups**: Use `pg_dump` for logical backups.

### SQLite Backup Strategies
- **File-level backup**: Simply copy the `.db` file (ensure no writes during copy).
- **Online backup**: Use `VACUUM INTO` or `.backup` command in SQLite.
- **WAL mode**: Enable for better concurrent read performance during backups.

### RPO/RTO Targets for Financial Systems
| Metric | Target | Strategy |
|--------|--------|----------|
| RPO (Recovery Point Objective) | < 1 hour | WAL archiving every 5 minutes + daily base backups |
| RTO (Recovery Time Objective) | < 4 hours | Automated restore scripts, warm standby |
| Backup frequency | Daily full + hourly WAL | `pg_basebackup` + `archive_command` |
| Retention | 7 years (compliance) | Immutable S3 storage with Object Lock |
| Testing | Monthly DR drill | Restore to isolated environment and verify |

### Backup Configuration
```bash
# PostgreSQL backup script
#!/bin/bash
BACKUP_DIR="/backups/postgres"
TIMESTAMP=$(date +%Y%m%d_%H%M%S)

# Base backup
pg_basebackup -h localhost -U backup_user -D ${BACKUP_DIR}/base_${TIMESTAMP} -Ft -z -P

# WAL archiving (already configured in postgresql.conf)
# archive_command = 'cp %p /backups/postgres/wal/%f'

# Retention: keep 30 days of base backups
find ${BACKUP_DIR}/base_* -maxdepth 0 -mtime +30 -exec rm -rf {} \;
```
