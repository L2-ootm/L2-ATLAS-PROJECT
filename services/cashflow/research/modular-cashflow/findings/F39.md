## 1. High Availability Targets

| Tier | Uptime | Annual Downtime | Typical Use |
|------|--------|-----------------|-------------|
| Standard | 99.9% | 8.76 hours | Internal tools, staging |
| High | 99.95% | 4.38 hours | Core SaaS products |
| Financial-grade | 99.99% | 52.56 minutes | Payment processing, ledger writes |
| Ultra | 99.999% | 5.26 minutes | Real-time settlement, trading |

**How financial platforms achieve 99.99%+**:
- Eliminate single points of failure at every layer (load balancer, app, database, network)
- Active-active or active-warm standby in multiple availability zones
- Automated failover with no human intervention (< 30s for DB, < 10s for app tier)
- Blue-green or canary deployments to avoid deployment-related downtime
- Chaos engineering to validate assumptions about redundancy
- Circuit breakers to isolate failures before they cascade

## 2. Database Resilience: PostgreSQL Replication and Failover

### Streaming Replication Modes

| Mode | Durability | Performance | Data Loss Risk |
|------|-----------|-------------|----------------|
| Asynchronous | Lower | Higher throughput | Up to `max_standby_streaming_delay` of WAL |
| Synchronous (1 standby) | Higher | Moderate latency increase | Zero data loss if standby is healthy |
| Synchronous (quorum) | Highest | Higher latency, lower throughput | Zero data loss (quorum write) |

**Recommendation for cashflow**: Use synchronous replication with at least 2 standbys in a quorum commit configuration (`synchronous_standby_names = 'ANY 1 (standby1, standby2)'`). This provides zero data loss while tolerating one standby failure.

### Patroni for Automatic Failover

Patroni is the de facto standard for PostgreSQL HA. Architecture:

```
┌─────────────────────────────────────────────┐
│              Distributed Config Store        │
│          (etcd / Consul / ZooKeeper)          │
└──────────┬──────────┬──────────┬────────────┘
           │          │          │
     ┌─────▼──┐  ┌────▼───┐  ┌──▼─────┐
     │Patroni 1│  │Patroni 2│  │Patroni 3│
     │(Leader) │  │(Replica)│  │(Replica)│
     │  PG +   │  │  PG +   │  │  PG +   │
     │  VIP    │  │  VIP    │  │  VIP    │
     └────┬────┘  └─────────┘  └─────────┘
          │
     ┌────▼────┐
     │ HAProxy  │
     │(traffic  │
     │ routing) │
     └─────────┘
```

**Key Patroni configuration for financial systems**:
- `loop_wait`: 10s (how often Patroni checks cluster state)
- `retry_timeout`: 10s (timeout for DCS operations)
- `maximum_lag_on_failover`: 1048576 (1MB max lag before failover refused)
- `postgresql.synchronous_mode: true`
- `postgresql.synchronous_mode_strict: true` (prevents writes if no sync standby)
- `postgresql.use_pg_rewind: true` (fast rejoin without full base backup)

### pg_auto_failover (Citus/PostgreSQL)

Alternative to Patroni. Simpler architecture:
- Uses a monitor node for health checking
- Automatic failover with `pg_autoctl`
- Better for teams already in the PostgreSQL ecosystem
- Less flexible than Patroni for complex topologies

**Recommendation**: Patroni for its battle-tested status, Kubernetes integration, and richer feature set.

## 3. Circuit Breaker Pattern

Based on Martin Fowler's implementation (source: martinfowler.com/bliki/CircuitBreaker.html):

### Three States

```
         ┌─────────┐  failures exceed  ┌─────────┐
         │ CLOSED   │ ─────────────────►│  OPEN    │
         │(normal)  │   threshold       │(failing) │
         └────┬─────┘                   └────┬─────┘
              │                              │
              │ success                      │ timeout expires
              │                              │
         ┌────▼─────┐                   ┌────▼──────┐
         │          │◄──────────────────│ HALF-OPEN │
         │ CLOSED   │  trial succeeds   │ (testing) │
         └──────────┘                   └───────────┘
```

### Parameters for Financial Services

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| Failure threshold | 5 consecutive failures OR 50% failure rate | Quick detection without false positives |
| Invocation timeout | 5s for payment APIs, 10s for banking | Match external SLA |
| Reset timeout | 30s | Allow recovery without hammering |
| Monitor callback | Alert on state change | Ops visibility |

### Implementation Recommendations

- **Hystrix** (Java/Netflix) - legacy but battle-tested
- **Resilience4j** (Java) - modern successor to Hystrix
- **pybreaker** (Python) - simple, works for Python services
- **opossum** (Node.js) - event-driven, good for async
- **Custom implementation** for Rust core

**Key insight from Fowler**: "Not all errors should trip the circuit. Some should reflect normal failures and be dealt with as part of regular logic." For a cashflow system:
- Trip on: connection refused, 5xx errors, timeout
- Do NOT trip on: 4xx errors (bad request), business validation failures

## 4. Retry Strategies

### Exponential Backoff with Jitter

```
delay = min(base_delay * 2^attempt + random_jitter, max_delay)
```

**Parameters for financial services**:
- `base_delay`: 500ms (start fast for transient issues)
- `max_delay`: 30s (don't retry indefinitely)
- `max_attempts`: 3 (balance between availability and load)
- `jitter`: uniform random [0, delay * 0.3]

### Retry Budget

Prevent retry storms:
- Max 20% of traffic can be retries per 10-second window
- If budget exceeded, fail fast without retry
- Critical for payment gateways that may already be degraded

### Circuit Breaker + Retry Combination

```
Request → Retry Policy → Circuit Breaker → External Service
                          │
                          └─ If OPEN → fail fast (no retry)
```

**Critical rule**: Never retry when circuit is open. Retrying through an open circuit just adds load to a failing system.

### Idempotency Requirement

All retryable operations MUST be idempotent for financial systems:
- Use idempotency keys for payment operations
- Database writes use unique constraints or upsert patterns
- Ledger entries have deterministic IDs (hash of business key)

## 5. Graceful Degradation

### Service Criticality Tiers

| Tier | Services | Degradation Behavior |
|------|----------|---------------------|
| P0 - Core | Transaction processing, ledger writes, balance reads | NEVER degrade. Reject with 503 if unable to serve. |
| P1 - Important | Payment initiation, reconciliation | Queue for retry. Show "processing" state. |
| P2 - Normal | Analytics, reporting, notifications | Degrade gracefully. Use cached data. Drop non-essential features. |
| P3 - Nice-to-have | Recommendations, audit logs, dashboards | Fail silently. Backfill later. |

### Degradation Patterns

1. **Feature flags**: Disable non-critical features when dependency is down
2. **Cached fallback**: Serve stale data with "(cached)" indicator
3. **Queue and retry**: Enqueue operations for later processing
4. **Partial results**: Return what you can, indicate what's missing
5. **Circuit breaker fallback**: Return a default/cached response

### Implementation for Cashflow

```python
class CashflowService:
    async def process_transaction(self, tx):
        # P0: Core - must succeed
        ledger_result = await self.ledger.write(tx)  # no fallback
        
        # P1: Important - queue if down
        try:
            await self.payment_gateway.initiate(tx)
        except CircuitOpenError:
            await self.queue.enqueue("payment_retry", tx)
            return {"status": "processing", "tx_id": tx.id}
        
        # P2: Normal - degrade gracefully
        try:
            await self.notification_service.send(tx)
        except Exception:
            pass  # Log and backfill later
        
        # P3: Nice-to-have - fire and forget
        try:
            await self.analytics.track(tx)
        except Exception:
            pass  # Best effort only
```

## 6. Chaos Engineering

### Netflix Chaos Monkey Patterns Adapted for Financial Systems

**What to inject (safely)**:
- Network latency on non-production databases
- Process kills on app tier (verify failover works)
- Disk space exhaustion on log volumes
- Certificate expiry simulation
- DNS resolution failures for non-critical services

**What NOT to inject (without extreme caution)**:
- Production database failures
- Payment gateway disconnections (use staging)
- Ledger corruption scenarios (use isolated test environments)
- Network partitions across regions

### Chaos Engineering Budget

| Environment | Frequency | Scope | Blast Radius |
|------------|-----------|-------|--------------|
| Dev | Daily | Any component | 100% |
| Staging | Weekly | All components | 50% per experiment |
| Production | Monthly | Non-critical only | 5% max, with rollback |

### Tools

- **Litmus Chaos** (Kubernetes-native) - good for container orchestration
- **Chaos Toolkit** (declarative experiments) - good for defining experiments as code
- **Gremlin** (commercial) - comprehensive platform
- **Custom scripts** - for financial-specific scenarios (e.g., simulate payment gateway 500 errors)

### Game Day Checklist

1. Pre-game: Define hypothesis, expected behavior, rollback plan
2. During: Monitor all P0 metrics, have incident commander
3. Post-game: Compare actual vs expected, update runbooks, create follow-up tasks

## 7. Incident Response

### Severity Levels

| Level | Definition | Response Time | Example |
|-------|-----------|---------------|---------|
| SEV1 | Complete service outage, data loss risk | 5 minutes | Payment processing down |
| SEV2 | Major feature degraded, >10% users affected | 15 minutes | Slow transaction processing |
| SEV3 | Minor feature issue, <10% users affected | 1 hour | Notification delays |
| SEV4 | Cosmetic issue, no user impact | Next business day | Dashboard display bug |

### On-Call Rotation

- Primary + secondary on-call, 1-week rotations
- Follow-the-sun if global (US, EU, APAC)
- Escalation path: on-call → team lead → engineering manager → VP
- Maximum 2 consecutive incidents before handoff

### Communication Templates

**SEV1 Initial (within 5 minutes)**:
```
[SEV1] Cashflow Platform - [Brief Description]
Status: Investigating
Impact: [Description of user impact]
Next update: [Time, typically 15 min]
Incident commander: [Name]
```

**SEV1 Resolution**:
```
[SEV1 RESOLVED] Cashflow Platform - [Brief Description]
Duration: [X minutes]
Root cause: [Brief description]
Impact: [Summary of affected users/transactions]
Follow-up: [Link to post-incident review]
```

### Post-Incident Review (PIR)

Required for all SEV1 and SEV2 incidents. Template:
1. Timeline of events
2. Impact summary
3. Root cause analysis (5 Whys)
4. What went well
5. What went poorly
6. Action items with owners and deadlines
7. Detection gap: How could we have detected this sooner?

## 8. SLA/SLO/SLI Definitions

### For Financial Cashflow Platform

| SLI | SLO Target | Measurement | Error Budget |
|-----|-----------|-------------|--------------|
| Availability | 99.95% | Successful requests / total requests | 21.9 min/month |
| Latency (p50) | < 100ms | Time to process transaction | N/A |
| Latency (p99) | < 500ms | Time to process transaction | N/A |
| Data durability | 99.999999% (8 9s) | Zero data loss on failover | N/A |
| Error rate | < 0.1% | 5xx responses / total requests | 0.1% of traffic |

### Error Budget Policy

- **Budget remaining > 50%**: Normal development velocity, can deploy freely
- **Budget remaining 25-50%**: Increased review, no risky changes
- **Budget remaining < 25%**: Freeze non-critical deployments, focus on reliability
- **Budget exhausted**: All hands on reliability, no feature work until budget restored

### Financial Compliance Considerations

- Transaction latency SLOs may be regulatory requirements (PSD2, open banking)
- Audit trail availability must be 100% (no error budget for audit logs)
- Data residency requirements may affect multi-region SLOs

## 9. Multi-Region Deployment

### Active-Active vs Active-Passive

| Pattern | Pros | Cons | Use Case |
|---------|------|------|----------|
| Active-Active | Lower latency globally, no failover delay | Complex data consistency, higher cost | Global payment platforms |
| Active-Passive | Simpler, lower cost | Failover delay (seconds to minutes), underutilized DR site | Regional financial platforms |

### Data Consistency Across Regions

**The CAP theorem reality**: For financial systems, you typically choose CP (consistency + partition tolerance) over AP.

**Approaches**:
1. **Synchronous replication** (same region only): Zero data loss, higher latency
2. **Asynchronous replication** (cross-region): Lower latency, potential data loss (< 1 second with good networking)
3. **CRDTs** (for non-critical data): Eventual consistency without conflicts
4. **Saga pattern** (for distributed transactions): Compensating transactions for cross-region operations

### For Cashflow Platform

**Recommended architecture**:
- **Primary region**: Full active deployment, synchronous replication within region
- **Secondary region**: Active-warm standby, asynchronous replication from primary
- **Failover**: Automated for infrastructure, manual approval for region-level failover (due to financial regulatory requirements)
- **Data**: Async replication with < 1 second lag, verify consistency on failover

### Latency Considerations

| Component | Target Latency | Impact if Exceeded |
|-----------|---------------|-------------------|
| Transaction write (same region) | < 50ms | User-facing delay |
| Transaction write (cross-region) | < 200ms | Acceptable for non-real-time |
| Balance read (same region) | < 10ms | User-facing delay |
| Balance read (cross-region) | < 50ms | Acceptable for most reads |
| Failover detection | < 10s | Minimal user impact |

## Summary of Recommendations for Cashflow Platform

1. **Target 99.99% uptime** for core transaction processing
2. **Use Patroni** with synchronous replication (quorum) for PostgreSQL HA
3. **Implement circuit breakers** with 5-failure threshold, 30s reset timeout
4. **Retry with exponential backoff** (500ms base, 30s max, 3 attempts, 30% jitter)
5. **Graceful degradation**: P0 services never degrade, P2/P3 degrade silently
6. **Chaos engineering monthly** in production, weekly in staging
7. **Incident response**: SEV1 within 5 min, SEV2 within 15 min
8. **SLI/SLO**: 99.95% availability, < 100ms p50 latency, 8 9s data durability
9. **Multi-region**: Active-warm standby, async replication, manual approval for region failover
