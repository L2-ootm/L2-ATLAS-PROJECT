**Date**: 2026-07-09
**Angle**: Data governance, data quality, MDM, validation, reconciliation, retention, archival, export, masking

## Finding 1: Data Lineage — End-to-End Tracking from Source to Report

Data lineage documents data's origins, transformations, and movements across systems over time. For a cashflow platform, every transaction must trace from source (bank feed, invoice, manual entry) through transformations (GL posting, allocation, consolidation) to final destination (reports, tax filings, dashboards).

**Key mechanisms:**
- **Backward tracing**: trace a report value back to source transactions (debugging)
- **Forward tracing**: track where a source change propagates (impact analysis)
- **Eager capture**: record full lineage at runtime (higher overhead, instant debugging)
- **Lazy capture**: record coarse lineage at runtime, replay for fine-grain (lower overhead, slower debugging)

**W3C PROV Data Model** (W3C recommendation): defines entities, activities, and agents in data production — interoperable provenance across heterogeneous systems. Adopt this for standardized lineage metadata.

**Application**: Implement eager lineage capture for critical financial paths (GL postings, reconciliation results) and lazy capture for analytical/reporting pipelines.

## Finding 2: Data Quality Dimensions — Measurement Framework for Financial Data

ISO 8000 defines data quality as "the degree to which inherent characteristics fulfill requirements." The core dimensions for financial data:

| Dimension | Financial Definition | Measurement |
|-----------|---------------------|-------------|
| **Accuracy** | Transaction amount matches source document | % records matching source within tolerance |
| **Completeness** | All required fields populated, no missing transactions | % mandatory fields non-null; record count reconciliation |
| **Consistency** | Same entity represented identically across modules | Cross-system match rate for master data |
| **Timeliness** | Data available within SLA (e.g., daily close by T+1) | Time from event to available in reporting |
| **Validity** | Values conform to business rules and formats | % records passing validation rules |
| **Uniqueness** | No duplicate transactions | Deduplication rate; duplicate detection accuracy |
| **Integrity** | Referential integrity maintained across tables | Orphan record count; FK violation rate |

**Critical**: Data quality checks should be placed at system boundaries (data entry) and after transformations, not only at the end. DQ checks on completeness at point of entry catch mandatory-field gaps immediately.

## Finding 3: Master Data Management — Chart of Accounts as Golden Record

MDM ensures "uniformity, accuracy, stewardship, semantic consistency, and accountability of the enterprise's official shared master data assets."

**Master data entities for cashflow platform:**
- **Chart of Accounts (CoA)**: the backbone of financial reporting; must be versioned, with effective dates for account hierarchy changes
- **Customer/Vendor Master**: single customer/vendor record federated across AP, AR, and CRM modules
- **Product Master**: product/service catalog with pricing, tax classification, and revenue recognition rules

**Golden Record pattern:**
- Single authoritative record per entity, created through matching/merging from multiple source systems
- MDM hub or registry pattern: consolidate master data from multiple sources into a central hub, replicate to downstream systems
- **Source of record model**: identify which system "owns" each attribute (e.g., billing address from ERP, email from CRM)

**Versioning**: Master data changes must be versioned with effective start/end dates. Account hierarchy reorganizations, customer name changes, and vendor mergers require historical preservation.

## Finding 4: Data Validation Rules — Cross-Field and Referential Integrity

Financial data validation must enforce invariants at write time, not just report time.

**Essential validation rules:**

| Rule Type | Example | Implementation |
|-----------|---------|----------------|
| **Double-entry** | SUM(debit) = SUM(credit) per transaction | Atomic write constraint; reject if unbalanced |
| **Range checks** | amount > 0 for invoices; amount can be negative for credits | CHECK constraints at DB level |
| **Referential integrity** | Every GL entry references valid CoA account | FK constraints; reject orphan entries |
| **Temporal validation** | Transaction date within fiscal period | Application-level check; reject post-close entries |
| **Cross-field validation** | Tax amount = rate × taxable amount | Computed column or trigger validation |
| **Uniqueness** | No duplicate invoice numbers per vendor | UNIQUE constraint; idempotent insert |
| **Business rule validation** | Payment cannot exceed open invoice balance | Application-level pre-commit check |

**Implementation priority**: DB-level constraints for structural invariants (FK, uniqueness, type checks); application-level for business rules (balance checks, period validation).

## Finding 5: Data Reconciliation — Automated Cross-Module Verification

Reconciliation verifies that related datasets agree across systems or time periods.

**Critical reconciliation pairs for cashflow:**

| Source A | Source B | Frequency | Tolerance |
|----------|----------|-----------|-----------|
| AP Sub-ledger | GL Control Account | Daily | $0 (must balance) |
| AR Sub-ledger | GL Control Account | Daily | $0 (must balance) |
| Bank Statement | Book Balance | Daily | $0 after reconciling items |
| Cash Position Report | Sum of All Cash Accounts | Real-time | $0 |
| Payroll Register | GL Payroll Expense | Per pay period | $0 |
| Tax Liability Register | GL Tax Payable | Monthly | $0 |

**Reconciliation workflow:**
1. Auto-match by key fields (transaction ID, date, amount)
2. Identify unmatched items (exceptions)
3. Flag items exceeding tolerance
4. Require manual resolution for exceptions
5. Generate reconciliation report with sign-off workflow

**Implementation**: Build reconciliation as a first-class service, not ad-hoc queries. Each module pair gets a dedicated reconciliation job with configurable matching rules and tolerance thresholds.

## Finding 6: Data Retention Policies — Jurisdiction-Specific Requirements

Financial data retention is governed by multiple overlapping regulations:

| Regulation | Jurisdiction | Minimum Retention | Scope |
|-----------|-------------|-------------------|-------|
| **Código Tributário Nacional (CTN)** | Brazil | 5 years from tax event | Tax records, invoices, receipts |
| **SOX (Sarbanes-Oxley)** | USA (public companies) | 7 years | Financial records, audit workpapers |
| **GDPR Art. 5(1)(e)** | EU | Purpose limitation — no fixed minimum, but must define retention periods | Personal data in financial records |
| **LRG (Lei de Responsabilidade Fiscal)** | Brazil | 5 years | Public sector fiscal documents |
| **CVM Instruction 480** | Brazil | 5 years from event | Securities market records |
| **Anti-Money Laundering** | Most jurisdictions | 5 years after relationship ends | Customer identification, transaction records |
| **Brazil Central Bank Res. 3.380** | Brazil | 10 years | Financial institution records |

**Policy design:**
- Tag every record with data classification and applicable regulation(s)
- Retention = max(all applicable minimums) per record
- Implement automated retention enforcement with hold/litigation support
- Document retention rationale for audit trail

## Finding 7: Data Archival — Hot/Warm/Cold Storage Tiers

Archival strategies manage the cost-performance tradeoff for historical financial data.

**Storage tier model:**

| Tier | Access Pattern | Data Age | Storage | Example |
|------|---------------|----------|---------|---------|
| **Hot** | Real-time query, write-heavy | Current fiscal year | SSD/primary DB | Active transactions, open AP/AR |
| **Warm** | Occasional query, read-heavy | 1-3 years | Compressed DB/tablestore | Closed fiscal periods, quarterly reports |
| **Cold** | Rare access, compliance only | 3+ years | Object storage (S3/GCS) | Archived tax records, audit evidence |

**Archival strategy for cashflow:**
- Automate tier migration based on fiscal period close
- Preserve query-ability for cold data (Parquet/Arrow format for analytical access)
- Maintain referential integrity metadata across tiers
- Ensure archival doesn't break downstream reports (materialized views or summary tables)

## Finding 8: Data Export and Portability — Vendor Lock-in Prevention

GDPR Article 20 grants data portability: "the data subject shall have the right to receive the personal data... in a structured, commonly used and machine-readable format."

**Export requirements for financial platform:**
- **Customer data export**: transaction history, account statements, contact information in machine-readable format (JSON, CSV, XML)
- **Vendor lock-in prevention**: use open standards for data formats (ISO 20022 for financial messaging, XBRL for financial reporting)
- **API-first design**: every data entity should have a REST/GraphQL export endpoint
- **Data model documentation**: publish schema definitions so consumers can parse exported data without proprietary tools

**Implementation considerations:**
- Export endpoints must respect access controls and audit logging
- Bulk export should support streaming for large datasets
- Financial data exports may need digital signatures for authenticity verification
- Cross-border data transfer restrictions (GDPR Chapter V) must be enforced at export time

## Finding 9: Data Masking — PII Protection in Non-Production Environments

Data masking modifies sensitive data to protect PII while preserving usability for testing.

**Masking techniques for financial data:**

| Technique | Use Case | Preserves Utility |
|-----------|----------|-------------------|
| **Substitution** | Replace real names/addresses with realistic fakes | High — looks like real data |
| **Shuffling** | Randomize values within a column | Medium — preserves distribution |
| **Number variance** | ±10% variance on financial amounts | High — preserves range/statistics |
| **Format-preserving encryption** | Encrypt PII while keeping format (e.g., credit card format) | High — valid format, unreadable content |
| **Nulling** | Remove sensitive fields entirely | Low — breaks referential integrity |
| **Masking out** | Show only partial data (e.g., last 4 digits) | High for display; low for testing |

**Static vs Dynamic masking:**
- **Static**: apply masking to a database copy before use in dev/test (one-time transformation)
- **Dynamic**: apply masking at query time based on user role/permissions (runtime transformation)

**For cashflow platform:**
- Apply static masking to any production data copied to dev/staging environments
- Use format-preserving encryption for customer identifiers (CPF/CNPJ in Brazil)
- Preserve transaction structure and amounts (within variance) for realistic testing
- Ensure masking is deterministic: same input always produces same output (maintains referential integrity across tables)

## Finding 10: Data Governance Organizational Model

Effective data governance requires clear roles and accountability:

| Role | Responsibility |
|------|---------------|
| **Data Owner** | Business leader accountable for data quality, security, and compliance for a domain (e.g., CFO owns financial data) |
| **Data Steward** | Operational role managing day-to-day data quality, resolving exceptions, enforcing rules |
| **Data Custodian** | Technical role responsible for data storage, security, backup, and archival infrastructure |
| **Data Protection Officer (DPO)** | GDPR-mandated role for EU personal data; oversight of privacy compliance |

**Governance processes:**
- Data quality monitoring with automated alerting on threshold breaches
- Steward review for data quality exceptions and master data changes
- Regular reconciliation sign-off workflow
- Audit trail for all data modifications (who, when, what, why)

## Finding 11: Financial Data Validation Architecture

Layer validation at multiple points to catch errors early:

```
Input Layer    → Format validation, type checking, required fields
Business Layer → Cross-field rules, referential integrity, period validation
Aggregation    → Balance checks (debits = credits), reconciliation
Reporting      → Completeness checks, consistency across reports
Audit          → Full lineage verification, exception review
```

**Key principle**: fail fast and fail loud. Reject invalid data at the earliest possible point rather than allowing it to propagate downstream.

## Finding 12: Audit Trail Requirements

Financial platforms must maintain immutable audit trails for regulatory compliance:

**Minimum audit trail fields:**
- Timestamp (UTC, millisecond precision)
- Actor (user ID or system process)
- Action (create, read, update, delete)
- Entity (table/entity modified)
- Before-state (for updates/deletes)
- After-state (for creates/updates)
- Reason/code (for changes — optional but recommended)
- Correlation ID (to link related changes)

**Implementation:**
- Append-only audit log (never update or delete audit records)
- Separate audit database/table with restricted write access
- Digital signatures on audit entries for tamper evidence
- Retention aligned with financial data retention requirements (minimum 7 years for SOX)

## Finding 13: Data Quality Scorecard Metrics

Track data quality over time with a scorecard:

| Metric | Target | Measurement |
|--------|--------|-------------|
| **Completeness Rate** | ≥ 99.5% | % mandatory fields populated |
| **Accuracy Rate** | ≥ 99.9% | % records matching authoritative source |
| **Consistency Rate** | ≥ 99.8% | % cross-system matches for shared entities |
| **Timeliness** | T+1 for daily close | Time from event to reporting availability |
| **Duplicate Rate** | ≤ 0.1% | Detected duplicates / total records |
| **Reconciliation Variance** | $0 for ledger pairs | Unresolved differences between reconciled datasets |
| **Validation Pass Rate** | ≥ 99.5% | Records passing all validation rules / total records |

Monitor these metrics dashboards; alert on degradation trends before thresholds are breached.

## Summary

A modular cashflow platform requires a layered data governance strategy:

1. **Lineage**: Track every value from source to report; implement W3C PROV-based metadata
2. **Quality**: Enforce six core dimensions (accuracy, completeness, consistency, timeliness, validity, uniqueness) with automated measurement
3. **MDM**: Treat CoA, customer/vendor, and product as golden records with versioned effective dates
4. **Validation**: DB-level constraints for structural invariants, application-level for business rules
5. **Reconciliation**: Automated daily reconciliation between all ledger pairs with zero-tolerance
6. **Retention**: Jurisdiction-specific policies; Brazil requires 5-10 years, SOX 7 years
7. **Archival**: Three-tier model (hot/warm/cold) with automated migration on fiscal close
8. **Export**: API-first, open standards, GDPR-compliant portability
9. **Masking**: Static masking for non-production; format-preserving encryption for PII
