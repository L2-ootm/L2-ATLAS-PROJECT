## Finding 1: Domain Event Schema Design in Finance

**Pattern**: Domain events in financial systems follow a `resource.action` naming convention, similar to Stripe's approach. Events represent meaningful state changes in the business domain.

**Event Naming Convention**:
```

payment.received

```

```json
  "version": "2024-01-01",
  "timestamp": "2026-07-09T10:30:00Z",
  "aggregate_type": "Invoice",

    "invoice_id": "inv_abc123",

    "currency": "USD",
    "customer_id": "cust_xyz789",
```

**Key Design Principles**:
- Events should be immutable facts, not commands
- Include correlation_id for tracing multi-step workflows
- Use semantic versioning for schema evolution
- Include enough data for consumers to act without callback queries

## Finding 2: Event Sourcing vs Event-Driven for Financial Ledgers

**Source**: microservices.io, Martin Fowler

**Event Sourcing**:
- Records every state change as an immutable event
- Event store is the single source of truth
- Current state derived by replaying events
- Provides complete audit trail (critical for financial compliance)

**Event-Driven**:
- Events notify systems of changes
- Primary database holds current state
- Events are secondary, for inter-service communication

**For Financial Ledgers - Event Sourcing is Superior**:

| Aspect | Event Sourcing | Event-Driven |
|--------|---------------|--------------|
| Audit Trail | Complete, immutable history | Limited to event logs |
| State Reconstruction | Replay events to any point in time | Requires snapshots |
| Double-Entry Accounting | Natural fit - each event is a journal entry | Requires separate audit system |
| Temporal Queries | Built-in capability | Requires custom implementation |
| Regulatory Compliance | Strong (SOX, GAAP, IFRS) | Additional work needed |

**Recommended Approach**: Use Event Sourcing for the core ledger, Event-Driven for cross-module communication.

**Example Event Sourcing for Journal Entries**:
```typescript

    entry_number: string;
```

**Source**: Stripe Documentation

**Event Types Structure**: Stripe uses `{resource}.{action}` naming with sub-resource events.

**Key Financial Events from Stripe**:

**Webhook Delivery Guarantees**:
- At-least-once delivery (duplicates possible)
- Automatic retries for 3 days with exponential backoff
- No ordering guarantees (consumers must handle)
- Signature verification required (HMAC-SHA256)

**Idempotency in Stripe Webhooks**:
```typescript
// Stripe recommends handling duplicate events
const processedEvents = new Set<string>();

async function handleWebhook(event: Stripe.Event) {
  // Use event.id for deduplication
  if (processedEvents.has(event.id)) {
    return; // Already processed
  }
  
  // Also check data.object.id for same logical event
  const logicalId = `${event.type}:${event.data.object.id}`;
  if (processedEvents.has(logicalId)) {
    return;
  }
  
  // Process event
  await processEvent(event);
  
  // Mark as processed
  processedEvents.add(event.id);
  processedEvents.add(logicalId);
}
```

**Stripe's Thin Events (v2)**:
- Reduced payload size
- Use `fetchRelatedObject()` to get full details
- Better for high-volume event streams

**Read Side (Queries)**:
```

```

**Financial CQRS Architecture**:

```
┌─────────────────────────────────────────────────────────────┐
│                      WRITE MODEL                            │
│  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐  │
│  │   Command    │───▶│   Domain     │───▶│  Event Store │  │
│  │   Handler    │    │   Logic      │    │  (Source of  │  │
│  └──────────────┘    └──────────────┘    │    Truth)    │  │
│                                          └──────────────┘  │
└─────────────────────────────────────────────────────────────┘
                              │
                              │ Events
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                      READ MODEL                             │
│  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐  │
│  │   Event      │───▶│  Projection  │───▶│  Read Store  │  │
│  │   Handler    │    │  Builder     │    │  (Denormalized│  │

│                                          └──────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

**Benefits for Financial Systems**:
- **Independent Scaling**: Writes (journal entries) vs reads (reports)
- **Optimized Schemas**: Write model enforces double-entry, read model optimized for queries
- **Audit Trail**: Event store provides complete history
- **Real-time Reporting**: Projections update read stores asynchronously

**Example Projections**:

```
│     └─▶ Emits: revenue.recognized                              │
│                                                                 │
│  5. Reporting Service updates financial statements             │
│     └─▶ Emits: report.updated                                  │
└─────────────────────────────────────────────────────────────────┘
```

**Compensating Transactions**:
```typescript

    {
      name: "create_journal_entry",
```

**Key Considerations for Financial Sagas**:
- **Isolation**: Use pessimistic locking for critical resources
- **Idempotency**: Each step must be idempotent
- **Audit Trail**: Log every step and compensation
- **Timeout Handling**: Define clear timeouts for each step

## Finding 6: Event Bus Implementations

**Source**: Redis Documentation, Various

### In-Process: Node.js EventEmitter
```typescript
import { EventEmitter } from 'events';

class FinancialEventBus extends EventEmitter {
  private static instance: FinancialEventBus;
  
  static getInstance(): FinancialEventBus {
    if (!FinancialEventBus.instance) {
      FinancialEventBus.instance = new FinancialEventBus();
      FinancialEventBus.instance.setMaxListeners(100);
    }
    return FinancialEventBus.instance;
  }
  
  async publish(event: FinancialEvent): Promise<void> {
    this.emit(event.event_type, event);
  }
  
  subscribe(eventType: string, handler: (event: FinancialEvent) => Promise<void>): void {
    this.on(eventType, async (event) => {

        await handler(event);

        this.emit('error', { eventType, event, error });
      }
    });
  }
}
```

**Limitations**: Single process, no persistence, no replay capability.

### Redis Streams
```typescript
import Redis from 'ioredis';

class RedisEventStore {
  private redis: Redis;
  
  constructor() {
    this.redis = new Redis();
  }
  
  async publish(stream: string, event: FinancialEvent): Promise<string> {
    const id = await this.redis.xadd(
      stream,
      '*',  // Auto-generate ID
      'event_type', event.event_type,
      'payload', JSON.stringify(event.data),
      'metadata', JSON.stringify(event.metadata)
    );
    return id;
  }
  
  async subscribe(
    stream: string, 
    group: string, 
    consumer: string,
    handler: (event: FinancialEvent) => Promise<void>
  ): Promise<void> {
    // Create consumer group if not exists
    try {
      await this.redis.xgroup('CREATE', stream, group, '0', 'MKSTREAM');
    } catch (e) {
      // Group already exists
    }
    
    while (true) {
      const results = await this.redis.xreadgroup(
        'GROUP', group, consumer,
        'COUNT', 10,
        'BLOCK', 5000,
        'STREAMS', stream, '>'
      );
      
      if (!results) continue;
      
      for (const [, messages] of results) {
        for (const [id, fields] of messages) {
          const event = this.parseEvent(fields);
          await handler(event);
          await this.redis.xack(stream, group, id);
        }
      }
    }
  }
}
```

**Redis Streams Advantages**:
- Persistence and durability
- Consumer groups for load balancing
- Message acknowledgment (XACK)
- Automatic claim of failed consumers (XAUTOCLAIM)
- Stream trimming for memory management

## Finding 7: Idempotency in Financial Event Processing

**Source**: Stripe Idempotency Documentation

**The Problem**: Financial systems must handle duplicate events safely due to:
- At-least-once delivery guarantees
- Network retries
- Consumer failures and replays

**Idempotency Key Patterns**:

### Pattern 1: Event ID Deduplication
```typescript
class IdempotentEventProcessor {
  private processedEvents: Map<string, Date> = new Map();
  
  async process(event: FinancialEvent): Promise<void> {
    const dedupeKey = `${event.event_type}:${event.event_id}`;
    
    if (this.processedEvents.has(dedupeKey)) {
      console.log(`Skipping duplicate event: ${dedupeKey}`);
      return;
    }
    
    await this.processEvent(event);
    this.processedEvents.set(dedupeKey, new Date());
  }
}
```

### Pattern 2: Database-Level Idempotency
```sql
-- Create idempotency table
CREATE TABLE processed_events (
  event_id VARCHAR(255) PRIMARY KEY,
  event_type VARCHAR(100) NOT NULL,
  processed_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  result JSONB
);

-- Use in transaction
BEGIN;
INSERT INTO processed_events (event_id, event_type, result)
VALUES ($1, $2, $3)
ON CONFLICT (event_id) DO NOTHING
RETURNING event_id;

-- If no rows returned, already processed
COMMIT;
```

### Pattern 3: Business Logic Idempotency
```typescript
async function processPaymentReceived(event: PaymentReceivedEvent) {
  const { payment_id, invoice_id, amount } = event.data;
  
  // Check if invoice already has this payment recorded
  const existingPayment = await db.query(
    'SELECT * FROM invoice_payments WHERE payment_id = $1',
    [payment_id]
  );
  
  if (existingPayment) {
    // Already processed, return success
    return { status: 'already_processed' };
  }
  
  // Process payment
  await db.query(
    'INSERT INTO invoice_payments (payment_id, invoice_id, amount) VALUES ($1, $2, $3)',
    [payment_id, invoice_id, amount]
  );
  
  return { status: 'processed' };
}
```

**Stripe's Approach**:
- Idempotency keys valid for 24 hours
- Same key returns same result (including errors)
- Parameters must match on retry
- Only applies to POST requests

## Finding 8: Event Schema Evolution and Versioning

**Pattern**: Use header-based versioning for event schemas.

```typescript
interface VersionedEvent {
  event_id: string;
  event_type: string;
  version: string; // "2024-01-01"
  timestamp: Date;
  data: any;
  metadata: Record<string, any>;
}

// Schema Registry Pattern
class SchemaRegistry {
  private schemas: Map<string, Map<string, z.ZodType>> = new Map();
  
  register(eventType: string, version: string, schema: z.ZodType) {
    if (!this.schemas.has(eventType)) {
      this.schemas.set(eventType, new Map());
    }
    this.schemas.get(eventType)!.set(version, schema);
  }
  
  validate(event: VersionedEvent): boolean {
    const schema = this.schemas.get(event.event_type)?.get(event.version);
    if (!schema) {
      throw new Error(`No schema for ${event.event_type}@${event.version}`);
    }
    return schema.safeParse(event).success;
  }
}
```

## Finding 10: Event Sourcing Implementation for Double-Entry Accounting

**Pattern**: Use event sourcing to maintain immutable audit trail for journal entries.

```typescript
// Event Types for Double-Entry Accounting
type AccountingEvent = 
  | JournalEntryCreated
  | JournalEntryPosted
  | JournalEntryVoided
  | AccountBalanceAdjusted;

interface JournalEntryCreated {
  event_type: 'journal_entry.created';
  aggregate_id: string;
  data: {
    entry_number: string;
    description: string;
    lines: Array<{
      account_id: string;
      debit: number;
      credit: number;
      memo?: string;
    }>;
    created_by: string;
  };
}

// Aggregate Root
class JournalEntry {
  private id: string;
  private status: 'draft' | 'posted' | 'voided';
  private lines: JournalEntryLine[];
  
  // Command: Create Journal Entry
  create(data: CreateJournalEntryData): JournalEntryCreated {
    // Validate: debits must equal credits
    const totalDebits = data.lines.reduce((sum, l) => sum + (l.debit || 0), 0);
    const totalCredits = data.lines.reduce((sum, l) => sum + (l.credit || 0), 0);
    
    if (totalDebits !== totalCredits) {
      throw new Error('Debits must equal credits');
    }
    
    return {
      event_type: 'journal_entry.created',
      aggregate_id: this.id,
      data: {
        entry_number: generateEntryNumber(),
        lines: data.lines,
        description: data.description,
        created_by: data.user_id
      }
    };
  }
  
  // Apply Event
  apply(event: AccountingEvent) {
    switch (event.event_type) {
      case 'journal_entry.created':
        this.lines = event.data.lines;
        this.status = 'draft';
        break;
      case 'journal_entry.posted':
        this.status = 'posted';
        break;
      case 'journal_entry.voided':
        this.status = 'voided';
        break;
    }
  }
}
```

## Finding 12: Event Store Query Patterns for Financial Reporting

**Pattern**: Use projections for real-time financial reporting.

```typescript
// Real-time Balance Projection
class BalanceProjection {
  async handleEvent(event: AccountingEvent) {
    switch (event.event_type) {
      case 'journal_entry.posted':
        for (const line of event.data.lines) {
          if (line.debit > 0) {
            await this.updateBalance(line.account_id, line.debit, 'debit');
          }
          if (line.credit > 0) {
            await this.updateBalance(line.account_id, line.credit, 'credit');
          }
        }
        break;
        
      case 'journal_entry.voided':
        // Reverse the entry
        for (const line of event.data.lines) {
          if (line.debit > 0) {
            await this.updateBalance(line.account_id, -line.debit, 'debit');
          }
          if (line.credit > 0) {
            await this.updateBalance(line.account_id, -line.credit, 'credit');
          }
        }
        break;
    }
  }
  
  async updateBalance(accountId: string, amount: number, type: 'debit' | 'credit') {
    const account = await this.readStore.get(accountId);
    
    if (account.type === 'asset' || account.type === 'expense') {
      // Debit increases, credit decreases
      account.balance += type === 'debit' ? amount : -amount;
    } else {
      // Credit increases, debit decreases (liability, equity, revenue)
      account.balance += type === 'credit' ? amount : -amount;
    }
    
    await this.readStore.update(accountId, account);
  }
}
```

## Finding 14: Webhook Security for Financial Events

**Pattern**: Verify webhook signatures and implement idempotency.

```typescript
import crypto from 'crypto';

class WebhookSecurity {
  verifyStripeSignature(
    payload: string,
    signature: string,
    secret: string
  ): boolean {
    const elements = signature.split(',');
    const timestamp = elements[0].split('=')[1];
    const signatureHash = elements[1].split('=')[1];
    
    // Check timestamp (prevent replay attacks)
    const currentTime = Math.floor(Date.now() / 1000);
    const tolerance = 300; // 5 minutes
    if (Math.abs(currentTime - parseInt(timestamp)) > tolerance) {
      return false;
    }
    
    // Compute expected signature
    const signedPayload = `${timestamp}.${payload}`;
    const expectedSignature = crypto
      .createHmac('sha256', secret)
      .update(signedPayload)
      .digest('hex');
    
    // Constant-time comparison to prevent timing attacks
    return crypto.timingSafeEqual(
      Buffer.from(signatureHash),
      Buffer.from(expectedSignature)
    );
  }
}
```

## Finding 15: Event-Driven Architecture Testing Patterns

**Pattern**: Test event handlers in isolation using in-memory event stores.

```typescript
describe('Invoice Payment Handler', () => {
  let eventBus: InMemoryEventBus;
  let handler: InvoicePaymentHandler;
  
  beforeEach(() => {
    eventBus = new InMemoryEventBus();
    handler = new InvoicePaymentHandler(eventBus);
  });
  
  it('should create journal entry when invoice is paid', async () => {
    const event: InvoicePaidEvent = {
      event_type: 'invoice.paid',
      event_id: 'evt_123',
      data: {
        invoice_id: 'inv_123',
        amount: 1500,
        payment_id: 'pay_123'
      }
    };
    
    await handler.handle(event);
    
    const publishedEvents = eventBus.getPublishedEvents();
    expect(publishedEvents).toHaveLength(1);
    expect(publishedEvents[0].event_type).toBe('journal_entry.created');
  });
  
  it('should be idempotent - skip duplicate events', async () => {
    const event: InvoicePaidEvent = {
      event_type: 'invoice.paid',
      event_id: 'evt_123',
      data: { invoice_id: 'inv_123', amount: 1500, payment_id: 'pay_123' }
    };
    
    await handler.handle(event);
    await handler.handle(event); // Duplicate
    
    const publishedEvents = eventBus.getPublishedEvents();
    expect(publishedEvents).toHaveLength(1); // Only processed once
  });
});
```

## Summary of Key Patterns for L2 Cashflow

1. **Event Naming**: Use `{resource}.{action}` convention (e.g., `invoice.created`, `payment.received`)
2. **Event Sourcing**: Core ledger uses event sourcing for complete audit trail
3. **CQRS**: Separate write model (journal entries) from read model (reports, balances)
4. **Saga Pattern**: Multi-step transactions (payment → reconciliation → revenue recognition)
5. **Event Bus**: Redis Streams for persistence, consumer groups for load balancing
6. **Idempotency**: Event ID deduplication + database constraints
7. **Schema Versioning**: Header-based versioning with schema registry
8. **Security**: HMAC-SHA256 signature verification for webhooks
