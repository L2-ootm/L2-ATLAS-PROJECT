## 1. Unit Testing Financial Calculations

### Brazilian Tax Calculations (Simples Nacional, ICMS, PIS/COFINS)

**Frameworks**: `pytest` + `decimal.Decimal` (mandatory for currency)

**Key patterns**:
- Use `Decimal` everywhere — never `float` for financial values. Configure `getcontext().prec = 10` and `ROUND_HALF_EVEN` (banker's rounding).
- Parameterize over tax brackets with known-correct outputs from official Receita Federal tables.

## 2. Property-Based Testing

**Framework**: `hypothesis` (Python) — generates random inputs and verifies invariants.

### Core Invariants for Financial Systems

```python
from hypothesis import given, strategies as st, assume, settings
from decimal import Decimal

# Custom strategy for monetary values
money = st.decimals(
    min_value=Decimal("-1000000"),
    max_value=Decimal("1000000"),
    places=2,
)

@given(st.lists(st.tuples(money, money), min_size=2, max_size=20))
def test_debits_always_equal_credits(lines):
    """Property: valid journal entries always balance."""
    # Generate balanced entries by pairing amounts
    balanced_lines = []
    for debit, _ in lines:
        balanced_lines.append(JournalLine(debit=abs(debit), credit=Decimal("0")))
    balanced_lines.append(JournalLine(debit=Decimal("0"), credit=sum(abs(d) for d, _ in lines)))

    entry = JournalEntry(lines=balanced_lines)
    assert entry.total_debit == entry.total_credit

@given(st.decimals(min_value=Decimal("0.01"), max_value=Decimal("9999999.99"), places=2))
def test_balance_never_goes_negative_after_payment(amount):
    """Property: paying an invoice never creates negative balance if sufficient funds exist."""
    ledger = create_ledger_with_balance(amount)
    invoice = create_invoice(amount)
    pay_invoice(ledger, invoice)
    assert ledger.balance >= Decimal("0")

@given(st.lists(money, min_size=1, max_size=100))
def test_running_balance_reconciles_with_entries(amounts):
    """Property: running balance always equals sum of all entries."""
    ledger = EmptyLedger()
    for amount in amounts:
        ledger.post(JournalLine(debit=max(amount, Decimal("0")), credit=abs(min(amount, Decimal("0")))))
    assert ledger.balance == sum(amounts)

@settings(max_examples=500)
@given(st.decimals(min_value=Decimal("0.01"), max_value=Decimal("999999.99"), places=2))
def test_tax_calculation_never_exceeds_revenue(taxable_revenue):
    """Property: total taxes on revenue never exceed the revenue itself."""
    total_tax = (
        calcular_simples_nacional(taxable_revenue) +
        calcular_pis(taxable_revenue) +
        calcular_cofins(taxable_revenue)
    )
    assert total_tax <= taxable_revenue

@given(st.decimals(min_value=Decimal("1"), max_value=Decimal("36500"), places=0))
def test_depreciation_never_exceeds_cost(useful_life_days):
    """Property: accumulated depreciation never exceeds depreciable cost."""
    cost = Decimal("100000.00")
    schedule = calculate_depreciation(cost, Decimal("0"), int(useful_life_days), "linear")
    accumulated = Decimal("0")
    for month in schedule:
        accumulated += month.value
        assert accumulated <= cost
```

**Workflow on snapshot diff**:
1. CI runs `pytest --snapshot-update` on a PR.
2. Snapshot file changes are committed alongside code changes.
3. PR reviewer inspects the snapshot diff — any unexpected change in P&L/Balance Sheet numbers is a red flag.
4. Use `--snapshot-warn-unused` to detect stale snapshots.

**Coverage target**: Every report-generating function has a golden master test. Snapshot files stored in `tests/snapshots/` and version-controlled.

            assert entry.ip_address is not None

    def test_no_entry_can_be_deleted(self, seeded_ledger):
        """Soft-delete only — entries must never be physically removed."""
        entry = seeded_ledger.post_journal_entry(lines=[...])
        seeded_ledger.void_entry(entry.id, reason="Correction")
        # Entry still exists, just marked void
        assert seeded_ledger.get_entry(entry.id) is not None
        assert seeded_ledger.get_entry(entry.id).status == EntryStatus.VOID

    def test_modification_creates_audit_log(self, seeded_ledger):
        """Any modification to a closed period must be logged."""
        entry = seeded_ledger.post_journal_entry(lines=[...], period="2026-01")
        seeded_ledger.close_period("2026-01")

        audit_log = seeded_ledger.get_audit_log(reference_id=entry.id)
        assert len(audit_log) >= 1  # at minimum: creation + closing

## Summary: Recommended Test Pyramid for Cashflow Platform

| Layer | Tool(s) | Coverage Target | Run Frequency |
|-------|---------|-----------------|---------------|
| **Unit** (calculations) | pytest, hypothesis, decimal | 100% line, 95% branch | Every commit |
| **Property-based** | hypothesis | 200+ examples per invariant | Every commit |
| **Golden master** | syrupy | All reports | Every commit |
| **Integration** | pytest, testcontainers | Key module interactions | Every PR |
| **Contract** | pact-python, jsonschema | All inter-module APIs | Every PR |
| **Regression** | Custom golden file comparison | 1000+ scenarios | Every PR |
| **Performance** | locust, pytest-benchmark | API latency < 200ms P99 | Weekly / pre-release |
| **Security** | OWASP ZAP, trufflehog, pip-audit | OWASP Top 10 | Weekly / pre-release |
| **Compliance** | lxml (XSD validation), custom | All SPED/NFe outputs | Every release |

### Key Principles

1. **Decimal everywhere** — `float` is forbidden in financial calculations. Configure `Decimal` context once at module level.
2. **Immutable audit trail** — journal entries are append-only; voiding creates a new entry, never deletes.
3. **Property-based > example-based** for invariants — a single `@given` test covers more ground than 100 parametrized cases.
4. **Golden master as specification** — when report output changes, the snapshot diff IS the review.
5. **Time-travel is mandatory** — fiscal period logic must be testable at arbitrary points in time.
6. **Contract-first** — define event schemas and API contracts before implementation; validate with Pact.
7. **Regression files as documentation** — golden test files serve as executable specification of expected tax calculations.
