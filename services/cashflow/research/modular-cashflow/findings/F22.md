## 1. Rule Engine Patterns for Financial Approvals

### Decision Tables
Decision tables encode IF-THEN logic in a tabular format. Ideal for expense approval thresholds:

```yaml
# Example: Expense Approval Decision Table
columns:
  - field: amount
    operators: [>, <=, ==]
  - field: department
    operators: [==]
  - field: currency
    operators: [==]
actions:
  - field: required_approvers
    values: ["manager", "director", "cfo", "board"]
  - field: sla_hours
    values: [24, 48, 72, 168]

rules:
  - condition: { amount: ">50000", department: "engineering" }
    action: { required_approvers: ["manager", "cfo"], sla_hours: 48 }
  - condition: { amount: ">100000" }
    action: { required_approvers: ["manager", "director", "cfo", "board"], sla_hours: 168 }
```

### Decision Trees
Tree-based branching for complex financial routing:

```
Expense Request
├─ Amount <= $1,000 → Auto-approve (manager notification only)
├─ Amount $1,001–$10,000
│   ├─ Category = "travel" → Manager → Finance review
│   └─ Category = "equipment" → Manager → IT review → Finance
├─ Amount $10,001–$50,000 → Manager → Director → Finance
└─ Amount > $50,000 → Manager → Director → CFO → Board (if > $200k)
```

### Rule Chains
Sequential rule evaluation where each rule can modify context:

1. **Validation rules** — input validation (required fields, format checks)
2. **Enrichment rules** — attach metadata (department budget remaining, YTD spend)
3. **Routing rules** — determine approvers and escalation paths
4. **Compliance rules** — regulatory checks (SOX, anti-fraud thresholds)
5. **Notification rules** — channel selection based on urgency/severity

## 2. Workflow Engine Comparison for Financial Platforms

### Temporal.io

**Architecture**: Durable execution engine. Code-first workflows in Go, Java, TypeScript, Python, .NET, Ruby, Rust.

**Key features for financial workflows**:
- **Durable execution**: Workflows survive process crashes, server failures, and can run for years
- **Event Sourcing**: Complete Event History of every state transition — perfect for audit trails
- **Replay-based recovery**: On failure, replays Event History to restore exact pre-failure state
- **Deterministic execution**: Workflow code must be deterministic; side effects go in Activities
- **Signals**: Inter-workflow communication for approval responses, escalation triggers
- **Cron Jobs**: Built-in scheduled workflow execution
- **Continue-As-New**: Chain workflow executions for long-running processes (e.g., monthly close)
- **Schedules**: Programmatic scheduled triggers with cron expressions

```typescript
// Temporal: Multi-level approval workflow
export async function expenseApprovalWorkflow(request: ExpenseRequest) {
  // Level 1: Manager approval (48h SLA)
  const managerApproved = await workflow.condition(
    `manager-approval-${request.id}`,
    48 * 60 * 60 * 1000, // 48h timeout
  );

  if (!managerApproved) {
    await escalateToDirector(request);
    return "escalated";
  }

  // Level 2: Finance review for amounts > $10k
  if (request.amount > 10000) {
    await activity.sendToFinanceReview(request);
    return "pending-finance";
  }

  return "approved";
}
```

**Strengths for finance**: Battle-tested at scale (Stripe, Netflix, Snap). Strongest audit/event history. Handles multi-year workflows. Best recovery guarantees.
**Weaknesses**: Operational complexity (requires Temporal Service infrastructure). Steeper learning curve. Self-hosted or Temporal Cloud (paid).

### Inngest

**Architecture**: Event-driven durable functions on your own compute. Functions triggered by events, cron, or webhooks.

**Key features**:
- **Step-based execution**: Each `step.run()` is a retriable checkpoint
- **Flow Control**: Built-in concurrency limits, throttling, rate limiting
- **Event-driven**: Functions respond to typed events (`expense.submitted`, `invoice.overdue`)
- **Multi-language**: TypeScript, Python, Go SDKs
- **Wait for human input**: `step.waitForEvent()` for approval gates

```typescript
// Inngest: Approval workflow with escalation
inngest.createFunction(
  { id: "expense-approval", triggers: { event: "expense.submitted" } },
  async ({ step, event }) => {
    const expense = event.data;

    // Wait for manager approval with timeout
    const approval = await step.waitForEvent("manager.decision", {
      timeout: "48h",
      if: `event.expense_id == '${expense.id}'`,
    });

    if (!approval?.data.approved) {
      await step.run("escalate-to-director", () =>
        notifyDirector(expense)
      );
    }

    if (expense.amount > 10000) {
      await step.run("finance-review", () =>
        sendToFinance(expense)
      );
    }
  }
);
```

**Strengths**: Developer-friendly (write normal code), event-first architecture, built-in flow control, self-hosted compute. Good for teams that want code-first without Temporal's infrastructure overhead.
**Weaknesses**: Newer ecosystem. Less battle-tested at extreme scale. No equivalent to Temporal's Event History for deep audit trails.

### n8n

**Architecture**: Visual workflow automation. Node-based, self-hosted or cloud. 400+ integrations.

**Key features**:
- **Visual editor**: Drag-and-drop workflow building
- **400+ app integrations**: SendGrid, Twilio, Slack, email, databases out of the box
- **Schedule triggers**: Cron-based execution
- **If/Switch nodes**: Visual branching logic
- **Expression engine**: JavaScript expressions for data transformation
- **Webhook nodes**: Built-in webhook triggers and HTTP requests

**Strengths**: Best for non-technical users. Rapid prototyping. Huge integration library. Self-hosted option.
**Weaknesses**: Not designed for complex long-running financial workflows. No durable execution guarantees. Limited state management. Better for integration glue than core workflow logic.

### Trigger.dev

**Architecture**: Open-source background jobs framework. TypeScript-first, runs on your infrastructure.

**Key features**:
- **Wait for token**: Human-in-the-loop approval workflows built-in
- **Scheduled tasks**: Cron-based recurring tasks
- **Realtime API**: Live task status with React hooks
- **Waits**: Pause execution until external conditions are met
- **Concurrency & Queues**: Configurable rate limiting

```typescript
// Trigger.dev: Approval with human-in-the-loop
export const expenseApproval = task({
  id: "expense-approval",
  retry: { maxAttempts: 3 },
  run: async (payload: ExpensePayload) => {
    // Wait for human approval token
    const approval = await waitForToken({
      token: `approval-${payload.id}`,
      timeout: { amount: 48, unit: "hours" },
    });

    if (!approval.approved) {
      await escalate(payload);
    }

    return { status: "approved", approver: approval.approver };
  },
});
```

**Strengths**: Simple API, good for background jobs with approval gates. Open-source self-hosted option. Nice DX with TypeScript.
**Weaknesses**: Less mature than Temporal for complex orchestration. Smaller ecosystem. Primarily TypeScript.

### Recommendation Matrix

| Criteria | Temporal | Inngest | n8n | Trigger.dev |
|---|---|---|---|---|
| Durable execution | ★★★★★ | ★★★★ | ★★ | ★★★ |
| Audit trail | ★★★★★ | ★★★ | ★★ | ★★★ |

## 3. Approval Workflows

### Multi-Level Approval Pattern

```
┌─────────────┐
│  Request     │
│  Submitted   │
└──────┬──────┘
       │
       ▼
┌─────────────┐     ┌──────────────┐
│  Level 1:   │────▶│  Escalation  │ (if SLA exceeded)
│  Manager    │     │  Manager     │
└──────┬──────┘     └──────────────┘
       │ Approved
       ▼
┌─────────────┐
│  Level 2:   │ (conditional: amount > threshold)
│  Director   │
└──────┬──────┘
       │ Approved
       ▼
┌─────────────┐
│  Level 3:   │ (conditional: amount > CFO threshold)
│  CFO/Board  │
└──────┬──────┘
       │ Approved
       ▼
┌─────────────┐
│  Executed   │
│  & Notified │
└─────────────┘
```

### Parallel Approvals
For cross-functional approvals (e.g., Finance + Legal + IT for large purchases):

```yaml
parallel_approval:
  type: all  # or "any" for single-approver-required
  approvers:
    - role: finance_director
      sla_hours: 48
      escalation: cfo
    - role: legal_counsel
      sla_hours: 72
      escalation: general_counsel
    - role: it_manager
      sla_hours: 24
      escalation: cto
  on_complete: proceed_to_execution
  on_timeout: escalate_all_pending
```

### Escalation Rules
Escalation triggers:
1. **Time-based**: SLA exceeded (configurable per approval level)
2. **Absence-based**: Approver is OOO (integrate with calendar/HR systems)
3. **Load-based**: Approver has > N pending approvals (auto-redistribute)
4. **Amount-based**: Amount exceeds approver's authority limit

### SLA Tracking
Track per-approval metrics:
- Time to first response
- Total approval cycle time
- Escalation rate
- Approval/rejection ratio
- Bottleneck identification (which level takes longest)

## 5. Scheduled Jobs

### Cron-Based Schedules
| Job | Cron Expression | Description |
|---|---|---|
| Monthly close reminder | `0 9 28 * *` | Remind team 2 days before month-end close |
| Weekly AR aging report | `0 8 * * 1` | Generate accounts receivable aging report |
| Daily bank reconciliation | `0 6 * * *` | Auto-reconcile bank transactions |
| Quarterly tax reminder | `0 9 1 1,4,7,10 *` | Quarterly tax filing reminder |
| Annual audit prep | `0 9 1 11 *` | Annual audit preparation checklist |

### Event-Triggered Sequences

**Invoice Overdue → Dunning Sequence:**
```
Invoice Overdue (Day 0)
├─ Day 1:  Email reminder (polite)
├─ Day 7:  Email + SMS (firm)
├─ Day 14: Email + SMS + Account flag
├─ Day 30: Email + SMS + Collections escalation
└─ Day 60: Final notice + Account suspension
```

### Temporal Schedules (code-based)
```typescript
await client.schedule.create({
  scheduleId: "monthly-close-reminder",
  spec: { cronExpressions: ["0 9 28 * *"], timezone: "America/Sao_Paulo" },
  action: { startWorkflow: { workflowType: "MonthlyCloseReminder" } },
});
```

### Inngest Cron Functions
```typescript
inngest.createFunction(
  { id: "monthly-close", triggers: { cron: "0 9 28 * *" } },
  async ({ step }) => {
    await step.run("send-reminders", () => sendCloseReminders());
    await step.run("generate-report", () => generateMonthlyReport());
  }
```

## 6. Webhook Management

### Outgoing Webhooks

**Payload structure for external systems:**
```json
{
  "id": "wh_abc123",
  "event": "payment.completed",
  "timestamp": "2026-07-09T10:30:00Z",
  "data": {
    "payment_id": "pay_xyz789",
    "amount": 15000.00,
    "currency": "BRL",
    "vendor": "Acme Corp",
    "status": "completed"
  },
  "metadata": {
    "triggered_by": "automated_reconciliation",
    "workflow_id": "wf_abc123"
  }
}
```

### Retry Logic
```
Attempt 1: Immediate
Attempt 2: +1 minute
Attempt 3: +5 minutes
Attempt 4: +30 minutes
Attempt 5: +2 hours
Attempt 6: +8 hours
Attempt 7: +24 hours (final)
```

### Payload Signing (HMAC)
```typescript
import crypto from 'crypto';

function signPayload(payload: string, secret: string): string {
  return crypto.createHmac('sha256', secret).update(payload).digest('hex');
}

// Send with header
headers['X-Webhook-Signature'] = `sha256=${signPayload(body, webhookSecret)}`;
headers['X-Webhook-Timestamp'] = Date.now().toString();
```

### Webhook Logs
Track per-webhook:
- Request/response headers and body
- HTTP status code
- Response time (latency)
- Retry count
- Delivery status (delivered/failed/pending)
- Error details on failure

## 7. Business Rule Configuration

### Rule Definition Schema

```json
{
  "rule_id": "EXP-001",
  "name": "CFO Approval for Large Expenses",
  "version": 3,
  "status": "active",
  "created_at": "2026-01-15T00:00:00Z",
  "updated_at": "2026-06-20T00:00:00Z",
  "conditions": {
    "all": [
      { "field": "expense.amount", "operator": "greater_than", "value": 10000 },
      { "field": "expense.currency", "operator": "equals", "value": "BRL" }
    ]
  },
  "actions": [
    { "type": "require_approval", "role": "cfo", "sla_hours": 72 },
    { "type": "notify", "channel": "email", "recipients": ["cfo@company.com"] },
    { "type": "flag", "tag": "high-value" }
  ],
  "metadata": {
    "regulation": "SOX-404",
    "owner": "finance-team"
  }
}
```

### Rule Versioning

```yaml
versioning:
  strategy: semantic  # major.minor.patch
  rules:
    - version 1: "Original: amount > 5000 requires CFO"
    - version 2: "Updated: amount > 10000 requires CFO (inflation adjustment)"
    - version 3: "Added: currency check for BRL"
  rollback: true  # can revert to any previous version
  audit: true     # every change logged with who/when/why
```

### Rule Testing

```
Test Suite: EXP-001 - CFO Approval Rule
├─ Case 1: amount=15000, currency=BRL → EXPECT: cfo_approval_required ✅
├─ Case 2: amount=5000, currency=BRL → EXPECT: no_cfo_required ✅
├─ Case 3: amount=15000, currency=USD → EXPECT: no_cfo_required ✅ (wrong currency)
├─ Case 4: amount=10000, currency=BRL → EXPECT: no_cfo_required ✅ (boundary, not >)
└─ Case 5: amount=10001, currency=BRL → EXPECT: cfo_approval_required ✅
```

### UI Configuration Pattern
For non-technical users, provide a visual rule builder:
1. **Field selector** (dropdown: amount, department, vendor, date, etc.)
2. **Operator selector** (dropdown: equals, greater than, contains, in list, etc.)
3. **Value input** (text/number/dropdown depending on field type)
4. **Action builder** (dropdown: approve, reject, notify, escalate, flag)
5. **Preview** — "When [amount] [>] [10000] and [currency] [=] [BRL], then [require CFO approval]"

### Debugging Failed Automations

Common failure modes and diagnostics:
1. **Rule evaluation error** — invalid condition syntax, missing fields
2. **Notification delivery failure** — provider outage, invalid email/phone
3. **Approval timeout** — approver unreachable, SLA too short
4. **External webhook failure** — target system down, payload too large
5. **State inconsistency** — concurrent modifications, race conditions

**Debugging tools needed**:
- Execution timeline view (visual trace)
- Per-step input/output inspector
- Error log with stack traces
- Replay capability (re-run with same inputs)
- Rule test runner (dry-run without side effects)

## 9. Recommended Architecture for Modular Cashflow Platform

### Core Components

```
┌─────────────────────────────────────────────────┐
│                  Cashflow Platform               │
├─────────────────────────────────────────────────┤
│                                                  │
│  ┌──────────────┐  ┌──────────────┐            │
│  │  Rule Engine  │  │  Workflow    │            │
│  │  (config DB)  │  │  Engine      │            │
│  │              │  │  (Temporal)  │            │
│  └──────┬───────┘  └──────┬───────┘            │
│         │                  │                     │
│         ▼                  ▼                     │
│  ┌──────────────┐  ┌──────────────┐            │
│  │  Approval    │  │  Scheduler   │            │
│  │  Manager     │  │  (cron +     │            │
│  │              │  │   events)    │            │
│  └──────┬───────┘  └──────┬───────┘            │
│         │                  │                     │
│         ▼                  ▼                     │
│  ┌──────────────┐  ┌──────────────┐            │
│  │  Notification│  │  Webhook     │            │
│  │  Router      │  │  Manager     │            │
│  │  (Novu)      │  │              │            │
│  └──────┬───────┘  └──────┬───────┘            │
│         │                  │                     │
│         ▼                  ▼                     │
│  ┌──────────────────────────────────┐          │
│  │        Audit Logger              │          │
│  │   (every action, every state)    │          │
│  └──────────────────────────────────┘          │
│                                                  │
└─────────────────────────────────────────────────┘
```

### Key Design Decisions

1. **Temporal for core workflows** — durable execution, built-in audit trail via Event History, proven at financial scale
2. **Rule engine as data** — store rules in DB, not code; enable UI configuration and versioning
3. **Inngest for event-driven glue** — lighter than Temporal for event-triggered side effects (notifications, webhooks)
4. **Novu for notifications** — unified multi-channel API, preference management, template engine
5. **PostgreSQL for rule storage** — JSONB for flexible rule conditions, versioned with temporal tables
6. **HMAC-signed webhooks** — HMAC-SHA256 signing with timestamp validation, retry with exponential backoff
