## Finding 1: Core SaaS Metrics — MRR and ARR

**Monthly Recurring Revenue (MRR)** is the normalized monthly revenue from all active subscriptions. It excludes one-time fees and is the single most important SaaS growth metric.

### MRR Formula

```
MRR = SUM(active_subscription_monthly_revenue)

# Breakdown:
MRR = New MRR + Expansion MRR - Contraction MRR - Churned MRR
```

- **New MRR**: Revenue from newly acquired customers
- **Expansion MRR**: Revenue gained from upgrades, upsells, cross-sells
- **Contraction MRR**: Revenue lost from downgrades
- **Churned MRR**: Revenue lost from cancellations

### ARR (Annual Recurring Revenue)

```
ARR = MRR × 12
```

ARR is used for larger enterprise SaaS valuations and annual contract reporting. It normalizes monthly fluctuations.

### Design implication for cashflow platform
The system must track MRR changes as discrete events (new, expansion, contraction, churn) to enable waterfall analysis. Store each subscription's contribution to MRR at the subscription-item level, not just the subscription level.

## Finding 2: Churn Metrics — Logo vs Revenue Churn

### Logo (Customer) Churn Rate

```
Logo Churn Rate = (Customers lost in period / Customers at start of period) × 100
```

### Revenue Churn Rate

```
Revenue Churn Rate = (MRR lost from churned customers / MRR at start of period) × 100
```

### Net Revenue Churn / Net Revenue Retention (NRR)

```
Net Revenue Retention = (Starting MRR + Expansion - Contraction - Churn) / Starting MRR × 100
```

- **NRR > 100%**: Net expansion (existing customers growing faster than losses)
- **NRR < 100%**: Net contraction (losing more revenue than expanding)
- **Best-in-class B2B SaaS NRR**: 120-130%+

Logo churn can be high while revenue churn is low (losing small accounts, retaining large ones). Always track both.

### Design implication
NRR requires cohort-based tracking — the platform must group MRR changes by the customer cohort's subscription start date.

## Finding 3: LTV, CAC, and Payback Period

### Customer Lifetime Value (LTV)

```
LTV = ARPA × Gross Margin / Churn Rate

# Where:
ARPA = Average Revenue Per Account (monthly)
Gross Margin = (Revenue - COGS) / Revenue
Churn Rate = Monthly customer churn rate
```

Alternatively:

```
LTV = ARPA / Monthly Churn Rate
```

### Customer Acquisition Cost (CAC)

```
CAC = Total Sales & Marketing Spend / New Customers Acquired (in period)
```

### LTV/CAC Ratio

```
Target: LTV/CAC ≥ 3.0 (3:1 ratio)
```

- **< 1**: Losing money on each customer
- **1-3**: Suboptimal, need to improve unit economics
- **> 3**: Healthy, potentially under-investing in growth

### Payback Period

```
Payback Period (months) = CAC / (ARPA × Gross Margin)
```

Target: < 12 months for SMB SaaS, < 18 months for enterprise.

### Design implication
CAC requires tracking marketing/sales spend alongside customer acquisition. The platform needs a cost attribution layer separate from revenue.

---

## Finding 4: MRR Quick Ratio

```
Quick Ratio = (New MRR + Expansion MRR) / (Contraction MRR + Churned MRR)
```

- **> 4**: Excellent growth efficiency
- **2-4**: Healthy
- **< 2**: Growth is barely outpacing losses — churn problem

The Quick Ratio measures growth efficiency. A SaaS company can have high MRR but a low Quick Ratio if it's "leaky" — growing fast but losing fast.

## Finding 5: Subscription Billing Models

### Model Taxonomy

| Model | Description | Example |
|-------|-------------|---------|
| **Flat-rate** | Fixed price per period | $29/mo for all features |
| **Tiered** | Price changes based on usage tier | $0-100 units @ $0.01, $101-1000 @ $0.008 |
| **Volume pricing** | Total volume determines unit price | 1-100 units @ $10 each, 101+ @ $8 each |
| **Stairstep** | Flat fee per tier threshold | $29 for <1000 events, $99 for 1001-10000 |
| **Usage-based** | Pay-per-use metered billing | $0.001 per API call |
| **Hybrid (base + overage)** | Base subscription + usage overage | $49/mo + $0.01 per request over 10k |
| **Per-seat** | Price per user/seat | $12/user/mo |
| **Per-transaction** | Fee per transaction processed | 2.9% + $0.30 per transaction |
| **Freemium** | Free tier with paid upgrade path | Free for <100 users, Pro at $49/mo |

### Stripe's Usage-Based Billing Architecture

Stripe offers two approaches (from Stripe docs):

1. **Metronome** (recommended for new integrations): Real-time metering, tiered/dimensional/composite pricing, prepaid credits, enterprise contracts, ramp schedules
2. **Basic Billing Meters**: Lower-level primitive for existing integrations, full compatibility with Connect/Checkout

### Design implication
The platform must support a **pricing rule engine** that accepts a pricing model definition (flat, tiered, volume, stairstep, usage, hybrid) and computes charges from raw metering events. The engine must handle:
- Multiple pricing models per subscription
- Tier transitions mid-period (prorations)
- Hybrid: base subscription + metered overage

## Finding 6: Usage Metering and Rating Engines

### Metering Pipeline

```
Event Ingestion → Aggregation Window → Rating Engine → Invoice Line Item
```

### Rating Engine Models

**Tiered pricing** (each tier priced independently):
```
Usage = 1,500 units
Tier 1: 0-1000 @ $0.01 = $10.00
Tier 2: 1001-1500 @ $0.008 = $4.00
Total = $14.00
```

**Volume pricing** (entire volume at best tier):
```
Usage = 1,500 units
If tier applies to whole volume:
  1-1000: $0.01 × 1,500 = $15.00
  1001+: $0.008 × 1,500 = $12.00 (best price applies)
Total = $12.00
```

**Stairstep** (flat fee per tier):
```
Usage = 1,500 units
If tier 1 covers 0-1000 and tier 2 covers 1001-5000:
  Flat fee at tier 2 = $49
Total = $49
```

### Aggregation Windows

- **Real-time**: Event-by-event (Stripe Metronome supports this)
- **Hourly/Daily/Monthly**: Aggregate within window, rate at period end
- **Rolling**: Sliding window aggregation

### Design implication
The rating engine must be a separate, stateless module. It receives (events, pricing_rules, period) and returns (rated_amount, tier_breakdown). This enables swapping rating strategies without changing the billing core.

## Finding 7: Revenue Recognition Under ASC 606

### Core Principle (from Stripe Revenue Recognition docs)

ASC 606 requires revenue to be recognized **ratably over the service period**, not at the point of payment. This is the fundamental rule for SaaS.

### The 5-Step Model

1. **Identify the contract** with a customer
2. **Identify performance obligations** (each subscription item = separate obligation)
3. **Determine the transaction price** (including variable consideration)
4. **Allocate the price** to each performance obligation
5. **Recognize revenue** as each obligation is satisfied (over time for SaaS)

### Journal Entry Pattern for SaaS Subscription

For a $120 annual subscription paid upfront:

```
At payment (Month 1):
  Dr. Cash                          $120
  Cr. Accounts Receivable           $120
  Dr. Accounts Receivable           $120
  Cr. Deferred Revenue              $120

Each month (Months 1-12):
  Dr. Deferred Revenue              $10
  Cr. Revenue                       $10
```

### Deferred Revenue Accounts

| Account | Type | Description |
|---------|------|-------------|
| Deferred Revenue | Liability (credit) | Invoiced but not yet recognized — services not yet delivered |
| Long-term Deferred Revenue | Liability (credit) | Non-current portion for periods >12 months |
| Unbilled Accounts Receivable | Asset (debit) | Revenue accrued but not yet invoiced (e.g., prorations) |
| Unbilled Deferred Revenue | Liability (credit) | Obligation for unbilled contracts |

### Revenue Contra Accounts

- **Refunds**: Portion of refunded amount previously recognized
- **Disputes**: Portion of disputed amount previously recognized
- **Credit Notes**: Adjustments to previously recognized revenue
- **Bad Debt**: Previously recognized revenue from uncollectible invoices
- **Voids**: Previously recognized revenue from voided invoices

### Negative Line Items (Downgrades)

When a customer downgrades mid-period, the billing system generates:
- Positive line item for remaining time on new plan
- Negative line item for remaining time on old plan (proration credit)

Revenue recognition must handle both positive and negative amounts independently.

### Design implication
The platform needs a **revenue schedule generator** that:
1. Takes an invoice with line items and service periods
2. Generates a recognition schedule (daily/monthly accrual)
3. Tracks deferred vs recognized balances
4. Handles prorations, refunds, and disputes as contra entries

## Finding 8: Dunning and Payment Recovery

### Stripe Smart Retries (from Stripe docs)

Smart Retries uses AI/ML to determine optimal retry times based on:
- Number of devices presenting the payment method in last N hours
- Best time-to-pay patterns (e.g., debit cards in certain countries succeed better at 12:01 AM local)
- Continuous learning from transaction data

**Default recommendation**: 8 retries within 2 weeks.

### Payment Method Ordering (Retry Priority)

```
1. Subscription default payment method
2. Subscription default payment source
3. Customer default payment method
4. (Legacy) Customer default payment source
```

### Hard Decline Codes (Non-Retryable)

These cannot be retried without a new payment method:
- `incorrect_number`
- `lost_card`, `pickup_card`, `stolen_card`
- `revocation_of_authorization`
- `authentication_required`
- `highest_risk_level`
- `transaction_not_allowed`

### Direct Debit Retry Limits

| Method | Max Retries | Max Period |
|--------|-------------|------------|
| ACH Direct Debit | 2 | 40 days |
| SEPA Direct Debit | 2 | 30 days |
| Bacs Direct Debit | 2 | 30 days |
| ACSS Direct Debit | 1 | 30 days |

### Recovery Outcomes (Post-Retry Failure)

| Setting | Behavior |
|---------|----------|
| Cancel subscription | Subscription → `canceled` after max retry period |
| Mark unpaid | Subscription → `unpaid`, invoices continue as draft |
| Leave past-due | Subscription stays `past_due`, invoices continue generating |

### Dunning Flow Pattern

```
Payment Fails → Smart Retry (T+1, T+3, T+7, T+14...)
  → Email notifications to customer
  → Customer updates payment method
  → Payment succeeds → Subscription stays active

If all retries fail:
  → Subscription transitions to past_due → unpaid → canceled
  → Access revoked
```

### Design implication
The platform must implement:
1. A **retry scheduler** that queues payment attempts with configurable delays
2. A **notification engine** that sends emails at each retry stage
3. **Hard decline detection** to stop retrying non-recoverable failures
4. **Grace period management** before transitioning subscription state

## Finding 9: Subscription Lifecycle State Machine

### Stripe Subscription Statuses (from Stripe docs)

```
                   ┌─────────────┐
                   │  (created)  │
                   │ incomplete  │
                   └──────┬──────┘
                          │ payment succeeds
                   ┌──────▼──────┐
                   │   active    │◄──────────────────┐
                   └──────┬──────┘                   │
                          │ payment fails             │ payment succeeds
                   ┌──────▼──────┐                   │
                   │  past_due   │────── retry ──────┘
                   └──────┬──────┘
                          │ all retries exhausted
                   ┌──────▼──────┐
                   │   unpaid    │
                   └──────┬──────┘
                          │ or manual cancel
                   ┌──────▼──────┐
                   │  canceled   │
                   └─────────────┘

  Alternative paths:
  • trial starts → trialing → (trial ends + payment) → active
  • trial ends without payment method → paused
  • incomplete_expired (23h timeout without payment)
```

### Status Definitions

| Status | Access Granted? | Invoices Generated? | Description |
|--------|----------------|---------------------|-------------|
| `trialing` | Yes | No (until trial ends) | In trial period, product provisioned |
| `active` | Yes | Yes | In good standing |
| `incomplete` | No | Yes (first invoice) | Awaiting first payment |
| `incomplete_expired` | No | No | First payment never completed |
| `past_due` | Depends on policy | Yes | Payment failed, retries ongoing |
| `unpaid` | No | Yes (draft) | Retries exhausted, subscription remains |
| `paused` | No | No | Trial ended without payment method |
| `canceled` | No | No | Terminal state |

### Grace Period Design

Many SaaS products implement a **grace period** between `past_due` and access revocation:
- Customer retains access for N days after payment failure
- Enables users to update payment without service interruption
- Typically 3-7 days for B2B, 0-3 days for B2C

### Design implication
The platform needs a **state machine engine** with:
- Configurable transitions (e.g., past_due → unpaid after X days, or past_due → canceled)
- Webhook events at each transition
- Grace period timers that trigger access changes
- Win-back campaign triggers on cancellation

## Finding 10: Pricing Architecture — Plans, Add-ons, Coupons, Tax

### Plan Definition Schema

```yaml
plan:
  id: "pro_monthly"
  name: "Pro"
  billing_period: monthly
  base_price: 49.00
  currency: USD
  items:
    - name: "base_access"
      type: fixed
      price: 49.00
    - name: "api_calls"
      type: metered
      unit_price: 0.001
      included: 10000  # free tier
  add_ons:
    - id: "extra_storage"
      name: "Extra Storage"
      price: 9.00/unit
      type: per_unit
```

### Coupon/Discount Engine

Discounts apply at the subscription or invoice level:

| Type | Behavior | Example |
|------|----------|---------|
| **Percentage** | % off total or line item | 20% off for 3 months |
| **Fixed amount** | Dollar amount off | $10 off per month |
| **Trial extension** | Free period | 30-day free trial |
| **Free tier credits** | Included usage | 10,000 API calls free |

### Tax-Aware Pricing

Two approaches:
1. **Tax-exclusive**: Base price + tax calculated at checkout (US model)
2. **Tax-inclusive**: Price includes tax (EU/VAT model)

The platform must support both modes and switch per-customer based on jurisdiction.

```
Tax-inclusive:  Price = €100 (VAT included)
  → Revenue = €100 / 1.21 = €82.64
  → Tax liability = €17.36

Tax-exclusive:  Price = $100 + $8.50 tax = $108.50
  → Revenue = $100.00
  → Tax liability = $8.50
```

### Design implication
The pricing engine must store:
- Plan definitions with metered and fixed items
- Add-on catalog with per-unit or flat pricing
- Coupon rules with stacking behavior (allow/prevent multiple coupons)
- Tax jurisdiction mapping and rate tables
- Price override capability for enterprise deals

## Finding 11: Subscription Lifecycle — Trials, Reactivation, Win-back

### Trial Management

- **Free trial**: No payment method required (Stripe: `trialing` status)
- **Trial with card**: Card on file, charged at trial end
- **Trial end behavior**: Convert to `active`, pause, or cancel
- **Trial conversion rate**: Trial-to-paid metric (typically 15-25% for B2B SaaS)

### Reactivation Flow

```
Canceled subscription → New subscription created
  → Same customer, new subscription ID
  → Historical data preserved on old subscription
  → New billing cycle starts
```

Stripe does not allow reactivation of a `canceled` subscription — a new one must be created.

### Win-back Campaign Triggers

| Trigger | Timing | Action |
|---------|--------|--------|
| Immediate post-cancel | 0-24 hours | "We're sorry to see you go" + discount offer |
| 7-day check-in | 7 days | "Here's what you're missing" email |
| 30-day win-back | 30 days | Special offer to return |
| Re-engagement | 90 days | "New features since you left" |
| Seasonal | Quarterly | Targeted promotions for churned cohort |

### Design implication
The platform needs:
- A **subscription history** table (not just current state) for cohort analysis
- A **reactivation policy** that handles billing continuity vs. fresh start
- Integration hooks for win-back campaign systems

## Finding 12: Metrics Dashboards — Cohort Analysis, Waterfalls, Tables

### MRR Waterfall

The MRR waterfall decomposes month-over-month MRR changes:

```
Starting MRR: $100,000
  + New MRR:       +$12,000  (new customers)
  + Expansion MRR:  +$5,000  (upgrades, add-ons)
  - Contraction MRR: -$3,000 (downgrades)
  - Churned MRR:    -$4,000 (cancellations)
  = Ending MRR:   $110,000
  = Net MRR Change: +$10,000 (+10%)
```

### Revenue Waterfall (Stripe-style)

```
Recognized Revenue (beginning of period)
  + New recognized
  + Expansion recognized
  - Refunds/disputes
  - Contraction recognized
  - Churn recognized
  = Recognized Revenue (end of period)

Deferred Revenue (beginning of period)
  + New invoiced
  - Revenue recognized
  = Deferred Revenue (end of period)
```

### Churn Cohort Table

```
Cohort      | Month 1 | Month 2 | Month 3 | Month 4 | Month 5
------------|---------|---------|---------|---------|--------
Jan 2026    | 100%    | 92%     | 88%     | 85%     | 83%
Feb 2026    | 100%    | 94%     | 90%     | 87%     | -
Mar 2026    | 100%    | 91%     | 86%     | -       | -
Apr 2026    | 100%    | 93%     | -       | -       | -
May 2026    | 100%    | -       | -       | -       | -
```

Key insight: Cohort analysis reveals whether product improvements are reducing churn over time (later cohorts retaining better).

### Expansion Waterfall

```
Starting expansion MRR
  + Seat expansion (existing accounts adding users)
  + Tier upgrades
  + Add-on purchases
  + Usage overage
  = Total expansion MRR
```

### Design implication
The platform must:
- Store MRR events (not just snapshots) so waterfalls can be reconstructed
- Tag every MRR change with a type (new/expansion/contraction/churn) and source
- Support cohort grouping by acquisition date, plan, geography, or segment
- Enable time-series aggregation for all metrics (daily, weekly, monthly granularity)

## Finding 13: Data Model Requirements for a Modular Cashflow Platform

Based on all findings, the core data entities are:

```
┌─────────────────┐     ┌──────────────────┐
│    Customer      │────▶│   Subscription    │
│  (account/tenant)│     │  (plan, status)   │
└─────────────────┘     └────────┬─────────┘
                                  │
                    ┌─────────────┼──────────────┐
                    ▼             ▼               ▼
            ┌──────────┐  ┌────────────┐  ┌──────────────┐
            │   Plan   │  │  Invoice   │  │ MRR Events   │
            │ (pricing)│  │ (line items│  │ (new/exp/con/ │
            └──────────┘  │  + status) │  │  churn)       │
                          └─────┬──────┘  └──────────────┘
                                │
                    ┌───────────┼────────────┐
                    ▼           ▼            ▼
            ┌──────────┐ ┌──────────┐ ┌──────────────┐
            │ Payment  │ │ Revenue  │ │  Metering    │
            │ Attempt  │ │ Schedule │ │  Events      │
            │ (+ retry)│ │ (deferred│ │  (usage)     │
            └──────────┘ │  /recog) │ └──────────────┘
                         └──────────┘
```

### Key tables for billing core

1. **customers** — tenant/account, payment methods, tax jurisdiction
2. **plans** — pricing model definition, items, billing period
3. **subscriptions** — links customer to plan, status, trial end, cancel date
4. **subscription_items** — individual line items (fixed + metered)
5. **invoices** — generated per billing period, line items, status
6. **invoice_items** — individual charges, prorations, credits
7. **payments** — payment attempts, status, retry count, failure reason
8. **coupons** — discount rules, validity, stacking
9. **metering_events** — raw usage events with timestamps
10. **mrr_events** — typed MRR deltas (new/expansion/contraction/churn)
11. **revenue_schedules** — deferred revenue accrual entries
12. **dunning_config** — retry policies, grace periods, notification templates

## Finding 14: Key Formulas Summary

| Metric | Formula | Target |
|--------|---------|--------|
| MRR | Sum of all active subscription MRR | Growing |
| ARR | MRR × 12 | Growing |
| Logo Churn | (Lost customers / Start customers) × 100 | < 5% monthly (SMB) |
| Revenue Churn | (Lost MRR / Start MRR) × 100 | < 3% monthly |
| NRR | (Start MRR + Expansion - Contraction - Churn) / Start MRR × 100 | > 100% |
| LTV | ARPA / Monthly Churn Rate | Varies |
| CAC | S&M Spend / New Customers | Minimize |
| LTV/CAC | LTV / CAC | > 3.0 |
| Payback | CAC / (ARPA × Gross Margin) | < 12 months |
| Quick Ratio | (New MRR + Expansion MRR) / (Contraction MRR + Churned MRR) | > 4.0 |
| Trial Conversion | (Trial → Paid) / Total Trials × 100 | 15-25% B2B |
