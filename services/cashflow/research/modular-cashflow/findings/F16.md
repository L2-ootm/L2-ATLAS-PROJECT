## 1. OFX/QFX Import Specification

### File Structure
- **OFX (Open Financial Exchange)**: Data-stream format for exchanging financial information
- **Versions**: OFXv1 (SGML-based, versions 1.0–1.6) and OFXv2 (XML-based)
- **QFX**: Proprietary Intuit variant of OFX for Web Connect (manual download)
- **Extension**: `.ofx` or `.qfx`
- **MIME type**: `application/x-ofx`

### OFX Transaction Fields (from ofxparse library)
```python
# Transaction object fields:
transaction.payee          # Payee name
transaction.type           # Transaction type (DEBIT, CREDIT, etc.)
transaction.date           # Transaction date
transaction.user_date      # User-facing date
transaction.amount         # Amount (Decimal type)
transaction.id             # Unique transaction ID (FITID)
transaction.memo           # Memo/description
transaction.sic            # Standard Industrial Code
transaction.mcc            # Merchant Category Code
transaction.checknum       # Check number
```

### Investment Transaction Fields
```python
transaction.type           # Transaction type
transaction.tradeDate      # Trade date
transaction.settleDate     # Settlement date
transaction.memo           # Memo
transaction.security       # Security object (uniqueid, name, ticker)
transaction.income_type    # Income type
transaction.units          # Number of units
transaction.unit_price     # Price per unit
transaction.comission      # Commission
transaction.fees           # Fees
transaction.total          # Total amount
transaction.tferaction     # Transfer action
```

### Account Types in OFX
- `AccountType.Bank` - Bank accounts
- `AccountType.CreditCard` - Credit card accounts
- `AccountType.Investment` - Investment/brokerage accounts
- `AccountType.Unknown` - Unknown type

### Account Object Fields
```python
account.account_id         # Account number
account.routing_number     # Bank routing number
account.branch_id          # Transit ID / branch number
account.type               # AccountType object
account.statement          # Statement object
account.institution        # Institution object
account.brokerid           # Investment broker ID (for investment accounts)
```

### Statement Object Fields
```python
statement.start_date       # Statement period start
statement.end_date         # Statement period end
statement.balance          # Ledger balance
statement.available_balance # Available balance
statement.transactions     # List of Transaction objects
```

### Parsing Libraries
- **ofxparse** (Python, MIT, v0.21): Simple parser, handles BankAccount/CreditAccount/InvestmentAccount
- **ofxtools** (Python, GPL-3.0, v1.1.1): Comprehensive OFXv1/v2 parser, covers OFX spec sections 7-13, zero external dependencies, requires Python 3.10+

### Key Parsing Pattern
```python
from ofxtools.Parser import OFXTree
parser = OFXTree()
with open('statement.ofx', 'rb') as f:
    parser.parse(f)
ofx = parser.convert()  # Converts to typed Python objects
# Navigate: ofx.statements[0].transactions, ofx.securities, etc.
```

## 2. QuickBooks Export Formats

### IIF (Intuit Interchange Format)
- **Legacy format** for QuickBooks Desktop
- Tab-delimited text format
- Record types: `!TRNS` (transaction header), `!SPL` (split line), `!ENDTRNS` (transaction end)
- Key fields: TRNSTYPE, DATE, ACCNT, NAME, AMOUNT, DOCNUM, MEMO
- **Limitation**: Does not support all QuickBooks data types

### QBO (QuickBooks Online XML)
- XML-based export format
- Used for bank/credit card statement downloads
- Contains transaction data with account mapping

### CSV Export
- QuickBooks Online supports CSV export for:
  - Transactions (with date, description, amount columns)
  - Chart of Accounts (name, type, number)
  - Journal entries
  - Reports (P&L, Balance Sheet, etc.)

### QFX (Intuit Web Connect)
- Proprietary variant of OFX
- Used for manual bank transaction downloads in Quicken/QuickBooks
- Same structure as OFX but with Intuit extension tags

## 4. CNAB 240/400 (Brazilian Bank File Formats)

### CNAB 240 Structure
- **Fixed-width format**, 240 characters per line
- **Encoding**: ASCII (some banks use EBCDIC for legacy systems)
- **Record Types**:
  - Header (Tipo Registro 0)
  - Transactions (Tipo Registro 1)
  - Trailer (Tipo Registro 9)

### CNAB 240 Record Layout (Generic)
```
Position 1-3:    Código do Banco (Bank Code)
Position 4-7:    Lote de Serviço (Service Batch)
Position 8:      Tipo de Registro (Record Type: 0=Header, 1=Transaction, 9=Trailer)
Position 9-14:   Número Sequencial do Registro (Sequential Record Number)
...
```

### Key Field Positions (vary by bank)
- **Header**: Bank code, service type, company name, CNPJ, date, sequence
- **Transactions**: Company account, beneficiary, transaction type, value, dates
- **Trailer**: Total records, total value, control fields

### CNAB 400 vs CNAB 240
- **CNAB 400**: 400 characters per line, older format, still used by some banks
- **CNAB 240**: 240 characters per line, newer standard, more structured

### Parsing Approach

## 6. Chart of Accounts Migration

### Mapping Between Systems
```python
# Account mapping configuration
ACCOUNT_MAP = {
    'source_system': {
        '4010': {'name': 'Cash', 'type': 'asset'},
        '4020': {'name': 'Accounts Receivable', 'type': 'asset'},
        '2010': {'name': 'Accounts Payable', 'type': 'liability'},
    },
    'target_system': {
        '1000': {'name': 'Cash and Cash Equivalents', 'type': 'current_asset'},
        '1100': {'name': 'Receivables', 'type': 'current_asset'},
        '2000': {'name': 'Payables', 'type': 'current_liability'},
    }
}
```

### Hierarchical Account Handling
- **Flat accounts**: Direct 1:1 mapping
- **Hierarchical accounts**: Parent-child relationships need level preservation
- **Numbering schemes**: Some systems use dot notation (1.1, 1.2), others use hierarchical codes (1000, 1100, 1200)

### Common Challenges
1. Account type mismatches (asset vs. current_asset)
2. Different granularity levels
3. Tax category mappings
4. Currency-specific accounts

## 8. Bulk Data Patterns

### Streaming Import
```python
# Generator pattern for large files
def stream_csv_import(file_path: str, batch_size: int = 1000):
    with open(file_path, 'r') as f:
        batch = []
        for line in csv.reader(f):
            batch.append(transform_row(line))
            if len(batch) >= batch_size:
                yield batch
                batch = []
        if batch:
            yield batch
```

### Progress Reporting
```python
# Progress tracking for imports
{
    "import_id": "imp_123",
    "status": "in_progress",
    "total_rows": 50000,
    "processed_rows": 25000,
    "percent_complete": 50.0,
    "errors": 12,
    "estimated_time_remaining": "5m 30s"
}
```

### Resumable Imports
- Track last successfully processed row
- Store import state in database
- On resume, skip processed rows
- Handle partial failures (skip bad rows, continue processing)

### Idempotent Imports
- Use unique transaction IDs or content hashes
- Check for existing records before insert
- Use upsert (INSERT ... ON CONFLICT UPDATE) patterns
- Maintain import run history for re-runs

## 9. Data Validation Rules

### Checksums
- **CNAB files**: Often include checksum fields at end of record
- **OFX**: No built-in checksum, but FITID (Financial Institution Transaction ID) provides uniqueness
- **CSV**: Row count validation, total amount verification

### Balancing Checks
```python
# Double-entry validation
def validate_journal_entry(entry: dict) -> bool:
    total_debits = sum(line['debit'] for line in entry['lines'])
    total_credits = sum(line['credit'] for line in entry['lines'])
    return abs(total_debits - total_credits) < Decimal('0.01')  # Penny tolerance
```

### Referential Integrity
- Account codes must exist in Chart of Accounts
- Contact IDs must reference valid contacts
- Currency codes must be valid ISO 4217

### Date Range Validation
- Transaction dates within fiscal year
- No future-dated transactions (configurable)
- Period close dates respected

## 11. Recommended Architecture for L2 Cashflow Import Module

### Module Structure
```
cashflow/
├── importers/
│   ├── __init__.py
│   ├── base.py           # Abstract importer interface
│   ├── ofx_importer.py   # OFX/QFX parser
│   ├── csv_importer.py   # Configurable CSV parser
│   ├── xlsx_importer.py  # Excel parser
│   ├── iif_importer.py   # QuickBooks IIF parser
│   └── cnab_importer.py  # CNAB 240/400 parser
├── validators/
│   ├── __init__.py
│   ├── base.py           # Abstract validator
│   ├── transaction_validator.py
│   ├── account_validator.py
│   └── balance_validator.py
├── transformers/
│   ├── __init__.py
│   ├── base.py           # Abstract transformer
│   ├── account_mapper.py # COA mapping
│   └── field_mapper.py   # Generic field mapping
├── loaders/
│   ├── __init__.py
│   ├── base.py           # Abstract loader
│   └── database_loader.py
└── pipeline/
    ├── __init__.py
    ├── orchestrator.py    # Main import orchestrator
    ├── progress.py        # Progress tracking
    └── audit.py           # Audit logging
```

### Key Interfaces
```python
# Base importer
class BaseImporter(ABC):
    @abstractmethod
    def can_import(self, file_path: str) -> bool:
        """Check if this importer handles the file format"""
        ...
    
    @abstractmethod
    def extract(self, file_path: str) -> Iterator[dict]:
        """Parse file and yield records"""
        ...

# Base validator
class BaseValidator(ABC):
    @abstractmethod
    def validate(self, record: dict) -> ValidationResult:
        """Validate a single record"""
        ...

# Base transformer
class BaseTransformer(ABC):
    @abstractmethod
    def transform(self, record: dict, context: dict) -> dict:
        """Transform record to target schema"""
        ...

# Base loader
class BaseLoader(ABC):
    @abstractmethod
    def load(self, records: list[dict]) -> LoadResult:
        """Load records into target system"""
        ...
```
