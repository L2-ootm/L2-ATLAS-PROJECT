## Finding 1: REST API Design for Financial Resources — Standard CRUD + Actions

**Source**: Stripe API reference (`/v1/invoices`, `/v1/payments`, `/v1/charges`)

Standard financial resource endpoints follow this pattern:

```
GET    /v1/invoices          — List all (paginated)
POST   /v1/invoices          — Create
GET    /v1/invoices/:id      — Retrieve
POST   /v1/invoices/:id      — Update (not PUT — partial updates preferred)
DELETE /v1/invoices/:id      — Cancel/delete
```

**Action endpoints** (non-CRUD operations) use nested paths:
```
POST /v1/invoices/:id/pay       — Pay an invoice
POST /v1/invoices/:id/finalize  — Finalize draft
POST /v1/invoices/:id/send      — Send to customer
POST /v1/invoices/:id/void      — Void
```

**Key pattern**: Financial APIs prefer POST over PUT for updates because financial mutations are rarely idempotent by default (double-posting creates duplicate transactions). Stripe explicitly does not support bulk updates — one object per request.

**Payload schema example** (Stripe invoice creation):
```json
{
  "customer": "cus_abc123",
  "currency": "usd",
  "auto_advance": true,
  "collection_method": "send_invoice",
  "days_until_due": 30,
  "metadata": {"order_id": "6735"},
  "lines": [
    {
      "description": "Consulting services",
      "quantity": 10,
      "unit_amount": 5000,
      "price": "price_xyz"
    }
  ]
}
```

**Response envelope** (Stripe standard):
```json
{
  "id": "in_abc123",
  "object": "invoice",
  "status": "open",
  "amount_due": 50000,
  "currency": "usd",
  "created": 1405641735,
  "livemode": false,
  "metadata": {}
}
```

## Finding 2: GraphQL vs REST for Financial Platforms

**Source**: Plaid API (REST-only), Stripe (REST-only, no GraphQL), general financial API landscape

**Current reality**: Major financial APIs (Stripe, Plaid, Xero, QuickBooks) are REST-only. None offer GraphQL as a primary interface.

**When GraphQL makes sense for financial platforms**:
- **Reporting dashboards**: Single GraphQL query can fetch invoices + line items + payments + journal entries in one round-trip, vs 4+ REST calls
- **Aggregated views**: Dashboards that combine data from multiple modules (cashflow + invoicing + reporting) benefit from GraphQL's ability to request exactly the fields needed
- **Client-driven field selection**: Mobile apps vs web dashboards vs PDF exports need different subsets of financial data

**When REST remains better**:
- **Mutations**: Financial mutations (create invoice, record payment) are better modeled as explicit POST endpoints with clear side effects
- **Webhooks**: Webhook payloads are naturally REST/JSON objects, not GraphQL subscriptions
- **Caching**: REST GET responses cache naturally; GraphQL queries are harder to cache at the HTTP layer

**Recommended hybrid approach**:
- REST for all CRUD operations and webhooks (module API namespace)
- GraphQL as a read-only query layer for cross-module dashboards and reporting
- REST webhook delivery for async events between modules

## Finding 3: API Versioning Strategies — What Major Players Use

**Source**: Stripe API versioning docs, Plaid API

### Stripe: Date-based versioning (header-based)
- Uses date-based versions: `2026-06-24.dahlia` (major codename + date)
- **Header versioning**: `Stripe-Version: 2026-06-24.preview` header
- Account-level default set in dashboard (Workbench)
- SDK versions pin to the API version current at SDK release time
- Monthly releases are backward-compatible; major releases require migration
- **No URL versioning** — all endpoints live under `/v1/` or `/v2/`

### Plaid: POST-only, no versioning in URL
- All requests are POST to endpoints like `https://production.plaid.com/auth/get`
- Versioning handled through request body parameters (`options` field)
- Separate sandbox/production hosts instead of URL versioning

### Xero: URL versioning
- Uses `/api.xro/2.0/` prefix for current version
- Version in URL path, not header

### QuickBooks: URL versioning
- Uses `/v3/company/{companyId}/` prefix
- Company ID in URL (multi-tenant isolation)

### Recommendation for cashflow platform:
```
Header versioning (Stripe model) + URL namespace (/v1/, /v2/)
```
Rationale:
- URL versioning for major breaking changes (simpler for clients)
- Header versioning for minor/patch changes within a major version
- Pin webhook event payloads to the API version at webhook creation time
- SDKs should pin their target version, like Stripe's SDKs do

## Finding 4: Rate Limiting and Pagination for Financial Data APIs

**Source**: Stripe pagination docs, Plaid API reference

### Cursor-based pagination (Stripe pattern):
```
GET /v1/invoices?limit=100&starting_after=in_abc123
```

Response:
```json
{
  "object": "list",
  "url": "/v1/invoices",
  "has_more": true,
  "data": [
    {"id": "in_def456", "object": "invoice", ...},
    {"id": "in_ghi789", "object": "invoice", ...}
  ]
}
```

Parameters:
- `limit` (optional, default 10, max 100)
- `starting_after` — cursor for forward pagination
- `ending_before` — cursor for backward pagination
- Mutually exclusive

**Why cursor-based over offset for financial data**:
- Financial transaction sets are large (millions of rows) and frequently mutating
- Offset pagination (`?page=5&per_page=20`) breaks when new rows are inserted between requests
- Cursor-based gives stable snapshots; the cursor is an opaque object ID
- Stripe explicitly uses `starting_after`/`ending_before` — never offset

**v2 API note**: Stripe's v2 namespace introduces a different pagination interface (not yet fully documented in fetched content).

**Rate limiting pattern**:
- Return `429 Too Many Requests` with `Retry-After` header
- Include rate limit headers: `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`
- Plaid uses client_id + secret in headers for rate limit tracking
- Consider per-tenant rate limits for multi-tenant platforms

## Finding 5: Multi-Tenant API Design — Tenant Identification and Data Isolation

**Source**: Azure multitenant architecture guide, Stripe Connect (organization/context pattern)

### Tenant identification methods:

1. **JWT claim** (recommended for API-first):
   ```json
   {
     "sub": "user_abc123",
     "tenant_id": "tenant_xyz789",
     "org_id": "org_def456",
     "scope": "invoices:read payments:write"
   }
   ```
   Every request carries the tenant context in the token. No extra header needed.

2. **Custom header** (X-Tenant-ID):
   ```
   X-Tenant-ID: tenant_xyz789
   ```
   Good for server-to-server calls where JWT isn't available.

3. **Subdomain** (for customer-facing apps):
   ```
   https://tenant-xyz.cashflow.app/api/v1/invoices
   ```
   Good for white-label SaaS, harder for API-first design.

4. **URL path prefix**:
   ```
   /api/v1/tenants/:tenant_id/invoices
   ```
   Explicit but noisy; rarely used in production APIs.

### Stripe's organization/context pattern:
- Stripe Connect uses `Stripe-Context` header to identify the originating account
- Webhook events include `context` field for multi-account routing
- Each connected account has its own API key for isolation

### Data isolation strategies:
- **Row-level security (RLS)**: All queries filtered by `tenant_id` at the database layer
- **Application-level filtering**: Middleware extracts tenant from JWT/header, adds WHERE clause
- **Schema-per-tenant**: For strict compliance requirements (rare for SaaS)
- **Database-per-tenant**: Maximum isolation, highest cost

### Cross-tenant queries (consolidated reporting):
```sql
-- Aggregated P&L across tenant group (parent org)
SELECT account_id, SUM(debit) - SUM(credit) as balance
FROM journal_entries
WHERE tenant_id = ANY(ARRAY['t1','t2','t3'])
  AND date BETWEEN '2026-01-01' AND '2026-06-30'
GROUP BY account_id
```
- Requires `org_id` or `group_id` above `tenant_id` in the hierarchy
- Cross-tenant queries must be explicitly authorized (separate permission scope)

## Finding 6: Webhook Delivery Patterns

**Source**: Stripe webhooks documentation (comprehensive)

### Retry logic (Stripe model):
- Live mode: retries for up to **3 days** with exponential backoff
- Sandbox: retries 3 times over a few hours
- Manual retry available via dashboard (15 days) or CLI (30 days)
- Events are retried even after a previous 2xx if the endpoint was previously failing

### Exponential backoff:
```
Attempt 1: immediate
Attempt 2: ~1 minute
Attempt 3: ~5 minutes
Attempt 4: ~30 minutes
Attempt 5: ~2 hours
Attempt 6: ~8 hours
Attempt 7: ~24 hours (max backoff)
Total window: ~3 days
```

### Signature verification (HMAC-SHA256):
```
Stripe-Signature: t=1492774577,v1=5257a869e7ecebeda32affa62cdca3fa51cad7e77a0e56ff536d0ce8e108d8bd
```

Verification steps:
1. Extract timestamp (`t=`) and signature (`v1=`)
2. Construct `signed_payload` = `{timestamp}.{raw_body}`
3. Compute HMAC-SHA256 with endpoint secret
4. Constant-time compare against received signature
5. Reject if timestamp is older than tolerance (default 5 minutes)

### Event replay:
- Stripe provides event retrieval API: `GET /v1/events/:id`
- Events are stored for 30 days
- Use `stripe events resend <event_id>` for manual replay
- Events are NOT guaranteed to be delivered in order — design handlers to be order-independent

### Idempotency:
- Log processed event IDs to prevent duplicate processing
- Two separate events can reference the same object — use `data.object.id` + `event.type` for dedup
- Return 2xx quickly; process asynchronously via queue

### Recommended webhook payload for cashflow platform:
```json
{
  "id": "evt_abc123",
  "type": "invoice.payment_received",
  "api_version": "2026-07-01",
  "created": 1689012345,
  "tenant_id": "tenant_xyz789",
  "data": {
    "object": {
      "id": "inv_abc123",
      "amount": 50000,
      "currency": "usd",
      "status": "paid",
      "paid_at": "2026-07-09T10:00:00Z"
    }
  }
}
```

## Finding 7: API-First Module Design

**Source**: Stripe architecture (REST namespace), Plaid API (POST-only), general microservice patterns

### Module namespace composition:
```
/api/v1/
├── invoices/          — Invoicing module
│   ├── GET    /list
│   ├── POST   /
│   ├── GET    /:id
│   ├── POST   /:id/update
│   ├── POST   /:id/pay
│   └── POST   /:id/finalize
├── payments/          — Payments module
│   ├── GET    /list
│   ├── POST   /
│   ├── GET    /:id
│   ├── POST   /:id/refund
│   └── POST   /:id/capture
├── journal-entries/   — Accounting module
│   ├── GET    /list
│   ├── POST   /
│   ├── GET    /:id
│   └── POST   /:id/post
├── accounts/          — Chart of accounts
├── reports/           — Reporting module (GraphQL candidate)
└── webhooks/          — Webhook management
    ├── GET    /list
    ├── POST   /
    ├── DELETE /:id
    └── POST   /:id/test
```

### Middleware chain pattern:
```
Request → Rate Limiter → Auth (JWT verify) → Tenant Context (extract tenant_id) → 
  CORS → Request Logger → Validator → Router → Handler → Response Serializer
```

### Router composition:
- Each module registers its own router with a prefix
- Main router composes module routers:
```python
app.include_router(invoices.router, prefix="/api/v1/invoices", tags=["invoices"])
app.include_router(payments.router, prefix="/api/v1/payments", tags=["payments"])
app.include_router(journal.router, prefix="/api/v1/journal-entries", tags=["journal"])
```

### OpenAPI-first workflow:
1. Define OpenAPI spec per module
2. Generate server stubs + client SDKs from spec
3. Validate requests against spec at runtime (middleware)
4. Publish spec as the contract between frontend and backend

## Finding 8: Financial API Security — OAuth 2.0 and Key Management

**Source**: Plaid API (client_id + secret), Stripe (API key model), QuickBooks OAuth, general banking API patterns

### OAuth 2.0 flows for banking integrations:

**Authorization Code Flow** (standard for server-to-server):
```
1. App redirects user to bank auth URL
   GET /authorize?
     response_type=code&
     client_id=APP_ID&
     redirect_uri=https://app.cashflow/callback&
     scope=accounts:read transactions:read&
     state=RANDOM_CSRF_TOKEN

2. User authenticates, bank redirects back
   GET /callback?code=AUTH_CODE&state=RANDOM_CSRF_TOKEN

3. App exchanges code for tokens
   POST /token
   Body: {
     grant_type: "authorization_code",
     code: "AUTH_CODE",
     client_id: "APP_ID",
     client_secret: "APP_SECRET"
   }

4. Response: {
     access_token: "at_xxx",
     refresh_token: "rt_xxx",
     expires_in: 3600,
     token_type: "Bearer"
   }

5. Use access_token for API calls
   Authorization: Bearer at_xxx

6. Refresh before expiry
   POST /token
   Body: {
     grant_type: "refresh_token",
     refresh_token: "rt_xxx",
     client_id: "APP_ID",
     client_secret: "APP_SECRET"
   }
```

### API key management (Stripe model):
- **Publishable key** (pk_...): Safe for client-side, identifies your account
- **Secret key** (sk_...): Server-side only, never expose
- **Restricted key** (rk_...): Scoped to specific resources/actions
- Keys rotated periodically; old keys can be deprecated with grace period

### Field-level encryption for PII/financial data:
- **At rest**: AES-256-GCM for SSN, bank account numbers, tax IDs
- **In transit**: TLS 1.2+ (Plaid requires TLS 1.2 minimum)
- **In application**: Encrypt before storage, decrypt on read; never log PII
- **Tokenization**: Replace sensitive fields with non-reversible tokens for logging/debugging

### Plaid security model:
- `client_id` + `secret` in headers (`PLAID-CLIENT-ID`, `PLAID-SECRET`)
- Tokens never exposed client-side (except `public_token` and `link_token`)
- Separate sandbox/production environments with distinct credentials
- Open Finance Data Security Standard for data handling

## Finding 9: Stripe Connect Multi-Tenant Pattern — Organization Architecture

**Source**: Stripe webhooks docs (organization context pattern)

Stripe's organization model is directly relevant to multi-tenant cashflow platforms:

```
Organization (org_id)
├── Account 1 (tenant_abc)
│   ├── API key: sk_test_123
│   └── Resources: invoices, payments, customers
├── Account 2 (tenant_def)
│   ├── API key: sk_test_456
│   └── Resources: invoices, payments, customers
└── Event Destination (webhook endpoint)
    ├── Listens to: organization events
    ├── Uses: Stripe-Context header to route
    └── Resolves account via context field
```

Key design decisions:
- **Event routing**: Webhook events carry `context` field identifying the originating account
- **API key per account**: Each tenant/account has its own API key
- **Organization-level event destination**: Single webhook endpoint receives events from all accounts
- **Context-aware processing**: Handler inspects `context` to determine which account the event belongs to

This pattern maps directly to:
```
Cashflow Platform:
├── Organization (parent company)
│   ├── Tenant A (subsidiary)
│   ├── Tenant B (subsidiary)
│   └── Tenant C (subsidiary)
├── Webhook endpoint receives all tenant events
└── Middleware resolves tenant from JWT/header
```

## Finding 10: Journal Entry and Double-Entry Accounting API Patterns

**Source**: Xero API structure, QuickBooks API, accounting standards

Standard journal entry endpoint structure:
```
POST /api/v1/journal-entries
{
  "date": "2026-07-09",
  "reference": "INV-2026-001",
  "narration": "Revenue from consulting services",
  "lines": [
    {
      "account_id": "acc_receivable",
      "debit": 50000,
      "credit": 0,
      "tenant_id": "tenant_abc"
    },
    {
      "account_id": "acc_revenue",
      "debit": 0,
      "credit": 50000,
      "tenant_id": "tenant_abc"
    }
  ],
  "metadata": {
    "source_module": "invoicing",
    "source_id": "inv_abc123"
  }
}
```

**Validation rules** (enforced at API layer):
- Total debits must equal total credits
- All line items must reference valid account_ids
- Period must be open (not closed for reporting)
- Tenant_id must match authenticated tenant

**Status workflow**:
```
draft → posted → (optionally) reversed
```

## Finding 11: API Error Response Standards

**Source**: Stripe, Plaid error patterns

Standard financial API error response:
```json
{
  "error": {
    "type": "invalid_request_error",
    "code": "amount_mismatch",
    "message": "Journal entry debits (45000) do not equal credits (50000)",
    "param": "lines[1].credit",
    "doc_url": "https://docs.cashflow.app/errors#amount_mismatch",
    "request_id": "req_abc123"
  }
}
```

Error types for financial APIs:
- `authentication_error` — invalid/missing credentials
- `authorization_error` — insufficient permissions for this tenant
- `invalid_request_error` — malformed request, validation failure
- `rate_limit_error` — too many requests
- `api_error` — internal server error (should not leak details)
- `balance_insufficient` — financial-specific: not enough funds
- `period_closed` — cannot modify entries in a closed accounting period

## Summary Table: Major API Decisions

| Decision | Recommended Approach | Rationale |
|---|---|---|
| REST vs GraphQL | REST for CRUD + webhooks; GraphQL for dashboards | Financial mutations need explicit semantics; dashboards benefit from flexible queries |
| Versioning | URL versioning (/v1/) + header versioning for minor changes | Stripe hybrid model — simple for major breaks, flexible for patches |
| Pagination | Cursor-based (`starting_after`/`ending_before`) | Stable for large, mutating transaction sets |
| Tenant ID | JWT claim + X-Tenant-ID header fallback | Token carries context; header for server-to-server |
| Data isolation | Application-level filtering + RLS | Cost-effective for SaaS; RLS as safety net |
| Webhooks | HMAC-SHA256 signature + exponential backoff (3 days) | Industry standard (Stripe); replay-safe with event ID dedup |
| API-first | OpenAPI spec per module → generate stubs + SDKs | Contract-first development |
| Auth | OAuth 2.0 Authorization Code for banking; API keys for platform | Banking integrations need user consent; platform uses key-based auth |
