## 1. SPED File Format Specifications

### 1.1 ECD — Escrituração Contábil Digital

**File structure:** Fixed-length records, pipe-delimited (`|`), one record per line.

**Block layout:**

| Block | Record | Purpose |
|-------|--------|---------|
| A | 0000 | Abertura (opening): version, CNPJ, company name, opening/closing date |
| A | 0001 | Abertura do Bloco A |
| A | 0007 | Dados Complementares da Entidade |
| A | 0150 | Tabela de Cadastro de Participante |
| A | 0175 | Alteração da Tabela de Cadastro de Participante |
| A | 0500 | Plano de Contas Contábil |
| A | 0990 | Encerramento do Bloco A |
| C | 1001 | Abertura do Bloco C |
| C | 1010 | Dados Complementares — Termo de Abertura |
| C | 1800 | Demonstração do Resultado do Exercício (DRE) |
| C | 1990 | Encerramento do Bloco C |
| E | 3001 | Abertura do Bloco E (Razão/Contas Contábeis) |
| E | 3500 | Balanço Patrimonial |
| E | 3990 | Encerramento do Bloco E |
| G | 4001 | Abertura do Bloco G (Escrituração Consolidada) |
| G | 4990 | Encerramento do Bloco G |
| H | 6001 | Abertura do Bloco H (Livro Caixa) |
| I | 8001 | Abertura do Bloco I (Parte A do Livro de Ativo Imobilizado) |
| J | 9001 | Abertura do Bloco J (Apuração do IRPJ/CSLL) |
| J | 9900 | Registros do Arquivo |
| J | 9990 | Encerramento do Bloco J |
| 9 | 9999 | Totalização do Arquivo |

**Key fields (Record 0000):**
```
|0000|LECD|DT_INI|DT_FIN|NOME_EMPR|CNPJ|UF|MUN|SUFRAMA|IND_NAT_PJ|IND_ATIV|
```
- `DT_INI` / `DT_FIN`: Date range (YYYYMMDD)
- `IND_NAT_PJ`: 01=ME, 02=EPP, 03=PJ em geral
- `IND_ATIV`: 0=Industrial, 1=Comércio, 2=Prestação de serviços

### 1.2 ECF — Escrituração Contábil Fiscal

**File structure:** Pipe-delimited records similar to ECD.

**Key blocks and record types:**

| Block | Record | Purpose |
|-------|--------|---------|

| C | 0400 | Demonstração do Lucro/Prejuízo |
| C | 0500 | Dados Consolidados |
| J | 9900 | Registros do Arquivo |
| 9 | 9999 | Totalização do Arquivo |

## 2. NFe XML Schema (NFe 4.00)

```xml

      <emit>          <!-- Emitente -->

      <retirada>      <!-- Retirada (opcional) -->

      <autXML>        <!-- Autorização XML (opcional) -->
      <det nItem="1"> <!-- Detalhamento por item -->
        <prod>        <!-- Produto/Serviço -->
        <Imposto>     <!-- Tributos -->
          <ICMS>
          <IPI>
          <II>
          <PIS>

          <ICMFCP>
        </Imposto>
        <extIPI>      <!-- Extensão IPI (opcional) -->
        <cProdANVISA> <!-- Registro ANVISA (opcional) -->
        <med>         <!-- Medicamentos (opcional) -->
        <arma>        <!-- Armas (opcional) -->
        <comb>        <!-- Combustíveis (opcional) -->
        <navRec>      <!-- Número da Autorização (opcional) -->
      </det>

      <transp>        <!-- Transporte -->
      <cobr>          <!-- Cobrança -->
      <pag>           <!-- Pagamento -->
      <infAdic>       <!-- Informações Adicionais -->

      <cana>          <!-- Cana (opcional) -->
      <infRespTec>    <!-- Responsável Técnico -->
    </infNFe>
    <infNFeSupl>      <!-- Informações Suplementares -->
```

### 3.2 Municipal Variations

| City | Standard | Key Differences |
|------|----------|-----------------|
| **São Paulo (SP)** | ABRASF 2.03 | ISS rate: 2-5% by CNAE; uses `ItemListaServico` codes; SEFAZ-SP digital certificate required |
| **Rio de Janeiro (RJ)** | ABRASF 2.04 | ISS rate: 2-5% by CNAE; specific RPS (Recibo Provisório de Serviço) flow; different XML namespace |
| **Belo Horizonte (MG)** | ABRASF 2.03 | ISS rate: 2-5% by LC 116/2003; specific validation rules for construction services |
| **Curitiba (PR)** | ABRASF 2.02 | ISS rate: 2-5%; unique `CodigoTributacaoMunicipio` mapping |
| **Porto Alegre (RS)** | ABRASF 2.03 | ISS rate: 2-5%; specific `Cnae` code requirements |

**Key variation areas:**
- `ItemListaServico` code mapping (LC 116/2003 list varies by municipality)
- `CodigoTributacaoMunicipio` mapping
- ISS aliquot ranges (2-5% typically)
- RPS (Recibo Provisório de Serviço) vs immediate NFS-e
- Deadline for RPS→NFS-e conversion (typically 1-30 days)
- Digital certificate requirements
- Web service endpoint URLs

## 5. CNPJ Validation Algorithm

### 5.1 CNPJ Format

```
XX.XXX.XXX/YYYY-ZZ
│   │       │   │
│   │       │   └── Check digit (2 digits)
│   │       └────── Branch suffix (4 digits, 0001 = headquarters)
│   └────────────── Base/Root (8 digits, identifies the company)
└────────────────── 14 total digits (alphanumeric as of July 2026)
```

### 5.2 Validation Algorithm (Pseudocode)

```
function ValidarCNPJ(cnpj[14]) -> bool:
    var v[2]

    // First check digit
    v[1] = 5×cnpj[1] + 4×cnpj[2] + 3×cnpj[3] + 2×cnpj[4]
    v[1] += 9×cnpj[5] + 8×cnpj[6] + 7×cnpj[7] + 6×cnpj[8]
    v[1] += 5×cnpj[9] + 4×cnpj[10] + 3×cnpj[11] + 2×cnpj[12]
    v[1] = 11 - (v[1] mod 11)
    if v[1] >= 10 then v[1] = 0

    // Second check digit
    v[2] = 6×cnpj[1] + 5×cnpj[2] + 4×cnpj[3] + 3×cnpj[4]
    v[2] += 2×cnpj[5] + 9×cnpj[6] + 8×cnpj[7] + 7×cnpj[8]
    v[2] += 6×cnpj[9] + 5×cnpj[10] + 4×cnpj[11] + 3×cnpj[12]
    v[2] += 2×cnpj[13]
    v[2] = 11 - (v[2] mod 11)
    if v[2] >= 10 then v[2] = 0

    return (v[1] == cnpj[13]) AND (v[2] == cnpj[14])
```

### 5.3 Alphanumeric CNPJ (from June 2026)

- Letters A-Z introduced; value = ASCII(char) - 48
  - A = 65-48 = 17, B = 18, ..., Z = 42
- Same check digit algorithm applies
- Existing numeric-only CNPJs remain valid

- 3 = Inapta
- 4 = Baixada

## 6. DAS MEI Generation

### 6.1 MEI Tax Structure

**Fixed monthly DAS value (2024):**
- Comércio: R$ 75.00 (ICMS) + R$ 5.00 (ISS) = **R$ 80.00**
- Indústria: R$ 75.00 (ICMS) = **R$ 75.00**
- Serviços: R$ 70.00 (ISS) = **R$ 70.00**
- Serviços + Comércio: R$ 75.00 (ICMS) + R$ 70.00 (ISS) = **R$ 145.00**

**Note:** INSS patronal (8%) is included in the DAS value. The employee INSS is NOT included.

### 6.2 PGMEI Calculation

**PGMEI (Programa Gerador de Pagamento do MEI):**
- Generates DAS boleto with barcode
- Due date: 20th of each month
- Barcode structure: bank code (237-Bradesco), due date, amount, CNPJ, check digit

**DAS barcode components:**
```
Fator de vencimento + Valor nominal + CNPJ + Código de barras
```

### 6.3 Calculation Method

The DAS MEI is a **fixed value** determined annually by the federal government (Lei Complementar 123/2006). No complex calculation is needed — it is a simple lookup based on:
1. CNAE primary activity code
2. Activity type (comércio/serviços/indústria)

**Excess revenue DAS:** If annual revenue exceeds R$ 81,000 (MEI limit), the excess is paid via PGMEI at the same fixed rates applied to the proportional period.

## 7. Tax Substitution (Substituição Tributária) and DIFAL

### 7.1 Substituição Tributária (ST)

**Mechanism:** The first seller in the chain collects ICMS for all subsequent stages.

**Calculation methods:**
1. **MVA (Margem de Valor Agregado):**
   ```
   BC_ST = (BC_ICMS * (1 + MVA%)) + Frete + Seguro + Outras despesas
   ICMS_ST = BC_ST * aliquota_interna
   ICMS_proprio = BC_ICMS * aliquota_interna
   ICMS_ST_a_recolher = ICMS_ST - ICMS_proprio
   ```

2. **IVA-ST (Índice de Valor Agregado para ST):**
   Similar to MVA but with state-specific adjustment factors.

3. **Pauta Fiscal (Preço Final ao Consumidor):**
   ```
   BC_ST = Preço final ao consumidor
   ICMS_ST = BC_ST * aliquota_interna
   ```

**Key fields (SPED Fiscal, Record E200/E210):**
- `CST_ICMS`: Código da Situação Tributária
- `BC_ICMS_ST`: Base de Cálculo do ICMS-ST
- `ALIQ_ICMS_ST`: Alíquota do ICMS-ST
- `VL_ICMS_ST`: Valor do ICMS-ST
- `VL_BC_RET`: Base de Cálculo Retida

### 7.2 DIFAL — Diferencial de Alíquotas

**Purpose:** Applied to interstate purchases by non-contributing establishments (e-commerce, marketplace).

**Calculation:**
```
DIFAL = (BC * aliquota_interna) - (BC * aliquota_interestadual)

Where:
  BC = Valor da operação (ou valor aduaneiro para importação)
  aliquota_interna = Alíquota ICMS do estado destino
  aliquota_interestadual = 4% (para estados Sul/Sudeste)
                           ou 7% (demais estados para Sul/Sudeste)
                           ou 12% (demais operações)
```

**Split (from 2019):**
- 100% to the state of destination (for ICMS-ST cases)
- Or split between origin and destination states

**Record types in EFD-ICMS IPI:**
- `E300`: DIFAL registration
- `E310`: DIFAL apuração
- `E311`: Ajuste da apuração
- `E312`: Processo referenciado

### 8.2 Fator R

**Purpose:** Determines if a service company (Anexo V) should be taxed under Anexo III.

**Formula:**
```
Fator R = (Folha de Pagamento últimos 12 meses) / (Receita Bruta últimos 12 meses)
```

**Rules:**
- If Fator R >= 28% → Company uses **Anexo III** (lower rates)
- If Fator R < 28% → Company uses **Anexo V** (higher rates)
- Folha de Pagamento includes: salaries, pro-labore, 13th salary, FGTS, INSS patronal

### 8.4 DAS Calculation Formula

```

DAS = RBT12 × Alíquota Efetiva / 12 (monthly payment)
```

## Implementation Notes for Cashflow Platform

### Module Architecture Requirements

1. **SPED Module:** Must support pipe-delimited file generation with record-type validation. Key validations: record ordering, block closure, field lengths, date formats (YYYYMMDD).

2. **NFe Module:** Must implement XML signing (XMLDSig), SOAP 1.2 client with X.509 certificate mutual TLS, and all 8 SEFAZ web services. Requires state-specific endpoint configuration.

3. **NFS-e Module:** Must handle ABRASF 2.02-2.04 variations per municipality. Use a municipality registry with endpoint URLs and XML namespace differences.

4. **eSocial Module:** Must implement event transmission with digital certificate. Events have strict deadlines and ordering dependencies (S-1299 must be last in period).

5. **CNPJ Validation:** Must implement both numeric (pre-July 2026) and alphanumeric (post-July 2026) validation. Use BrasilAPI or Receita Federal WS for live lookups.

6. **MEI Module:** Simple DAS generation with CNAE-based fixed value lookup. No complex calculation needed.

7. **ST/DIFAL Module:** Must maintain per-state aliquot tables and MVA adjustment factors. Calculation varies significantly by product (NCM) and state pair.

8. **Simples Nacional Module:** Must implement all 5 annexes with fator R calculation and state-specific sublimite handling.
