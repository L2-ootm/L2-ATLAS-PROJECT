## Marketplace Models Compared

### 1. Shopify App Store
- **Model**: Open marketplace, free to list, 0% revenue share on first $1M/year (since 2021), then standard 20% commission (Shopify takes 20%, developers keep 80%).
- **Scale**: 10,000+ apps (as of 2024), $230M+ earned by app partners in 2020 alone.
- **Listing**: Free, no upfront fees. Developers publish directly.
- **Review process**: Curated — apps reviewed for compliance with quality checklist, support responsiveness, prohibited actions (fake reviews, misleading listings).
- **Discovery**: Search, categories, featured banners, app listings with screenshots/video, merchant reviews/ratings.
- **Key insight**: Shopify's 2021 decision to go 0% on first $1M was a major developer incentive that accelerated ecosystem growth. Typical Shopify merchant uses 6 apps.

### 2. Salesforce AppExchange
- **Model**: Curated enterprise marketplace. Two tiers: free listing + 25% revenue share for paid apps, or 15% for ISV partners who meet volume thresholds.
- **Listing fee**: $2,750 for security review + listing fee (varies by certification type).
- **Review process**: Mandatory security review (PMD static analysis, manual review, penetration testing). ISV partner certification required.
- **Discovery**: AppExchange search, categories, featured apps, customer reviews, implementation partner directory.
- **Key insight**: Enterprise buyers trust AppExchange because of the rigorous security review. Financial services customers especially require SOC2, PCI compliance proof.

### 3. Xero App Store
- **Model**: Free listing, partner program with tiered benefits (Silver/Gold/Platinum based on installs).
- **Revenue share**: No commission — developers sell directly to customers, Xero takes 0%.
- **Discovery**: Categorized by function (accounting, payroll, CRM), integration partner directory, API partner badges.
- **Key insight**: Xero's 0% revenue share model attracted thousands of integrations. Monetization comes from the core platform subscription, not marketplace fees.

### 4. Odoo Apps Marketplace
- **Model**: Open-source community marketplace. Free apps in Community Edition; paid apps in Enterprise Edition marketplace.
- **Revenue share**: 70/30 or 85/15 depending on developer tier. Odoo takes commission on paid marketplace listings.
- **Architecture**: Modular — each Odoo app is an independent module with defined API surface.
- **Discovery**: Search, categories, installation count, ratings, version compatibility badges.
- **Key insight**: Odoo's modular architecture (each feature = separate installable module) is the closest parallel to a plugin-based cashflow platform. Community modules extend functionality without touching core.

**Recommendation for L2 cashflow**: Start with 0% commission (like Xero) to attract initial developers. Once ecosystem reaches ~50 active plugins, introduce 15% commission on paid plugins only. Free plugins remain 0% forever.

## Plugin Discovery Mechanisms

1. **Search**: Full-text search across name, description, tags, author. Critical — 60%+ of plugin installs come from search.
2. **Categories**: Functional categories (e.g., "Cashflow Forecasting", "Tax Integration", "Bank Connection", "Reporting", "AI Analysis").
3. **Ratings & Reviews**: 1-5 star system with written reviews. Only verified users can review.
4. **Featured/Staff Picks**: Curated editorial selection. Drives 10-30x installs for featured plugins.
5. **Installation Count**: Social proof — show number of active installs.
6. **Compatibility Badges**: "Works with v2.x", "Maintained", "Last updated X days ago".
7. **Author Profiles**: Developer/company pages showing all their plugins and credibility.

## Plugin Quality Assurance

### Review Process (Best Practices)
1. **Automated scan**: Static analysis, dependency audit, license compliance check.
2. **Security review**: OWASP Top 10 scan, authentication/authorization review, data handling audit.
3. **Functional testing**: Automated test suite required, minimum 80% code coverage.
4. **Manual review**: Human review of code quality, UX consistency, documentation.
5. **Sandbox testing**: Plugin tested in isolated sandbox environment before approval.

### Certification Programs
- **Shopify**: "Built for Shopify" badge — requires meeting performance, UX, and support standards.
- **Salesforce**: ISVForce certification — security review + compatibility testing.
- **Xero**: "Xero App Partner" badge — integration testing + support SLA.

**Recommendation for L2 cashflow**: Implement a 3-tier certification:
- **Community**: Self-service, automated scan only (fast listing).
- **Verified**: Security review + integration tests (for paid plugins).
- **Certified**: Full audit + performance benchmarks + support SLA (for "L2 Certified" badge).

## Plugin SDK Design Patterns

### API Surface Design
1. **Core API**: Stable, versioned interface for essential operations (read/write data, trigger events, render UI).
2. **Extension Points**: Named hooks where plugins can inject behavior (e.g., `on_transaction_created`, `before_reconcile`, `render_dashboard_widget`).
3. **Data Access Layer**: Sandboxed access to specific data models. Plugins cannot access arbitrary data.
4. **Event System**: Publish-subscribe for cross-plugin communication.

### Versioning
- **Semantic versioning** (MAJOR.MINOR.PATCH) for plugin releases.
- **Core API versioning**: `v1`, `v2` — with deprecation windows.
- **Compatibility matrix**: Explicit declaration of supported core versions.
- **Migration guides**: Provided for breaking changes in core API.

### SDK Package
```
cashflow-plugin-sdk/
  ├── manifest.json          # Plugin metadata, version, dependencies
  ├── src/
  │   ├── index.ts           # Main entry point
  │   ├── hooks/             # Lifecycle hooks
  │   ├── api/               # Core API client
  │   ├── events/            # Event subscriptions
  │   └── ui/                # UI component library
  ├── tests/                 # Required test suite
  ├── docs/                  # Plugin documentation
  └── README.md
```

## Plugin Governance

### Deprecation Policies
- **6-month deprecation window**: Announce deprecated API, provide migration guide, warn in logs.
- **3-month grace period**: After deprecation, plugins using old API get warnings but continue working.
- **Removal**: After grace period, deprecated API removed. Plugins must update or break.

### Breaking Change Management
- **Core API changes**: Announce 6 months in advance via changelog + developer newsletter.
- **Security patches**: Immediate, no deprecation window (security takes priority).
- **Feature removal**: 12-month notice with migration tooling provided.

### Compatibility Matrices
- **Version matrix table**: Core version x Plugin version compatibility.
- **CI/CD pipeline**: Each plugin tested against supported core versions automatically.
- **Long-term support (LTS)**: Core versions supported for 2 years after release.

## Ecosystem Growth Strategy

### Phase 1: Seeding (Months 0-6)
Build 15-20 essential plugins yourself:
- Bank connections (Brazil, US, EU)
- Basic cashflow forecasting
- Tax integration (Simples Nacional, ICMS, ISS)
- Payment processors (Stripe, Mercado Pago, PagSeguro)
- Accounting export (OFX, CSV, integrations with QuickBooks/Xero)
- Dashboard/reporting widgets
- Notification plugins (email, Telegram, WhatsApp)
- Currency conversion
- Invoice generation

### Phase 2: Developer Outreach (Months 6-12)
- **Developer documentation**: Comprehensive SDK docs, tutorials, example plugins.
- **Developer portal**: Dashboard for plugin submission, analytics, revenue tracking.
- **Slack/Discord community**: Direct line to core team for plugin developers.
- **Hackathons**: Quarterly virtual hackathons with prizes for best plugins.
- **Co-marketing**: Feature top plugins in newsletter, social media, conference talks.

### Phase 3: Ecosystem Maturity (Months 12-24)
- **Revenue share program**: Optional monetization for developers.
- **Partner program**: Tiered benefits (early access to beta APIs, co-development opportunities).
- **Certification program**: "L2 Certified" badge for quality plugins.
- **Plugin analytics**: Public install counts, ratings, trends.

## Financial Plugin Monetization Models

| Model | How it Works | Best For |
|-------|-------------|----------|
| **Free** | No cost, community goodwill | Utility plugins, core integrations |
| **Paid** | One-time or monthly subscription | Specialized analytics, premium bank connections |
| **Freemium** | Free tier + paid premium features | Most common — free basic, paid advanced |
| **Usage-based** | Pay per API call / transaction | High-volume integrations |
| **SaaS add-on** | Plugin is separate SaaS, platform takes referral fee | Complex plugins needing own infrastructure |

**Recommendation**: Default to **freemium** for financial plugins. Core bank connections should be free. Advanced forecasting, AI analysis, compliance tools can be paid.

## Key Patterns from Successful Marketplaces

1. **Start with free**: Shopify and Xero both offer free listings. Removes barrier to entry.
2. **Build the first 20 yourself**: Don't wait for third-party developers. Seed the ecosystem with essential plugins.
3. **Security is non-negotiable for financial platforms**: Mandatory security review before listing. SOC2/PCI compliance for paid plugins.
4. **Developer experience is the product**: SDK, docs, sandbox, and support determine ecosystem health.
5. **Featured listings drive discovery**: Editorial curation is more important than algorithmic ranking for B2B plugins.
6. **Ratings build trust**: Only verified users can review. Prevent fake reviews aggressively.
7. **Version compatibility matters**: Financial software changes slowly. Clear compatibility matrices prevent support burden.
8. **Revenue share should be generous initially**: 0% commission until ecosystem is established. Then introduce modest fees.
