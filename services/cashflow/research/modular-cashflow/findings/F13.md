# F13: Security Architecture for Modular Financial Platforms

**Date**: 2026-07-09
**Category**: Security Architecture
**Confidence**: High
**Sources**: AWS KMS Documentation, Azure Encryption Overview, OWASP Cryptographic Storage Cheat Sheet, OWASP Logging Cheat Sheet, PCI DSS v4.0 Requirements Guide, GDPR Art. 17, OWASP ASVS v5.0

## Findings

### F13.1 — AES-256-GCM as Default Encryption Standard
**Source**: OWASP Cryptographic Storage Cheat Sheet, Azure Encryption Overview
**Relevance**: Core encryption at rest

Use AES-256 with GCM (Galois/Counter Mode) as the primary cipher for all PII and financial data at rest. GCM provides authenticated encryption, guaranteeing both confidentiality and integrity in a single operation. Key requirements:
- Minimum 256-bit key length for symmetric encryption
- GCM or CCM authenticated modes always preferred over CBC/CTR
- If GCM unavailable, use CTR or CBC with separate Encrypt-then-MAC authentication
- Never use ECB mode for any production data
- Generate IVs/nonces using CSPRNG (`secrets` in Python, `crypto.randomBytes()` in Node.js, `rand::thread_rng()` in Rust)

### F13.2 — Envelope Encryption Pattern (DEK/KEK)
**Source**: OWASP Cryptographic Storage, AWS KMS Documentation, Azure Key Vault
**Relevance**: Key management architecture

Implement envelope encryption with a two-tier key hierarchy:
- **Data Encryption Key (DEK)**: Encrypts actual data, generated per-record or per-column
- **Key Encryption Key (KEK)**: Encrypts the DEK, stored separately in a vault (AWS KMS, HashiCorp Vault, Azure Key Vault)
- DEK is generated as a random 256-bit key, encrypted under the KEK, and stored alongside the ciphertext
- KEK never leaves the HSM/vault boundary
- This means a database breach alone does not expose data — attacker also needs vault access
- For column-level financial sensitivity (e.g., PAN, SSN, bank account), use separate DEKs per sensitivity class

### F13.3 — Key Management with AWS KMS / HashiCorp Vault
**Source**: AWS KMS Documentation, OWASP Cryptographic Storage
**Relevance**: Key lifecycle and vault selection

AWS KMS provides three key tiers:
- **Customer Managed Keys (CMK)**: Full control over rotation, policies, deletion. Monthly fee + per-use fee. Required for SOC 2 compliance evidence.
- **AWS Managed Keys**: Auto-rotated annually, no lifecycle control, free. Acceptable for non-sensitive workloads.
- **AWS Owned Keys**: Managed by the service, no customer visibility. Use only for default encryption.

AWS KMS key hierarchy uses 256-bit AES-GCM domain keys (rotated daily) wrapping HSM backing keys (rotated yearly), which in turn encrypt Customer Data Keys (CDKs).

For non-AWS environments, HashiCorp Vault provides equivalent functionality with transit secrets engine for DEK wrapping and auto-rotation policies. Recommended key rotation intervals:
- KEK: every 90 days or on suspected compromise
- DEK: every 30 days or per-record for high-sensitivity data
- Document all key management processes per PCI DSS Req. 3.6

### F13.4 — TLS 1.3 and mTLS for Transit Encryption
**Source**: Azure Encryption Overview, PCI DSS Req. 4
**Relevance**: Encryption in transit

- Enforce TLS 1.3 minimum for all client-facing connections; TLS 1.2 acceptable only as fallback with strong cipher suites
- Perfect Forward Secrecy (PFS) required — use ECDHE key exchange
- Certificate configuration: RSA 2048-bit minimum or ECC P-256/P-384
- mTLS (mutual TLS) required for all service-to-service communication in microservices architecture
- Azure applies MACsec (IEEE 802.1AE) data-link layer encryption by default for inter-datacenter traffic
- PCI DSS Req. 4.1: Use strong encryption for all cardholder data transmission over public networks
- Disable all SSL versions, TLS 1.0, TLS 1.1, and weak cipher suites (3DES, RC4, export ciphers)
- HSTS header with `max-age=31536000; includeSubDomains` on all domains

### F13.5 — OAuth 2.0 + PKCE for Web App Authentication
**Source**: OWASP ASVS, OWASP OAuth2 Cheat Sheet
**Relevance**: Authentication architecture

Implement OAuth 2.0 Authorization Code flow with PKCE (Proof Key for Code Exchange) for all web and mobile applications:
- PKCE prevents authorization code interception attacks (RFC 7636)
- Use `code_verifier` (43-128 character random string) and `code_challenge` (S256 hash)
- Access tokens: JWT with short expiry (5-15 minutes), signed with RS256 or ES256
- Refresh tokens: opaque, long-lived (30-90 days), rotated on use, bound to client
- Never store tokens in localStorage; use httpOnly, secure, SameSite=strict cookies
- For financial SaaS multi-tenancy: embed tenant_id in JWT claims and enforce at middleware layer

### F13.6 — API Key Management: Rotate, Revoke, Scope
**Source**: PCI DSS Req. 8, OWASP ASVS
**Relevance**: API security

API keys for machine-to-machine (M2M) communication:
- Scoped to minimum required permissions (read-only for analytics, read-write for transaction processing)
- Rotated automatically every 90 days maximum
- Revocation endpoint with immediate effect (< 60 second propagation)
- Rate limiting per key: enforce at gateway level
- Never embed API keys in source code or client-side code
- Log all API key usage with caller IP, timestamp, endpoint, and response code
- Store hashed API keys only — treat like passwords

### F13.7 — RBAC with Row-Level Security for Multi-Tenant Financial Systems
**Source**: PCI DSS Req. 7, OWASP Access Control Cheat Sheet
**Relevance**: Authorization model

- **RBAC (Role-Based Access Control)** for platform-level roles: `admin`, `accountant`, `viewer`, `auditor`
- **ABAC (Attribute-Based Access Control)** for fine-grained conditions: time-of-day, IP range, transaction amount thresholds
- **Row-Level Security (RLS)** enforced at database layer for multi-tenancy — each query filtered by `tenant_id` automatically
- Enforce least-privilege: no role should have access to data outside its business function
- PCI DSS Req. 7.1: Restrict access to cardholder data based on business need-to-know
- Separation of duties: the user who initiates a transaction should not be the same user who approves it (dual-control)
- Implement `authorization` middleware that verifies both role permissions AND tenant ownership on every request

### F13.8 — SOC 2 Type II Compliance Requirements
**Source**: SOC 2 Trust Service Criteria (AICPA)
**Relevance**: Compliance architecture

SOC 2 Type II covers five Trust Service Criteria (TSC):
- **Security (Common Criteria)**: Logical and physical access controls, system operations, change management, risk mitigation. This is the only mandatory criterion.
- **Availability**: Uptime SLAs, DR/BCP testing, capacity monitoring. For SaaS: typically 99.9% uptime commitment.
- **Processing Integrity**: Data processing accuracy, completeness, timeliness. Financial data must balance, reconcile, and have error correction mechanisms.
- **Confidentiality**: Encryption of confidential data at rest and in transit, access controls.
- **Privacy**: Notice, choice/consent, collection, use/retention/disposal, access, disclosure, quality.

Audit evidence collection requirements:
- Continuous monitoring with automated evidence collection (not point-in-time snapshots)
- 6-12 month audit observation period
- Evidence of: access reviews (quarterly), key rotation logs, change management tickets, incident response drill results, vulnerability scan reports, penetration test findings
- Map controls to TSC categories; maintain control matrix document

### F13.9 — PCI DSS Requirements for Card Data Handling
**Source**: PCI DSS v4.0 Requirements Guide
**Relevance**: Card data compliance

PCI DSS has 12 requirements across 6 goals. Key items for a financial platform:
- **Req. 3**: Never store sensitive authentication data post-authorization (CVV, PIN, magnetic stripe data). Mask PAN (show only first 6 / last 4 digits). Render PAN unreadable at rest via encryption, truncation, or hashing.
- **Req. 4**: Encrypt PAN transmission over public networks using TLS 1.2+ with strong crypto.
- **Req. 8**: Unique user IDs, MFA for all CDE access, password complexity (min 12 chars), account lockout after 10 failed attempts.
- **Req. 10**: Track and monitor all access to network resources and cardholder data. Log retention: minimum 1 year, 3 months immediately available.
- **Scope Reduction Strategy**: Use tokenization (via payment processor like Stripe, Adyen) so PAN never touches your servers. This can reduce SAQ from SAQ D (300+ controls) to SAQ A or SAQ A-EP (~20-30 controls).

SAQ types:
- **SAQ A**: Card-not-present only, fully outsourced to PCI-compliant third party
- **SAQ A-EP**: E-commerce with redirect but your domain handles the page
- **SAQ B**: Imprint machine or standalone dial-out terminal
- **SAQ D**: Any environment that stores, processes, or transmits cardholder data (most comprehensive)

### F13.10 — Fraud Detection Patterns
**Source**: OWASP Business Logic Security, PCI DSS Req. 10
**Relevance**: Transaction monitoring

Implement layered fraud detection:
- **Velocity Checks**: Rate-limit transactions per user/account (e.g., max 5 transfers/hour, max 3 international wires/day). Use sliding window counters in Redis.
- **Anomaly Detection**: Flag transactions that deviate from historical patterns — unusual amounts (> 2x average), unusual times (3 AM vs. 9 AM business hours), unusual destinations.
- **Duplicate Detection**: Idempotency keys on all financial operations. Hash (user_id + amount + recipient + timestamp_bucket) to detect near-duplicates within a configurable window.
- **Separation of Duties**: Different users for transaction creation, approval, and settlement. Enforce via RBAC + workflow state machine.
- **Amount Thresholds**: Auto-flag transactions exceeding configurable limits for manual review. Implement tiered approval: <$100 auto-approve, $100-$10K manager approval, >$10K executive approval.
- **Geographic Checks**: Flag cross-border transactions, transactions from high-risk jurisdictions (per FATF grey/black lists).

### F13.11 — Immutable Audit Trails and Tamper-Evident Logs
**Source**: OWASP Logging Cheat Sheet, PCI DSS Req. 10
**Relevance**: Audit logging architecture

Build a three-tier audit logging system:
- **Application Logs**: Structured JSON with timestamp (UTC), user_id, tenant_id, action, resource, outcome, IP, request_id. Never log: passwords, session tokens, full PAN, encryption keys.
- **Audit Trail (Financial)**: Append-only ledger of all financial mutations. Each entry includes: previous_entry_hash, current_entry_hash (hash chain). This creates a tamper-evident chain similar to blockchain.
- **Compliance Logs**: Access logs, permission changes, configuration changes. Retained per PCI DSS Req. 10.7: minimum 1 year, 3 months immediately available for analysis.

Log protection requirements:
- Store logs in separate partition/database from application data
- Write-only access for application services; read-only for auditors
- Forward to centralized SIEM (e.g., Datadog, Splunk, CloudWatch) in real-time
- Synchronize all system clocks via NTP (PCI DSS Req. 10.4)
- Implement log integrity verification: hash-based or cryptographic signatures
- Never store logs in web-accessible locations

### F13.12 — Secrets Management: Zero-Trust Architecture
**Source**: OWASP Secrets Management, OWASP Cryptographic Storage
**Relevance**: Secrets lifecycle

Zero-trust secrets management:
- **Never use plain environment variables for production secrets** — they leak through `/proc/self/environ`, `phpinfo()`, crash dumps, and logging
- Use dedicated vault: AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, or Supabase Vault
- Secrets injection at runtime via vault API or sidecar proxy (Vault Agent)
- Automatic rotation: database credentials rotated every 24 hours, API keys every 90 days, TLS certificates via ACME/Let's Encrypt auto-renewal
- Audit every secret access (who, when, which secret, from which IP)
- Emergency revocation: one-click rotation for all secrets in case of breach
- Developer secrets: use `.env.example` with placeholders, never commit actual values. Pre-commit hooks to scan for leaked secrets (gitleaks, truffleHog).

### F13.13 — Input Validation and Injection Prevention
**Source**: OWASP ASVS, PCI DSS Req. 6
**Relevance**: Application security

Prevent all OWASP Top 10 injection vectors:
- **SQL Injection**: Use parameterized queries/prepared statements exclusively. Never concatenate user input into SQL. ORM parameter binding.
- **XSS**: Output encoding based on context (HTML, URL, JavaScript, CSS). Use CSP headers with `default-src 'self'`. Sanitize rich text input with DOMPurify.
- **CSRF**: Anti-CSRF tokens on all state-changing operations. SameSite=strict cookies. Verify `Origin` and `Referer` headers.
- **Input Validation**: Whitelist validation on all inputs. Validate type, length, range, and format. Financial amounts: validate to 2 decimal places, positive values, within reasonable bounds.
- **PCI DSS Req. 6.5**: Address common vulnerabilities in development process: injection flaws, broken auth, XSS, insecure direct object references, security misconfiguration, sensitive data exposure, broken session management.

### F13.14 — Data Retention, Right to Deletion (GDPR/LGPD)
**Source**: GDPR Art. 17, PCI DSS Req. 3.1
**Relevance**: Privacy compliance

GDPR Art. 17 (Right to Erasure) and LGPD (Brazilian equivalent) require:
- Erase personal data without undue delay when: no longer necessary, consent withdrawn, unlawful processing, legal obligation
- Financial exception: Transaction records may need retention for regulatory compliance (anti-money laundering laws typically require 5 years minimum)
- Implementation: Soft-delete with automated hard-delete after retention period. Pseudonymize data after retention period if audit trail needed.
- Data retention policy by category:
  - Transaction data: 5-7 years (varies by jurisdiction)
  - PII (names, addresses): Duration of business relationship + 1 year
  - Authentication logs: 1 year minimum (PCI DSS)
  - Marketing consent: Until withdrawn + 30 days
- Build "data subject access request" (DSAR) automation: export all user data in machine-readable format (JSON/CSV) within 30 days (GDPR) or 15 days (LGPD)
- Right to deletion must cascade: delete from primary DB, replicas, backups (or mark for deletion at next backup rotation), search indexes, analytics pipelines
- Document all data processing activities per GDPR Art. 30 (Records of Processing Activities)

### F13.15 — Row-Level Security and Multi-Tenant Data Isolation
**Source**: OWASP Multi-Tenant Security Cheat Sheet, PCI DSS Req. 7
**Relevance**: Multi-tenant isolation

For a modular financial platform serving multiple businesses:
- Database-level RLS policies: `WHERE tenant_id = current_setting('app.current_tenant_id')::uuid`
- Every API request sets the tenant context in a session variable before query execution
- Separate encryption keys per tenant for maximum isolation (optional, adds cost)
- No cross-tenant data leakage: test with automated authorization regression tests
- Audit trail must log tenant_id on every operation for forensics
- Consider database-per-tenant for highest-security customers (regulatory requirement in some jurisdictions)

## Summary

This research covers the complete security architecture stack for a modular financial platform. The 15 findings span encryption at rest (AES-256-GCM, envelope encryption), encryption in transit (TLS 1.3, mTLS), authentication (OAuth 2.0 + PKCE), authorization (RBAC + RLS), compliance frameworks (SOC 2, PCI DSS, GDPR/LGPD), fraud detection patterns, secrets management, input validation, audit logging, and data retention. The architecture follows a defense-in-depth model with envelope encryption as the foundational pattern, vault-backed key management, and immutable audit trails for regulatory compliance.
