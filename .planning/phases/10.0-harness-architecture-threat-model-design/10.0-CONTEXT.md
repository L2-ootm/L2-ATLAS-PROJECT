## Phase Boundary

Phase 10.0 is a design/enabling phase. It commits ‚Äî as reviewable documents and a drafted SQL schema ‚Äî the four foundational decisions that the rest of v1.1 (10.1‚Äì10.6) builds against, **before any harness code is written**:

1. Auth-store layout (`~/.atlas/auth.json`).
2. The ATLAS-adapter / vendored-foundation boundary.
3. The `0004_registry_v2.sql` provider/model/route schema.
4. OAuth-callback + native-IPC threat models and the fallback-cascade error-classification spec.

**In scope:** decisions, drafted schema DDL, drafted threat models, the error-classification table, and the Pydantic-schema direction. **Out of scope:** implementing the auth store, the adapter, the model-discovery pipeline, the TUI, or the native shell ‚Äî those are owned by 10.1‚Äì10.6. This phase owns no v1 REQ-IDs (precedent: v1.0 Phase 7); it de-risks the REQ-IDs those phases own. Where this phase's output is a `.sql` file or schema, the file is *drafted/committed* here; its *application* and reader-cutover are owned downstream.
</domain>

<decisions>
## Implementation Decisions

### Area 1 ‚Äî Auth-Store Layout (enables AUTH-01/02, SEC-03; success criterion 1)

- **Location:** flat `~/.atlas/auth.json`, alongside the existing `~/.atlas/atlas.db`. `~/.atlas/` is already the ATLAS home (gateway `default_db_path()` resolves there); auth.json is a peer, not a new home.
- **Single path resolver:** all auth-store path resolution goes through one function (e.g. `auth_store_path()` in `services/agent-runtime/atlas_runtime/`). Future per-profile layout (`~/.atlas/profiles/<name>/auth.json`) becomes a one-function change, not a scatter of `Path.home()` calls. The resolver honors an `ATLAS_HOME` (or equivalent) env override so tests never touch the real home dir.
- **File format:** JSON object with a top-level `"version": 1` field (forward-migration hook, mirrors Hermes `AUTH_STORE_VERSION`) and a `"providers"` map keyed by `provider_id`. Each provider entry holds auth material + metadata (e.g. `auth_type`, `base_url`, `created_at`, `last_refresh_at`).
- **Atomic write + cross-process lock:** ATLAS owns its own implementation ‚Äî temp-file + `os.replace`, guarded by a sidecar lock file with a ~15s timeout. It **mirrors the Hermes pattern** (`atomic_replace`, fcntl/msvcrt lock, `AUTH_LOCK_TIMEOUT_SECONDS`) but does **not import** `foundation/atlas-hermes` internals, so the vendored foundation stays re-syncable against upstream and the ATLAS store has no foundation dependency.
- **Permissions:** the store is created current-user-only ‚Äî `0600` on POSIX, `icacls` current-user grant on Windows (SEC-03).
- **Secrets at rest (v1.1):** file-store-first ‚Äî keys are stored in the user-only-perms file; OS keychain is deferred (AUTHX-01, v1.2+). A computed `redacted_hint` (e.g. last 4 chars) is kept for display; **full secret values are never returned by any status/list command, never written to SQLite, and never written to audit JSONL.**
- **Credential boundary (cross-area, links to Area 3):** credentials live **only** in the file store. The SQLite registry references `provider_id` and *derives* `auth_status`; it never holds secret material.

### Area 2 ‚Äî Adapter / Foundation Boundary (enables AGNT-01..06; success criterion 2)

- **Location:** the ATLAS runtime adapter lives in `services/agent-runtime/atlas_runtime/` (new module, e.g. `chat/adapter.py` or `agent_adapter.py`). It wraps the Hermes `AIAgent` defined in `foundation/atlas-hermes/run_agent.py` by import + instantiation.
- **One-way dependency:** `services/agent-runtime` depends on `foundation/` and never the reverse. The only path by which foundation references ATLAS is the already-established registered-plugin shim precedent (DIV-F-002: `foundation/atlas-hermes/plugins/atlas_audit/` delegating into `services/agent-runtime/atlas_audit`). This keeps the foundation upstream-mergeable.
- **`foundation/` changes are extension-points only:** any change to the vendored tree is recorded in `foundation/atlas-hermes/DIVERGENCE_LOG.md` and is a hook/extension, never a rewrite of the Hermes agent loop (AGNT-01; hard gate: "no parallel second chat runtime if the Hermes foundation can be wrapped").
- **Adapter contract (the boundary this phase commits):** the adapter (a) resolves `(provider, base_url, api_key, model, api_mode)` from the auth store + route policy and constructs `AIAgent` with them; (b) owns the fallback cascade (Area 4); (c) emits ATLAS audit events (`model_call_start` / `model_call_end` with provider, model, run_id, token usage; `provider_fallback`) on the ATLAS event bus (AGNT-04, AUD-01/02); (d) **preserves** Hermes tool-approval gates rather than bypassing them (AGNT-05); (e) runs the transcript through the redaction filter before any persistence (AGNT-06).

### Area 3 ‚Äî `0004_registry_v2.sql` Schema (enables PROV-01, MOD-01/03/04/05; success criterion 3)

- **New additive migration** `infra/migrations/0004_registry_v2.sql`, drafted/committed in this phase. Three tables (from `.planning/prep/v1.1-provider-model-registry-spec.md ¬ß8`):
  - `provider_registry(provider_id PK, display_name, auth_type, default_base_url, api_modes_json, source, status, last_checked, last_error)` ‚Äî models a provider distinctly from credential/runtime/model/route (PROV-01).
  - `model_registry_v2(model_id, provider_id, source, api_mode, status, auth_status, context_window, metadata_json DEFAULT '{}', first_seen, last_seen, deactivated_at, PRIMARY KEY(model_id, provider_id, source))` ‚Äî the **composite key** gives idempotent multi-source discovery (MOD-03); **source-scoped `deactivated_at`** means one provider going offline never deactivates another source's models (MOD-04); `status` and `auth_status` are **distinct columns** (MOD-05).
  - `route_policy(task_class PK, provider_id, model_id, fallback_policy_json DEFAULT '{}', updated_at)` ‚Äî task-class ‚Üí model routing (forward-looking for ROUTE-01/02, table landed now to avoid a later migration churn).
- **Backward compatibility:** the migration is purely **additive**. The existing `0003` `model_registry` table is **not dropped**. The Rust gateway currently reads `model_registry` directly for `/v1/models`; that reader cutover to `model_registry_v2` (and cockpit truth, MOD-06) is owned by **Phase 10.3**, after which the legacy table can be retired. This satisfies the seed's "migrate or view over it; do not destroy old `model_registry`" and keeps the gateway green mid-milestone.
- **`auth_status` derivation:** computed at discovery time from the file auth store (credential present? reachable?) ‚Äî `available` / `auth_present` / `needs_api_key` / `needs_login` / `offline` / `rate_limited` / etc. per the spec's status vocabulary. Never a stored secret.
- **Schema source of truth:** new Pydantic v2 models (`Provider`, `ModelV2`/registry row, `RoutePolicy`) are added under `packages/atlas-core/atlas_core/schemas/` per D-012; the DDL mirrors them, consistent with how `0001`/`0003` mirror `core.py` / `model_registry.py ensure_schema()`.
- **Migration mechanism:** unchanged ‚Äî sorted-glob `executescript` apply + `ensure_schema()` parity (the existing convention); no new runner is introduced.

### Area 4 ‚Äî Threat Models + Fallback-Cascade Spec (enables SEC-04/05, AGNT-02/03, AUD-01; success criterion 4)

- **OAuth-callback flow ‚Äî DEFERRED for v1.1 (documented decision, SEC-05).** ATLAS ships no OAuth callback flow in v1.1 (locked scope: Codex read-only, no OpenAI/Codex OAuth-protocol reuse, file-store-first). The OAuth-callback threat-model **draft** records the hard requirements any *future* callback flow must meet before it can ship, so a later spike inherits a gate spec rather than a blank page: PKCE (S256) mandatory; `state` generated + validated; an **ephemeral loopback listener** bound to `127.0.0.1` on an OS-assigned port that accepts **exactly one** callback then closes; short overall timeout; exact redirect-URI match; **no token or code in any log/URL persisted**.
- **Native-IPC threat model ‚Äî DRAFT enumerating the planned Tauri 2 surface (SEC-04).** The shell is built in 10.5, but 10.0 drafts the threat model so 10.5 builds against it. Principles: capability-scoped **allowlist, deny-by-default**; the **PTY channel transports keystrokes/bytes, never command strings** ‚Äî no arbitrary-exec bridge (NAT-03); cockpit‚Üîgateway stays loopback HTTP (gateway is already 127.0.0.1-only); `open_external` restricted to an allowlisted scheme set. Draft IPC command inventory, each row = `{command, params, caller, privilege, threat, mitigation}` ‚Äî e.g. `get_readiness()` (read-only auth/model status), `pty_open(profile_id)` (spawns `atlas tui`; no arg string injection), `pty_write(bytes)` (keystrokes only), `pty_resize(rows, cols)`, `open_external(url)` (scheme-allowlisted).
- **Fallback-cascade error-classification table (the canonical halt-vs-cascade contract; AGNT-02/03, AUD-01):**
  - `401` / `403` auth ‚Üí **HALT**, never cascade; surface remediation (`atlas auth add <provider>` / `atlas auth doctor`); emit audit. Auth failures must never silently fall through (AGNT-03).
  - Network / connection refused / DNS / timeout ‚Üí **CASCADE** to next live provider; emit `provider_fallback{from,to,reason}` (AUD-01).
  - `5xx` server error ‚Üí **CASCADE**.
  - `429` rate-limited ‚Üí **CASCADE** (mark provider temporarily unavailable / `rate_limited`).
  - `404` model-not-found / `400` unsupported-model ‚Üí **model-scoped**: cascade only to a source that actually carries that model; otherwise HALT with "model unavailable" remediation.
  - `400` malformed-request (client/our-bug) ‚Üí **HALT** (cascading cannot help); surface as internal error.
  - Provider content refusal / valid-but-declined response ‚Üí **NOT an error**; return to the user, do not cascade.
  - **Order:** OpenAI/Codex-compatible lane **first**, then remaining configured providers ordered by **liveness/health, not fixed priority** (AGNT-02). Each provider attempted at most once per request; a bounded total-attempt budget prevents long hangs. The provider actually used is surfaced on the response (AGNT-03).

### Cross-Cutting Locked Decisions

- Secrets never enter SQLite or audit JSONL; the file store is the only secret home; a redaction filter covers CLI output, logs, and audit (SEC-01).
- Codex is detection-only and read-only; `~/.codex` is byte-identical (mtime+hash) after any ATLAS command (SEC-02, hard gate).
- Rust-first for native components; Python stays confined to the foundation surface + LLM adapters + scripts (D-022). No Electron (D-005, hard gate).

### Claude's Discretion

- Exact module/file names within `services/agent-runtime/atlas_runtime/` (e.g. `auth/store.py`, `chat/adapter.py`), exact Pydantic field types/validators, exact lock-file naming, and the precise document filenames/locations for the two threat-model drafts (suggested `docs/operations/` or a `docs/security/` home) are left to the planner/executor within the patterns decided above.
- Whether the schema draft is committed as the literal `0004_registry_v2.sql` file or as a design note that the executor turns into the `.sql` ‚Äî the planner may choose, provided the committed artifact contains the exact column/key definitions decided here.

## Existing Code Insights

### Reusable Assets
- **Hermes auth reference** ‚Äî `foundation/atlas-hermes/hermes_cli/auth.py`: `AUTH_STORE_VERSION`, `AUTH_LOCK_TIMEOUT_SECONDS=15.0`, `~/.hermes/auth.json`, `ProviderConfig` dataclass + `PROVIDER_REGISTRY`, atomic-write + fcntl/msvcrt locking. Pattern to mirror (not import) for the ATLAS-owned store.
- **Hermes agent** ‚Äî `foundation/atlas-hermes/run_agent.py`: `class AIAgent` (ctor ~L350‚Äì399) takes `base_url, api_key, provider, api_mode, model, max_iterations, enabled_toolsets, ‚Ä¶`; `run_conversation()` + per-provider credential-refresh methods. This is what the adapter wraps.
- **Service-layer analog** ‚Äî `services/wiki-runtime/atlas_wiki/` (pure service fns + thin CLI + DB-passed-in) is the structural analog for the new agent-runtime adapter/auth/chat modules.
- **Model registry (v1)** ‚Äî `services/agent-runtime/atlas_runtime/model_registry.py`: `fetch_gateway_models`, `refresh(conn, lock, source, fetcher)`, `list_models`, `RefreshResult`. Evolves into the v2 discovery pipeline (10.3).
- **Redaction** ‚Äî `packages/atlas-core/atlas_core/schemas/core.py` `SECRET_PATTERNS` (token/api_key/secret/password/bearer regexes) is the existing redaction primitive to reuse for SEC-01.
- **Gateway** ‚Äî `native/atlas-core-rs/crates/atlas-gateway/src/lib.rs`: loopback-only (`127.0.0.1`, port 8484), read-only REST + CLI-dispatch writes, CORS already allows `tauri://localhost`, SSE via rowid poll. The IPC threat model builds on this loopback posture.

### Established Patterns
- Migrations: `infra/migrations/000N_*.sql` applied by sorted-glob `executescript`; DDL mirrors Pydantic / `ensure_schema()` (D-012). `0001_core`, `0002_wiki_provenance`, `0003_model_registry` exist.
- Foundation divergence discipline: `foundation/atlas-hermes/DIVERGENCE_LOG.md` (DIV-F-001..006); the `atlas_audit` bundled plugin (DIV-F-002) is the precedent for any foundation extension point.
- ATLAS home is `~/.atlas/` (gateway `default_db_path()` ‚Üí `~/.atlas/atlas.db`); `ATLAS_DB` / `ATLAS_CLI` env overrides exist.

### Integration Points
- Auth store ‚Üê read by adapter (Area 2), discovery `auth_status` (Area 3), `atlas auth*` CLI (10.1).
- `model_registry_v2` ‚Üê written by discovery (10.3), read by Rust gateway `/v1/models` after the 10.3 cutover, surfaced on cockpit Models page (MOD-06).
- Adapter ‚Üê emits to the ATLAS event bus (`services/agent-runtime/atlas_audit`), preserving Hermes approval gates.
- Native-IPC threat model ‚Üê consumed by the 10.5 Tauri shell build.
