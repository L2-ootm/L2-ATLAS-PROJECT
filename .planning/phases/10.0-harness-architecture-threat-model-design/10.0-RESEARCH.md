**Researched:** 2026-06-15

## Summary

Phase 10.0 commits four foundational designs before any harness code is written. This research **validates each committed decision against (a) the live Hermes reference implementation already vendored in the repo and (b) current authoritative external sources (RFC 8252/7636, SQLite UPSERT docs, Tauri 2 security docs, OpenAI SDK error map, Windows file-API semantics).** The headline finding: the committed decisions are sound, but **two carry real landmines** that the planner must surface in the drafted artifacts ŌĆö they are not blockers, they are caveats the design must record so downstream phases inherit the warning rather than rediscover it.

1. **Auth store** ŌĆö The "temp + `os.replace` + sidecar lockfile (fcntl/msvcrt), 15s timeout, 0600/icacls perms" pattern is **exactly what Hermes' own `hermes_cli/auth.py` does** and is the right pattern to mirror. **LANDMINE:** `os.replace` on Windows (`MoveFileEx`) is **not guaranteed atomic** when the target exists ŌĆö it can silently fall back to a non-atomic `CopyFile`. Hermes accepts this; the ATLAS draft must record it as a known limitation and lean on the lockfile + fsync + `.corrupt`-preservation fallback (which Hermes already implements) rather than claim hard atomicity.
2. **Adapter boundary** ŌĆö Composition (import + instantiate `AIAgent`), one-way `services/agent-runtime ŌåÆ foundation`, extension-points-only via the `atlas_audit` plugin-shim precedent (DIV-F-002, verified present) is **upstream-mergeable and correct**. The `AIAgent.__init__` surface is verified: it is a keyword forwarder into `agent.agent_init.init_agent` with `base_url, api_key, provider, api_mode, model, max_iterations, enabled_toolsets, ŌĆ”` plus a native `fallback_model` / `credential_pool` hook the adapter can exploit.
3. **`0004_registry_v2.sql`** ŌĆö Composite PK `(model_id, provider_id, source)` + source-scoped `deactivated_at` is the correct idempotent-discovery shape; SQLite `INSERT ŌĆ” ON CONFLICT(model_id, provider_id, source) DO UPDATE` with `excluded.` references preserves `first_seen` and bumps `last_seen`. Additive migration is safe alongside `0003`. **The lower-risk compatibility path is a VIEW** (verified: the Rust gateway reads `model_registry` with named columns `model_id, provider, source, first_seen, last_seen, active`, so a view exposing those columns is a drop-in) ŌĆö but the committed deferred-cutover-in-10.3 plan is also valid and keeps `0003` untouched. Recommend: draft both, let 10.3 choose.
4. **Threat models + fallback** ŌĆö OAuth loopback draft requirements map 1:1 onto RFC 8252 ┬¦7.3 + RFC 7636 (PKCE S256, any-port loopback, exact redirect match, open-port-only-during-flow). Tauri 2's capability/permission system is real and deny-by-default ŌĆö the PTY-as-byte-channel design is the correct primitive. The fallback errorŌåÆaction table maps cleanly onto the **OpenAI Python SDK exception hierarchy** (`AuthenticationError`/401, `PermissionDeniedError`/403, `NotFoundError`/404, `BadRequestError`/400, `RateLimitError`/429, `InternalServerError`/Ōēź500, `APIConnectionError`/network), making it directly implementable.

**Primary recommendation:** Proceed with all four committed decisions as drafted. Record the two landmines (Windows `os.replace` non-atomicity; VIEW-vs-deferred-cutover tradeoff) directly in the committed artifacts so 10.1/10.3 inherit them. Draft the schema as the literal `0004_registry_v2.sql` (matching the seed ┬¦8 column-for-column) plus mirrored Pydantic v2 models.

# Area 1 ŌĆö Auth File-Store (cross-platform)

### VALIDATION VERDICT: Ō£ģ SOUND, with one recorded landmine

The committed pattern ŌĆö flat `~/.atlas/auth.json`, single `auth_store_path()` resolver honoring an `ATLAS_HOME` override, JSON `{"version":1,"providers":{ŌĆ”}}`, temp-file + `os.replace` + sidecar lockfile (fcntl/msvcrt) with 15s timeout, 0600/icacls perms, mirror-not-import Hermes ŌĆö **matches the live Hermes implementation almost exactly** and is the correct, battle-tested shape. Mirroring it (without importing) is sound and keeps the vendored foundation re-syncable.

### How Hermes actually does it (the pattern to mirror)

Verified from `foundation/atlas-hermes/hermes_cli/auth.py` and helpers:

- **Constants** (`auth.py:70-71`): `AUTH_STORE_VERSION = 1`, `AUTH_LOCK_TIMEOUT_SECONDS = 15.0`. [VERIFIED: codebase]
- **Lock** (`_file_lock`, `auth.py:968-1037`): a **sidecar lock file** (separate from `auth.json`), reentrant per-thread via a `threading.local` depth counter, `fcntl.flock(LOCK_EX|LOCK_NB)` on POSIX / `msvcrt.locking(LK_NBLCK, 1)` on Windows, polled at 50ms until a `time.monotonic()` deadline, raising `TimeoutError`. Falls back to a depth-only guard when neither module exists. [VERIFIED: codebase]
  - **Windows-specific quirk it already handles:** `msvcrt.locking` requires the lock file to have Ōēź1 byte and the file pointer at position 0 ŌĆö Hermes writes a single space byte if the lock file is empty (`auth.py:1004-1005`) and `seek(0)` before lock/unlock. The ATLAS mirror **must replicate this** or Windows locking silently no-ops. [VERIFIED: codebase]
- **Atomic save** (`_save_auth_store`, `auth.py:1099-1145`):
  1. `secure_parent_dir(auth_file)` chmods the parent dir to `0o700` (no-op on Windows; refuses `/` and top-level dirs ŌĆö see `hermes_constants.py:257`). [VERIFIED: codebase]
  2. Temp file named `{name}.tmp.{pid}.{uuid4hex}` created via **`os.open(O_WRONLY|O_CREAT|O_EXCL, S_IRUSR|S_IWUSR)`** ŌĆö this closes the TOCTOU window where a default umask (0644) would briefly expose tokens between `open()` and `chmod()`. [VERIFIED: codebase]
  3. `handle.write` ŌåÆ `flush` ŌåÆ **`os.fsync(fileno())`**.
  4. **`atomic_replace(tmp, auth_file)`** = `os.replace`, but resolves symlinks first so a symlinked `auth.json` is written in-place rather than detached (`utils.py:61-82`). [VERIFIED: codebase]
  5. **`os.fsync` the parent directory fd** (POSIX) so the rename is durable.
  6. Best-effort `auth_file.chmod(S_IRUSR|S_IWUSR)` after replace.
- **Corruption resilience** (`_load_auth_store`, `auth.py:1059-1096`): on `json.loads` failure it copies the file to `auth.json.corrupt` and returns an empty store rather than crashing. The ATLAS mirror should keep this behavior. [VERIFIED: codebase]

### LANDMINE 1 ŌĆö `os.replace` is NOT guaranteed atomic on Windows when the target exists

`os.replace` ŌåÆ Win32 `MoveFileEx(MOVEFILE_REPLACE_EXISTING)`, which **"is NOT guaranteed to be atomic. Under certain circumstances it may silently fall back to a non-atomic `CopyFile`."** [VERIFIED: multiple sources ŌĆö python-atomicwrites docs, MS Win32 fileio docs]

Implications for the drafted design:
- The committed phrasing "atomic write via temp + os.replace" is **POSIX-true, Windows-best-effort**. The draft must say so explicitly.
- Mitigations Hermes already provides, which the ATLAS draft should adopt as the *real* durability story on Windows: (a) the **cross-process lockfile serializes writers** so two `os.replace` calls never race; (b) **fsync before replace** means the temp file is fully on disk; (c) the **`.corrupt` preservation fallback** means a torn write is recoverable, not silently lost.
- **Do NOT** claim hard atomicity in AUTH-02's acceptance text. AUTH-02 says "atomic (temp + replace) and guarded by a cross-process lock; concurrent writers never corrupt or silently lose data" ŌĆö the lock is what guarantees the no-corruption property on Windows, *not* `os.replace`. The draft should make the lock the load-bearing element, not the rename.
- Optional hardening for the planner to consider (not required): `MoveFileTransacted`/`NtSetInformationFile` are "more atomic" but unsupported/Vista-only; the `atomicwrites` PyPI library exists but **adds a dependency** and is in maintenance ŌĆö mirroring Hermes (zero new deps) is the better fit given D-022's dependency discipline.

### Permissions ŌĆö correct cross-platform approach

- **POSIX:** `os.chmod(path, stat.S_IRUSR | stat.S_IWUSR)` = `0o600`. Create the temp via `os.open(..., 0o600)` to avoid the umask TOCTOU window (as Hermes does). [VERIFIED: codebase + Python stdlib]
- **Windows:** `os.chmod` is effectively a **no-op for ACLs** (Python only toggles the read-only bit on Windows). SEC-03 explicitly requires `icacls`. Correct command shape [VERIFIED: ss64.com/icacls, 4sysops]:
  ```
  icacls "<path>\auth.json" /inheritance:r /grant:r "%USERNAME%:(F)" /Q
  ```
  - `/inheritance:r` strips all inherited ACEs (otherwise Users/Authenticated Users inherit read).
  - `/grant:r "<user>:(F)"` replaces explicit grants with current-user Full only.
  - Resolve the current user robustly (not just `%USERNAME%`): prefer the SID via `os.environ` or `win32security`/`GetUserNameEx`; for a draft, `%USERNAME%` is acceptable but note that domain accounts may need `DOMAIN\user`. [VERIFIED: external] ŌĆö exact identity-resolution call is an open question for 10.1.
  - Run via `subprocess` with the path as a list arg (no shell-string interpolation) to avoid injection.

### Format & resolver (validated against seed ┬¦5/┬¦6)

- `{"version": 1, "providers": {<provider_id>: {auth_type, base_url, created_at, last_refresh_at, redacted_hint, ŌĆ”}}, "metadata": {ŌĆ”}}` ŌĆö matches seed ┬¦6 and Hermes' `{"version", "providers"}` shape. The `version` field is the forward-migration hook (Hermes migrates an older `"systems"` format ŌĆö `auth.py:1087-1094` ŌĆö proving the hook earns its keep).
- **Single resolver** honoring `ATLAS_HOME` (env override so tests never touch real home) is the right call. Hermes uses `HERMES_HOME` identically (`hermes_constants.py:294`). Profile-ready layout (`~/.atlas/profiles/<name>/auth.json`) becomes a one-function change. [VERIFIED: codebase + seed ┬¦5]
- **`redacted_hint`** (e.g. last 4 chars) for display; **full secrets never** returned by status/list, never in SQLite, never in audit JSONL. Reuse `packages/atlas-core/atlas_core/schemas/core.py` `SECRET_PATTERNS` for the redaction filter (SEC-01). [VERIFIED: codebase, per CONTEXT code_context]

### LANDMINE 2 ŌĆö lock-file staleness on crash

A sidecar lockfile held by a crashed process: `fcntl.flock` and `msvcrt.locking` are **advisory OS-level locks released automatically when the file handle/process dies**, so a crash does **not** leave a permanently-held lock (unlike a naive "create lock file, delete on exit" scheme). Hermes relies on this. The ATLAS mirror must use the **same OS-lock-on-an-open-handle** approach (not a presence-based lockfile) to inherit crash-safety. The draft should state this explicitly so 10.1 doesn't "simplify" it into a presence-based lock that deadlocks after a crash. [VERIFIED: codebase pattern + POSIX/Win32 lock semantics]

2. Whether the ATLAS lock file is `auth.json.lock` as a sibling (Hermes uses a dedicated lock path). Naming left to executor per Claude's-Discretion; recommend a sibling `auth.lock` to keep it out of any `*.json` glob.

# Area 3 ŌĆö `0004_registry_v2.sql` Schema

### VALIDATION VERDICT: Ō£ģ SOUND (composite PK + source-scoped deactivation + additive migration all correct)

### Composite PK + idempotent UPSERT ŌĆö validated

`PRIMARY KEY(model_id, provider_id, source)` is the correct conflict target for idempotent multi-source discovery (MOD-03). SQLite UPSERT against a composite PK using `excluded.` to reference incoming values, while **preserving `first_seen` by simply not updating it**:

```sql
INSERT INTO model_registry_v2
  (model_id, provider_id, source, api_mode, status, auth_status,
   context_window, metadata_json, first_seen, last_seen, deactivated_at)
VALUES
  (:model_id, :provider_id, :source, :api_mode, :status, :auth_status,
   :context_window, :metadata_json, :now, :now, NULL)
ON CONFLICT(model_id, provider_id, source) DO UPDATE SET
  api_mode       = excluded.api_mode,
  status         = excluded.status,
  auth_status    = excluded.auth_status,
  context_window = excluded.context_window,
  metadata_json  = excluded.metadata_json,
  last_seen      = excluded.last_seen,     -- bump
  deactivated_at = NULL;                   -- reactivate on re-discovery
  -- first_seen intentionally NOT in SET ŌåÆ original preserved
```
[VERIFIED: sqlite.org/lang_upsert ŌĆö "Column names in DO UPDATE refer to the original unchanged value; use `excluded.` for the would-be-inserted value." Composite conflict target supported.]

**Note:** Bare `ON CONFLICT DO UPDATE` (no target) also works, but for a composite-PK table the **explicit `ON CONFLICT(model_id, provider_id, source)` target is preferred** for clarity and to fail loudly if the PK ever changes. The current v1 `model_registry.py refresh()` uses a read-diff-then-INSERT/UPDATE three-pass approach (`model_registry.py:132-166`); the v2 path can either keep that pattern or switch to UPSERT. **Recommendation:** UPSERT for the per-row upsert + a separate source-scoped deactivation pass (below) ŌĆö fewer round-trips, naturally idempotent.

### Source-scoped soft-delete (MOD-04) ŌĆö validated

Deactivation must be scoped to the source so one provider going offline never touches another source's rows. **Soft-delete via `deactivated_at` timestamp (not a hard DELETE)** is correct ŌĆö it preserves routing history (matching the v1 module's `active=0` rationale: "deactivated, not deleted, so routing history stays resolvable," `model_registry.py:6-7`). The deactivation pass:

```sql
UPDATE model_registry_v2
SET deactivated_at = :now
WHERE source = :source
  AND deactivated_at IS NULL
  AND (model_id, provider_id) NOT IN (
        SELECT value_model_id, value_provider_id FROM <incoming-this-source> );
```
(Implementation detail ŌĆö the executor will materialize "incoming for this source" however the discovery pipeline supplies it; the load-bearing facts are: **filter by `source`**, set a **timestamp** not a delete, and **only touch currently-active rows**.) [VERIFIED against v1 pattern + UPSERT semantics]

### `status` vs `auth_status` distinct columns (MOD-05) ŌĆö validated

Two columns is correct: `status` (model reachability/lifecycle: `available`/`offline`/`unsupported`/ŌĆ”) is orthogonal to `auth_status` (credential state: `auth_present`/`needs_api_key`/`needs_login`/`rate_limited`/`expired`/ŌĆ”). A model must never show `available` when auth is missing ŌĆö distinct columns make that a query-time AND, not a conflated single field. `auth_status` is **computed at discovery time from the file auth store, never a stored secret** (Area-1 credential boundary). [VERIFIED: CONTEXT + seed ┬¦3/┬¦4 status vocab]

### Backward-compat: additive migration + the VIEW option

- **The Rust gateway read path (verified):** `atlas-gateway/src/db.rs:255` `list_models()` runs `SELECT model_id, provider, source, first_seen, last_seen, active FROM model_registry ORDER BY model_id LIMIT ?1`, and returns `Ok(vec![])` if the table is missing. **Because it selects named columns, a compatibility VIEW named `model_registry` exposing exactly those six columns over `model_registry_v2` would be a transparent drop-in** ŌĆö the gateway code wouldn't change. [VERIFIED: codebase `db.rs:253-275`]
  - VIEW sketch (for the draft to record as the *option*, mapping v2 ŌåÆ legacy shape):
    ```sql
    -- Only viable AFTER 0003 model_registry is renamed/retired, since a view
    -- and a table cannot share the name. Hence this is a 10.3-cutover artifact.
    CREATE VIEW model_registry AS
    ```

# Area 4 ŌĆö Threat Models + Fallback-Cascade Spec

## 4a. OAuth loopback-callback threat model (DRAFT only ŌĆö OAuth deferred per SEC-05)

### VALIDATION VERDICT: Ō£ģ The drafted gate requirements map 1:1 onto RFC 8252 + RFC 7636

ATLAS ships **no** OAuth callback flow in v1.1; 10.0 only drafts the gate any future flow must pass. Every committed requirement is directly traceable to current authoritative native-app OAuth guidance:

| Drafted requirement | Authoritative basis | Provenance |
|---------------------|---------------------|------------|
| PKCE **S256** mandatory | RFC 8252 requires PKCE [RFC 7636]; S256 is the secure transform (plain only when S256 unsupported) | [CITED: datatracker.ietf.org/doc/html/rfc8252 ┬¦8.1; RFC 7636] |
| `state` generated + validated (CSRF) | RFC 6749/8252 anti-CSRF; one-time high-entropy value compared on callback | [CITED: RFC 8252] |
| Ephemeral loopback listener on `127.0.0.1`, **OS-assigned port** | RFC 8252 ┬¦7.3 ŌĆö loopback redirect, "any port" must be allowed; bind 127.0.0.1 only | [CITED: RFC 8252 ┬¦7.3] |
| Accept **exactly one** callback then close; open port only during the flow | RFC 8252 ┬¦8.3 / ┬¦7.3 ŌĆö "open the port only when starting the request and close once the response is returned; listen on loopback only" | [CITED: RFC 8252] |
| **Exact** redirect-URI match | RFC 8252 ŌĆö exact string match of redirect URI (modulo the port-flexibility allowance for loopback) | [CITED: RFC 8252] |
| **No token/code** in any log or persisted URL | RFC 8252 ┬¦8.x + ATLAS SEC-01 redaction | [CITED: RFC 8252 + REQUIREMENTS SEC-01] |
| Loopback uses `http` (no TLS) ŌĆö acceptable | RFC 8252 ┬¦7.3 ŌĆö request never leaves the device | [CITED: RFC 8252 ┬¦7.3] |
| Manual-paste fallback for SSH/headless | Native-app guidance for browserless environments (Hermes implements this) | [CITED: seed ┬¦12 + RFC 8252 device-flow alternative] |

**Implementation hint for the future spike (record in draft):** Python stdlib `http.server` bound to `("127.0.0.1", 0)` (port 0 = OS-assigned) + a single-request handler that captures `code`+`state` then shuts down via `server.shutdown()`. Read the actual port from `server.server_address[1]` and use it to build the exact redirect URI. No third-party dep needed.

### Landmine (OAuth draft)
- **`state` comparison must be constant-time** (`hmac.compare_digest`) and the listener must reject any callback whose `state` ŌēĀ the issued value **before** processing `code`. A naive `==` plus "first callback wins" is the classic loopback hijack vector. Record explicitly.
- **Port-zero binding is right, but the redirect URI registered with the AS must then be the *exact* `http://127.0.0.1:<chosen-port>/callback`** ŌĆö RFC 8252 ┬¦7.3 port flexibility means the AS must tolerate the port varying; not all providers honor this (the gate must list "provider supports loopback port flexibility" as a precondition).

## 4b. Native-IPC threat model (DRAFT ŌĆö Tauri 2 surface, shell built in 10.5)

### VALIDATION VERDICT: Ō£ģ Tauri 2 capability/permission system is real and is the correct primitive; PTY-as-byte-channel is sound

Tauri 2 (not Tauri 1) is required and its security model directly supports the committed design [CITED: v2.tauri.app/security/capabilities + /security/permissions]:

- **Deny-by-default, capability-scoped allowlist.** Tauri 2 treats the webview as **untrusted by default**; every IPC command (built-in, plugin, or custom) must be **explicitly permitted** in a capability file (`src-tauri/capabilities/*.json`), scoped to specific windows/webviews. This is a deliberate inversion from Tauri 1's flat allowlist. This **is** the "allowlist, deny-by-default" the committed model names. [CITED: v2.tauri.app/security/capabilities]
- **Per-command permissions.** Each custom command auto-generates `allow-<command>` / `deny-<command>` permission identifiers; plugins ship their own (e.g. `shell:allow-open`, `dialog:default`). The draft's IPC inventory rows map directly onto these grants. [CITED: v2.tauri.app/security/permissions]
- **Scopes** further constrain a permitted command (e.g. `open_external` restricted to an allowlisted URL scheme set is a Tauri **scope** on the opener/shell permission). [CITED: v2.tauri.app]
- **No Electron** (D-005) ŌĆö Tauri uses the OS webview + a Rust core, so the trust boundary is the IPC bridge, which capabilities lock down. Ō£ģ

**The committed IPC command inventory is the right shape.** Each row = `{command, params, caller, privilege, threat, mitigation}`. Validated rows:

| command | params | privilege | threat | mitigation (Tauri 2 primitive) |
|---------|--------|-----------|--------|--------------------------------|
| `get_readiness()` | ŌĆö | read-only | info disclosure | capability allows it for cockpit window only; returns redacted status (no secrets) |
| `pty_open(profile_id)` | profile id (enum/validated) | spawn `atlas tui` | arbitrary-exec if it took a command string | **spawn a fixed argv (`atlas tui --profile <id>`), profile_id validated against known profiles; NO free-form command param** (NAT-03 hard gate) |
| `pty_write(bytes)` | raw bytes | write to PTY stdin | command injection if interpreted | **bytes are keystrokes to an already-running `atlas tui`, not a shell ŌĆö the child is `atlas tui`, never `bash`/`cmd`** (NAT-02/03) |
| `pty_resize(rows, cols)` | ints | resize | DoS via huge values | clamp to sane bounds |
| `open_external(url)` | url | OS opener | open arbitrary scheme (e.g. `file://`, custom handlers) | **scheme allowlist (`https`, maybe `mailto`) via Tauri scope**; reject everything else |

### LANDMINE 5 ŌĆö the PTY-as-byte-channel is the whole security argument; do not weaken it

The single most important native-IPC invariant: **the PTY transports keystrokes/bytes to a pre-spawned `atlas tui` child, never a command string the frontend supplies.** If any IPC command ever accepts a command/argv string from the webview, the entire deny-by-default model collapses into an arbitrary-exec bridge (NAT-03 explicitly forbids this). The draft must state this as a **hard gate**, and 10.5's `pty_open` must spawn a **fixed program with validated, enumerated args** ŌĆö `profile_id` is an *identifier looked up against known profiles*, not a path or command. [CITED: NAT-02/03 + Tauri 2 scope model]

Also record:
- **No remote origin in production** (NAT-01) ŌĆö the cockpit is loaded from a local bundle; a remote origin in the webview would let a compromised page invoke permitted IPC commands.
- **Cockpit Ōåö gateway stays loopback HTTP** (gateway already `127.0.0.1:8484`, CORS already allows `tauri://localhost`) ŌĆö this is **not** an IPC surface and doesn't need a capability grant; keep it as HTTP so it isn't conflated with the privileged IPC channel. [VERIFIED: codebase gateway posture per CONTEXT]
- **No PTY plugin is chosen yet** ŌĆö Tauri 2 has no first-party PTY; 10.5 will pick one (community `tauri-plugin-pty` or a custom command over `portable-pty`). That choice carries its own package-legitimacy audit at 10.5. Flag, don't decide here. [ASSUMED ŌĆö PTY plugin landscape from training; verify at 10.5]

## 4c. Fallback-cascade error-classification table

### VALIDATION VERDICT: Ō£ģ Directly implementable against the OpenAI Python SDK exception hierarchy

The committed errorŌåÆaction contract maps cleanly onto the OpenAI SDK's typed exceptions (the SDK most OpenAI-compatible providers' clients mirror). Verified mapping [CITED: platform.openai.com/docs/guides/error-codes + github.com/openai/openai-python]:

| Condition | HTTP | OpenAI SDK exception | Committed action | Verdict |
|-----------|------|----------------------|------------------|---------|
| Auth invalid/expired | 401 | `openai.AuthenticationError` | **HALT**, remediation (`atlas auth add`/`doctor`), audit; never cascade | Ō£ģ AGNT-03 |
| Permission denied | 403 | `openai.PermissionDeniedError` | **HALT** | Ō£ģ |
| Network/DNS/timeout/conn-refused | ŌĆö | `openai.APIConnectionError` (+ `APITimeoutError` subclass) | **CASCADE**; emit `provider_fallback{from,to,reason}` | Ō£ģ AUD-01 |
| Server error | Ōēź500 | `openai.InternalServerError` | **CASCADE** | Ō£ģ |
| Rate limited | 429 | `openai.RateLimitError` | **CASCADE**; mark provider `rate_limited`/temp-unavailable | Ō£ģ |
| Model not found | 404 | `openai.NotFoundError` | **model-scoped** ŌĆö cascade only to a source carrying that model, else HALT "model unavailable" | Ō£ģ |
| Unsupported/malformed model param | 400 | `openai.BadRequestError` | model-scoped (unsupported-model) **or** HALT (our-bug) ŌĆö distinguish by error body | ŌÜĀ’ĖÅ see landmine |
| Malformed request (client/our bug) | 400 | `openai.BadRequestError` | **HALT** (cascade can't help); surface as internal error | Ō£ģ |
| Content refusal / valid-but-declined | 200 | *no exception* (normal response) | **NOT an error** ŌĆö return to user, do not cascade | Ō£ģ |

Parent class for any non-2xx is `openai.APIStatusError` exposing `.status_code` and `.response` ŌĆö the classifier can branch on `isinstance` against the typed subclasses (precise) and fall back to `.status_code` ranges for non-OpenAI-SDK clients. [VERIFIED: error map]

**Ordering (AGNT-02):** OpenAI/Codex-compatible lane **first**, then remaining configured providers ordered by **liveness/health, not fixed priority**; each provider attempted **at most once per request**; a **bounded total-attempt budget** prevents long hangs; the **provider actually used is surfaced on the response** (AGNT-03). This is a standard health-aware bounded-cascade pattern and is implementable as: ordered candidate list ŌåÆ for each, try once ŌåÆ on a CASCADE-class error record reason + emit `provider_fallback` + advance ŌåÆ on a HALT-class error stop immediately ŌåÆ on success annotate response with provider. [VERIFIED: pattern + SDK error types]

### LANDMINE 6 ŌĆö `400` is overloaded (unsupported-model vs our-bug)

A single `400 BadRequestError` covers both "this provider doesn't support that model/param" (ŌåÆ model-scoped cascade) **and** "we sent a malformed request" (ŌåÆ HALT). The classifier **cannot** distinguish these from the status code alone ŌĆö it must inspect the error body/`code`/`message`. The committed table already splits them, but the draft must state that **the 400 branch requires body inspection** and define a **safe default: when ambiguous, HALT** (cascading a malformed request to N providers just multiplies the same error and burns the attempt budget). Record the disambiguation heuristic as an open implementation detail for 10.2. [VERIFIED: SDK semantics ŌĆö 400 is not self-describing]

### LANDMINE 7 ŌĆö OpenAI-compatible providers vary in error fidelity

Local sidecars (LM Studio, Ollama, FreeLLMAPI) and OpenRouter do **not** always return the exact OpenAI status code/body for a given failure (e.g. a down sidecar yields a connection error, not a clean 503; some return 200 with an error JSON body). The classifier must therefore **treat any non-conforming/garbled response as a CASCADE-class transport failure**, not assume the OpenAI SDK exception will always fire. The `urllib`-based v1 `fetch_gateway_models` already only trusts loopback hosts (`model_registry.py:13-14`); the v1.1 cascade must keep that defensive posture. [VERIFIED: codebase + general OpenAI-compat ecosystem behavior]

### Open questions (Area 4)
1. Exact `400` disambiguation heuristic (unsupported-model vs our-bug) ŌĆö defer to 10.2, draft a safe-default-HALT rule.
2. Bounded total-attempt budget value (e.g. N=3 distinct providers, or a wall-clock cap) ŌĆö pick a default in the draft; tune in 10.2.
3. Which providers constitute the "OpenAI/Codex-compatible lane" that goes first ŌĆö the draft should enumerate the lane membership rule (api_mode Ōłł {chat_completions, responses, codex_responses}), not hardcode provider names.
