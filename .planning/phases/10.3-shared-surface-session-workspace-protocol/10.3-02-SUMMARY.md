# Plan 10.3-02 Summary — fail-closed workspace resolver

## What shipped

`services/agent-runtime/atlas_runtime/workspace_service.py`:
- `WorkspaceError(ValueError)` carrying a `reason` in `{traversal, symlink_escape,
  stale_root, unregistered, cross_project}`.
- `global_root()` — derives from `db.DEFAULT_DB_PATH.parent` so the workspace root and DB
  home never diverge (Pitfall 4); testable by monkeypatching `db.DEFAULT_DB_PATH`.
- `resolve_workspace(conn, *, kind, project_id=None)` — global → `global_root()`; project →
  registry `get_project`, raising `unregistered` on miss.
- `assert_contained(root, target)` — `.resolve()` + `os.path.commonpath` containment (no
  prefix matching); `stale_root` when the root is gone (no auto-reregister); mixed-drive
  `commonpath` ValueError mapped to a typed error (Pitfall 1); `symlink_escape` vs
  `traversal` distinguished via lexical-`abspath` vs resolved comparison (Pitfall 5).
- `assert_write_allowed(conn, lock, *, session_id, run_id, declared_root, target)` — denies a
  contained-but-different-registered-project target with the distinct `cross_project` reason
  and an audited `permission_transition` denial (SURF-03, AUD-01).

## Decisions

- Global root bound to DB home rather than adding a separate `ATLAS_HOME` env path that
  `db.py` does not honor — adding one would reintroduce the divergence the design forbids.
  Test isolation monkeypatches `db.DEFAULT_DB_PATH`, never the live `~/.atlas`.
- `cross_project` is a genuinely distinct typed reason from `traversal`: a traversal target
  escapes the declared root; a cross-project target stays inside its own root but is not the
  declared workspace. Both fail closed.
