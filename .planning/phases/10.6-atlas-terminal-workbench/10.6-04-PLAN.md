<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Operator keypress -> resolve_approval_choice | Untrusted UI selection crosses into broker decision authority |
| permission_ui.py -> permission_broker.claim/record_allow_rule | TUI never holds decision authority itself; broker is sole authority per PERM-04/PERM-07 |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10.6-11 | Elevation of Privilege | tui/permission_ui.py "session" branch | mitigate | record_allow_rule is called via the broker's own validated entrypoint with arg_pattern derived ONLY from allow_pattern_for_args (same normalization as the real invoke() path) — TUI never constructs or caches its own allow-rule matching logic, preventing a scope-bypass via a looser TUI-local pattern |
| T-10.6-12 | Repudiation | tui/permission_ui.py "cancel" branch | accept | Cancel intentionally makes no broker call and leaves the approval pending; this is the only valid mapping since _VALID_CLAIM_DECISIONS has no cancel value — operator can revisit the same pending approval later, no silent state loss |
| T-10.6-13 | Tampering | resolve_approval_choice swallowing broker exceptions | mitigate | Every claim()/record_allow_rule() exception (WrongSessionError, NotActiveSessionError, StaleApprovalError, AlreadyDecided, ApprovalChannelMissingError) is allowed to propagate unmodified — no try/except wraps the dispatch, so a headless/no-channel request fails closed rather than appearing to silently succeed |
| T-10.6-14 | Information Disclosure | render_approval_panel approval.summary display | accept | approval.summary/args already passed through the same audit/persistence redaction boundary (D-013) before being stored; permission_ui renders as-is |
</threat_model>
