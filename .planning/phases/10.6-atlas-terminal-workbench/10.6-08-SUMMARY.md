## Gap Closure (post-review fixes, 2026-06-27)

Code review (`10.6-REVIEW.md`, deep depth, 24 files) found the individual TUI
modules pass their isolated unit tests but were never assembled into a working
product: the transcript read from a table nothing wrote to, the command
palette was dead code, and the permission group hard-coded an empty session
id. This section documents the fixes, each proven by a test that drives the
REAL path (audit_events ledger, the assembled `run_workbench` entrypoint),
not a seeded shortcut.

### CR-01 (HIGH): Transcript read an unwritten `surface_events` table

- **Root cause:** Migration `0018_surface_events.sql` created a physical
  table that nothing in production code ever wrote to (only the unit test
  seeded it directly). `transcript.poll_and_render` read from that table, so
  the production transcript was permanently empty.
- **Fix:** Rewired `transcript.poll_and_render` to the locked design —
  `audit_service.get_events_for_session(conn, session_id)` →
  `surface_events.normalize_surface_events(audit_events, run_outcome=None,
  session_id=session_id)` → `surface_events.replay_since(events, last_seq)`.
  No persistence layer; the projection is recomputed from the immutable
  `audit_events` ledger on every poll.
- **Deleted:** `infra/migrations/0018_surface_events.sql` (confirmed via
  repo-wide grep that nothing else referenced the table).
- **Fixed false docstring** in `transcript.py` claiming a writer existed.
- **Found + fixed an additional latent bug while rewiring:**
  `normalize_surface_events` assigns the FIRST event `seq=0`;
  `poll_and_render`'s (and `app.py`'s `_poll_loop`'s) `last_seq=0` default
  meant `replay_since(events, 0)` excluded `seq=0` — the first event was
  silently dropped on every session's first poll. Fixed by defaulting
  `last_seq=-1` in both places (Rule 1 — correctness bug found while
  proving the fix with a real test).
- **Test:** Rewrote `tests/tui/test_transcript_poll.py` to seed `runs` +
  `audit_events` rows via `audit_service.emit` (the real write path) instead
  of seeding the deleted `surface_events` table directly. Verifies the first
  poll returns both seeded events, a second poll with the new `last_seq`
  returns nothing, and a THIRD poll after a fresh `audit_service.emit` call
  returns exactly the one new event — proving the append-only gap-free
  contract end-to-end, not just dedup-to-empty.
- **Files:** `atlas_runtime/tui/transcript.py`, `atlas_runtime/tui/app.py`,
  `tests/tui/test_transcript_poll.py`, `infra/migrations/0018_surface_events.sql` (deleted)
- **Commit:** (transcript rewire + migration delete + off-by-one fix)

### CR-02 (HIGH): Command palette was dead code

- **Root cause:** `command_dispatch.dispatch_command` was never imported or
  called from `app.py`; the composer's `/`-prefixed branch printed a static
  placeholder string referencing a "later plan" that had already landed.
- **Fix:** `run_workbench`'s composer loop now calls
  `command_dispatch_module.dispatch_command(eff_conn, line,
  surface_session_id=session.id)` for `/`-prefixed lines and prints
  `result.text`. Non-`/` lines route to `_submit_to_agent` (see below).
- **Test:** Added `tests/tui/test_composer_routing.py`, which drives the
  ASSEMBLED `run_workbench` entrypoint (stubbing `prompt_toolkit` with a
  scripted fake session, not calling `dispatch_command` directly) and
  asserts a `/`-prefixed line invokes `dispatch_command` with the real
  session id while a non-`/` line invokes `_submit_to_agent` instead.
- **Files:** `atlas_runtime/tui/app.py`, `tests/tui/test_composer_routing.py`

### CR-03 (HIGH): Permission group hard-coded an empty session id

- **Root cause:** `_handle_permission`'s `list`/`decide` subcommands passed
  `surface_session_id=""` to `permission_broker.list_actionable` /
  `permission_ui.resolve_approval_choice` — a predicate that can never match
  a real UUID-shaped session id.
- **Fix:** `dispatch_command`'s signature is now
  `dispatch_command(conn, command_text, *, surface_session_id)`; the
  `Handler` type and every registered handler (`project`, `focus`, `config`,
  `permission`, the deferred seams) now take `surface_session_id` as a third
  positional argument. `_handle_permission` passes the real threaded id
  through to both broker calls.
- **Test:** Added `test_permission_list_scopes_to_threaded_session_id`
  (asserts the real id reaches `list_actionable`, not `""`) and
  `test_permission_list_only_returns_owning_session_rows` (seeds one
  approval owned by the active session and one owned by a different
  session; asserts `/permission list` returns only the owning session's
  row) to `tests/tui/test_command_dispatch.py`.
- **Files:** `atlas_runtime/tui/command_dispatch.py`,
  `tests/tui/test_command_dispatch.py`

### `_submit_to_agent` wiring (part of CR-01/CR-02 root cause)

- **Root cause:** `app.py`'s `_submit_to_agent` was an explicit no-op stub —
  nothing ever produced the audit events CR-01's transcript fix needed to
  render.
- **Fix:** `_submit_to_agent(conn, lock, *, session_id, line,
  start_and_execute=run_executor.start_and_execute_async)` now creates a
  `pending` Mission via `mission_service.create_mission` and drives it via
  `run_executor.start_and_execute_async(..., session_id=session_id, ...)` on
  a background daemon thread (non-blocking for the composer prompt). The
  default `"native"` `AgentRuntime` (`atlas_runtime.agents.native`) never
  requires `claude_agent_sdk` — with no provider configured it falls back to
  a deterministic, clearly-labeled mock response (the existing honest-failure
  contract), so this path is real and testable in this environment.
  `start_and_execute` is injectable for tests (a stub/mock executor, not a
  real background thread).
- **Found + fixed an additional latent bug while wiring this in:**
  `run_workbench` created the surface session but never transitioned it out
  of `"starting"`. `handle_cancel`'s Ctrl-C/EOF unwind calls
  `transition_session(..., "cancelling")`, which `_ALLOWED_FROM` only
  permits from `"active"`/`"suspended"` — so the Ctrl-C path always raised
  `ValueError` in production. Fixed by transitioning the session to
  `"active"` immediately after creation, before the composer loop begins
  (Rule 1 — found while driving the real `run_workbench` path in the new
  composer-routing test, which surfaced the `ValueError` on its EOF branch).
- **Human-verify (deferred to UAT):** a live end-to-end agent run against a
  configured real provider (not the mock fallback) was not exercised in this
  fix — `_submit_to_agent` is unit-tested with `start_and_execute` injected
  as a stub. Driving an actual provider call is out of scope for this
  gap-closure pass; the wiring itself (session → mission → run →
  audit_events → transcript) is what was broken and is now real and tested.
- **Files:** `atlas_runtime/tui/app.py`
