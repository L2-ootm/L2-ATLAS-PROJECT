# Phase 10.6: Code Review Report

**Reviewed:** 2026-06-27T21:03:42Z
**Depth:** deep
**Files Reviewed:** 24
**Status:** issues_found

## Summary

Reviewed the ATLAS Terminal Workbench (Phase 10.6) against the locked CONTEXT decisions and
the 10.3 SurfaceEvent contracts. The individual modules (`header.py`, `permission_ui.py`,
`resume.py`, `capabilities.py`, `session_select.py`) are clean, well-documented, and faithfully
implement their narrow contracts (masked header, fail-closed resume, broker-mapped permission
dialog). Tests for those modules genuinely exercise the locked behaviors and were not weakened.

However, the streaming/transcript path and the command-palette path have a severe integration
gap that tests do not catch because they exercise the units in isolation, never the assembled
`run_workbench` entrypoint against a real, unseeded database. Concretely:

1. **The production transcript is permanently empty.** A brand-new `surface_events` table was
   created (migration `0018`) in direct contradiction of the locked "no new persistence, reuse
   the 10.3 read-projection" decision, and nothing in production code ever writes to it.
2. **The TUI-07 command palette is entirely dead code.** `command_dispatch.dispatch_command` is
   never imported or called from `app.py` or `main.py`; the composer's slash-command branch
   prints a static placeholder instead.
3. **The permission command group, even if wired, would be non-functional** — it hard-codes
   `surface_session_id=""`, which can never match a real approval row.

These three findings combine into a single structural problem: TUI-04 (streaming transcript),
TUI-06 (permission dialog reachability via the palette), and TUI-07 (command palette) are not
actually usable end-to-end in the shipped `atlas` / `atlas tui` entrypoint, despite each unit
passing its own isolated tests. De-brand (TUI-11), resume fail-closed (TUI-09), and the
permission-broker mapping logic itself (TUI-06's `permission_ui.py`) are sound.

## Critical Issues

### CR-01: Transcript reads from a persisted `surface_events` table that nothing writes to — production transcript is permanently empty

**File:** `services/agent-runtime/atlas_runtime/tui/transcript.py:36-58`, `services/agent-runtime/atlas_runtime/tui/app.py:172-183`, `infra/migrations/0018_surface_events.sql:1-23`

**Issue:** The locked CONTEXT decision ("Streaming / Refresh Model") is explicit: poll via
`audit_service.get_events_for_session` → `surface_events.normalize_surface_events` →
`replay_since(last_seq)`, **append-only**, and "Reuse the 10.3 contracts exactly — no new event
bus, no new normalizer." The 10.3 design doc confirms `surface_events.py` is a pure in-process
**read projection** with "no DB writes, no pub/sub, no new event bus" (`surface_events.py:8`).

Instead, migration `0018_surface_events.sql` creates a brand-new physical `surface_events`
table, and `transcript.py`'s `poll_and_render` / `_events_for_session` reads rows directly out of
that table via `SELECT ... FROM surface_events WHERE session_id = ?`. The module docstring
(`transcript.py:5-6`) explicitly — and falsely — claims this table is "the persisted projection
cache written by the surface-session run loop."

A full-repo search confirms there is no `INSERT INTO surface_events` anywhere in production code
(`atlas_runtime/`); the only `INSERT INTO surface_events` statement in the entire diff is the
test-only seed helper in `tests/tui/test_transcript_poll.py:16-21`. `app.py`'s background
`_poll_loop` (`app.py:172-183`) calls `transcript_module.poll_and_render(eff_conn, console,
session_id=session.id, last_seq=last_seq)` against the real connection every 0.5s forever — it
will always return `[]` because the table is never populated outside of unit tests that seed it
directly. The agent-invocation path that would presumably emit real events
(`_submit_to_agent`, `app.py:73-84`) is an explicit unimplemented no-op stub.

Net effect: the shipped native workbench has a transcript window that will never render a single
agent/tool/task event in production — TUI-04 ("streaming transcript") and TUI-05 (normalized
event rendering) are not met by the assembled system, even though `render_event` and
`poll_and_render` pass their isolated unit tests (which seed the table by hand).

**Fix:** Remove the persisted `surface_events` table and migration `0018` entirely, and rewire
`transcript.py`/`poll_and_render` to the locked design:
```python
def poll_and_render(
    conn: sqlite3.Connection,
    console: Console,
    *,
    session_id: str,
    last_seq: int = 0,
    run_outcome: Optional[RunOutcome] = None,
) -> list[SurfaceEvent]:
    audit_events = audit_service.get_events_for_session(conn, session_id)
    events = surface_events_module.normalize_surface_events(
        audit_events, run_outcome, session_id=session_id
    )
    return list(surface_events_module.replay_since(events, last_seq))
```
This also requires deciding how `run_outcome` (the terminal completion event) is threaded once
the real run loop is wired — out of scope for this phase per `app.py`'s own `_submit_to_agent`
docstring, but the table-based design must not ship as a stand-in; it actively contradicts the
locked decision and produces a non-functional feature, not merely an incomplete one.

### CR-02: TUI-07 command palette is unreachable dead code — `dispatch_command` is never called from the workbench

**File:** `services/agent-runtime/atlas_runtime/tui/app.py:204-212`, `services/agent-runtime/atlas_runtime/tui/command_dispatch.py` (entire module)

**Issue:** `run_workbench`'s composer loop branches on a leading `/`:
```python
if line.startswith("/"):
    # Slash-command dispatch lands in a later plan
    # (10.6-07's command_dispatch.dispatch_command);
    # no dispatcher exists yet in this plan's scope.
    console.print(f"(unrecognized command: {line})")
```
This comment is stale — plan 10.6-07 already landed `command_dispatch.py` (commit `3ec8fe4`),
but `app.py` was never updated to call it. Nowhere in `app.py` or `cli/main.py` is
`command_dispatch` imported or `dispatch_command` invoked (confirmed by repo-wide grep — the only
hit for `command_dispatch.dispatch_command` outside the module itself is this stale comment).
Every `/project`, `/focus`, `/config`, `/permission`, `/wiki`, `/subagent`, `/context` command
typed into the actual shipped workbench prints the static placeholder string, regardless of
whether the command is real, deferred, or garbage. The entire TUI-07 command-palette feature set
delivered in plan 08-07 is unreachable in the assembled product.

The unit tests in `tests/tui/test_command_dispatch.py` are sound (they correctly exercise
`dispatch_command` directly), but no test exercises `run_workbench`'s composer loop end-to-end
with a `/`-prefixed line, so this integration gap is invisible to CI.

**Fix:** Wire the dispatcher into the composer branch:
```python
if line.startswith("/"):
    result = command_dispatch_module.dispatch_command(eff_conn, line)
    console.print(result.text)
```
and add an integration test that drives `run_workbench`'s loop (or a refactored extraction of the
composer-line-handling logic into a separately-testable function) with a `/project list` input
and asserts the dispatcher's output reaches `console.print`.

### CR-03: `/permission` command group hard-codes an empty `surface_session_id` — non-functional even once wired

**File:** `services/agent-runtime/atlas_runtime/tui/command_dispatch.py:150-180`

**Issue:** `_handle_permission`'s `list` and `decide` subcommands both pass
`surface_session_id=""` to `permission_broker.list_actionable` / `permission_ui.resolve_approval_choice`:
```python
approvals = permission_broker.list_actionable(conn, surface_session_id="")
...
result = permission_ui.resolve_approval_choice(
    conn, approval_id=approval_id, surface_session_id="", choice=choice,
)
```
`permission_broker.list_actionable` filters with `WHERE surface_session_id = ?` bound to the
literal empty string (`permission_broker.py:232-236`); real surface sessions are created with
UUID-shaped ids (`surface_session_service.create_session`), so this predicate can never match a
real row. The command always reports "no actionable approvals" regardless of what is actually
pending for the active session. `decide` would similarly always be evaluated against the wrong
(nonexistent) session scope, so any decision attempted through this path that doesn't 404 inside
`tool_service._load` would still fail the broker's `WrongSessionError`/`NotActiveSessionError`
ownership checks.

The in-line comment acknowledges this is a known gap ("the composer-facing entry point is
expected to be extended with session context in the Wave 4 wiring pass"), but CR-02 confirms Wave
4 (`d0a0f88`, `a7eb0bd`, `6c95f3c`) never actually threaded session context through — the
dispatcher signature (`dispatch_command(conn, command_text)`) still carries no session parameter
at all. This is doubly dead: even if CR-02 is fixed and the dispatcher is wired into the
composer, `/permission list` and `/permission decide` remain non-functional until the session id
is threaded through.

**Fix:** Thread the active `surface_session_id` through `dispatch_command`'s signature (e.g.
`dispatch_command(conn, command_text, *, surface_session_id)`) and into the registry's `Handler`
type, then pass the real id from `app.py`'s `session.id` at the call site fixed in CR-02. Do not
ship `_handle_permission` as "scopes to empty session id, which resolves to no rows" — that
phrasing in the code comment frames a functional gap as a safety property; it is not a safety
mitigation, it is the command being broken.

## Warnings

### WR-01: `transcript.py` module/function docstrings assert false provenance for the `surface_events` table

**File:** `services/agent-runtime/atlas_runtime/tui/transcript.py:5-9, 39-41`

**Issue:** Independent of CR-01's functional bug, the docstrings make specific, false claims:
"the persisted projection cache written by the surface-session run loop" and "rows here are
already-normalized SurfaceEvents (one row per projected event)". No such writer exists anywhere
in the codebase. This will mislead the next engineer who reads the module in isolation (e.g.
during the CR-01 fix) into believing the table is populated by an existing, working pipeline
elsewhere, rather than realizing it needs to be removed.

**Fix:** When CR-01 is fixed (table removed, transcript rewired to `normalize_surface_events` +
`get_events_for_session`), this docstring naturally goes away. If for some reason the table-based
design is intentionally kept (it should not be, per the locked decision), the docstring must be
corrected to state plainly that no writer currently exists.

### WR-02: `_handle_mission` permanently dead-ends "list"/"show" with an error, contradicting its own registration as a working core group

**File:** `services/agent-runtime/atlas_runtime/tui/command_dispatch.py:115-126`

**Issue:** `_handle_mission` is registered in `COMMAND_REGISTRY` as a non-deferred core handler
(`_register("mission", _handle_mission)`, line 206), but its body for the only two subcommands it
recognizes (`list`, `show`) unconditionally returns `ok=False` with a message stating the feature
"is wired by the app.py mission-submission integration (Wave 4), not by command_dispatch
directly." That integration was never built (CR-02 confirms `app.py` never calls
`command_dispatch` at all). This handler is registered as a "core" group per the CONTEXT
decision's must-have list, but every code path through it returns failure — it should have been
registered with `deferred=True` like `wiki`/`subagent`/`context`, or removed, rather than
presented as delivered-but-broken.

**Fix:** Either implement read-only `mission list`/`mission show` against `mission_service`
directly (consistent with `_handle_project`'s pattern), or register it via
`_make_deferred_handler("mission")` until the Wave-4 integration referenced in the comment
actually exists.

### WR-03: `test_headless_missing_channel_fails_closed` asserts only `Exception`, not the specific typed error

**File:** `services/agent-runtime/tests/tui/test_permission_ui.py:100-112`

**Issue:** The test asserts `with pytest.raises(Exception):` rather than
`pytest.raises(permission_broker.ApprovalChannelMissingError)`. The source code
(`permission_ui.py:165-168`) does raise the specific typed error, so this is not currently
masking a real defect, but the bare `Exception` assertion would silently continue to pass if a
future refactor accidentally raised a different, unrelated exception (e.g. an `AttributeError`
from a typo) instead of the intended fail-closed error — defeating the purpose of this
regression test.

**Fix:**
```python
with pytest.raises(permission_broker.ApprovalChannelMissingError):
    resolve_approval_choice(..., headless=True)
```

### WR-04: `dispatch_command`'s catch-all `except Exception` in the handler-call path can mask programming errors as user-facing command failures

**File:** `services/agent-runtime/atlas_runtime/tui/command_dispatch.py:253-256`

**Issue:** The broad `except Exception as exc: return CommandResult(ok=False, text=f"command
'{command}' failed: {exc}")` is intentional and documented (T-10.6-22, never let one bad command
kill the asyncio loop) — that goal is legitimate. However, as written it also silently swallows
genuine bugs in handler code (e.g. a `TypeError` from a signature mismatch, or a future handler
that accidentally raises `KeyboardInterrupt`-adjacent control-flow exceptions) and reports them
identically to expected domain errors, with no logging. An operator sees the same generic
"command failed" text whether the command was malformed or the handler itself has a bug, and
there is no audit trail to distinguish the two after the fact.

**Fix:** At minimum, log the exception (e.g. via the existing audit/logging convention used
elsewhere in the codebase) before returning the `CommandResult`, so a real handler bug is
distinguishable from a textbook the typed-error path during later debugging:
```python
except Exception as exc:  # noqa: BLE001 - last line of defense, see T-10.6-22
    logger.exception("tui command dispatch failed: %s", command)
    return CommandResult(ok=False, text=f"command '{command}' failed: {exc}")
```

## Info

### IN-01: `_poll_loop`'s `last_seq` recompute uses `max(last_seq, *(...))` where the unpacked generator could be more directly expressed

**File:** `services/agent-runtime/atlas_runtime/tui/app.py:181-182`

**Issue:** `last_seq = max(last_seq, *(event.seq for event in events))` is correct (guarded by
`if events:` so the generator is never empty), but it is a slightly obscure way to express "take
the max seq seen so far." Not a bug, purely a readability nit.

**Fix:** Optional simplification: `last_seq = max(last_seq, events[-1].seq)` since `events` is
already guaranteed ascending-by-seq via `replay_since`'s pass-through ordering — slightly clearer
intent (last event's seq) than an unpacked-generator `max()`.
