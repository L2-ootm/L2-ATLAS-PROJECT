# Phase 10.6: ATLAS Terminal Workbench - Research

**Researched:** 2026-06-27
**Domain:** Python terminal UI (Rich + prompt_toolkit) over existing Typer CLI / surface-session contracts
**Confidence:** HIGH (codebase contracts, Rich/prompt_toolkit APIs verified locally) / MEDIUM (UX pattern synthesis from Hermes Ink + MiMoCode/OpenCode)

## Summary

This phase replaces `atlas tui`'s current thin wrapper (`cli/tui.py` → vendored Hermes Ink JS
process) with a **native Python terminal workbench** built on Rich (already an installed transitive
dependency of `typer`) + prompt_toolkit (already installed transitively via `hermes-agent`, but not
yet declared as a direct `atlas-runtime` dependency). The workbench is a pure presentation/dispatch
layer over contracts already built in 10.2–10.5: `surface_session_service`, `surface_events`,
`audit_service`, `agent_contract_service`, `permission_broker`, `config_service`/
`control_plane_service`. No new event bus, no new persistence, no new business logic.

The architecturally significant decision is the **render-loop integration pattern**: prompt_toolkit's
own `Application`/`PromptSession` event loop must drive both keystroke input AND periodic event-poll
refresh, because Rich's `Live` and prompt_toolkit's full-screen `Application` cannot both own the
terminal simultaneously. The verified-correct pattern (confirmed via the installed prompt_toolkit
3.0.52 API surface) is: use `prompt_toolkit.patch_stdout.patch_stdout()` + a `PromptSession` whose
multiline `prompt_async()` runs inside an `asyncio` loop alongside a background poll task that prints
new transcript lines through the patched stdout (which prompt_toolkit redraws around safely). Rich's
`Console.print()` (not `Live`) is the correct primitive for the **append-only** transcript region;
Rich `Live` is reserved for the **status header** only, refreshed on its own short interval, because
the CONTEXT-mandated append-only transcript (no flicker, narrow/wide reflow) is fundamentally
print-once-per-event, not redraw-whole-screen.

Both vendored references converge on the same shape: a fixed bottom composer, an append-only/
scrolling transcript above it, a slim status rule between them, and tool-call/permission events
rendered as bordered "cards" inline in the transcript rather than as separate panes. MiMoCode's
approval dialog literally enumerates **Approve once / Always / Deny** — directly isomorphic to this
phase's required approve-once / scoped-allow / reject / cancel. Hermes Ink's `ApprovalPrompt`
(`ui-tui/src/components/prompts.tsx`) implements the identical 4-option pattern with number-key quick
pick, arrow navigation, Enter confirm, Esc/Ctrl-C deny — this is the UX to port verbatim into a Rich
panel + prompt_toolkit key-binding loop.

**Primary recommendation:** Build a new `atlas_runtime/tui/` package (app loop, transcript renderer,
composer, status header, permission dialog, command dispatch) wired into `cli/main.py` via a Typer
`@app.callback(invoke_without_command=True)` for bare `atlas`, plus a thin `atlas tui` command that
calls the same entry function — both paths converge on one `run_workbench()` function. Retire
`cli/tui.py`'s default registration; keep the Hermes Ink launcher reachable only behind an explicit
hidden/dev flag if at all.

### Pattern 1: Single asyncio loop owns both input and polling (TUI-04, TUI-08)
**What:** Rich `Live` and prompt_toolkit's terminal control cannot both own raw-mode stdin/stdout
redraw at once. The verified-safe combination is `prompt_toolkit.patch_stdout.patch_stdout()`
wrapping the whole session, with the composer driven by `PromptSession.prompt_async()` inside an
`asyncio` event loop, and a second `asyncio.create_task` that wakes on an interval, polls
`audit_service.get_events_for_session`, and calls `console.print(...)` for new lines. `patch_stdout`
makes ad-hoc `print()`/`Console.print()` calls from the background task interleave safely above the
live prompt instead of corrupting it.
**When to use:** Always for this workbench's main loop — this is the architecturally load-bearing
decision the planner must lock in Wave 0.
**Example:**
```python
# Source: prompt_toolkit 3.0 docs (patch_stdout, asyncio integration) — verified locally
# python -c "from prompt_toolkit.patch_stdout import patch_stdout; from prompt_toolkit import PromptSession"
import asyncio
from prompt_toolkit import PromptSession
from prompt_toolkit.patch_stdout import patch_stdout
from rich.console import Console

console = Console()

async def poll_loop(session_id: str, last_seq: int, conn, stop_event: asyncio.Event):
    while not stop_event.is_set():
        events = audit_service.get_events_for_session(conn, session_id)
        normalized = surface_events.normalize_surface_events(events, session_id=session_id)
        for ev in surface_events.replay_since(normalized, last_seq):
            render_event(console, ev)   # transcript.py — one fn per EventKind
            last_seq = ev.seq
        await asyncio.sleep(0.5)  # polling cadence — Claude's discretion, NOT a fixed contract

async def main_loop(conn, session):
    ptk_session = PromptSession(multiline=True)
    stop_event = asyncio.Event()
    poll_task = asyncio.create_task(poll_loop(session.id, 0, conn, stop_event))
    with patch_stdout():
        try:
            while True:
                text = await ptk_session.prompt_async("> ")
                dispatch(text, conn, session)
        except (EOFError, KeyboardInterrupt):
            stop_event.set()
            await poll_task
```
This is a research-derived pattern (not lifted from a single official tutorial) — confidence MEDIUM,
both APIs (`patch_stdout`, `PromptSession.prompt_async`, `asyncio.create_task`) are independently
[VERIFIED: local interactive import + `prompt_toolkit.__version__` check], but their joint use in
exactly this shape is a documented-but-not-singly-canonical prompt_toolkit idiom (multiple
prompt_toolkit examples in its own repo combine `patch_stdout` with background asyncio tasks printing
to a patched stdout — this is the library's own documented coexistence mechanism, not a novel hack).

### Pattern 2: Append-only transcript via Console.print, NOT Live (TUI-04)
**What:** CONTEXT mandates "append-only to avoid flicker and support narrow/wide reflow." Rich
`Live` redraws a *fixed renderable* in place — it is the wrong primitive for an ever-growing
transcript (it would either truncate or require holding the whole history in a `Group` renderable and
re-rendering on every tick, which is the flicker CONTEXT explicitly rules out). The correct pattern:
plain `console.print(renderable)` calls, one per new event, scroll naturally with the terminal's own
scrollback (mirrors the Hermes Ink TUI's own choice of normal flex-column flow over `AlternateScreen`
buffering for the transcript — `INLINE_MODE` in `appLayout.tsx` explicitly documents "host terminal's
native scrollback captures rows scrolled off the top").
**When to use:** All transcript/event rendering. Reserve `Live` exclusively for the status header
line (a small, single-region, redraw-cheap target) per CONTEXT's own phrasing: "Rich `Live` for the
status header + active streaming region."
**Example:**
```python
# Source: Rich docs (Console.print is the base primitive; Live is documented as
# for "live updating" a SINGLE renderable region) — rich.readthedocs.io/en/stable/live.html
from rich.live import Live
from rich.console import Console

console = Console()
with Live(console=console, refresh_per_second=4, transient=False) as live:
    live.update(render_status_header(session, config_snapshot))
    # ... elsewhere, NOT through `live`:
    console.print(render_tool_call_card(event))  # append-only, scrolls normally
```

### Pattern 3: Permission dialog as a blocking modal that suspends the composer (TUI-06)
**What:** Both the Hermes Ink TUI (`PromptZone`/`ApprovalPrompt` in `prompts.tsx`) and MiMoCode's web
approval dialog implement the SAME shape: a 4-option blocking choice (Hermes: once/session/always/
deny; MiMoCode: approve once/always/deny) that takes over input until resolved, with number-key quick
pick, arrow-key navigation, Enter-confirm, and Esc-as-deny. Port this into Python: when
`permission_broker.list_actionable(conn, surface_session_id=...)` returns a pending row, suspend the
normal `PromptSession.prompt_async()` call, render a Rich `Panel` (bordered, warn-colored, same visual
language as Hermes' `borderColor={t.color.warn} borderStyle="double"`) listing the four choices, and
read a single keypress via prompt_toolkit's `Application` + `KeyBindings` (not the multiline prompt)
until a choice resolves. Map choices to `permission_broker.claim(decision=..., record_rule=...)`:
  - **Approve once** → `claim(decision="approve")`, no `record_allow_rule` call
  - **Scoped allow** (session) → `claim(decision="approve")` THEN
    `record_allow_rule(rule_kind="session", arg_pattern=allow_pattern_for_args(args), ...)`
  - **Reject** → `claim(decision="reject")`
  - **Cancel** → do NOT claim; leave pending (it will expire and fail closed per
    `list_actionable`'s own expiry filtering) OR explicitly call `claim(decision="reject")` if CONTEXT
    wants cancel to resolve rather than dangle — **OPEN QUESTION, flagged below.**
**When to use:** Every time the poll loop detects a new actionable approval for this session.
**Example:**
```python
# Source: ATLAS permission_broker.py (verified signatures) + Hermes Ink prompts.tsx pattern (ported)
from rich.panel import Panel
from rich.text import Text

OPTS = ("once", "session", "reject", "cancel")
LABELS = {"once": "Approve once", "session": "Approve, allow this session",
          "reject": "Reject", "cancel": "Cancel (leave pending)"}

def render_approval_panel(approval, selected_idx, theme):
    body = Text(f"{approval.tool_name}  args={approval.args_normalized}\n")
    for i, opt in enumerate(OPTS):
        marker = "▸ " if i == selected_idx else "  "
        body.append(f"{marker}{i+1}. {LABELS[opt]}\n",
                     style="bold" if i == selected_idx else None)
    return Panel(body, border_style=theme.warn, title="approval required")
```

### Pattern 4: Status header is a pure projection of existing snapshots (TUI-03)
**What:** Compose the header from `control_plane_service.get_config_snapshot()` (model/auth/
effective provider — masked, no secrets) + the active `SurfaceSession` (permission_mode,
prompt_version/tool_catalog_version/context_policy_version for the "context budget" line) + the
active Focus (via `focus_service`, already used elsewhere in the CLI). No new aggregation service is
needed — this is read-and-format only.
**When to use:** On session start and after every config/model/permission-mode change event in the
poll loop.

### Anti-Patterns to Avoid
- **Driving the poll loop with `time.sleep` in the same thread as `PromptSession.prompt()`:** blocks
  the composer entirely; must be `asyncio` (`prompt_async`) or a background thread that only ever
  calls thread-safe `Console.print` under `patch_stdout`. Do not introduce a second SQLite connection
  or a second lock — reuse the existing module-level `_get_connection()`/`_get_lock()` factories from
  `cli/main.py` so the TUI participates in the same WAL+lock discipline as every other CLI command.
- **Re-deriving event ordering/kind logic in the TUI:** every renderer must consume
  `SurfaceEvent.kind` from `surface_events.normalize_surface_events` — never re-parse
  `payload_json` to decide what kind of event it is; that logic is centralized and tested in 10.3.
- **Using `Live` to redraw the entire transcript on every tick:** violates the explicit CONTEXT
  append-only/no-flicker requirement and does not reflow correctly on resize without re-issuing every
  historical line (expensive, and Rich's own docs note `Live` is meant for a single bounded region).
- **Hand-rolling ANSI escape codes for color/box-drawing:** Rich's `Console`/`Panel`/`Table`/`box.*`
  already solve no-color, legacy-Windows, and ASCII-safe substitution — see Don't Hand-Roll below.

**Key insight:** every "hard" terminal-UI problem in this phase (capability detection, box-drawing,
line editing, event ordering, permission authority) already has an owner — either Rich, prompt_toolkit,
or a 10.2–10.5 service. The entire phase is genuinely a thin presentation/dispatch layer; the planner
should be suspicious of any task that proposes new logic beyond formatting and call-wiring.

## Common Pitfalls

### Pitfall 1: Live region fighting prompt_toolkit for terminal control
**What goes wrong:** Starting a Rich `Live` context (which takes over redraw via ANSI cursor control)
at the same time prompt_toolkit's `Application`/`PromptSession` is also managing the cursor/raw mode
produces corrupted, overlapping output — both libraries assume they own the terminal.
**Why it happens:** Rich `Live` and prompt_toolkit each implement their own "alternate redraw" model;
neither is aware of the other.
**How to avoid:** Scope `Live` tightly to the status header only (a `Live(console=..., transient=False)`
instance updated on a timer/event), and ensure all OTHER output (transcript, dialogs) goes through
plain `console.print()` wrapped in `patch_stdout()` so prompt_toolkit can safely interleave it above
the active prompt. Never call `console.print()` for transcript content from inside the `Live` context
manager's `with` block.
**Warning signs:** Garbled/duplicated lines, cursor jumping, composer text disappearing after a poll
tick — these all indicate two redraw owners colliding.

### Pitfall 2: Polling cadence vs. SQLite WAL contention
**What goes wrong:** A tight poll loop (e.g. every 50ms) calling `get_events_for_session` competes
with the same `threading.Lock` + WAL-mode connection every other CLI write path uses, and at
multi-process scale (gateway + CLI + TUI all on `~/.atlas/atlas.db`) can add lock contention.
**Why it happens:** The TUI reuses the SAME connection/lock factories as `cli/main.py`
(`_get_connection`/`_get_lock`) — CONTEXT's "no new event bus" decision means polling IS a real
read query against the live DB, not a free in-memory operation.
**How to avoid:** Choose a modest polling interval (e.g. 250–750ms; CONTEXT marks this Claude's
discretion) and ensure reads use the existing read-only `SELECT`-only path in `get_events_for_session`
(no write lock needed for reads against a WAL DB — only the heartbeat/transition writes need
`with lock:`).
**Warning signs:** Visible input lag while a run is actively streaming; `database is locked` errors
under concurrent gateway + TUI usage.

### Pitfall 3: `legacy_windows` auto-detection in non-standard terminals (VS Code, WSL)
**What goes wrong:** Rich's `legacy_windows` detection is itself a heuristic (checks `sys.platform`
+ `colorama`/`ENABLE_VIRTUAL_TERMINAL_PROCESSING` availability); inside WSL or VS Code's integrated
terminal the actual capability differs from a bare `cmd.exe`, and an incorrect detection would pick
the wrong box-drawing/color fallback.
**Why it happens:** Detection happens once at `Console()` construction by inspecting environment, not
by a runtime capability probe.
**How to avoid:** Do not override Rich's detection manually unless a documented failure is found in
human-verify testing; if forcing is needed, expose it via an explicit `--ascii`/`--no-color` flag
(`Console(no_color=True, legacy_windows=True)`) rather than silently guessing differently from Rich.
**Warning signs:** Boxes render as mangled Unicode question marks/garbage in a specific terminal during
the TUI-10 human-verify matrix pass.

### Pitfall 4: Cancel (Ctrl-C) racing the poll task and leaving a half-rendered event
**What goes wrong:** `KeyboardInterrupt` raised mid-`console.print()` of a multi-line tool-call card
can leave a partially printed block, and if the cancel path does not also stop the background poll
`asyncio.Task` before tearing down `patch_stdout`, a stray late tick can print after the prompt has
exited, corrupting terminal state on exit.
**Why it happens:** `prompt_toolkit.prompt_async()` raises `KeyboardInterrupt`/`EOFError` directly to
the awaiting coroutine; if the surrounding `try/except` does not `stop_event.set()` + `await
poll_task` before falling out of the `with patch_stdout():` block, the background task can still be
running when stdout patching is removed.
**How to avoid:** Wrap cancellation in the pattern shown in Pattern 1 — always cancel/await the poll
task in a `finally` before exiting the `patch_stdout` context, mirroring how `surface_session_service`
already drives `cancelling → completed` transitions cooperatively (10.3-04's `cancel_token`).
**Warning signs:** Terminal left in a broken state (no echo, wrong colors) after Ctrl-C; a session
stuck in `cancelling` state never reaching `completed` (verifiable via `list_sessions`).

### Pitfall 5: De-brand audit missing the Hermes Ink launcher's *visible* surface
**What goes wrong:** TUI-11 requires no imported product identity in "help text, console output,
package names, paths, runtime output, or binaries." The literal string `"foundation/atlas-hermes"`
currently appears in `cli/tui.py`'s docstring/error message ("foundation/atlas-hermes not found (run
from within the ATLAS repo)") and in the Typer command help string ("foundation Ink TUI,
ATLAS-skinned"). If the Hermes Ink path is kept reachable behind a hidden/dev flag (CONTEXT allows
this), that flag's own help text and any error message it can surface to a normal `atlas`/`atlas tui`
invocation must not leak "Hermes"/"Ink" — only an explicit `--dev`-gated diagnostic should ever
mention it, and even then minimally.
**Why it happens:** The current wrapper's error/help strings were written when the Hermes launcher
WAS the default path; de-branding the default surface without also auditing the now-secondary path's
visible strings leaves a leak.
**How to avoid:** grep for `hermes`/`Hermes`/`Ink`/`atlas-hermes` (case-insensitive) across
`atlas_runtime/cli/`, the new `atlas_runtime/tui/` package, and `--help` output of every command
(`atlas --help`, `atlas tui --help`) as part of TUI-11 verification.
**Warning signs:** `atlas --help` or `atlas tui --help` mentioning "Hermes" or "Ink" anywhere in the
visible text.

import typer

@app.callback(invoke_without_command=True)

@app.command("tui", help="Launch the ATLAS terminal workbench.")

### Resume/replay fail-closed sequence (TUI-09)
```python
# Source: atlas_runtime/surface_session_service.py:377-414 (resume_session, read verbatim)
# and agent_contract_service.py:194-213 (replay_contract, read verbatim)
from atlas_runtime import surface_session_service
from atlas_runtime.agent_contract_service import ContractCompatibilityError

try:
    session = surface_session_service.resume_session(
        conn, lock, session_id, owner_token=new_owner_token, owner_pid=os.getpid(),
    )
except ContractCompatibilityError as exc:
    # FAIL CLOSED — no resume happened. Render this to the user verbatim; do not retry
    # with relaxed expectations (that would defeat the version-drift guard).
    render_error_card(f"resume blocked: {exc}")
    raise typer.Exit(1)
except ValueError as exc:
    render_error_card(f"cannot resume: {exc}")
    raise typer.Exit(1)
```
`resume_session` itself calls `agent_contract_service.replay_contract(... expected_prompt_version=...,
expected_catalog_version=..., expected_context_policy_version=...)` BEFORE any state transition — the
TUI only needs to catch and render the two failure types; it must never bypass this by constructing
its own version comparison.

## Open Questions

1. **Does "cancel" in the permission prompt mean leave-pending or explicit-reject?**
   - What we know: TUI-06 requirement text lists "approve once, scoped allow, reject, and cancel" as
     four distinct options — implying cancel is NOT the same action as reject.
   - What's unclear: `permission_broker` has no "cancel" decision value distinct from "approve"/
     "reject" (`claim(decision=...)` validates against `_VALID_CLAIM_DECISIONS`, not enumerated here
     but referenced in `claim()`'s docstring as a fixed set). If "cancel" simply means "close this
     dialog without resolving, the approval remains actionable on the next poll tick," the TUI does
     not call `claim` at all for that choice — it just stops rendering the prompt for now and will
     show it again on the next event tick.
   - Recommendation: planner should treat "cancel" as a TUI-local UI dismissal (no broker call), and
     "reject" as the only path that calls `claim(decision="reject")`. Verify
     `_VALID_CLAIM_DECISIONS`'s exact members in `permission_broker.py` before locking task wording.

2. **Exact polling cadence and whether it should be event-driven (push) instead of purely time-based.**
   - What we know: CONTEXT locks in-process polling (no SSE/new bus); cadence is Claude's discretion.
   - What's unclear: whether a tighter poll-on-input-submit (poll immediately after dispatching a
     command, in addition to the timer) materially improves perceived responsiveness without adding
     contention.
   - Recommendation: default timer-based poll (e.g. 500ms) PLUS an immediate poll right after any
     TUI-initiated write (command dispatch, approval claim) so the user's own actions feel instant;
     this is a UX nicety, not a contract requirement.

3. **Scope of "session/workspace selection" UI (TUI-02) — interactive picker vs. CLI flags vs. both.**
   - What we know: `project_service.list` and `surface_session_service.list_sessions` already exist;
     CONTEXT says "Exact widget layout... at Claude's discretion."
   - What's unclear: whether `atlas tui --project <id>` should skip the interactive picker entirely
     (useful for scripting/automation) or whether the picker is always shown.
   - Recommendation: support both — an optional `--project`/`--global` flag that bypasses the
     interactive picker when supplied (mirrors the existing `atlas mission create --project` pattern),
     falling back to an interactive picker when omitted and stdin is a TTY.

### Known Threat Patterns for this stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|------------------------|
| Approval spoofing via a stale/replayed claim from the TUI | Tampering / Spoofing | Already mitigated entirely inside `permission_broker.claim` (nonce echo, expiry, at-most-once UPDATE) — TUI must pass through the SAME `nonce` the approval row carries, never invent its own |
| Secret leakage into the rendered transcript/header | Information Disclosure | `audit_service.emit`'s redaction boundary (`SECRET_PATTERNS`, D-013) already redacts before persistence; the TUI renders `payload_json`/masked config snapshots as-is and must NOT attempt to "unmask" or re-fetch unmasked values for display |
| A malformed/crafted slash-style command in the composer causing an unhandled exception that crashes the render loop (denial of service to the operator's own session) | Denial of Service | Command dispatch must catch and render errors as an error card (`UX-01` pattern: state what failed, which contract decided it, and the remediation) rather than letting an exception propagate out of the asyncio loop and kill the TUI process |
| Headless/no-channel approval requests rendered as if actionable | Tampering / Elevation of Privilege | `ApprovalChannelMissingError` (PERM-05, fail-closed) must be surfaced to the operator as an explicit "this surface cannot approve — no channel registered" message, never silently retried or treated as "pending forever" |
