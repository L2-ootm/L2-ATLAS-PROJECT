## Decisions Made

- **Ground-truth divergence (test vs. plan prose):** The plan's `<action>` text described `resolve_approval_choice(conn, lock, *, approval, surface_session_id, choice, nonce)` taking a `ToolApproval` object, an explicit `lock`, and an explicit `nonce`. The actually-committed `tests/tui/test_permission_ui.py` (ground truth per the execution contract) calls `resolve_approval_choice(db, approval_id=approval_id, surface_session_id=session_id, choice="approve_once")` — passing only `approval_id` (a string) and no `lock`/`nonce`. Implemented to match the test: the function loads the `ToolApproval` itself via `tool_service._load`, owns a module-level `_LOCK`, and reads `nonce` off the loaded row. This is the SEC-02-correct behavior anyway (the TUI should never invent or cache a nonce independent of the persisted row).
- **Choice vocabulary:** The test uses `"approve_once"` / `"allow_scoped"` while the plan's `_OPTS` constant uses `"once"` / `"session"`. Added a `_CHOICE_ALIASES` map so both vocabularies resolve identically rather than picking one and breaking the other description.
- **Capabilities field name:** Used the real `Capabilities.box_style` (`"ascii"`/`"unicode"`) field instead of the plan prose's assumed `caps.ascii_only` boolean — `capabilities.py` (read during execution) does not define `ascii_only`.
