# Phase 10.6 Plan 07: TUI Command Dispatch (TUI-07 core-first) Summary

Registry-based `dispatch_command(conn, command_text)` that parses composer slash-command
strings and routes core groups (project/focus/mission/config/permission) to existing service
functions, with wiki/subagent/context registered as explicit deferred extension seams.

## What Was Built

`services/agent-runtime/atlas_runtime/tui/command_dispatch.py`:

- `CommandResult` — frozen dataclass (`ok: bool`, `text: str`, `deferred: bool = False`), with
  an `is_error` property (`not ok`) to satisfy both vocabularies callers might use.
- `COMMAND_REGISTRY` — `dict[str, _RegistryEntry]` built via a single `_register(name, handler,
  *, deferred=False)` helper. Core entries: `project`, `focus`, `mission`, `config`,
  `permission`. Deferred entries: `wiki`, `subagent`, `context`.
- `dispatch_command(conn, command_text)` — parses the slash-command string (`_parse`, strips
  a leading `/`, shlex-splits), looks up the command in `COMMAND_REGISTRY`, and either returns
  an unknown-command `CommandResult` or calls the handler inside an outer `try/except Exception`
  that converts any handler bug into a `CommandResult(ok=False, ...)` rather than raising
  (T-10.6-22).
- Core handlers delegate immediately to `project_service`, `focus_service`,
  `config_service.get_value`/`set_value`, `permission_broker.list_actionable`, and
  `tui.permission_ui.resolve_approval_choice` — no new business logic.
- `mission` handler explicitly defers full run-dispatch to the Wave 4 app.py
  agent-submission integration per the plan's instruction not to duplicate that path.
- Deferred handlers (`wiki`/`subagent`/`context`) return `CommandResult(ok=False,
  deferred=True, text=...)` naming the group and stating it is deferred to a follow-up phase —
  distinguishable from the unknown-command message.

## Known Stubs

- **`permission` command group `surface_session_id` scoping** —
  `services/agent-runtime/atlas_runtime/tui/command_dispatch.py:_handle_permission` currently
  passes `surface_session_id=""` to `permission_broker.list_actionable` and
  `permission_ui.resolve_approval_choice` because `dispatch_command`'s signature (matching the
  committed test) does not carry session context. This is documented inline in the code as a
  fail-closed placeholder (empty session id yields zero rows under the broker's strict-scope
  filter, never another session's approvals) pending Wave 4's app.py wiring, which is expected
  to either extend `dispatch_command`'s signature or bind session context via a closure when
  registering composer key bindings. Not a security regression (fails closed), but the
  `permission decide` subcommand is not yet exercised by any test and should be covered when
  Wave 4 wires the composer to a live `surface_session_id`.
- **`mission` command group** — intentionally returns an informational, non-deferred `ok=False`
  message pointing at the Wave 4 app.py integration; this is per-plan scope (not a stub to
  resolve in this plan), not a TUI-07-deferred group, and not covered by the committed test
  suite (no test exercises `mission` in this plan).
