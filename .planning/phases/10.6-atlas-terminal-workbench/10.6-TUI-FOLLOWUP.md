# 10.6 TUI — Follow-up: Live-Launch Defects + MiMo-Code SOTA Upgrade

**Date:** 2026-06-27
**Trigger:** First real interactive launch of `atlas tui` (operator, pre-demo).
**Status of phase 10.6:** functionally "complete" on paper (45 unit tests green)
but the interactive composer had **never been run live** — the launch was
env-gated to `10.6-HUMAN-UAT.md`. Running it surfaced defects the unit tests
had mocked away.

---

## 1. Defects found and fixed this session

All three were in the *interactive* path only (gateway + cockpit unaffected).
Each was hidden because the unit tests stubbed out the exact primitive that was
broken — the recurring 10.6 lesson: **per-module green ≠ assembled product
works.**

| # | File | Defect | Fix |
|---|------|--------|-----|
| 1 | `atlas_runtime/tui/app.py` (`_main_loop`) | `async with patch_stdout():` — prompt_toolkit's `patch_stdout` is a **synchronous** context manager (`__enter__`, no `__aenter__`). Crashed on launch: `TypeError: '_GeneratorContextManager' object does not support the asynchronous context manager protocol`. | `with patch_stdout():` |
| 2 | `tests/tui/test_composer_routing.py` | Test stub `_NullAsyncCtx` was **async-only** — it was written to match the buggy `async with`, which is *why the bug shipped green*. | Replaced with sync `_NullSyncCtx` (`__enter__`/`__exit__`) mirroring the real library. |
| 3 | `atlas_runtime/tui/app.py` (`_main_loop`) | Rich `Live(refresh_per_second=4)` wrapped around the `prompt_async` loop inside `patch_stdout`. `Live` repaints on a timer; co-owning the terminal with `patch_stdout` flooded the screen with repeated headers and leaked raw cursor/erase escape codes. **Directly violated the module's own docstring** ("Live never co-owns the terminal with patch_stdout"). The snapshot was static anyway (`live.update` called once) — `Live` added zero value. | Removed `Live` entirely; header now printed once via `header.render_status_header(...)` under `patch_stdout`. Deleted dead `from rich.live import Live` import + `_render_header_for_live` adapter; corrected both docstrings. |

**Verification:** `tests/tui/` → 40 passed; module imports clean. Full visual
confirmation still requires a real TTY (cannot be driven from a non-interactive
harness).

## 2. Residual known issues (NOT yet fixed — superseded by §3)

The current workbench now launches and is *usable*, but the operator verdict is
that it is **well below the ATLAS quality bar** ("looks like pure trash"). Open
defects/limitations:

- **Rich SGR escapes render literally** — the header shows `?[1mATLAS?[0m`
  instead of bold "ATLAS". Probable cause: Rich-emitted ANSI SGR codes are
  written through prompt_toolkit's `StdoutProxy` on Windows where VT processing
  is not applied to that proxied stream. Needs confirmation. **Do not invest in
  fixing on the current stack** — see §3.
- **Rudimentary single-line UX** — a one-line status header + a bare `>` prompt.
  No panes, no scrollback transcript region, no syntax/diff rendering, no
  status/permission affordances, no theming to the L2 design system. Far from
  the MiMo-Code / opencode bar.
- The Rich + prompt_toolkit combination is fundamentally fragile on Windows
  (two libraries contending for terminal control). This is an architectural
  ceiling, not a polish gap.

## 3. Decision: bring the MiMo-Code TUI to reach SOTA

**Intent (operator directive, 2026-06-27):** replace/upgrade the ATLAS terminal
workbench so it reaches current state-of-the-art, using **XiaomiMiMo/MiMo-Code**
as the reference/donor.

Provenance is already established (Phase 10.1 intake):
- Upstream: `https://github.com/XiaomiMiMo/MiMo-Code` v0.1.2, commit
  `86d95a79bf0879bcb442ffe6b12914f6d8e68a4e`, MIT.
- Reference checkout (gitignored): `_EXTERNAL_REPOS/mimo-code/`.
- Attribution/notices: `ATTRIBUTION.md` (§"MiMo-Code Terminal UI Reference"),
  `docs/third-party/ATLAS_TUI_UPSTREAM_NOTICE.md`,
  `docs/imports/ATLAS_TUI_PROVENANCE.md`, `docs/imports/ATLAS_TUI_INTAKE_DECISIONS.md`,
  `docs/imports/ATLAS_TUI_SOURCE_INVENTORY.csv`.
- Note: MiMo-Code carries a `USE_RESTRICTIONS.md` beyond the MIT text — review
  before any vendoring decision.

**Stack reality to design around:**
- MiMo-Code is a **TypeScript/Bun** monorepo (`packages/opencode`, TS-only in
  this fork). Upstream opencode's celebrated TUI is **Go + Charm/BubbleTea**.
- Either way it is **not Python**, so this cannot be a Rich rewrite. Per **D-022**
  (Python only for the Hermes foundation surface; product surfaces are
  Rust/Zig or vendored sidecars), the right shape is almost certainly a
  **TUI sidecar** that talks to the ATLAS gateway/CLI over the existing
  surface-session + permission-broker contracts — the same pattern as the
  Discord and Cashflow sidecars.

**Open design questions (for a dedicated phase — candidate v1.2):**
1. Adapter vs. vendor: thin MiMo/opencode TUI binary as a sidecar driving the
   ATLAS surface-session protocol, vs. a from-scratch Charm/BubbleTea (Go) or
   Rust (ratatui) workbench that *reimplements* MiMo's patterns under L2 brand.
   v1.2-ROADMAP-DRAFT already constrains "external MiMo/OpenCode/Codex
   integrations are adapters only — no donor runtime/config/auth/identity
   imported." That constraint points to **reimplement-patterns**, not vendor the
   donor's TUI wholesale.
2. Contract surface: the SOTA TUI must consume the 10.3 surface-session, 10.5
   permission-broker, and 10.6 transcript/audit-event normalization already
   built — those are the integration seams; do not rebuild them.
3. Branding: must apply the L2 design-system tokens (Electric Violet / Cyber
   Blue / Titanium White, HUD voice) — the CLI skin engine already encodes these
   for the Hermes CLI; the new TUI should match.
4. Disposition of the current Rich/prompt_toolkit workbench: keep as a
   minimal fallback, or retire once the sidecar lands.

**Do NOT** spend further effort polishing the Rich/prompt_toolkit workbench
beyond keeping it launchable — it is interim and slated for replacement.
