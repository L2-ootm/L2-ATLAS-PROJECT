### CR-01 — Transcript reads a persisted `surface_events` table nothing writes to

**Re-verified: RESOLVED.**
- `infra/migrations/0018_surface_events.sql` — confirmed absent (`ls` returns nothing).
- Repo-wide grep for `FROM surface_events`, `INSERT INTO surface_events`, `CREATE TABLE ... surface_events` across `atlas_runtime/` and `infra/migrations/` — zero matches.
- `atlas_runtime/tui/transcript.py:41-69` (`poll_and_render`) now calls `audit_service.get_events_for_session(conn, session_id)` -> `surface_events_module.normalize_surface_events(audit_events, run_outcome, session_id=session_id)` -> `surface_events_module.replay_since(events, last_seq)` — exactly the locked CONTEXT.md design, no DB read of a persisted projection table.
- Off-by-one fix verified: `last_seq` default is `-1` in both `transcript.poll_and_render` (line 46) and `app.py`'s `_poll_loop` (line 209), correctly accounting for `normalize_surface_events` assigning the first event `seq=0`.
- `tests/tui/test_transcript_poll.py` seeds via `audit_service.emit(...)` (real write path), not a hand-seeded table — confirmed by reading the file (lines 37, 45, 75).
- Module docstring (WR-01) no longer claims a "persisted projection cache written by the surface-session run loop" — current docstring (lines 1-20) explicitly states "There is no separate persisted `surface_events` table — the projection is recomputed from the immutable `audit_events` ledger on every poll."

### CR-02 — `dispatch_command` never called from the workbench (dead code)

**Re-verified: RESOLVED.**
- `atlas_runtime/tui/app.py:243-247` — the composer loop's `if line.startswith("/")` branch calls `command_dispatch_module.dispatch_command(eff_conn, line, surface_session_id=session.id)` and prints `result.text`. The stale "lands in a later plan" comment cited in the review is gone.
- `command_dispatch` is imported at module scope (`app.py:40`) and genuinely invoked at call time, not just imported.
- `tests/tui/test_composer_routing.py` exists (3.8K) with `test_slash_line_routes_to_dispatch_command_with_session_id` and `test_non_slash_line_routes_to_submit_to_agent` — drives the assembled `run_workbench` entrypoint with a scripted fake `prompt_toolkit` session, not calling `dispatch_command` directly.
- `_submit_to_agent` (app.py:77-112) is no longer a no-op stub: it creates a real `mission_service.create_mission(...)` and drives `start_and_execute_async(..., session_id=session_id, ...)`, closing the session -> run -> audit_events -> transcript loop CR-01 depends on. Confirmed by reading the function body directly, not the docstring alone.
- Additional latent bug found+fixed during this wiring (session never transitioned out of `"starting"`, breaking Ctrl-C unwind) is present in current code: `app.py:202` calls `surface_session_service.transition_session(eff_conn, eff_lock, session.id, "active")` immediately after session creation, before the composer loop begins.

### CR-03 — `/permission` hard-codes empty `surface_session_id`

**Re-verified: RESOLVED.**
- `command_dispatch.dispatch_command`'s signature is `dispatch_command(conn, command_text, *, surface_session_id)` (line 248-249) — required keyword-only, no default.
