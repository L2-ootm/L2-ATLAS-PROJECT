# Phase 10.6 Plan 02: Terminal Workbench Foundation Layer Summary

**Capability probing, masked status header, and workspace picker (atlas_runtime/tui/) turning Wave 0's capabilities/session_select/header RED stubs GREEN.**

## Performance

- **Duration:** ~35 min
- **Tasks:** 3
- **Files modified:** 6 (5 created, 1 modified)

## Accomplishments
- `tui/capabilities.py`: `probe_capabilities()` wraps `rich.console.Console` detection into a frozen `Capabilities` dataclass (`no_color`, `box_style`, `width`, `layout`, `color_system`), honoring `NO_COLOR`, legacy-Windows ASCII fallback, and narrow (<80 col) width without clamping.
- `tui/theme.py`: `safe_style(name, caps)` is the single point where no-color mode is enforced across the whole `tui/` package; `TOKENS` hand-picks Rich color names approximating the bronze/celestial ATLAS brand marks (no machine-readable palette exists in `brand/` yet).
- `tui/session_select.py`: `select_workspace()` resolves `--project`/`--global` flags directly (no interactive prompt attempted), fails closed with `RuntimeError` on no-flags + non-TTY instead of blocking on input, and falls back to a `prompt_toolkit` radiolist picker over registered projects when interactive.
- `tui/header.py`: `render_status_header(console, snapshot)` prints a compact status line (model id/provider, permission mode, context budget, focus, session state) from an explicit field allowlist that never reads `resolved_api_key` or any other secret-shaped key.
- `workspace_service.resolve_workspace` gained a `use_global=True` alias for `kind="global"` (kind is now optional, defaulting from `use_global`) so the tui picker's call convention works against the real production function, not just the test's monkeypatched fake.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] `workspace_service.resolve_workspace` lacked the `use_global` kwarg the new tui call convention requires**
- **Found during:** Task 2 (session_select implementation)
- **Issue:** The Wave-0 test fakes for `select_workspace`'s global-flag path assert `workspace_service.resolve_workspace` is called with a `use_global=True` kwarg, but the real function only accepted `kind="global"|"project"`. Calling the real (unmocked) function with `use_global=True` in production would have raised `TypeError`.
- **Fix:** Made `kind` optional on `resolve_workspace`, defaulting to `"global"` if `use_global` is True else `"project"`, so both call conventions work identically in production and under test mocks.
- **Files modified:** `services/agent-runtime/atlas_runtime/workspace_service.py`
- **Verification:** `tests/test_workspace_service.py` (9 tests, pre-existing) still pass unchanged; `tests/tui/test_session_select.py` (3 tests) pass.
- **Committed in:** `9a5c281` (Task 2 commit)

**2. [Rule 1 - Bug] `select_workspace` would have returned a bare path string in production, not the dict shape its own tests/callers expect**
- **Found during:** Task 2 (session_select implementation)
- **Issue:** The real `workspace_service.resolve_workspace` returns a plain `str` (canonical root path); the Wave-0 tests' monkeypatched fakes return a dict with `kind`/`project_id`/`root_path` and assert `result["kind"]`. A pass-through implementation would only work under test mocks and crash on `result["kind"]` against the real function's return value.
- **Fix:** Added `_as_selection_dict()` to normalize either a bare string or an already-dict-shaped return value into the consistent `{"kind", "project_id", "root_path"}` shape `select_workspace` always returns.
- **Files modified:** `services/agent-runtime/atlas_runtime/tui/session_select.py`
- **Verification:** `tests/tui/test_session_select.py` (3 tests) pass.
- **Committed in:** `9a5c281` (Task 2 commit)
