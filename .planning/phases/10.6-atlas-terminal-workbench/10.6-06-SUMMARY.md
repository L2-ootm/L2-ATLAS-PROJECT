# Phase 10.6 Plan 06: Single Asyncio Loop Workbench Backbone Summary

**`run_workbench` wires prompt_toolkit's composer and a 500ms transcript-poll task into one `asyncio.run()` call, with Rich `Live` scoped exclusively to the status header and `handle_cancel` driving fail-closed Ctrl-C session-state unwind (TUI-04/TUI-08).**

## Performance

- **Duration:** ~25 min
- **Tasks:** 2/2 completed
- **Files modified:** 1 (created)

## Accomplishments
- `handle_cancel(conn, lock=None, *, session_id)` performs exactly one `transition_session(conn, lock, session_id, "cancelling")` call, propagating `ValueError` unswallowed on illegal/terminal-state transitions (TUI-08). `tests/tui/test_cancel_unwind.py` is GREEN (2/2).
- `run_workbench` is the single-asyncio-loop backbone: resolves workspace via `session_select.select_workspace`, creates a `starting` surface session, probes terminal capabilities, then runs one `asyncio.run(_main_loop())` that owns the `PromptSession.prompt_async` composer and a background `_poll_loop` task driving `transcript.poll_and_render`/`render_event`.
- `Live` is constructed exactly once and only ever updated with header content (via a `console.capture()` adapter over `header.render_status_header`'s print-side-effect contract); transcript/event output is always plain `console.print(...)` under `patch_stdout`, never a second `Live` region.
- The poll task is always `.cancel()`'d and `await asyncio.gather(..., return_exceptions=True)`'d in a `finally` block before `patch_stdout`'s context exits, so it never outlives the loop (RESEARCH Pitfall 4 / T-10.6-19).
- The non-slash-command composer path routes to a clearly-named `_submit_to_agent` stub (not silently dropped) pending the agent-invocation wiring of a later plan (T-10.6-21).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Adapted `header.render_status_header`'s return contract for `Live.update`**
- **Found during:** Task 2
- **Issue:** The plan's `<interfaces>` section describes `render_status_header(snapshot, session, *, caps, focus=None) -> RenderableType`, but the actually-committed Wave-1 function is `render_status_header(console, snapshot) -> None` — it prints directly via `console.print(text)` and returns nothing. Passing `None` to `Live.update(...)` would either raise or render nothing, defeating the plan's `must_haves` truth that the header renders inside the `Live` region.
- **Fix:** Added a small module-level adapter, `_render_header_for_live(console, snapshot)`, that calls `header.render_status_header` inside `console.capture()` and replays the captured ANSI text as a `rich.text.Text` object for `Live.update`. `header.py` itself was left untouched — it remains the single source of header-formatting logic, fully respecting its own already-tested contract.
- **Files modified:** `services/agent-runtime/atlas_runtime/tui/app.py` (no `header.py` changes)
- **Verification:** `python -c "from atlas_runtime.tui.app import run_workbench, handle_cancel; ... assert 'patch_stdout' in src; assert 'poll_task.cancel' in src"` passes; full `tests/tui/` suite (excluding the separately-scoped `test_command_dispatch.py`) passes 30/30 with no regressions.
- **Committed in:** `b6b61ee` (Task 2 commit)
