# Phase 10.6 Plan 03: Transcript Poll/Replay + Per-EventKind Renderers Summary

**Gap-free SurfaceEvent poll/replay primitive plus a 10-branch Rich renderer dispatch (text/reasoning/tool_call/tool_result/task/retry/retrieval/approval/error/completion), backed by a new `surface_events` migration the Wave-0 tests required but no prior migration provided.**

## Performance

- **Duration:** 25 min
- **Tasks:** 2 completed
- **Files modified:** 2 (1 created module, 1 created migration)

## Accomplishments
- `poll_and_render(conn, console, *, session_id, last_seq=0, run_outcome=None)` reads the persisted `surface_events` table for one session and returns only unseen events (`seq > last_seq`), delegating gap-detection to `surface_events.replay_since` — zero Rich Console/Live calls.
- `render_event(event, *, caps=None)` dispatches all 10 `EventKind` literals to distinct, non-empty Rich renderables; malformed `payload_json` never raises (falls back to `{}`); an unrecognized kind raises `ValueError` naming the kind.
- Added `infra/migrations/0018_surface_events.sql` so the durable per-session event table the Wave-0 test fixtures assume actually exists.

## Task Commits

1. **Task 1 + Task 2 (transcript.py is one file; tests share one commit):** `59f3db0` (feat) — both `poll_and_render` and `render_event` landed together since Task 2 appends to the same file Task 1 created, and both Wave-0 test files (`test_transcript_poll.py`, `test_event_renderers.py`) needed the migration fix simultaneously to even collect.

## Files Created/Modified
- `services/agent-runtime/atlas_runtime/tui/transcript.py` — `poll_and_render` (gap-free poll/slice) + `render_event` (per-kind Rich dispatch)
- `infra/migrations/0018_surface_events.sql` — persisted `surface_events(id, session_id, seq, kind, run_id, occurred_at, payload_json)` table + `(session_id, seq)` index

## Decisions Made
- Read directly from a new `surface_events` table rather than routing through `audit_service.get_events_for_session` + `surface_events.normalize_surface_events`: the Wave-0 RED test seeds rows directly into `surface_events` with no corresponding `runs`/`audit_events` rows, so the durable-table read path is the one the locked test fixtures actually exercise. The in-process `normalize_surface_events`/`replay_since` contracts remain the canonical projection logic for the live run loop (Wave 3); this table is the pollable read-side cache for reconnect/cross-process polling.
- `poll_and_render` returns `list[SurfaceEvent]` (adapting `replay_since`'s tuple) because the locked test compares the second poll's result to `[]`.
- `render_event`'s `caps` parameter defaults to `probe_capabilities()` since the locked test calls it without `caps`.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Added migration 0018_surface_events.sql for the missing `surface_events` table**
- **Found during:** Task 1 (gap-free poll/replay primitive)
- **Issue:** `tests/tui/test_transcript_poll.py`'s `_seed_events` helper inserts directly into a `surface_events` table via raw SQL, but no migration in `infra/migrations/` defines that table — every prior migration only adds `surface_sessions` (0016) and `tool_approvals`/`approval_channels` (0017). Without this table the test fixture's INSERT would raise `sqlite3.OperationalError: no such table: surface_events` before `poll_and_render` is ever reached.
- **Fix:** Added `infra/migrations/0018_surface_events.sql` defining `surface_events(id, session_id, seq, kind, run_id, occurred_at, payload_json)` plus a `(session_id, seq)` index, mirroring the `SurfaceEvent` Pydantic contract field-for-field and the soft-FK precedent used by `surface_sessions`/`tool_approvals`.
- **Files modified:** `infra/migrations/0018_surface_events.sql` (new)
- **Verification:** `tests/tui/test_transcript_poll.py` and `tests/tui/test_event_renderers.py` both pass (12/12); broader `tests/tui` suite (excluding other plans' still-RED stubs) passes (21/21).
- **Committed in:** `59f3db0` (Task 1/2 commit)

**2. [Rule 1 - Bug] Adapted poll_and_render's return type to list, not tuple**
- **Found during:** Task 1 verification run
- **Issue:** Plan prose specifies `poll_and_render(...) -> tuple[SurfaceEvent, ...]`, but the locked test asserts `second == []` on the no-new-events branch — a tuple never equals a list in Python (`() == []` is `False`), so following the plan's literal type would fail the locked Wave-0 test.
- **Fix:** `poll_and_render` wraps `surface_events.replay_since(...)`'s tuple result in `list(...)` before returning. Internal ordering/gap-detection logic is unchanged (still delegated to `replay_since`); only the final adaptation changed.
- **Files modified:** `services/agent-runtime/atlas_runtime/tui/transcript.py`
- **Verification:** `test_replay_since_dedup_on_second_poll` passes.
- **Committed in:** `59f3db0` (Task 1/2 commit)
