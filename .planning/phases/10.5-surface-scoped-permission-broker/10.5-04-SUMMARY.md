# Phase 10.5 Plan 04: Fail-Closed Headless Gate + Replay/Stale Defenses Summary

Closed the two security paths of the surface-scoped permission broker: the PERM-05
fail-closed headless/API gate in `tool_service.invoke()` (an `ask` decision for a
headless `api` surface with no unrevoked approval channel denies without queuing a
pending row) and the SEC-02 replay/stale defenses in `permission_broker.claim()` (nonce
echo verification, expiry enforcement, and malformed-contract rejection — all resolved
as typed errors before the at-most-once UPDATE). Every failure path resolves to
deny/typed-error; nothing silently auto-approves.

## What Was Built

### Task 1 — Channel registration + PERM-05 fail-closed gate (`2cbe179`)
- `permission_broker.register_channel(conn, lock, *, surface_session_id, surface_kind)`:
  construct-validate-then-write — builds the frozen `ApprovalChannel` model first, then
  UPSERTs the `approval_channels` row (`ON CONFLICT(surface_session_id) DO UPDATE ...
  revoked_at=NULL`) so a re-register re-opens the gate.
- `permission_broker.revoke_channel(...)`: sets `revoked_at=now` on the unrevoked row;
  idempotent (returns `True` only when a row was actually revoked).
- `permission_broker.has_open_channel(conn, surface_session_id) -> bool`: pure read for
  an unrevoked row.
- `ApprovalChannelMissingError` typed error added (and re-exported in `__all__`).
- `tool_service.invoke()`: in the `requires_approval` branch, BEFORE building the
  pending `ToolApproval`, when `surface_kind == "api"` and no unrevoked
  `approval_channels` row exists, emit an `approval` audit event
  (status='rejected', reason='no_approval_channel', surface_kind + workspace_root in
  data; `policy_result='denied'`) AFTER the lock and RETURN a denied `ToolApproval`
  (status='rejected', reason set) WITHOUT inserting a pending row. The channel check is
  an inline SELECT, not an import of `permission_broker`, to avoid the
  broker→tool_service import cycle. Non-`api` surfaces (cli/tui/webui) are unaffected.

**Denial provenance (Warning-4):** the headless deny persists NO `tool_approvals` row.
The denial record lives entirely in the emitted `approval`/`rejected` audit event. This
is by design — PERM-01 record-completeness applies to approvals that enter the queue,
not to fail-closed denials.

### Task 2 — Nonce / expiry / malformed-claim rejection in `claim()` (`d091211`)
- `StaleApprovalError(ValueError)` added (re-exported in `__all__`); closed
  `_VALID_CLAIM_DECISIONS` decision-label set.
- In `claim()`, BEFORE the decision-column UPDATE and the at-most-once delegate:
  1. **Malformed guard** — a blank/unknown `decision` label or a blank `nonce` raises
     `StaleApprovalError` before any SQL (the unknown-decision path now raises
     `StaleApprovalError` instead of a bare `ValueError`).
  2. **Nonce echo** — `approval.nonce is not None and approval.nonce != nonce` raises
     `StaleApprovalError` with no row mutation.
  3. **Expiry** — `now > approval.expiry_at` (naive expiry normalized to UTC) emits a
     deny `approval` audit event (reason 'expired') AFTER the lock, then raises
     `StaleApprovalError`.
- These run after the Plan-03 ownership + active-session checks but before the
  rowcount-gated UPDATE, so a stale/replayed claim can never win the at-most-once race.
  The Plan-03 AlreadyDecided / at-most-once behavior is untouched.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Unknown-decision path raised bare `ValueError`, not the SEC-02 typed error**
- **Found during:** Task 2
- **Issue:** Plan 03's `claim()` raised a bare `ValueError("unknown claim decision")`.
  The SEC-02 malformed-claim contract requires a typed error and the plan specifies a
  closed decision-label set surfacing as `StaleApprovalError`.
- **Fix:** The malformed guard now raises `StaleApprovalError` (a `ValueError` subclass,
  so the test's accepted `(StaleApprovalError, ValueError, AlreadyDecided)` set remains
  satisfied) and runs before any SQL.
- **Files modified:** `services/agent-runtime/atlas_runtime/permission_broker.py`
- **Commit:** `d091211`

**2. [Rule 2 - Missing critical functionality] Naive expiry_at would raise TypeError mid-claim**
- **Found during:** Task 2
- **Issue:** A hand-seeded or legacy naive `expiry_at` compared against a tz-aware `now`
  raises `TypeError: can't compare offset-naive and offset-aware datetimes`, which would
  surface as an untyped crash instead of a fail-closed deny.
- **Fix:** Normalize a naive stored `expiry_at` to UTC before the comparison so expiry
  always resolves to a clean `StaleApprovalError`.
- **Files modified:** `services/agent-runtime/atlas_runtime/permission_broker.py`
- **Commit:** `d091211`
