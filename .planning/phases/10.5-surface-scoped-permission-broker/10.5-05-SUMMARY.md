# Phase 10.5 Plan 05: PERM-07 Scope Guard + AUD-02 Read-Only Outcome Projection Summary

Closed the broker's last two requirements. PERM-07: session-scoped allow-once/session/always
rules now have a real store/consume path bounded to a (workspace_root, surface_kind, tool_name,
arg_pattern) 4-tuple, keyed to one surface session, and provably unable to widen into global
policy/config. AUD-02: the cross-surface outcome projection is a pure read of terminal outcomes
that grants observers no decision authority. The full broker contract is complete; all 9
`test_permission_broker.py` tests are green.

## What Was Built

### Task 1 — Session-scoped allow-rule store + consume with hard scope bound (`d6f1651`)
- `record_allow_rule(conn, lock, *, surface_session_id, workspace_root, surface_kind, tool_name,
  arg_pattern, rule_kind)`: construct-validate-then-write — builds the frozen `SessionAllowRule`
  (rejecting a malformed `rule_kind` before any SQL), then INSERTs into `session_allow_rules`
  inside `with lock: with conn:`. Writes ONLY to that table; no policy.py/config.yaml touch.
- `match_allow_rule(conn, *, surface_session_id, workspace_root, surface_kind, tool_name,
  args_normalized) -> Optional[SessionAllowRule]`: session-scoped SELECT requiring an exact
  4-tuple match (`arg_pattern == args_normalized`), newest first, no wildcard. Pure read.
- `allow_pattern_for_args(args)`: convenience wrapper delegating to `tool_service._normalize_args`
  — the SAME helper `invoke()` uses for `args_normalized` — so a rule's `arg_pattern` is
  byte-identical to a real approval's key (Warning-5 cross-path parity).
- `_consume_allow_once(conn, lock, rule_id)`: deletes a matched allow_once rule (bounded to the id).
- `claim(... record_rule=None)`: optional `SessionAllowRuleKind`. On a winning approve claim by the
  owning active session, records a rule bound to the resolved terminal approval's exact 4-tuple;
  allow_once is consumed immediately, allow_session/allow_always persist. Back-compatible (default
  None; existing callers unaffected).
- Module-level PERM-07 hard-rule comment + `_VALID_RULE_KINDS`. There is NO import of a
  config-write API and no `policy.decide(` call — the only `policy.decide`/`patch_config` strings
  in the module are in explanatory comments.

**Warning-5 parity proven:** the same args run through `tool_service._normalize_args` (the invoke
insert path) and through `allow_pattern_for_args` yield an identical normalized+redacted string
(`{"a": 1, "b": 2, "token": "[REDACTED]"}`), so a recorded rule actually matches a real approval.

### Task 2 — Cross-surface read-only outcome projection (`78ffe31`)
- `list_outcomes(conn, *, limit=None)`: terminal outcomes (executed | rejected | failed) with
  surface provenance, newest decided first, optional cap. Pending rows excluded by design.
- `get_outcome(conn, approval_id) -> Optional[ToolApproval]`: single terminal-outcome view, or
  None if unknown/pending/executing.
- Both are pure reads: no `lock` parameter, no INSERT/UPDATE/DELETE, no call into
  claim/approve/reject — verified by AST body inspection (docstring stripped). An observer gains
  no claim handle and cannot act via the projection (T-10.5-05-OBSERVER). `__all__` updated.
