# Phase 10.5: Surface-Scoped Permission Broker - Research

**Researched:** 2026-06-26
**Domain:** Server-authoritative approval broker — SQLite concurrency, surface-session routing, fail-closed authorization
**Confidence:** HIGH (grounded entirely in this repo's verified files; no external dependencies)

## Summary

Phase 10.5 extends the existing `tool_service.invoke()` chokepoint and the `tool_approvals`
table so that every approval is anchored to the surface session that initiated it, resolvable
only by that live session, atomically claimable at most once, and fail-closed for headless/API.
Every primitive this phase needs already exists in the codebase: the atomic
`UPDATE … WHERE status='pending'` rowcount guard (`tool_service.approve`/`reject`), the
`with lock: with conn:` write discipline, the surface-session state machine with an `active`
state and terminal immutability trigger (`surface_session_service`), the audit `emit()` write
boundary, the migration runner with a `schema_migrations` tracker, and a control-plane config
with optimistic-revision patching. **This is an extension phase, not a greenfield one** — the
work is additive columns, one new registration table, a routing parameter on `invoke()`, an
authority-checked claim API, and a read-only cross-surface projection.

The single largest design decision (Claude's discretion per CONTEXT.md) is **service layout**:
extend `tool_service.py` in place vs. add a dedicated `permission_broker.py`. The recommendation
below is to **add a thin `permission_broker.py` that owns the surface-scoped queue + authority-
checked claim API, while the atomic state transitions stay in `tool_service`** — because the
existing `approve()`/`reject()` already hold the verbatim TOCTOU guard and moving them risks
regressing the Phase 10.0.4 / Phase C guarantee.

**Primary recommendation:** One new migration (`0017_permission_broker.sql`) adds surface
columns + expiry + nonce to `tool_approvals` and creates an `approval_channels` table; extend
`ToolApproval` and add an `ApprovalChannel` frozen model; add `surface_session_id`/`surface_kind`
to `invoke()`; gate `approve()`/`reject()` on a matching `active` session via a new
`broker.claim()` authority check that reuses the existing rowcount-gated atomic UPDATE.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**Approval Record Model**
- Bind approvals to their surface by adding `surface_session_id` (FK → surface_sessions) and
  `surface_kind` columns to `tool_approvals` (immutable anchor, not a run_id-derived join).
- Expiry is a configurable default TTL surfaced through the 10.4 control-plane config
  (default 300s); expiry is fail-closed (denies the pending request).
- Reuse the existing audit `_redact()` plus a canonical arg-normalization that doubles as the
  policy match key — no separate redaction path.
- Risk classification reuses `policy.py` `ToolRiskLevel` (read/write/shell); no new taxonomy.

**Surface Ownership & Resolution Authority**
- A decision claim requires a matching `surface_session_id` AND the session being in `active`
  state; claims from a different or terminal session are rejected.
- 10.5 ships the broker contract + queue/claim API only. The TUI native prompt consumes it in
  Phase 10.6; the WebUI sidebar/queue consumes it in Phase 10.7.
- Cross-surface visibility is a read-only audit projection of terminal outcomes, kept separate
  from the actionable claim query — observers never gain decision authority (AUD-02).
- The actionable pending-queue query is scoped strictly by `surface_session_id`; it never
  returns another surface's actionable items (PERM-02/PERM-04).

**Headless/API & Fail-Closed Policy**
- Headless/API `ask` decisions deny by default unless an explicit approval channel was
  registered for that session (PERM-05).
- Disconnect/timeout policy is fail-closed (deny) by default, configurable via the control plane.
- Approval-channel registration is an explicit record keyed by `surface_session_id`; presence of
  that registration is the gate (not an implicit connected socket).
- Replay/spoof defense uses a revision/nonce plus a session-state check; stale or malformed
  requests are rejected as typed errors (SEC-02).

**Exactly-Once Claim & Scope Guard**
- At-most-once via an atomic `UPDATE … WHERE status='pending'` rowcount-gated claim inside a
  SQLite transaction (first writer wins); concurrent losers receive a typed "already decided"
  result carrying the winning decision (PERM-06).
- allow-once / allow-session / allow-always rules are session-scoped and held on the session
  row; they are never written to global policy/config. allow-always remains bounded to
  workspace + surface + tool + argument-pattern and cannot widen to global scope (PERM-07).
- Pending rows survive process restart in SQLite; an orphaned `executing` row with no result
  reconciles to a fail-closed (failed) outcome; live surface sessions are reclaimed via the
  Phase 10.3 lifecycle state machine.

### Pattern 1: Atomic rowcount-gated claim (the verbatim TOCTOU guard)
**What:** First-writer-wins transition out of `pending`, guarded by `rowcount == 1`.
**When to use:** Every state transition that must execute the deferred action at most once.
**Example (existing — reuse verbatim, add authority pre-check):**
```python
# Source: services/agent-runtime/atlas_runtime/tool_service.py:232-244 (approve)
with lock:
    with conn:
        cur = conn.execute(
            "UPDATE tool_approvals SET status = 'executing' "
            "WHERE id = ? AND status = 'pending'",
            (approval_id,),
        )
        claimed = cur.rowcount == 1
if not claimed:
    current = _load(conn, approval_id)
    raise ToolApprovalError(f"approval {approval_id!r} is {current.status!r}, not pending")
```
For 10.5 the loser must receive a **typed AlreadyDecided result carrying the winning decision**
(CONTEXT PERM-06), not just an exception — read the current row after `claimed is False` and
return `{status, decision, reason}` as a typed result.

### Anti-Patterns to Avoid
- **Reading liveness via `os.kill(pid, 0)`:** broken on Windows. The session service already
  uses owner-token + heartbeat-TTL liveness — read `surface_sessions.state`, never probe PIDs.
  [VERIFIED: surface_session_service.heartbeat docstring "RESEARCH Pitfall 2"]
- **`;`-splitting migration scripts:** the migration runner uses `executescript` and must NOT
  split trigger bodies (`BEGIN … END`). If 10.5 adds a trigger, keep the body intact.
  [VERIFIED: db.py `_apply_sql_tolerant`, 0016 trigger comment]
- **Writing session-scoped allow rules to `policy.py` or `config.yaml`:** that silently widens
  global policy (PERM-07 violation). Session rules live on session-keyed state only.
- **A second redaction path:** reuse `audit_service._redact()` + one canonical arg-normalization
  that doubles as the policy match key (CONTEXT locked).
- **Emitting audit inside `with lock:`** — deadlocks (`emit()` re-acquires the lock).

## Atomic At-Most-Once Claim (PERM-06)

**The correct SQLite pattern under WAL** is exactly what the codebase already uses:
a single `UPDATE … WHERE status='pending'` inside `with lock: with conn:`, gated on
`cur.rowcount == 1`. [VERIFIED: tool_service.py approve/reject]

- **Why it is correct under WAL:** the `with conn:` block is a transaction; the UPDATE takes a
  write lock; the `WHERE status='pending'` predicate makes the transition conditional, so only
  the first committing transaction matches a row (`rowcount==1`) and every subsequent one sees
  `rowcount==0`. WAL changes concurrency for *readers*, not the single-writer serialization that
  guarantees this — only one writer commits the `pending→executing` flip.
- **In-process serialization:** the shared `threading.Lock` (`with lock:`) serializes writers in
  the same process before they even reach SQLite, so the rowcount guard primarily defends against
  cross-process / cross-connection races and is belt-and-suspenders in-process. Keep both.
- **How losers detect loss (CONTEXT requires a typed result, not just an exception):** after
  `claimed = cur.rowcount == 1` is False, re-`_load` the row and return a typed
  `AlreadyDecided(winning_status, decision, reason)` carrying the winner's outcome. The existing
  code raises `ToolApprovalError`; 10.5 should return the typed result for the broker API while
  still raising for the internal `tool_service.approve` contract (or wrap it).
- **`executing` is the transient claim state** so two concurrent approvers cannot both run the
  deferred adapter. [VERIFIED: tool.py:28-30 comment]

## Surface-Scoped Routing & Authority (PERM-02)

**Where the session id comes from in the call path:** `tool_service.invoke()` currently takes
`tool_name, args, mode, ctx, reason`. [VERIFIED: tool_service.py:93-102] It has **no surface
session today** — the only callers are golden_workflows and discord_service, which pass keyword
args. [VERIFIED: grep of invoke call sites] The clean seam is to add **optional keyword params**
`surface_session_id: Optional[str] = None` and `surface_kind: Optional[str] = None` to `invoke()`
(back-compatible — existing callers default to None / treated as `operator`/non-routed). The
caller that owns the surface (the TUI/WebUI/API entrypoints in 10.6/10.7, and the agent harness
loop) passes the active session id it already holds.

**Binding at insert time:** when `decision.requires_approval`, construct the `ToolApproval` with
`surface_session_id`, `surface_kind`, `workspace_root` (from the session or `ctx`), `expiry_at`,
and `nonce`, then INSERT (Pattern 2). The anchor is immutable for the row's life.

**Authority enforcement in `claim()`:** before the atomic UPDATE, (a) the claimant's
`surface_session_id` must equal `approval.surface_session_id`, and (b) that session must be in
`state='active'` per `surface_sessions`. A claim from a different surface, or from a terminal/
suspended session, is rejected with a typed error — no row is touched. The **actionable queue
query** (`list_actionable`) filters strictly `WHERE surface_session_id = ? AND status='pending'
AND expiry_at > now`; it never returns another surface's rows (PERM-02/04). The cross-surface
**outcome projection** is a separate read that returns terminal outcomes only and exposes no
claim path (AUD-02).

## Fail-Closed Headless/API (PERM-05)

- **Gate:** in `invoke()`, when `surface_kind == "api"` (headless) and the policy result is
  `ask`/`requires_approval`, check `approval_channels` for an unrevoked row keyed by
  `surface_session_id`. **Absent → immediate deny** (emit `approval` with `status` denied/rejected
  + reason `no_approval_channel`); the deferred action never runs. Present → create the pending
  approval normally.
- **Registration model:** an explicit `approval_channels` record, registered by the API surface
  when it is prepared to service approvals. Presence (not a live socket) is the gate.
- **Disconnect/timeout/expiry:** all three resolve to **deny** (fail-closed), configurable via the
  control plane. Expiry is enforced at read/claim time: `now > expiry_at` ⇒ the row is treated as
  denied (and a sweep can flip it terminal with `reason=expired`).
- **Restart reconciliation:** at startup, sweep `tool_approvals` for `status='executing'` rows
  with no `result` (a crash mid-execution) and flip them to `failed` (fail-closed) — mirroring
  `surface_session_service.reconcile_orphans` for sessions. `pending` rows survive restart
  untouched (PERM-06); live sessions are reclaimed by the existing session reconcile sweep, and a
  reclaimed session can no longer satisfy the `state='active'` authority check, so its orphaned
  pendings become unclaimable until expiry denies them.

## Scope-Bounding allow-once/session/always (PERM-07)

Session-scoped rules are held on session-keyed state (`session_allow_rules` table keyed by
`surface_session_id`, or a JSON column on `surface_sessions`) — **never** in `policy.py` or
`config.yaml`. The boundary:

- `policy.py.decide()` stays the **global** authority and is untouched (still read-only-posture:
  write/shell always `requires_approval`). [VERIFIED: policy.py:43-61]
- The broker consults session-scoped rules **only as a per-claim shortcut** for that specific
  session — it can auto-resolve a pending approval for the initiating session but cannot change
  what `policy.decide()` returns for any other session or globally.
- `allow-always` is bounded to the 4-tuple `workspace_root + surface_kind + tool_name +
  arg_pattern`; it cannot match across workspaces, surfaces, or as a wildcard tool, and is never
  promoted to global config. This is the hard PERM-07 guarantee and must be test-covered with a
  negative case (a rule in session A does not affect session B / a different workspace).

## Audit Provenance & AUD-02

- Emit approval lifecycle via `audit_service.emit(conn, lock, run_id=…, event_type="approval",
  session_id=surface_session_id, data={approval_id, tool_name, status, decision, reason,
  surface_kind, workspace_root})`. The enriched `data` carries requesting-surface provenance;
  `_redact()` runs automatically. [VERIFIED: audit_service.emit, existing approval emit in tool_service:152]
- The cross-surface **read projection** queries terminal approval events / terminal
  `tool_approvals` rows and returns outcome-only views. It must expose **no** function that flips
  a status — observers see *what happened*, never *decide*. Keep it in `permission_broker` as a
  pure `list_outcomes()`/`get_outcome()` with no `conn`-mutating path. This is the AUD-02 line.

## SEC-02 Adversarial Concerns

| Pattern | STRIDE | Standard Mitigation (in this phase) |
|---------|--------|-------------------------------------|
| Approval spoofing (a surface resolves another's request) | Spoofing / Elevation | `claim()` requires `approval.surface_session_id == caller` AND session `state='active'`; cross-surface claim → typed reject (PERM-02) |
| Replay (re-submitting a stale claim) | Tampering | Per-request `nonce` stored on the row; claim must echo the matching nonce; mismatch → typed stale error |
| Stale request (claim after expiry) | Tampering | `expiry_at` check at claim time; `now > expiry_at` → deny |
| Malformed request | Tampering | Pydantic validation on the claim contract before any SQL; typed `ValidationError` |
| Double-execution race | Tampering | Atomic rowcount-gated claim (PERM-06) |
| Privilege widening via allow-always | Elevation | 4-tuple bound + never written to global policy (PERM-07) |
| Secret leakage in approval args | Information disclosure | `_redact()` once at the audit boundary; `args_normalized` stored redacted |

**Tests must cover:** wrong-session claim rejected; terminal/suspended-session claim rejected;
nonce mismatch rejected; expired-request denied; concurrent claim → exactly one winner + typed
AlreadyDecided to losers; restart leaves pending intact and flips orphaned `executing` → failed;
session-A allow-always does not affect session-B/other-workspace.

## Common Pitfalls

### Pitfall 1: Emitting audit inside the lock (deadlock)
**What goes wrong:** `audit_service.emit()` re-acquires the same `threading.Lock`; calling it
inside `with lock:` deadlocks. **How to avoid:** emit AFTER releasing the lock (Pattern 3).
**Warning signs:** test hangs. [VERIFIED: surface_session_service.py:226 "Pitfall 6"]

### Pitfall 2: `_COLS` / `_row_to_approval` drift
**What goes wrong:** `tool_service._COLS` and `_row_to_approval` zip a fixed column order into
`ToolApproval(**dict(...))`. Adding columns to the model/table without updating both string +
zip silently mismaps fields. **How to avoid:** update `_COLS`, the INSERT column list, the
VALUES placeholders, and `_row_to_approval` in lockstep with the migration. [VERIFIED: tool_service.py:36-53,144-151]

### Pitfall 3: Hard FK on `surface_session_id` breaks operator/legacy rows
**What goes wrong:** a `REFERENCES surface_sessions(id)` FK would reject the legacy `operator`
sentinel and NULL-session rows under `PRAGMA foreign_keys=ON`. **How to avoid:** soft FK (stored
+ indexed, not enforced), per the 0016/0013 precedent.

### Pitfall 4: Migration trigger body split by `executescript`
**What goes wrong:** if 10.5 adds a trigger (e.g. immutable terminal approvals), naive `;`
splitting corrupts the `BEGIN … END` body. **How to avoid:** the runner uses `executescript`
and never splits — keep the trigger body intact, as 0016 does. [VERIFIED: db.py:64-83, 0016:36-44]

### Pitfall 5: Allow-rule leakage into global policy
**What goes wrong:** persisting an allow-always to `config.yaml`/`policy.py` widens scope for all
sessions (PERM-07 breach). **How to avoid:** session-keyed storage only; `policy.decide()`
untouched; negative cross-session test.
