# Phase 10.5: Surface-Scoped Permission Broker - Context

**Gathered:** 2026-06-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Route every approval request to the surface session that caused it, while ATLAS remains the
sole authority for policy, persistence, audit, and exactly-once execution. This phase delivers
the broker contract and server-side routing/claim logic: approval records anchored to a surface
session, an actionable pending-queue query scoped per surface, atomic at-most-once decision
claiming, fail-closed headless/API behavior, and scope-bounded allow-once/session/always rules.

Out of scope (deferred to later phases): the ATLAS TUI native blocking prompt UI (Phase 10.6)
and the WebUI conditional header/sidebar queue UX (Phase 10.7). 10.5 ships the broker API and
contracts those surfaces will consume — not the surface UIs themselves.

</domain>

<decisions>
## Implementation Decisions

### Approval Record Model
- Bind approvals to their surface by adding `surface_session_id` (FK → surface_sessions) and
  `surface_kind` columns to `tool_approvals` (immutable anchor, not a run_id-derived join).
- Expiry is a configurable default TTL surfaced through the 10.4 control-plane config
  (default 300s); expiry is fail-closed (denies the pending request).
- Reuse the existing audit `_redact()` plus a canonical arg-normalization that doubles as the
  policy match key — no separate redaction path.
- Risk classification reuses `policy.py` `ToolRiskLevel` (read/write/shell); no new taxonomy.

### Surface Ownership & Resolution Authority
- A decision claim requires a matching `surface_session_id` AND the session being in `active`
  state; claims from a different or terminal session are rejected.
- 10.5 ships the broker contract + queue/claim API only. The TUI native prompt consumes it in
  Phase 10.6; the WebUI sidebar/queue consumes it in Phase 10.7.
- Cross-surface visibility is a read-only audit projection of terminal outcomes, kept separate
  from the actionable claim query — observers never gain decision authority (AUD-02).
- The actionable pending-queue query is scoped strictly by `surface_session_id`; it never
  returns another surface's actionable items (PERM-02/PERM-04).

### Headless/API & Fail-Closed Policy
- Headless/API `ask` decisions deny by default unless an explicit approval channel was
  registered for that session (PERM-05).
- Disconnect/timeout policy is fail-closed (deny) by default, configurable via the control plane.
- Approval-channel registration is an explicit record keyed by `surface_session_id`; presence of
  that registration is the gate (not an implicit connected socket).
- Replay/spoof defense uses a revision/nonce plus a session-state check; stale or malformed
  requests are rejected as typed errors (SEC-02).

### Exactly-Once Claim & Scope Guard
- At-most-once via an atomic `UPDATE … WHERE status='pending'` rowcount-gated claim inside a
  SQLite transaction (first writer wins); concurrent losers receive a typed "already decided"
  result carrying the winning decision (PERM-06).
- allow-once / allow-session / allow-always rules are session-scoped and held on the session
  row; they are never written to global policy/config. allow-always remains bounded to
  workspace + surface + tool + argument-pattern and cannot widen to global scope (PERM-07).
- Pending rows survive process restart in SQLite; an orphaned `executing` row with no result
  reconciles to a fail-closed (failed) outcome; live surface sessions are reclaimed via the
  Phase 10.3 lifecycle state machine.

### Claude's Discretion
- Concrete service/file layout (e.g., a dedicated broker service vs. extending tool_service),
  exact migration column types, error code naming, and test fixture structure — within the
  decisions above and existing conventions.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `services/agent-runtime/atlas_runtime/policy.py` — `PolicyDecision` (allowed/requires_approval/
  denied) and `ToolRiskLevel`; the decision source 10.5 routes on.
- `services/agent-runtime/atlas_runtime/tool_service.py` — single chokepoint: `invoke()` calls
  `policy.decide()`, emits `tool_requested`, creates pending `ToolApproval` for write/shell;
  `approve()` / `reject()` handle deferred execution. The natural seam for routing + claim.
- `packages/atlas-core/atlas_core/schemas/tool.py` — `ToolApproval` (frozen), `ToolApprovalStatus`
  (pending/executing/executed/rejected/failed); extend with surface fields.
- `packages/atlas-core/atlas_core/schemas/surface_session.py` + `agent_contract.py` —
  `SurfaceSession`, `SurfaceIdentity` (kind cli/tui/webui/api, session_id), `WorkspaceIdentity`,
  `PermissionMode`. The routing target identity (Phase 10.3).
- `services/agent-runtime/atlas_runtime/surface_session_service.py` — session lifecycle + state
  machine; source of the `active`/terminal state the claim check reads.
- `services/agent-runtime/atlas_runtime/audit_service.py` — `emit()` single write boundary with
  `_redact()`; approval events ride `event_type` on the audit bus (extend with surface provenance).
- `services/agent-runtime/atlas_runtime/secure_store.py` — cross-platform locking primitive (10.4)
  if any durable non-SQLite state is needed.
- `foundation/atlas-hermes/tools/approval.py` — legacy contextvar-based per-session approval;
  pattern reference for session-keyed prompting (to be bridged to surface_session_id).

### Established Patterns
- Atomic writes inside `with lock: with conn:`; lock re-acquired between transactions.
- Construct + validate Pydantic model before INSERT (D-013).
- SQLite WAL, FK ON, migration runner in `db.py`; migrations in `infra/migrations/` (latest:
  0016_surface_sessions.sql; 0013_tool_approvals.sql is the table to extend).
- Audit-backed change events as the cross-surface visibility mechanism (no pub/sub bus; D-022).

### Integration Points
- Extend `infra/migrations/` with a new migration adding surface FK + expiry + registration to
  tool_approvals (and any approval-channel registration table).
- Wire `tool_service.invoke()` to capture the initiating `surface_session_id` and route the
  pending approval; add claim/decision API enforcing the active-session authority check.
- Emit enriched audit events (requesting surface provenance) via `audit_service.emit()`.

</code_context>

<specifics>
## Specific Ideas

Security posture is conservative throughout: deny-by-default for headless/API, fail-closed on
disconnect/timeout/expiry, typed rejection of stale/malformed/replayed requests, and a hard
guarantee that session-scoped allow rules cannot widen global policy or cross workspace/surface
boundaries. The broker is server-authoritative; surfaces are clients that request and claim,
never sources of policy truth.

</specifics>
