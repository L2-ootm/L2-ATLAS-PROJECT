# Phase 10.5: Code Review Report

**Reviewed:** 2026-06-26
**Depth:** standard
**Files Reviewed:** 7
**Status:** issues_found

## Summary

Reviewed the surface-scoped permission broker (Phase 10.5): migration 0017, the
control-plane/tool schemas, the broker core, the tool-service chokepoint, and the
Wave-0 contract tests. The security-critical invariants the prompt called out hold
in the code as written: `claim()` issues no status-flip UPDATE (only a non-status
`decision` write gated on `status='pending'`), the at-most-once guard lives solely in
`tool_service.approve()/reject()` (rowcount==1), authority reads the
`surface_sessions.state` column rather than probing a PID, audit `emit` is invoked
outside the `with lock:` block, args route through the single `_redact` boundary, and
the broker contains no policy/config-write import (PERM-07 holds).

However, two correctness defects undercut the SEC-02 stale/expiry guarantees:

1. **CR-01** — the configured `approval_ttl_seconds` is silently ignored because the
   TTL resolver reads a non-existent attribute (`.permissions` vs the schema's
   `.permission`), masked by a bare `except Exception`. Every approval gets the
   hardcoded 300s default regardless of operator config.
2. **CR-02** — the nonce replay guard is bypassable for any approval persisted with a
   NULL `nonce` (legacy rows, or any non-broker insert path), because the check is
   gated on `approval.nonce is not None`.

Plus a fail-closed gap on expiry (NULL `expiry_at` never expires) and a `list_actionable`
query that silently drops NULL-expiry rows. Details below.

## Critical Issues

### CR-01: Configured `approval_ttl_seconds` is silently ignored (wrong attribute name, swallowed by bare except)

**File:** `services/agent-runtime/atlas_runtime/tool_service.py:115`
**Issue:** `_resolve_approval_ttl_seconds` reads
`config_service.load_config().permissions.approval_ttl_seconds`, but the config field
defined in `control_plane.py:99` is `permission` (singular):
`AtlasConfig.permission: PermissionConfig`. `AtlasConfig` is frozen with
`extra="forbid"`, so `.permissions` does not exist and the attribute access raises
`AttributeError`. That exception is caught by the broad
`except Exception` on line 117, so the function silently returns
`_DEFAULT_APPROVAL_TTL_SECONDS` (300) every time. The operator-configurable
`PermissionConfig.approval_ttl_seconds` (added this phase, SEC-02) is therefore dead
code — the durable config path never takes effect, and an operator who shortens the TTL
to harden the stale gate gets no change in behavior. The only TTL override that works is
the test/`ctx['approval_ttl_seconds']` path. The bug is invisible because the failure is
absorbed as a "safe default."
**Fix:**
```python
ttl = config_service.load_config().permission.approval_ttl_seconds
```
Additionally, narrow the `except` so a real config-read failure is distinguishable from
a coding error (catch `(ImportError, OSError, ControlPlaneError, ValueError)` rather than
bare `Exception`, or log before falling back). A bare `except Exception` over an
attribute access permanently hides this class of typo.

### CR-02: Nonce replay guard is bypassed when the persisted nonce is NULL

**File:** `services/agent-runtime/atlas_runtime/permission_broker.py:486`
**Issue:** The SEC-02 replay defense is:
```python
if approval.nonce is not None and approval.nonce != nonce:
    raise StaleApprovalError(...)
```
Gating the comparison on `approval.nonce is not None` means any approval row whose
`nonce` column is NULL accepts **any** caller-supplied nonce, including a forged or
replayed one. The malformed-claim guard at line 463 rejects a *blank* incoming nonce,
but it does not require the stored nonce to be present — so a claim with
`nonce="anything-nonblank"` against a NULL-nonce approval passes the guard entirely. All
0017 columns are nullable by design (migration comment lines 10, 96–97), and approvals
can be inserted by any path that omits the nonce (legacy `invoke()` callers that pass no
surface kwargs still insert a row; only the `surface_kind=='api'` deny branch and the
fresh-nonce branch are exercised — a write/shell approval with `surface_kind` other than
`api` and no nonce supplied is possible). The contract claims (docstring line 454) "a
stale/replayed claim can never win the race," but a NULL-nonce row has no replay
protection at all.
**Fix:** Fail closed when the persisted nonce is absent on a surface-scoped claim:
```python
if approval.nonce is None or approval.nonce != nonce:
    raise StaleApprovalError(
        f"nonce mismatch for approval {approval_id!r} (replay/stale/missing)"
    )
```
If legacy NULL-nonce rows must remain claimable through a different (non-replay-guarded)
path, make that an explicit, separately-audited branch rather than an implicit hole in
the guard. As written, the security control is opt-out by data state.

## Warnings

### WR-01: NULL `expiry_at` never expires — fail-open hole in the SEC-02 stale gate

**File:** `services/agent-runtime/atlas_runtime/permission_broker.py:493`
**Issue:** The expiry check is wrapped in `if approval.expiry_at is not None:`. Any
approval persisted with a NULL `expiry_at` (every legacy row, and any non-`invoke()`
insert) bypasses the expiry deny entirely and remains claimable forever. Combined with
CR-02, a NULL-nonce + NULL-expiry approval has neither replay nor staleness protection.
The phase intent (SEC-02) is a fail-closed stale gate; "no deadline recorded" should be
treated as expired/untrusted for a surface claim, not as "valid indefinitely."
**Fix:** Treat a missing expiry as fail-closed for surface-scoped claims:
```python
if approval.expiry_at is None:
    raise StaleApprovalError(f"approval {approval_id!r} has no expiry (fail-closed)")
# ... then the now > expiry_dt check
```
or, if legacy rows are intentionally exempt, require a `surface_session_id` to be present
(which this path already implies) and document the exemption explicitly.

### WR-02: `list_actionable` drops NULL-expiry pending rows via `expiry_at > ?`

**File:** `services/agent-runtime/atlas_runtime/permission_broker.py:225`
**Issue:** The actionable query filters `... AND expiry_at > ?`. In SQLite, any
comparison with NULL yields NULL (not true), so a pending approval whose `expiry_at` is
NULL is silently excluded from the actionable queue. A surface that legitimately queued a
pending approval without an expiry would never see it surface in the sidebar/prompt — a
silent disappearance, not an error. This is the inverse risk of WR-01 (there NULL is
fail-open; here NULL is invisible), so the two NULL semantics are also internally
inconsistent. If the policy is "every surface approval must have an expiry," enforce it at
insert/schema level; otherwise handle NULL explicitly here.
**Fix:**
```sql
WHERE surface_session_id = ? AND status = 'pending'
  AND (expiry_at IS NULL OR expiry_at > ?)
```
(or, paired with WR-01's enforcement, make NULL expiry impossible so the query is
unambiguous).

### WR-03: Expiry comparison is lexical on ISO strings, not on datetimes

**File:** `services/agent-runtime/atlas_runtime/permission_broker.py:225`, `:498`
**Issue:** Two different comparison strategies exist for the same expiry concept. In
`claim()` (line 494–498) the comparison is a proper timezone-aware `datetime` compare
(`now_dt > expiry_dt`, with a `tzinfo` backfill). In `list_actionable` (line 225) the
comparison is a raw SQL string compare `expiry_at > _now_iso()`. ISO-8601 string ordering
only equals chronological ordering when both strings use the same offset (e.g. both
`+00:00`) and identical formatting. `_now_iso()` emits `...+00:00`; the seeded test rows
also use `+00:00`. But nothing enforces that every persisted `expiry_at` is UTC with a
`+00:00` suffix (a row written with `Z`, a different offset, or naive form would sort
incorrectly). The lexical path can therefore include an actually-expired row or exclude a
live one. The two code paths should share one comparison contract.
**Fix:** Normalize all persisted `expiry_at` to a single canonical UTC `+00:00` ISO form
at write time (already true for `invoke()`), and document the invariant; or compare in
Python after loading. At minimum, assert the canonical form so a drifted writer fails
loudly rather than mis-sorting.

### WR-04: `claim()` re-reads the row after `ToolApprovalError` outside the lock — loser can observe a transient `executing` status

**File:** `services/agent-runtime/atlas_runtime/permission_broker.py:541-548`
**Issue:** On a lost race, the loser catches `ToolApprovalError` and re-reads via
`tool_service._load(conn, approval_id)` (no lock held), then raises `AlreadyDecided` with
`current.status`. Between the winner's atomic flip to `executing` and its later
`_set_terminal` to `executed/failed`, the loser's re-read can observe `status='executing'`
and `decision`/`reason` still unset. `AlreadyDecided` then carries `status='executing'`
(non-terminal) and `decision=None`, contradicting the docstring promise (lines 15–16,
447–448) that the loser receives the "winning outcome." The PERM-06 test only asserts the
loser's status is `not in (None, 'pending')`, so `executing` passes the test while
violating the documented contract. A surface rendering the loser's result would show a
transient/incomplete state.
**Fix:** Either have the loser poll/re-read until the row is terminal
(`status in ('executed','rejected','failed')`) before constructing `AlreadyDecided`, or
document that `AlreadyDecided.status` may be the transient `executing` and have callers
treat it as "decided, outcome pending." The current docstring overstates the guarantee.

### WR-05: `record_allow_rule` / consume path runs after the winner emits, with no scope re-validation of `terminal.surface_kind`/`workspace_root`

**File:** `services/agent-runtime/atlas_runtime/permission_broker.py:558-574`
**Issue:** When `record_rule` is requested, the rule's `arg_pattern` is set from
`terminal.args_normalized` and its scope from `terminal.workspace_root` /
`terminal.surface_kind`. If the approval was persisted with NULL `workspace_root`,
`surface_kind`, or `args_normalized` (all nullable per 0017), the resulting
`SessionAllowRule` is constructed with `None` for `NOT NULL` columns
(`workspace_root`, `surface_kind`, `arg_pattern` are `NOT NULL` in the DDL, lines 46–49).
The Pydantic model `SessionAllowRule` declares these as `str` (non-Optional), so model
construction raises a `ValidationError` — but only at claim time, after the approval has
already executed and the winner audit event is about to be emitted. The result is a claim
that succeeds and executes the side effect, then throws while recording the convenience
rule, surfacing as an unexpected exception to the caller after an irreversible action.
Pattern-matching reliability (`match_allow_rule` exact `==`) also silently fails if
`args_normalized` was NULL.
**Fix:** Validate that `terminal.workspace_root`, `terminal.surface_kind`, and
`terminal.args_normalized` are all non-NULL before attempting `record_allow_rule`, and
skip rule recording (or raise before execution) when they are absent. Recording a
convenience rule must never be able to throw after the gated side effect has run.

## Info

### IN-01: `list_outcomes` casts `limit` to `int()` without bounds, accepting negatives

**File:** `services/agent-runtime/atlas_runtime/permission_broker.py:254`
**Issue:** `params = (int(limit),)` — a negative limit is passed straight to SQL
`LIMIT ?`. SQLite treats a negative LIMIT as "no limit," so `limit=-1` returns all rows,
the opposite of a caller's likely intent to cap output. Not a security issue (read-only
projection), but a surprising contract.
**Fix:** `params = (max(0, int(limit)),)` or reject negative values with a `ValueError`.

### IN-02: `match_allow_rule` constructs `SessionAllowRule(created_at=row[7])` from a raw string

**File:** `services/agent-runtime/atlas_runtime/permission_broker.py:392-401`
**Issue:** `created_at=row[7]` passes the stored ISO string to a field declared as
`datetime.datetime`. Pydantic v2 will coerce a valid ISO string, so this works, but it is
inconsistent with the explicit `.isoformat()` round-trip used elsewhere and relies on
implicit coercion. If a row ever stored a non-ISO value the failure would be opaque.
**Fix:** Parse explicitly (`datetime.datetime.fromisoformat(row[7])`) for symmetry with
the write path, matching the `_row_to_approval` convention.

### IN-03: Broker docstring claims nonce/expiry are "not enforced yet" but the code does enforce them

**File:** `services/agent-runtime/atlas_runtime/permission_broker.py:36-38`
**Issue:** The module docstring states "The nonce/expiry replay guard (SEC-02) ... land in
Plans 04/05; `claim` accepts the `nonce` param now and threads it through without
enforcing a mismatch yet." But `claim()` (lines 484–519) *does* enforce nonce mismatch and
expiry. Stale documentation about a security control's status is a maintenance hazard — a
future reader may "add" enforcement that already exists, or assume it is absent. Reconcile
the docstring with the implemented behavior (and with CR-02/WR-01 once the NULL holes are
closed).
**Fix:** Update the module docstring to reflect that nonce and expiry are enforced in
`claim()` as of this phase.

### IN-04: `reconcile_executing_orphans` reads orphan ids outside the lock (benign TOCTOU, but worth a note)

**File:** `services/agent-runtime/atlas_runtime/permission_broker.py:615-632`
**Issue:** The initial `SELECT id ... WHERE status='executing' AND result IS NULL` runs
before acquiring the lock; the per-row UPDATE inside the lock re-checks
`status='executing' AND result IS NULL`, so a row that became terminal in between is
correctly skipped (rowcount 0). This is safe as written. The note: the early-return
`if not orphans: return 0` can race a row that *becomes* an orphan immediately after the
read, but reconcile is a startup sweep and idempotent, so the residual risk is acceptable.
No fix required; documenting that the guarded UPDATE is what makes this correct so a future
refactor does not "optimize away" the redundant `WHERE` predicate.
**Fix:** None required. Keep the `WHERE status='executing' AND result IS NULL` guard on the
UPDATE (it is load-bearing, not redundant).
