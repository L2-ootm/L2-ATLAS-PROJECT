## Summary

Final re-review (iteration 3) confirming the WR-01 fix. **WR-01 is resolved** and no
Critical or Warning regression was introduced. All standing security invariants continue
to hold. The phase is clean: only three non-blocking Info items remain (two carried over,
one newly noted timeout edge that is benign).

**WR-01 resolution (verified).** The lost-race loser no longer reads the race-prone
`decision` column when constructing `AlreadyDecided`. It now derives both `decision` and
`reason` from the authoritative terminal status the winner converged to
(`permission_broker.py:629-639`):

- `status == 'rejected'` -> `decision='reject'`, `reason=current.reason` (the row's reason
  is authoritative: only `tool_service.reject()` produces `rejected`, and it persists the
  winning reject reason at `tool_service.py:409-412`).
- `status in ('executed','failed')` -> `decision='approve'`, `reason=None` (only
  `tool_service.approve()` produces these states; it never writes `decision` and never
  writes a row `reason`, so any persisted reason would be stale and is correctly dropped).

The mapping is exhaustive and sound against every terminal state `tool_service` can write:
`executed`/`failed` originate only from `approve()` (`tool_service.py:377`, and the
adapter-exception path `:369`), and `rejected` originates only from `reject()`
(`tool_service.py:409`). `reconcile_executing_orphans` also yields `failed`, which still
maps correctly to an `approve` decision (the action was authorized and attempted). The
prior contradiction — a winning `executed` status paired with a loser's stale
`decision='reject'` — can no longer occur. The non-status `decision`-column write at
`:581-585` remains, but it is now decoupled from the `AlreadyDecided` contract, so its
last-writer-wins behavior no longer leaks to a caller.

**No regression introduced.** The WR-01 change is confined to the loser branch
(`:629-639`); it adds no SQL, no lock acquisition, and no new control flow. The
bounded-poll WR-04 logic above it is unchanged.

**Standing invariants re-verified (all hold):**

- **No status-flip UPDATE in `claim()`.** The broker's only direct UPDATE is the non-status
  `decision`-column write gated on `status='pending'` (`permission_broker.py:581-585`). The
  atomic at-most-once guard (`UPDATE ... WHERE id=? AND status='pending'`, `rowcount==1`)
  lives only in `tool_service.approve()` (`tool_service.py:352-357`) and
  `tool_service.reject()` (`:408-413`).
- **Fail-closed expiry/nonce.** Nonce: NULL/blank-stored or mismatch all rejected before any
  UPDATE (`:533`). Expiry: NULL `expiry_at` treated as expired, strict `>` with tz-aware
  normalization, deny audit emitted before failing closed (`:543-575`); `list_actionable`
  applies the identical contract (`:243-249`).
- **Parameterized SQL.** Every query uses `?`/named placeholders. The only string
  interpolation is the static `tool_service._COLS` column list (not user-derived).
- **`_redact` secret boundary.** Single redaction at the `_normalize_args`/`invoke`
  boundary (`tool_service.py:93,166,251`); the broker adds no second redaction path.
- **PERM-07 no policy/config-write import.** The broker imports only `audit_service`,
  `mission_service`, `tool_service`. Allow-rules write exclusively to `session_allow_rules`,
  bounded to the per-session 4-tuple, no wildcard, no global policy mutation.
- **Audit emit outside the lock.** Expiry deny (`:551-568`, no lock held), winner provenance
  (`:685-701`, after all rule-record lock blocks released), reconcile provenance
  (`:742-753`, after the `with lock:` block).

The broker contract suite (`test_permission_broker.py`, 9 tests) covers PERM-01/02/05/06/07,
SEC-02, and AUD-02, including the concurrent at-most-once race.

## Info

### IN-01: `list_outcomes` casts `limit` to `int()` without bounds, accepting negatives

**File:** `services/agent-runtime/atlas_runtime/permission_broker.py:275`
**Issue:** `params = (int(limit),)` passes a negative limit straight to SQL `LIMIT ?`.
SQLite treats a negative LIMIT as "no limit," so `limit=-1` returns all rows — the opposite
of a caller's likely intent to cap output. Read-only projection, so not a security issue,
but a surprising contract. (Carried over; non-blocking.)
**Fix:** `params = (max(0, int(limit)),)` or reject negatives with a `ValueError`.

### IN-02: `match_allow_rule` constructs `SessionAllowRule(created_at=row[7])` from a raw string

**File:** `services/agent-runtime/atlas_runtime/permission_broker.py:436-445`
**Issue:** `created_at=row[7]` passes the stored ISO string to a `datetime.datetime` field.
Pydantic v2 coerces a valid ISO string so this works, but it is inconsistent with the
explicit `.isoformat()` write path and relies on implicit coercion; a non-ISO stored value
would fail opaquely. (Carried over; non-blocking.)
**Fix:** Parse explicitly (`datetime.datetime.fromisoformat(row[7])`) for symmetry with the
write path and the `_row_to_approval` convention.

### IN-03: loser's `AlreadyDecided` can carry a non-terminal `status='executing'` if the winner crashed mid-flight

**File:** `services/agent-runtime/atlas_runtime/permission_broker.py:610-639`
**Issue:** If the winner dies between the atomic flip to `executing` and `_set_terminal`, the
loser's bounded 5s poll (`:610`) exits with `current.status == 'executing'`. The `else`
branch (`:632-634`) then constructs `AlreadyDecided(status='executing', decision='approve',
reason=None)`. The `status` is non-terminal, which is slightly looser than the WR-04 intent
of always carrying a terminal outcome. This is benign — `executing` paired with `approve` is
not a contradiction (an executing claim is an approve in flight), and
`reconcile_executing_orphans` sweeps such orphans to `failed` on the next startup. It is a
pre-existing timeout edge, not introduced or worsened by the WR-01 fix, and does not affect
the decision/reason consistency WR-01 addressed. Noted for completeness only.
**Fix (optional):** On poll timeout, either raise a distinct typed error (e.g.
`StaleApprovalError`/a "winner indeterminate" signal) rather than an `AlreadyDecided` with a
non-terminal status, or document `executing` as an accepted `AlreadyDecided.status` value.
