developer workflows and proves it is an extensible harness, not a closed demo"? **Yes.**

## Success criteria

### SC1 — Workspace/GitHub/WebFetch/Webhook work end-to-end; read-only default, write/shell gated — ✅
- Four adapters shipped (stdlib + `gh` only): `workspace` (boundary-bounded), `github` (gh argv,
  honest-failure), `web_fetch` (GET, SSRF-guarded, size-capped), `webhook_notify` (POST, SSRF-guarded).
- Single chokepoint `tool_service.invoke` → `policy.decide`: read auto-runs; write/shell short-circuit
  to a pending `tool_approvals` row.
- **Live**: `atlas tools call … -- workspace` returned a ToolResult; `… -- webhook_notify` produced a
  PENDING approval and did NOT execute; **approve** ran the deferred adapter (terminal `failed` = honest
  404 from the test URL); **reject** left it `rejected` with no execution.
- Tests: `test_tool_adapters` (13), `test_tool_service` (7), `test_policy` decide (4).

### SC2 — Tool Manifest v0 implemented + documented; adding a tool = manifest + adapter — ✅
- Frozen `ToolManifest` (name/description/risk_level/permissions/inputs/outputs/audit_events) in
  atlas-core; `tool_registry` loads + validates YAML manifests (fail-fast) and binds name→adapter.
- Four manifests validate; `docs/tools.md` documents the schema + the manifest+adapter workflow;
  README points to it.
- Tests: `test_tool_manifest` (5), `test_tool_registry` (5). **Live**: `GET /v1/tools/manifests` → 4 tools.

### SC3 — Policy/permissions visible in the WebUI; read-only default, approval gate, no-sensitive-data posture — ✅ (1 confirmatory visual deferred)

  no inline execution). tsc + `npm run build` green; feeding endpoints verified returning correct data.
- **Deferred (confirmatory only):** the pure visual eyeball of the rendered panels in a browser. Stack
  is live at `http://127.0.0.1:5173` (System page). Functional behavior is already proven via CLI+HTTP.

- `AuditEvent.event_type` extended with snake_case `tool_requested`/`tool_completed`/`tool_failed`
