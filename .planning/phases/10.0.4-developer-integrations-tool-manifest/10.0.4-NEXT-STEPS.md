## Execution Sequence

Plans execute in wave order. Dependency graph: `00 → 01 → {02, 03} → 04 → 05`.

### Wave 0 — Plan 00: Test Stubs + Temp-DB Conftest

**What:** Write failing test stubs for every plan's verification. Extend `audit_events.event_type` TEXT column (no CHECK constraint) to confirm `tool_requested`/`tool_completed`/`tool_failed` persist without migration.

**Verify after:** `pytest packages/atlas-core/tests services/agent-runtime/tests -q` — all new stubs should FAIL (RED). This is the Nyquist gate — stubs must land red before any implementation.

**Caution:**
- Do NOT let stubs pass accidentally by importing existing code that already satisfies them. Each stub must assert something that doesn't exist yet.
- Confirm `event_type` column has no CHECK constraint by querying the schema (`PRAGMA table_info(audit_events)`). If a CHECK exists, the three new verbs need a migration — which would change the plan.

### Wave 1 — Plan 01: Core Schema + Policy

**What:** `ToolManifest` / `ToolResult` / `ToolApproval` frozen Pydantic models in `packages/atlas-core`. Extend `AuditEvent.event_type` Literal with three tool verbs. Implement `policy.decide(manifest, mode) → PolicyDecision` (read=allow, write/shell=needs-approval). Add `pyyaml>=6,<7` dependency.

**Verify after:**
- `pytest packages/atlas-core/tests/test_tool_manifest.py -q` — manifest validation (bad input → rejection)
- `pytest services/agent-runtime/tests/test_policy.py -q` — policy.decide returns correct decisions

**Caution:**
- `ToolManifest` must be frozen (D-012). No mutable defaults.
- `policy.decide` is the single chokepoint entry — every tool call flows through it. Get the signature right: `decide(manifest: ToolManifest, mode: PolicyMode) → PolicyDecision`. The `PolicyMode` enum (read/write/shell) maps from the manifest's `risk_level`.
- pyyaml is the only new dep. Verify it's already in the venv (`pip show pyyaml`). If not, add to `pyproject.toml` under `[project] dependencies` — not as an optional extra.

### Wave 2 — Plans 02 + 03: Adapters + Tool Service (parallel)

**What:**
- **Plan 02:** Four adapters (`workspace`, `github`, `web_fetch`, `webhook_notify`) + YAML manifests for each. SSRF guards on web_fetch. Workspace bounded by `check_workspace_boundary`. GitHub shells to `gh` CLI (argv-only, no shell=True). Webhook is POST with approval gate.
- **Plan 03:** `tool_registry` (loads manifests, binds adapters, fail-fast on invalid). `tool_service.invoke()` chokepoint — calls `policy.decide`, routes reads to immediate execution, routes writes/shell to pending approval row. Migration `0013_tool_approvals.sql`. TOCTOU-safe atomic UPDATE.

**Verify after:**
- `pytest services/agent-runtime/tests/test_tool_adapters.py -q` — four adapters incl. SSRF (5 cases), workspace escape, gh failure
- `pytest services/agent-runtime/tests/test_tool_registry.py -q` — fail-fast load, adapter binding
- `pytest services/agent-runtime/tests/test_tool_service.py -q` — invoke/propose/approve/reject + audit emit + redaction + concurrent approve

**Caution (critical):**
1. **SSRF guard must be bulletproof.** web_fetch `_assert_safe` blocks: non-http(s) schemes, loopback (127.x, ::1, 0.0.0.0), RFC1918 (10.x, 172.16-31.x, 192.168.x), link-local, and DNS rebinding. Size cap + timeout. GET-only. Test with real malicious URLs.
2. **TOCTOU in approve()** — use the exact Phase C pattern: `UPDATE tool_approvals SET status='executing' WHERE id=? AND status='pending'` + check `rowcount==1`. If rowcount != 1, the approval was already claimed.
3. **Redact-once at boundary** — args and results are JSON-stringified and passed through `SECRET_PATTERNS` redaction ONCE in `tool_service.invoke()` before persisting. Never redact twice (double-redaction breaks audit readability).
4. **0013 migration** must be additive only (CREATE TABLE IF NOT EXISTS). No DROP, no ALTER existing tables. Mirror 0012 pattern. `run_id` is NOT FK-enforced (the audit event carries the FK).
5. **gh CLI auth delegated** — ATLAS never handles tokens. The adapter shells to `gh` and trusts the user's existing login. If `gh` isn't authenticated, the adapter returns a clear error, not a crash.

### Wave 3 — Plans 04 + 05: CLI + Gateway + Cockpit (sequential)

**What:**
- **Plan 04:** `atlas tools` CLI group (list/invoke/approve/reject). Gateway `/v1/tools/*` + `/v1/tools/approvals/*` routes (dispatch-only, D-022).
- **Plan 05:** Cockpit System page: POLICY panel (read-only badge, risk-tier legend, no-sensitive-data posture), TOOLS list from manifests, Tool Approvals queue (Approve/Reject). api.ts client. `docs/tools.md` authoring doc. Human-verify checkpoint.

**Verify after:**
- `pytest services/agent-runtime/tests/test_cli_tools.py -q` — CLI commands
- `cargo test -p atlas-gateway` — gateway routes (if Rust changed)
- `npm run build` — cockpit tsc/lint/build green
- **Human-verify:** `atlas up` → System page → confirm read-only badge, risk legend, tools list, approval queue → approve/reject a pending tool call

**Caution:**
1. **Gateway dispatch-only** — Rust routes must: validate request body → call `dispatch_atlas` → parse JSON response → return. No DB access, no state mutation in Rust. This is D-022.
2. **Gateway 400 on invalid body** — cargo test must assert empty/invalid body → 400. Don't let malformed requests reach Python.
3. **Human-verify is blocking** — Plan 05 has a manual checkpoint that cannot be automated. The operator must visually confirm the POLICY panel renders literal text ("Read-only by default", risk legend, "No sensitive data stored"). Plan cannot be marked done without this.
4. **webhook_notify POST path** — only exerciseable against a real reachable endpoint. Unit tests mock `urlopen`. Accept mocked-only verification for POST unless a localhost mock target is set up.

## Risk Register

| # | Risk | Severity | Mitigation |
|---|------|----------|------------|
| 1 | SSRF bypass in web_fetch (DNS rebinding, IPv6) | HIGH | `_assert_safe` blocks all private/loopback/link-local; test with `http://127.0.0.1`, `http://[::1]`, `http://10.0.0.1`, `http://169.254.x.x` |
| 2 | TOCTOU race in tool approval | HIGH | Atomic UPDATE WHERE status='pending' + rowcount==1; unit test with concurrent approve |
| 3 | Double redaction of args/results | MEDIUM | Redact once in `tool_service.invoke()` boundary; never in adapters or CLI |
| 4 | Gateway D-022 violation (Rust doing business logic) | HIGH | Plan 04: dispatch-only pattern; cargo test asserts 400 on invalid body; no DB imports in Rust |
| 5 | `event_type` CHECK constraint blocks new verbs | LOW | Confirm no CHECK in Wave 0; if exists, add migration |
| 6 | pyyaml not in venv | LOW | Check `pip show pyyaml` before Plan 01; add to pyproject.toml if missing |
| 7 | `gh` not authenticated on operator machine | LOW | Adapter returns clear error; not a crash; human-verify step tests with real auth |
| 8 | 0013 migration conflicts with 0012 | LOW | Additive only (CREATE TABLE IF NOT EXISTS); no DROP/ALTER; mirror 0012 pattern |
| 9 | Plan 05 human-verify unfulfilled | MEDIUM | Operator must visually confirm panels; cannot skip this gate |
| 10 | temp-DB tests accidentally hit live DB | MEDIUM | All tests use `db.connect(tmp_path)` fixture; NEVER drive live CLI in smoke tests (memory: `cli-db-path-not-atlas-home`) |
