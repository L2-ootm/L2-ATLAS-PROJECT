# Phase 10.0.4: Developer Integrations & Tool Manifest - Context

**Gathered:** 2026-06-22
**Status:** Ready for planning

<domain>
## Phase Boundary

ATLAS proves it is an extensible harness â€” not a closed demo â€” by shipping four real
developer-workflow integrations behind a uniform, auditable, policy-gated tool layer.

Delivers:
- A **Tool Manifest v0** contract (YAML schema: name/description/risk_level/permissions/
  inputs/outputs/audit_events) + a `tool_registry` that loads manifests and binds adapters.
  Adding a tool = write a manifest + adapter.
- Four **integration adapters**: Local Workspace (read), GitHub (read via the `gh` CLI),
  Web Fetch (GET), Webhook Notification (POST) â€” read-only by default; write/shell/POST are
  approval-gated.
- A **policy + approval pipeline** generalizing the Phase C Discord proposeâ†’approveâ†’execute
  state machine to all tools (approval state in Python/SQLite; gateway dispatch-only, D-022).
- **Audit events** `tool_requested` / `tool_completed` / `tool_failed` on the existing bus.
- **WebUI policy visibility**: a System-page POLICY panel + manifest-driven tools list +
  a Tool Approvals queue (reusing the Discord Pending-Approvals UI pattern).

Out of scope: a full in-app tool-authoring UI; new third-party Python deps; any edit to the
D-001-locked `foundation/atlas-hermes/` tree.
</domain>

<decisions>
## Implementation Decisions

### Tool Manifest v0 (Area 1)
- One YAML manifest per tool under `services/agent-runtime/atlas_runtime/tools/manifests/*.yaml`,
  loaded by a `tool_registry` that mirrors the existing `agents/registry.py` pattern.
- `risk_level` taxonomy is `read | write | shell` â€” maps 1:1 to SC1 (read = default-allowed;
  write & shell = approval-gated).
- An adapter binds to its manifest by `name`; the registry resolves it to a Python adapter
  exposing `run(args, ctx) -> ToolResult`. The manifest declares inputs/outputs/permissions/
  audit_events.
- A new **frozen Pydantic `ToolManifest`** model lives in `packages/atlas-core` (D-012); manifests
  are validated at load time and an invalid/unknown manifest is a fail-fast load error.

### Integration adapters (Area 2)
- Four adapters, **no new Python dependencies** (anti-bloat): `workspace` (read file / list dir /
  grep, bounded by `policy.check_workspace_boundary`), `github` (shell out to the already-installed
  `gh` CLI â€” repo/issue/PR read), `web_fetch` (stdlib `urllib` GET), `webhook_notify` (stdlib HTTP
  POST).
- GitHub auth is **delegated to the user's existing `gh` login** â€” ATLAS handles no token. If a
  token is ever needed it is an `env:VAR` reference only (D-002).
- `web_fetch` safety: GET only; size cap + timeout; block non-`http(s)` schemes and
  localhost/private-IP targets (SSRF guard); accept text/JSON content.
- **Read-only is the global default.** Workspace writes, shell actions, and webhook POSTs are
  write-class and require explicit operator approval (SC1).

### Policy & approval pipeline (Area 3)
- Generalize the Phase C Discord pipeline into a tool-agnostic one: a `tool_approvals` table
  (migration **0013**, additive, mirroring `0012_discord_approvals`) + `tool_service.propose/
  approve/reject`. Approval state lives in Python/SQLite; the Rust gateway stays dispatch-only
  (D-022); lifecycle is audit-first (D-002).
- Extend `PolicyDecision` with `requires_approval: bool` (the `ToolCall` docstring already promises
  it) and add `policy.decide(manifest, mode) -> PolicyDecision` returning allow / deny /
  needs-approval from `risk_level` + the read-only mode flag.
- **Single chokepoint:** every tool call flows through `tool_service.invoke()`, which calls
  `policy.decide()` before execution; write/shell short-circuit to a pending approval row.
- Tool args and results are secret-**redacted** via the existing `SECRET_PATTERNS` at the audit
  boundary (reuse the Phase C redaction); args/results persist as JSON strings (D-013).

### Audit events + WebUI policy surface (Area 4)
- Extend `AuditEvent.event_type` Literal with `tool_requested`, `tool_completed`, `tool_failed`
  (snake_case, matching `tool_call`/`discord_action`; TEXT column â†’ **no migration**). The SC's
  dotted `tool.requested` form is the conceptual/external label.
- SC3 visibility: extend the existing **System** page with a POLICY panel â€” read-only-by-default
  badge, risk-tier legend, explicit no-sensitive-data posture statement â€” plus a Tools/Integrations
  list rendered from the manifests (name/risk_level/permissions).
- Approval queue UI reuses the Discord **Pending Approvals** pattern (Approve/Reject); proposals
  never execute inline.
- In scope this phase: read-only policy/tools visibility + the approval queue. A full tool-authoring
  UI is deferred.

### Claude's Discretion
- Exact manifest field names/ordering within the agreed schema, adapter file layout, gateway route
  shapes, and CLI command surface â€” choose to match existing `agents/`, `discord_*`, and gateway
  conventions.
</decisions>

## Existing Code Insights

### Reusable Assets
- `services/agent-runtime/atlas_runtime/policy.py` â€” `PolicyDecision` (allowed/reason),
  `check_workspace_boundary` (CWD-escape-safe, cross-platform), `check_tool_allowed` (allowlist).
  Foundation for the workspace adapter + the policy chokepoint. Needs `requires_approval` + a
  `decide()` aggregator.
- `packages/atlas-core/atlas_core/schemas/core.py` â€” `AuditEvent` (event_type Literal, `tool_name`,
  `policy_result`, JSON-string `data`), `ToolCall` (policy_allowed/requires_approval fields already
  defined). Add `ToolManifest` here; extend the Literal with the three tool verbs.
- **Phase C Discord write surface** â€” `discord_service.propose/approve/reject/list`,
  `DiscordApproval` schema, `0012_discord_approvals.sql`, `atlas discord approvals/approve/reject`,
  gateway `POST /v1/discord/writes` + approvals routes, and the cockpit Pending-Approvals panel.
  This is the end-to-end template for the tool approval pipeline (state machine, CLI, gateway,
  cockpit, redaction).
- `agents/registry.py` â€” registry pattern to mirror for `tool_registry`.
- `audit_service.emit(conn, lock, run_id, event_type, data, policy_result)` â€” the audit bus entry
  point; emits append-only, fail-open.

### Established Patterns
- D-022 dispatch-only gateway: Rust validates + forwards to the `atlas` CLI; Python owns all writes
  and state. D-002 audit-first: every state transition emits an audit event; secrets are `env:VAR`
  refs only and are redacted before persistence.
- Additive migrations follow `0012_discord_approvals.sql` (CREATE TABLE IF NOT EXISTS, `run_id` not
  FK-enforced; the emitted audit event carries the `runs(id)` FK). Migration runner =
  `db.apply_migrations` (`atlas db init`), drift-tolerant, tracked in `schema_migrations`.
- CLI sub-apps wired into `cli/main.py` via try/except import (e.g. the discord/wiki groups).

### Integration Points
- New: `atlas_runtime/tools/` (registry + adapters + manifests), `tool_service.py`, `tool_approvals`
  migration, atlas-core `ToolManifest`, `atlas tools` CLI group, gateway tool/approval routes,
  cockpit System POLICY panel + Tool Approvals queue.
- `gh` CLI is present on this machine (used elsewhere in the project) â€” the GitHub adapter shells to
  it; no PyGithub.
