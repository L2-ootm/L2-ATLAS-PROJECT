# Plan 10.0.4-03 — Summary

**Wave:** 3 (registry + chokepoint + migration) · **Status:** complete · **Executed:** 2026-06-22 (inline)

## What shipped
The single policy chokepoint + approval state machine — a near-verbatim generalization of Phase C.

- **`infra/migrations/0013_tool_approvals.sql`** — additive `tool_approvals` table (mirrors 0012;
  run_id stored, not FK-enforced; the audit event carries the runs(id) FK) + status index. Applies
  via `db.apply_migrations` (confirmed: table present through the real runner).
- **`tools/registry.py`** — `load_manifests` (validate-or-fail-fast), `ToolRegistry` with
  `.manifests`, `.known_tools()`, `.resolve(name)` (raises ValueError on unknown), `get_registry()`
  singleton. Binds the four manifest names to their adapter `run` callables. Does **not** probe `gh`
  at load.
- **`tool_service.py`** — cloned from `discord_service.py`:
  - `invoke(conn, lock, *, tool_name, args, mode, ctx, reason)` — the single chokepoint: emits
    `tool_requested`, calls `policy.decide`; read-class runs the adapter now and emits
    `tool_completed` (or `tool_failed`) + a linked redacted `ToolCall`; write/shell short-circuits to
    a `pending` tool_approvals row (emits `approval`) and returns the `ToolApproval` — **no inline exec**.
  - `approve` — atomic `UPDATE … WHERE status='pending'` (rowcount==1 TOCTOU claim), runs the deferred
    adapter, flips executed/failed, emits `tool_completed`/`tool_failed`.
  - `reject` — atomic claim to `rejected`, no execution.
  - `list_approvals`/`get_approval`/`_set_terminal`/`_load`/`_row_to_approval` + `ToolApprovalError`.
  - Redact-once at the boundary (`_redact(json.dumps(args))`); `ensure_operator_run` FK bootstrap.

## Deviation
- Per the Plan-00 test, `invoke` **returns** `ToolResult(ok=False)` on adapter error (emitting
  `tool_failed`) rather than re-raising — cleaner for the CLI/gateway callers. An adapter that returns
  `ok=False` (honest failure, e.g. gh non-zero) also emits `tool_failed`.
