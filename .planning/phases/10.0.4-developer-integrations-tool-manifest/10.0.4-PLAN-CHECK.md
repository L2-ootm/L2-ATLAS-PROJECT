## Warnings (non-blocking)

**W-1 [nyquist_compliance] VALIDATION.md frontmatter flags are stale**
- nyquist_compliant: false and wave_0_complete: false in 10.0.4-VALIDATION.md frontmatter, and the Validation Sign-Off checklist is all-unchecked with Approval: pending.
- The plans themselves DO satisfy Nyquist (Wave 0 stubs precede impl; automated verify per task). The flag is a process-tracking artifact, not a plan defect, but it should be flipped to true once Wave 0 (Plan 00) completes or the verifier will read it as non-compliant.
- Fix: set nyquist_compliant: true / wave_0_complete: true and tick the sign-off after Plan 00 executes.

**W-2 [research_resolution] RESEARCH.md Open Questions lacks the (RESOLVED) marker**
- The section heading is '## Open Questions' (no (RESOLVED) suffix) and each of OQ1-OQ3 carries a Recommendation line but no explicit RESOLVED marker.
- All three are in fact resolved and honored in the plans: OQ1 (mode = code default read_only) -> Plan 01 Task 2; OQ2 (webhook SSRF-guard + approval, no allowlist) -> Plan 02 Task 2; OQ3 (copy discord _get_connection, keep monkeypatch seam) -> Plan 04 Task 1. So this is a marker/hygiene gap, not an unresolved blocker.
- Fix: rename to '## Open Questions (RESOLVED)' and prefix each with RESOLVED: for an unambiguous verifier trace.

**W-3 [dependency_correctness] Wave numbers in plan frontmatter do not match VALIDATION.md wave column**
- Plan frontmatter waves run 0,1,2,3,4,5 (one wave per plan). VALIDATION.md Per-Task map uses waves 0,1,2,3 and assigns plan IDs 00..06 (it references a Plan 06 and an SC mapping pre-dating the final 6-plan split into 00-05). The depends_on edges (the source of truth) are correct and acyclic: 00 -> 01 -> {02, 03} -> 04 -> 05.
- Plan 03 frontmatter correctly lists depends_on: [10.0.4-01, 10.0.4-02], so the real edge is 02 -> 03 and ordering is safe. The frontmatter wave integers are cosmetic and slightly over-serialized but not wrong.
- Fix (optional): reconcile VALIDATION.md task-ID/wave table to the final 00-05 numbering; drop the stale Plan 06 row.

**W-4 [context_compliance] webhook_notify POST path is exercised end-to-end only by the human-verify checkpoint**
- CONTEXT lists Webhook Notification (POST) as write-class/approval-gated, in scope. The plans implement it (Plan 02) and gate it (Plan 03). Compliant. Flagging only that the human-verify checkpoint (Plan 05 step 6) is the ONLY end-to-end exercise of the POST path, depending on the operator having a reachable webhook target. Unit tests cover it with mocked urlopen.
- Fix (optional): add a localhost-mock note to the checkpoint, or accept mocked-only verification for the POST path and state it in the checkpoint resume-signal.

**W-5 [task_completeness] Plan 00 Task 1 verify uses a grep-on-pytest-output one-liner that can pass spuriously**
- The verify pipes pytest -q output through grep -E for tokens including tool_requested|fail|error. That matches the literal token tool_requested in the test source/IDs even when the assertion does not run RED as intended. The acceptance_criteria correctly require RED, but the automated command does not strictly prove RED vs GREEN.
- Fix (optional): assert a non-zero pytest exit or check explicit FAILED/passed counts rather than grepping tokens.
