# Phase 10.7: Web Agent Surface & Permission Queue UX — Specification

**Created:** 2026-06-29
**Ambiguity score:** 0.12 (gate: ≤ 0.20)
**Requirements:** 12 locked

## Goal

Replace the React Console's ad-hoc agent path with the canonical shared surface-session
protocol and deliver one configurable, audited permission system whose policy core is shared
by every usable ATLAS surface while each surface owns only its presentation adapter.

## Background

ATLAS already has the stronger persistence and authority foundations: frozen surface-session
models, normalized surface events, a surface-scoped permission broker, atomic claims,
session-scoped allow rules, masked config contracts, and audited cross-surface outcomes. The
React Console currently bypasses those contracts through `/v1/console/stream`, keeps transient
turn state locally, and exposes global tool approvals on the System page without actionable
surface-session ownership.

Hermes provides the donor-quality behavior model in `foundation/atlas-hermes/tools/approval.py`
and its surface adapters: one guard pipeline, an unbypassable hardline floor, manual/smart/off
policy modes, once/session/always decisions, per-session blocking queues, timeout-denies,
ContextVar/thread-local isolation, and native channel renderers. Phase 10.7 adapts those proven
mechanisms into ATLAS contracts; it does not introduce a parallel Hermes runtime or replace the
ATLAS broker with an opaque dependency.

## Requirements

1. **Canonical Web agent surface**: The existing React Console becomes the canonical WebUI
   client of the shared surface-session protocol.
   - Current: Console posts directly to `/v1/console/stream` and stores transient messages
     without an ATLAS surface-session identity.
   - Target: Global and registered-project Web sessions create, execute, reconnect, resume,
     cancel, and terminate through the same session/run contracts used by other ATLAS surfaces.
   - Acceptance: Integration tests start both global and project Web sessions and prove their
     session, workspace, mission, run, agent/model, permission mode, and lifecycle identities
     remain stable through reconnect, resume, cancellation, and terminal completion.

2. **Normalized event parity**: WebUI renders the canonical normalized surface-event stream.
   - Current: Console consumes a separate NDJSON event shape with text, tool, result, and failure
     events only.
   - Target: WebUI consumes text, activity, tool calls/results, diffs, tasks/subagents, retries,
     retrieval provenance, approvals, errors, cancellation, and completion from the shared
     stream without surface-specific runtime events.
   - Acceptance: One reference event fixture containing every registered event kind renders in
     both WebUI and TUI conformance tests with matching sequence and terminal outcome.

3. **Single permission authority**: ATLAS exposes one policy-and-broker authority to every
   surface.
   - Current: The persisted ATLAS broker is surface-scoped, while several existing entry points
     still use separate or global approval paths.
   - Target: CLI, Go TUI, React WebUI, API/headless execution, enabled messaging channels, and
     future desktop adapters all invoke the same policy decision and broker lifecycle contracts;
     adapters only render and submit decisions.
   - Acceptance: Contract tests prove every current surface adapter calls the shared authority
     and that no production entry point can execute a guarded action through a parallel approval
     engine.

4. **Master policy with narrowing profiles**: A global master policy is the safety ceiling and
   surface profiles may only narrow it.
   - Current: ATLAS policy is effectively fixed read-only for write/shell tools and has no
     operator-configurable surface profile hierarchy.
   - Target: The master policy supports easy presets and explicit rules; surface, workspace,
     project, agent, and channel profiles can be stricter but cannot widen the master ceiling.
   - Acceptance: Policy-matrix tests cover master/profile precedence and prove every attempted
     widening override is rejected with a typed, audited error.

5. **Open and expressive operator configuration**: Operators can choose manual, smart, or
   full-autonomy behavior and express allow/deny exceptions.
   - Current: No complete shared configuration can state "allow everything except X/Y" or
     constrain autonomy to the active workspace.
   - Target: Versioned config supports presets plus ordered rules over tool/capability, risk,
     normalized arguments/command patterns, path/workspace, surface, agent, and decision scope.
     Full autonomy can allow all non-hardline actions while explicit deny rules still win.
   - Acceptance: Tests demonstrate manual, smart, and full-autonomy presets; allow-all-except
     tool/path rules; workspace-only rules; and deterministic explain output naming the matched
     rule, source, and effective decision.

6. **Unbypassable hardline floor**: Catastrophic actions remain denied under every policy mode.
   - Current: ATLAS risk metadata and path guards fail closed but do not yet expose a
     Hermes-quality, explicitly tested hardline command/action floor across all surfaces.
   - Target: A compact, versioned hardline ruleset blocks catastrophic filesystem, block-device,
     privilege-escalation, fork-bomb, shutdown/reboot, and equivalent actions before smart or
     full-autonomy evaluation.
   - Acceptance: The hardline corpus is denied under manual, smart, full-autonomy, cron,
     allow-once/session/always, and forged surface-profile cases; no user-configured allow rule
     can override it.

7. **Workspace and ATLAS self-management scopes**: Policy can confine agents to the active
   workspace while preserving explicit user-directed ATLAS maintenance.
   - Current: Workspace path guards exist, but permission configuration cannot express the full
     operator policy and the ATLAS installation/config scope is not a first-class capability.
   - Target: Operators can deny actions outside the active workspace. The ATLAS installation,
     update, and non-secret config locations are modeled as a protected maintenance scope that
     remains addressable for explicit user requests to upgrade or reconfigure ATLAS.
   - Acceptance: Tests prove workspace-only mode denies unrelated paths, explicit user-directed
     maintenance can reach only declared ATLAS maintenance roots, and all maintenance mutations
     pass path validation, masked config/update validation, audit logging, and secret redaction.

8. **Complete decision lifecycle**: Permission decisions support deny, allow once, allow for the
   owning session, and durable scoped allow without widening authority.
   - Current: The broker implements atomic claim and scoped allow primitives, but WebUI and other
     adapters do not expose one complete lifecycle.
   - Target: All adapters share expiry, nonce, claim, timeout, heartbeat, decision, execution,
     recovery, and terminal-outcome semantics. Silence and timeout always deny.
   - Acceptance: Concurrency/restart tests prove at-most-once execution, stale/foreign claims
     fail closed, each scope expires as defined, blocked runs stay alive while awaiting a live
     channel, and timeout produces a terminal denial.

9. **Conditional WebUI chrome**: Agent controls appear only when operationally relevant.
   - Current: Session state is embedded inside the Console page and approvals are permanently
     located on the System page.
   - Target: A minimal header appears for active/reconnectable agent state. A right queue sidebar
     appears only for actionable approvals or queued work owned by the matching WebUI session,
     or when the operator explicitly pins it.
   - Acceptance: Component tests prove header/sidebar visibility for inactive, active,
     reconnectable, matching-pending, foreign-pending, pinned-empty, and resolved states.

10. **Accessible responsive queue UX**: The Web queue is operable across supported viewport and
    assistive-technology conditions.
    - Current: The global approval list is pointer-oriented and is not a conditional,
      focus-managed session queue.
    - Target: Queue items expose risk, tool/action, normalized/redacted arguments, workspace,
      expiry, provenance, and decision choices with keyboard navigation, predictable focus,
      responsive layout, and screen-reader announcements.
    - Acceptance: Automated accessibility tests cover names/roles/states, keyboard-only
      approve/reject/cancel flows, focus restoration, live-region announcements, narrow/wide
      layout, and reduced-motion behavior with no critical violations.

11. **Shared operational controls**: Web project, model, config, permission, cancellation,
    reconnect, and resume controls use shared contracts.
    - Current: Web controls are split across Console, Models, and System routes and some use
      route-specific API shapes.
    - Target: The canonical Web surface reads and mutates only the masked config, project,
      provider/model, surface-session, run, and permission contracts; optimistic conflicts and
      restart-required settings are surfaced without silent overwrite.
    - Acceptance: Contract tests cover successful reads/writes, config revision conflicts,
      invalid project roots, model/auth remediation, cancellation, reconnect, resume, and
      permission-mode changes without exposing secret values.

12. **Lean compatibility and provenance**: The implementation reuses proven local mechanisms
    without adding a framework or hiding Hermes-derived behavior.
    - Current: ATLAS and Hermes contain overlapping approval capabilities with different
      persistence and surface contracts.
    - Target: Reused Hermes code/patterns are transformed behind ATLAS-owned, JSON-stable
      contracts with documented provenance; no new runtime framework, message bus, ORM, or
      approval database is introduced.
    - Acceptance: Dependency and provenance checks show no new production framework, one
      authoritative approval store, no imported product identity in user-facing output, and no
      unexplained regression greater than 5% in measured gateway startup, idle memory, or Web
      bundle size.

## Boundaries

**In scope:**
- Replace the React Console's ad-hoc agent transport with shared surface sessions and events.
- Extend the existing ATLAS policy/config/broker contracts with master policy, narrowing
  profiles, expressive rules, a hardline floor, and protected maintenance scope.
- Wire the shared non-visual permission contract into every currently usable ATLAS surface.
- Deliver the conditional Web session header and matching-session queue sidebar.
- Deliver keyboard, screen-reader, responsive, reconnect/resume, and conflict/error behavior.
- Adapt and document compatible Hermes approval mechanisms and tests.

**Out of scope:**
- Building the future native desktop shell — it consumes these contracts after ATLAS stabilizes.
- Replacing the existing ATLAS agent, executor, audit ledger, config store, or permission store —
  those remain authoritative.
- Adding a new policy framework, message bus, ORM, service mesh, or database — existing local
  contracts are sufficient.
- Making foreign-surface approvals actionable in WebUI — only terminal read-only outcomes may
  cross surface boundaries.
- Foundation-vendored developer TUI removal — only the rejected ATLAS OpenTUI and Rich
  generations are removed here; the foundation source remains available for provenance work.
- Foundation-wide de-branding — tracked separately as deferred release-gate work.

## Constraints

- The hardline floor is evaluated before every configurable allow path and cannot be disabled.
- Surface profiles may narrow but never widen the global master ceiling.
- Permission identity and mutable state remain JSON-stable and auditable; secrets and raw
  chain-of-thought never enter config, events, or approval records.
- Python changes stay within the Hermes-derived plugin/LLM foundation surface; new
  infrastructure and gateway contracts follow the Rust-first rule.
- No new production framework or persistence layer is permitted without a new recorded decision.
- Timeouts and unavailable approval channels deny by default.
- The same normalized decision and event contracts must remain usable by CLI, Go TUI, React
  WebUI, messaging channels, API/headless clients, and a future native desktop adapter.

## Acceptance Criteria

- [ ] React Console creates global/project surface sessions instead of calling the ad-hoc console
      stream endpoint.
- [ ] WebUI renders every canonical event kind with TUI-equivalent sequence and outcome.
- [ ] Every current execution surface routes guarded actions through one ATLAS policy/broker.
- [ ] Master policy presets and ordered allow/deny rules produce deterministic explain output.
- [ ] Surface/workspace/project/agent profiles cannot widen the master policy.
- [ ] Full autonomy executes permitted actions without prompts while hardline and explicit deny
      rules still block.
- [ ] Workspace-only policy and explicit user-directed ATLAS maintenance scope pass path,
      validation, audit, and secret-redaction tests.
- [ ] Allow once/session/durable scope, timeout-deny, heartbeat, restart, stale nonce, foreign
      surface, and at-most-once execution tests pass.
- [ ] Conditional header and sidebar visibility match active/pinned/owned queue state.
- [ ] Queue workflows pass keyboard, focus, screen-reader, narrow/wide, and reduced-motion tests.
- [ ] Config/project/model/permission/cancel/reconnect/resume controls use only shared contracts.
- [ ] No new production framework/store is added and measured startup/memory/bundle size remain
      within the 5% regression budget.
