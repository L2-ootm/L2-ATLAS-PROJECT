# Phase 10.7 Research: Shared Permission Policy and Canonical Web Surface

**Researched:** 2026-06-29  
**Status:** Complete  
**Scope:** Existing implementation seams, contract risks, test strategy, and performance budgets

## Executive Summary

Phase 10.7 is large but does not require a second approval system. ATLAS already owns the
durable authority needed for a Hermes-quality design:

- frozen surface-session and normalized-event schemas;
- lifecycle, heartbeat, reconnect, resume, cancellation, and orphan reconciliation;
- surface-owned approval channels and queues;
- nonce-bound expiry, atomic claims, at-most-once execution, and scoped allow rules;
- masked, revisioned configuration; and
- audited terminal outcomes.

The missing backend is a deterministic policy layer in front of the existing broker. The
missing transport is a first-class session/scoped-permission API through the CLI and Rust
gateway. The missing frontend is migration of the current React Console away from
`/v1/console/stream`, followed by conditional shell chrome for the active session and its owned
queue.

The safest delivery shape is one roadmap phase with four gated waves. Wave 1 freezes the data
contracts and policy fixtures before any frontend work. The frontend consumes fixtures and
gateway contracts only after those pass.

## Existing Authority and Reusable Seams

### Surface sessions

`services/agent-runtime/atlas_runtime/surface_session_service.py` already provides:

- `create_session`;
- `get_session` and `list_sessions`;
- validated state transitions;
- `heartbeat`;
- orphan reconciliation; and
- `resume_session`.

The service emits audit records carrying the stable surface-session identity. The core schema
already validates surface kind, workspace kind, lifecycle state, and normalized event kinds.
Phase 10.7 should expose these services, not recreate them in the gateway or React.

### Permission broker

`services/agent-runtime/atlas_runtime/permission_broker.py` already provides the authoritative
approval lifecycle:

- register/revoke/verify an approval channel for a surface session;
- list actionable approvals only for the owning surface session;
- list read-only terminal outcomes;
- record and match workspace/surface/tool/argument-scoped allow rules;
- claim an approval atomically with its nonce; and
- reconcile orphaned approvals after disconnect or restart.

`services/agent-runtime/atlas_runtime/tool_service.py::invoke` is the single execution
chokepoint. It currently calls `policy.decide`, verifies that API/headless requests have a live
approval channel, persists an approval with normalized arguments, and executes only through the
broker path. The new policy evaluator belongs immediately before the existing approval creation
branch.

### Current policy gap

`services/agent-runtime/atlas_runtime/policy.py::decide` currently allows read tools and requires
approval for write/shell tools. Its `mode` argument does not yet implement configurable
semantics. This is a small replacement seam, but it is security-critical.

The policy result must become a JSON-stable explanation containing:

- effective decision (`allow`, `ask`, or `deny`);
- reason code;
- matched rule id and source layer;
- effective preset/profile;
- risk/capability and normalized target facts; and
- whether a protected maintenance scope was used.

No rule may contain executable code. Rules are validated data and evaluated in fixed precedence.

### Configuration gap

`packages/atlas-core/atlas_core/schemas/control_plane.py::PermissionConfig` currently exposes
only mode, approval TTL, and disconnect behavior. The control-plane setting registry currently
surfaces only `permission.mode`.

Extend the existing versioned config rather than adding a policy file or database. The minimal
new shape is:

- master preset: `manual | smart | full_autonomy`;
- master ordered rules;
- named narrowing profiles with typed selectors;
- workspace-only and protected-maintenance controls;
- TTL/disconnect/timeout values; and
- hardline ruleset version as read-only metadata.

Profiles must be validated against the master ceiling during config mutation. An attempted
widening returns a typed conflict and emits an audit event.

## Hermes Donor Findings

The operator-provided architecture map and
`foundation/atlas-hermes/tools/approval.py` establish the behavior floor:

1. hardline checks precede every configurable allow;
2. dangerous-action evaluation is centralized;
3. once/session/always scopes are identity-bound;
4. channel adapters block through bounded waits while keeping the owning session alive;
5. timeouts and silence deny;
6. surface identity is isolated with context-local state; and
7. manual, smart, and off/full-autonomy modes share one guard path.

ATLAS should adapt these semantics and selected test cases. It should not import Hermes approval
state, global variables, product identity, or storage. Provenance belongs in the phase summary
and source comments at transformed logic.

## Hardline and Path Model

The hardline floor is a compact immutable corpus evaluated before configured rules. It covers at
least:

- destructive filesystem roots and recursive deletion of protected roots;
- block-device formatting/writes and partition-table destruction;
- shutdown, reboot, firmware, and boot mutation;
- fork bombs and equivalent resource-exhaustion primitives;
- unsafe privilege escalation and credential-store mutation; and
- attempts to disable or rewrite the hardline evaluator itself through an agent action.

Matching must use normalized tool facts, not raw string search alone:

- tool/capability and declared risk;
- parsed or normalized arguments where available;
- canonical filesystem paths;
- active workspace root;
- declared ATLAS install/update/non-secret config roots; and
- explicit user-directed maintenance intent supplied by trusted runtime context.

The maintenance scope is not a blanket path exception. It permits only declared ATLAS
maintenance capabilities on declared roots, still subject to hardline, path validation, masked
config/update validation, redaction, and audit.

## Transport and Surface Findings

### CLI and Rust gateway

The Python CLI is already the service boundary used by Rust gateway mutations. The Rust gateway
should remain thin:

- CLI commands create/get/list/heartbeat/resume/cancel surface sessions;
- scoped approval commands list/claim/reject with session id and nonce;
- policy explain returns the stable receipt; and
- the gateway maps these JSON contracts to HTTP without policy logic.

The current global `/v1/tools/approvals` routes and Go client omit session ownership/nonce.
Keep read-only compatibility briefly, but migrate all actionable clients to session-scoped
routes. Foreign approvals must never be actionable through the new routes.

### Go TUI and other adapters

The Go TUI already polls approvals and submits decisions. Its client only needs the frozen
session id, approval nonce, and scoped endpoints. The interactive renderer remains local.

CLI, API/headless, and messaging integrations should share a small adapter contract:

- announce/poll an owned pending request;
- heartbeat the session/channel during bounded waits;
- submit one nonce-bound decision; and
- render the terminal receipt.

Adapters may differ visually. They may not classify risk, override policy, or execute a tool.

### React Console

`services/web-ui-react/src/routes/Console.tsx` is the correct canonical page but currently posts
to `consoleChatStream` and keeps its own transient event model. `Layout.tsx` is the correct
location for a session coordinator because the conditional header and queue must survive route
composition. `System.tsx::ToolApprovalsPanel` is a global legacy presentation and must cease to
be an actionable approval path.

No new frontend state framework is needed. Use:

- a shell-level React context/coordinator;
- existing API helpers and standard hooks;
- the shared session and normalized-event DTOs;
- existing `TopoField`, glass/HUD components, tokens, and Lucide icons; and
- native dialog/focus behavior plus small tested utilities.

The legacy `/v1/console/stream` endpoint remains only as a hidden Phase 10.8 rollback seam. New
React code must contain no call to it.

## Test Strategy

Each wave begins with failing tests.

### Wave 1: contracts and policy

- schema round trips and backward-compatible defaults;
- master/profile narrowing validation;
- manual/smart/full-autonomy matrix;
- allow-all-except rules;
- deterministic precedence and explain receipts;
- hardline corpus under every mode/scope/surface;
- workspace escape and protected-maintenance cases; and
- tool-service integration proving no guarded bypass.

### Wave 2: service and transport adapters

- CLI JSON contract tests;
- Rust API and contract tests for session ownership, nonce, expiry, and errors;
- Go client tests for scoped queues/decisions;
- API/headless live-channel and timeout tests;
- bounded heartbeat/cancellation tests; and
- a static/contract assertion that production guarded entry points reach the same authority.

### Wave 3: Web migration and UX

- TypeScript contract fixture parsing for every normalized event kind;
- global/project create, reconnect, resume, cancel, and terminal flows;
- conditional header/sidebar state matrix;
- owned versus foreign approval visibility;
- keyboard-only decision flows and focus restoration;
- live-region terminal announcements;
- narrow/wide layout and reduced motion; and
- config revision-conflict and provider remediation states.

The React project currently has no component-test runner. Add only the smallest development-only
test stack required for DOM behavior; do not add a production dependency or state framework.

### Wave 4: conformance and budgets

- shared fixture parity across runtime, Go, Rust, and React;
- exactly-once/restart/concurrency regression suite;
- provenance/dependency audit;
- secret and product-identity scans;
- full Python, Rust, Go, and Web builds; and
- before/after performance comparison.

## Risks and Mitigations

| Risk | Mitigation |
|---|---|
| Policy and frontend drift | Freeze schemas, reason codes, fixtures, and route DTOs in Wave 1 |
| A profile silently widens authority | Validate on config mutation and re-check at evaluation |
| Full autonomy bypasses hardline | Hardline is code-owned first stage with adversarial corpus |
| Maintenance becomes a general escape hatch | Require trusted explicit intent + declared capability + declared roots |
| Foreign surface claims an approval | Session ownership + nonce + active channel checked atomically |
| Blocking approval stalls an async runtime | Bounded adapter waits/heartbeats; no blocking Rust runtime thread |
| A second approval engine survives | Tool-service chokepoint tests and production-path audit |
| Frontend scope adds a framework | Standard hooks; dev-only minimum test dependencies |
| Ambitious phase obscures failures | Four gates with independently runnable verification and summaries |

## Entropy Findings (codebase memory graph analysis, 2026-06-29)

### Wiring gaps confirmed

**1. React Console legacy transport** — `Console.tsx` calls `consoleChatStream` (api.ts:406)
which hits `/v1/console/stream`. The Rust gateway dispatches to `atlas console chat --stream`,
a separate CLI path that bypasses the surface-session protocol. The Go TUI already uses the
correct path: surface session → run stream. Plan 05 replaces this.

**2. `policy.decide` has exactly one caller** — `tool_service.invoke`. No other code path
reaches the policy engine. The new policy engine from Plan 02 replaces this one call site.
This also means no policy evaluation happens at the Rust gateway level (correct per D-022,
but worth noting for the conformance audit in Plan 07).

**3. `check_all_command_guards` has zero callers in ATLAS code** — the Hermes approval guard
exists in `foundation/atlas-hermes/tools/approval.py` but nothing in `services/agent-runtime/`
calls it. ATLAS has its own permission system (broker + tool_service). The Hermes behavior
model (4-option prompt, session-scoped YOLO, smart approval) has not been adapted yet.
Plan 02 and Plan 04 Task 2 bridge this gap.

### Dead code inventory (graph-verified)

| Target | Graph evidence |
|--------|---------------|
| `apps/atlas-tui/` (Gen 1 TUI) | No inbound callers from any active code |
| `atlas_runtime/tui/` (Gen 2 TUI) | Called only by `dev-rich-tui` hidden command |
| `/v1/console/stream` endpoint | Called only by React Console via `consoleChatStream` — dead after Plan 05 |
| `/v1/console/chat` endpoint | Called only by CLI `console chat` — dead after Plan 05 or kept for CLI |
| `console_service.py` | Orphaned if console endpoints are removed |
| `console_stream` Rust handler | Dispatches to `atlas console chat --stream` — dead after Plan 05 |

### Structural observations

**Two parallel console endpoints** exist: `/v1/console/chat` (single-shot) and
`/v1/console/stream` (NDJSON streaming). Both dispatch through the CLI, not through
`tool_service.invoke`. After Plan 05, both become dead code for the React surface.

**`console_service.py` is an orphan path** — it runs agent execution directly without
going through the tool policy chokepoint. This is not a security issue (chat is not tool
invocation), but it's a code smell that the cleanup should address.

**Go TUI wiring is clean** — surface session, approval polling, run streaming all use
the canonical contracts. No issues found.

### Current TUI state (three generations)

| Gen | Location | Stack | Status |
|-----|----------|-------|--------|

### What the Go TUI has today

Header, provider modes board, missions list, permissions panel, scrollback viewport, composer (textarea), settings form with provider probe. All wired to the real gateway via HTTP/SSE. ASCII/Unicode safe. 7.82 MiB stripped binary.

### What is missing versus SOTA

The Phase 10.1 intake inventory (`docs/imports/ATLAS_TUI_SOURCE_INVENTORY.csv`) classified 180 donor files: 4 adopt-pattern, 80 rewrite, 96 reject. The 80 rewrite candidates define the gap between the current dashboard and a complete workbench.

**Structural gaps (from opencode/MiMo rewrite candidates):**

- Sidebar with context, files, goals, tasks, and LSP information (11 sidebar components)
- Session management: picker, timeline, fork, rename, delete (10 session route components)
- Rich dialogs: command palette, help, select, confirm, prompt, alert, status, export (10 UI dialog components)
- Composer upgrades: autocomplete, history, frecency, stash, part-based input (7 prompt components)
- Subagent and task visibility: task items, subagent footers, todo items
- Thread model: conversation threading, fork from timeline

**Behavioral gaps (from Hermes TUI analysis):**

- Approval prompt with 4 options (once/session/always/deny), arrow/number/enter navigation, command preview up to 10 lines, Esc=deny. The Go TUI currently only offers `a`=approve and `x`=reject with no scope choice and no command preview.
- Clarify prompt for multi-choice or free-text agent clarification
- Confirm prompt for destructive operations (Yes/No)
- Overlay state machine: any active overlay blocks normal input
- Gateway event routing for `approval.request`, `sudo.request`, `secret.request`, `clarify.request` with distinct overlay handlers per event type

**Visual gaps:**

- No theme system (Hermes has `skin_engine` with 4 built-in skins and user YAML skins)
- No markdown or diff rendering in the scrollback
- No structured tool call/result display (currently raw text lines only)
- No activity indicators during streaming beyond a text status

### Reference transformation rules

Every reference implementation is a behavior or pattern source, never a copy target. The transformation boundary is:

**opencode / MiMo-Code:**
- Study the **interaction pattern** (session lifecycle, sidebar layout, dialog flow, composer behavior)
- Reimplement against ATLAS surface-session contracts (Phase 10.3), normalized events (Phase 10.3), and permission broker (Phase 10.5)
- Never import: provider/model selection, auth, config, storage, plugin, MCP, telemetry, product identity, paths, URLs, package names
- The pinned source is commit `86d95a79` (v0.1.2) with `USE_RESTRICTIONS.md` beyond MIT; only patterns are reused

**Hermes foundation:**
- Study the **behavior model** (guard pipeline, 4-option approval flow, overlay state machine, session-scoped YOLO, smart approval, timeout semantics)
- Adapt the Python logic into Go against the same ATLAS broker contracts
- Never import: Hermes approval state, global variables, product identity, or storage
- Provenance belongs in source comments at transformed logic and in the phase summary

**Both references share one rule:** ATLAS owns the agent, tools, policy, audit, Project registry, Current Focus, wiki/Brain memory, configuration, provider/model registry, and approval state. A terminal client must not create a second authority for any of them.

### TUI approval UX convergence (Wave 2 scope)

The current Go TUI sends `"once"` hardcoded on approve (client.go `ApproveTool`). Wave 2 (Plans 03–04) must include:

1. Replace the `a`/`x` approval with the 4-option overlay from Hermes (once/session/always/deny)
2. Add session scope to approval decisions (nonce-bound, session-owned)
3. Add command preview (up to 10 lines) to the approval prompt
4. Wire `clarify.request` and `confirm.request` gateway events to overlay handlers
5. Block normal input while any overlay is active

This is directly portable — the Go TUI already has the broker client and surface session. The rendering and key dispatch need the richer overlay pattern.

### Cleanup ordering

1. Audit with `l2-entropy-reduction` — identify all dead code, stale references, and unused dependencies
2. Remove Gen 1 (`apps/atlas-tui/`) and Gen 2 (`atlas_runtime/tui/`) after verifying no active import paths
3. Remove legacy console stream and global approval routes after Wave 3 migration is verified
4. Run `ultrareview` to confirm no remaining callers of removed code
5. Run full test suites (Python, Rust, Go, React) to verify no regressions
6. Run `anti-bloat-engineering` checks for dependency and binary size
7. Commit atomically with provenance in the phase summary
