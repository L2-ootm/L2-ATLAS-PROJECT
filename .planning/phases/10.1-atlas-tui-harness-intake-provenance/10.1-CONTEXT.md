# Phase 10.1: ATLAS TUI Harness Intake & Provenance — Context

**Discussed:** 2026-06-23
**Prepared for planning:** 2026-06-24
**Status:** Decisions locked

## Phase Boundary

Phase 10.1 establishes a legally attributable, technically isolated, measurable
ATLAS terminal foundation. It does not wire the agent, sessions, projects,
configuration, permissions, Brain/wiki retrieval, or WebUI parity. Those belong
to Phases 10.2–10.8.

The phase may prove that OpenTUI/Solid can render a small ATLAS shell on the
supported terminal stack. It must not import a second agent runtime or preserve
the donor application's product architecture.

## Decisions

### Locked Decisions

- ATLAS keeps one existing agent/runtime. The terminal project is a UI donor and
  design reference only.
- No donor-specific agent, provider layer, SDK client, config store, memory
  store, telemetry, updater, share service, account flow, hosted-service
  coupling, or plugin runtime enters shipped ATLAS code.
- Shipped packages, symbols, commands, environment variables, state paths, URLs,
  generated artifacts, and UI copy use ATLAS identity only.
- Donor identity is permitted only in attribution, license, provenance,
  design-history, and intake-inventory documentation.
- Intake is deny-by-default. Every reviewed source file is classified as
  `adopt-pattern`, `rewrite`, or `reject`; unclassified files are rejected.
- The source checkout remains external or under ignored raw-source storage. A
  raw snapshot is never committed into the ATLAS repository.
- `apps/atlas-tui/` is the destination for the new experience-layer package.
  The existing Hermes-derived Ink TUI remains available until Phase 10.8 cutover.
- Use OpenTUI + Solid as the rendering substrate for the intake proof. Pin the
  audited dependency line first; upgrade only through an explicit measured
  change.
- Keep the first shell deliberately small: compact ATLAS text mark, workspace
  placeholder, status line, empty transcript region, composer placeholder, and
  clean exit. No artistic ASCII-art dependency and no backend calls.
- English-only copy is sufficient for v1.1 intake. Donor i18n, voice, image,
  audio, provider, auth, plugin, update, and hosted-service assets are rejected.
- Public distribution of derivative code is gated on preserving MIT notices and
  resolving the relationship between the repository's MIT license and its
  separately published use-restrictions document. Planning may proceed; public
  release may not silently assume the question is settled.
- Tests and scans are load-bearing deliverables, not cleanup work.

### Agent's Discretion

- Exact internal component names and folder layout inside `apps/atlas-tui/`.
- Whether the compact ATLAS mark uses plain text only or a Unicode border with
  an ASCII fallback.
- Exact implementation of the generic boundary scanner and inventory generator,
  provided their executable source contains no donor identity.
- Exact cold/warm startup and idle-memory measurement mechanism on Windows,
  provided results are reproducible and machine metadata is recorded.

### Deferred Ideas (Out of Scope)

- Agent/runtime wiring, normalized event streaming, project selection, and
  session resume (Phase 10.3 and Phase 10.6).
- Global config/auth/model editing (Phase 10.4).
- Actionable permission prompts (Phase 10.5 and Phase 10.6).
- Full transcript, tool, diff, task, subagent, retrieval, and command renderers
  (Phase 10.6).
- WebUI header/sidebar permission queue (Phase 10.7).
- Legacy TUI retirement and native/Tauri shell work (Phase 10.8 or later).
- Voice, audio, hosted account flows, updater, sharing, marketplace/plugins, and
  multilingual UI.

## Acceptance Shape

Phase 10.1 is complete when:

1. all 180 audited donor TUI files are represented in a provenance inventory;
2. every item has a classification and rationale, with no default-adopt path;
3. required copyright/license notices and the separate restrictions issue are
   documented;
4. a small ATLAS-owned OpenTUI/Solid shell builds, starts, renders, and exits;
5. automated boundary scans find no imported product identity or prohibited
   network/runtime coupling in code or built artifacts;
6. dependency, file-count, source-size, binary-size, startup, and idle-memory
   baselines are recorded against explicit budgets.
