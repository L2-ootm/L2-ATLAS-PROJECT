<research_summary>
## Summary

The audited upstream release is v0.1.2 at commit
`86d95a79bf0879bcb442ffe6b12914f6d8e68a4e` (2026-06-23). Its TUI subtree alone
contains 180 files and 2,630,082 bytes. The largest areas are assets (1.41 MB),
components (270.8 KB), contexts (266.1 KB), i18n (247.5 KB), and routes
(162.7 KB). Coupling is not confined to a logo: 63 files contain upstream
product-name references, 32 refer to the upstream SDK, 42 are provider-related,
20 auth-related, 18 share-related, 26 update-related, and 10 voice-related.

The useful core is narrower: OpenTUI/Solid rendering, layout primitives,
composer interaction patterns, dialogs, routing, themes/keybindings, and
terminal-friendly transcript renderers. The safe approach is therefore not a
repository merge or mechanical rebrand. It is a classified, deny-by-default
transplant into a new ATLAS package, starting with a minimal shell and adding
only reviewed components in Phase 10.6.

OpenTUI's official documentation describes a native Zig renderer with
TypeScript bindings and first-class Solid support. Native rendering uses FFI;
Bun is the straightforward supported runtime. The audited donor pins Bun
1.3.11, OpenTUI core/solid 0.1.101, and Solid 1.9.10. Preserve that known line
for the proof, but do not carry the donor's large dependency graph or its
Solid patch unless an ATLAS test reproduces the patched defect.

**Primary recommendation:** build a clean `apps/atlas-tui` package with only
OpenTUI core, OpenTUI Solid, and Solid; generate a complete external-source
inventory; enforce generic forbidden-term/dependency/network scans; and prove a
small offline ATLAS shell before any runtime integration.
</research_summary>

|---|---|---|

- Upstream SDK/plugin/provider/auth/config/memory/session runtime.

development dependencies. Any increase needs a measured reason in the phase
summary.

### Pattern 1: Deny-by-default intake

Every source item gets a path, hash, size, area, classification, rationale, and
destination/ATLAS analogue. Unknown is a failing classification, not an implicit
adopt.

### Pattern 2: Pure shell model plus native renderer

Keep brand text, layout state, and snapshot output in pure functions testable
without FFI. The normal entrypoint creates the OpenTUI renderer; `--snapshot`
prints deterministic output and exits for CI.

### Pattern 3: Data-driven forbidden terms

Executable scanner code stays product-neutral. Donor names, old commands,
state paths, environment variables, domains, and hosted endpoints live in a
documentation-owned terms file, which the generic scanner consumes.

### Anti-Patterns to Avoid

- **Fork-and-rebrand:** identity and runtime coupling are too distributed for
  reliable search/replace.
- **Whole-tree import followed by deletion:** prohibited code can survive in
  bundles, lockfiles, tests, assets, or dormant routes.
- **Treating a legal notice as a runtime exception:** attribution belongs in
  docs/notices, not product strings or network behavior.
- **Performance promises without machine metadata:** startup and memory numbers
  are meaningless unless the command, sample count, runtime, OS, and hardware
  are recorded.
- **Adding a backend client in the shell proof:** it would prematurely cement
  the Phase 10.3 protocol.
</architecture_patterns>

| Problem | Don't build | Use instead | Why |
|---|---|---|---|
| Terminal layout/input renderer | ANSI cursor and resize engine | OpenTUI core | FFI renderer already handles layout, input, lifecycle, and terminal differences. |
| Reactive UI binding | Custom observable layer | OpenTUI Solid + Solid | Keeps the component model small and proven. |
| Raw-source tracking | Copy of upstream repo in ATLAS | External checkout + hashed CSV | Avoids source contamination and repository bloat. |
| Brand/prohibited behavior audit | Manual grep checklist only | Generic repeatable scanner + terms file | Must also inspect metadata and compiled artifacts. |
| Agent/session/config abstractions | Donor SDK wrappers | Later ATLAS contracts | ATLAS already owns these authorities. |

<common_pitfalls>
## Common Pitfalls

### Pitfall 1: License oversimplification

**What goes wrong:** the repository is described as MIT-only while its README
also subjects use and derivatives to a separate restrictions document.
**Avoidance:** preserve MIT notices, document the restrictions separately, and
gate public derivative distribution on explicit review/upstream clarification.

### Pitfall 2: Hidden identity in build products

**What goes wrong:** source looks clean while lockfiles, source maps, snapshots,
assets, URLs, environment variables, or bundle strings retain old identity.
**Avoidance:** scan source, metadata, lockfile, tests, and compiled output.

### Pitfall 3: Pulling in the runtime through a UI component

**What goes wrong:** a useful dialog or transcript component imports SDK,
provider, sync, auth, storage, or plugin contexts transitively.
**Avoidance:** copy concepts only until dependencies are mapped; rewrite against
ATLAS contracts in later phases.

### Pitfall 4: Carrying upstream patches blindly

**What goes wrong:** ATLAS inherits maintenance debt for a patch whose triggering
behavior is absent.
**Avoidance:** start unpatched; add a patch only after a focused regression test
fails and the upstream issue is documented.

### Pitfall 5: Bundled runtime becomes a system tax

**What goes wrong:** every CLI operation pays Bun/OpenTUI startup or install cost.
**Avoidance:** keep the TUI as an optional experience package and do not load it
for non-TUI CLI commands.
</common_pitfalls>

<code_examples>
## Verified OpenTUI Pattern

Official OpenTUI documentation uses `createCliRenderer()` from
`@opentui/core` and `render()` from `@opentui/solid`. The ATLAS shell should
wrap renderer creation and always expose deterministic cleanup. Snapshot mode
must avoid native renderer creation so CI can test structure without a TTY.
</code_examples>

<open_questions>
## Open Questions

1. **Can the derivative be publicly distributed under ATLAS's intended terms
   without accepting the separate upstream restrictions?**
   - Known: source files carry MIT notices; the upstream README separately says
     use and derivatives are subject to restrictions.
   - Recommendation: treat as a release/legal gate. Technical internal intake
     may proceed with notices preserved.

2. **Is the upstream Solid 1.9.10 patch needed?**
   - Known: the patch addresses transition computation behavior; the minimal
     shell does not yet exercise that path.
   - Recommendation: do not adopt it unless an ATLAS regression test reproduces
     the issue.

3. **What is the final standalone artifact size on Windows?**
   - Known: the full upstream binary is much larger than the desired shell.
   - Recommendation: measure during Plan 03 and block above 140 MB rather than
     guessing.
