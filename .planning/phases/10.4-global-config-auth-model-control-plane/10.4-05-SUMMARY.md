# Phase 10.4 Plan 05: Gateway PATCH Delivery and Error Mapping Summary

**Rust gateway gains a PATCH /v1/config dispatch adapter with safe single-argv changes serialization and structured 409/400/500 error mapping, with no config/auth/model logic added to Rust**

## Performance

- **Duration:** 5 min
- **Tasks:** 3
- **Files modified:** 3

## Accomplishments

- Added `ConfigPatchBody`/`config_patch` handler and wired `PATCH /v1/config`, dispatching `atlas config patch --expected-revision N --changes-json <json>` with `changes` serialized via `serde_json::Value::to_string()` into exactly one argv element (verified against a deliberately shell-metacharacter-laden value).
- Added `dispatch_atlas_raw` to preserve full child `Output` (status/stdout/stderr) and an `ApiError::Structured(StatusCode, Value)` variant that forwards the CLI's stderr JSON body verbatim, mapping `config_revision_conflict` to 409 and validation/unknown-key codes to 400 while leaving unexpected failures at 500 — without the gateway ever inspecting config field values.
- Proved GET/PATCH response bodies never contain resolved secret values via a secret-sentinel regression test, and proved cross-surface conformance: the same fixture snapshot is accepted by CLI JSON and gateway GET, a gateway-shaped patch is visible to a subsequent independent CLI read without restart, and a stale second writer is rejected (409-equivalent exit 2) rather than silently overwriting the first writer.

## Decisions Made

- Used a single argv element (one serde_json-serialized string) for `--changes-json` rather than expanding individual `--set key=value` flags, matching the existing CLI contract and making the "no shell interpolation" invariant trivially provable with one regression test.
- Mapped HTTP status purely from the CLI's structured `code` field (never from exit code alone, never from field content), keeping the mapping logic a flat lookup table independent of config schema knowledge.
- Deferred full end-to-end live-process testing (real `atlas` binary subprocess across CLI and gateway in the same run) to Phase 10.8, per the plan's explicit scope boundary; this plan proves the transport/parsing contract via stub-based HTTP tests and CLI-level service tests, not live cross-process integration.
