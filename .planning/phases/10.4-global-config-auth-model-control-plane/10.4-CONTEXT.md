**Gathered:** 2026-06-25

<domain>
## Phase Boundary

Turn the existing `~/.atlas/config.yaml`, dormant v2 provider/model registry, and committed
`~/.atlas/auth.json` design into one versioned, secret-safe control plane shared by CLI, gateway,
and the later TUI/WebUI surfaces.

In scope: frozen public contracts; config migration/versioning; cross-process locking and durable
writes; optimistic revision conflicts; masked GET/PATCH semantics; change/audit events; ATLAS-owned
auth storage and read-only external auth detection; provider/model/runtime/permission/context
effective-status projection; gateway and CLI adapters; hot-reload vs restart-required metadata.

Out of scope: OAuth/device-code flows, OS keychain integration, browser delivery UX, TUI/WebUI
rendering, permission-request routing, provider fallback execution changes, and changing an active
session's frozen prompt/model contract.

</domain>

<decisions>
## Implementation Decisions

### Config Transaction Contract

- `config.yaml` gains explicit `schema_version` and monotonic `revision` fields. Existing unversioned
  files migrate in memory to v1 and are rewritten only through the normal locked mutation path.
- One OS-handle advisory lock primitive protects both config and auth stores (`fcntl.flock` on
  POSIX, `msvcrt.locking` with the required one-byte/seek-zero behavior on Windows). No presence
  locks, watcher daemon, or new dependency.
- Writes use same-directory temp files, explicit owner-only creation, flush + file `fsync`,
  `os.replace`, parent-directory `fsync` where available, and permission hardening. The lock—not
  `os.replace`—is the Windows no-concurrent-corruption guarantee.
- PATCH accepts `{expected_revision, changes}` where `changes` is a dotted-key map. A stale
  revision raises a typed conflict and never writes; successful patches increment revision once.

### Shared Surface API and Reload Semantics

- The canonical read result is one masked snapshot containing revision/schema metadata plus each
  setting's configured value, effective value, source, validation state, hot-reload/restart
  requirement, and remediation.
- CLI and Rust gateway are thin adapters over the same service: `atlas config json/patch` and
  `GET/PATCH /v1/config`. Later TUI/WebUI clients consume this contract rather than reimplementing
  config logic.
- Hot-reloadable values become visible on the next service read or new execution because the
  runtime resolves from disk at the boundary; no process-global config cache is introduced.
- Active surface sessions retain their Phase 10.2/10.3 frozen model and prompt snapshots. Config
  changes affect new sessions/runs unless a field is explicitly safe to apply live.

### Auth Boundary and External Detection

- ATLAS-owned secrets live only in flat `~/.atlas/auth.json`, resolved through one
  `ATLAS_HOME`-aware function and protected by the same durability/permission rules as config.
- Masked APIs return provider id, auth type/status, source, health, timestamps, redacted hint, and
  remediation only. Raw keys/tokens never enter config responses, SQLite registry rows, session
  events, audit data, or gateway payloads.
- `env:VAR` remains a valid non-secret config reference. Effective auth status may report whether
  the referenced variable is present, but never its value.
- External Codex/Claude-style stores are presence-detected read-only. ATLAS does not parse secret
  payloads, copy credentials, mutate external files, or treat them as ATLAS-owned auth.

### Provider and Model Status Plane

- One control-plane projection joins configured provider/model selection, ATLAS auth status,
  external read-only detection, and provider/model registry health into a frozen JSON-stable
  status contract.
- Provider/model mutation is a config PATCH, not a second registry-specific settings path.
  Registry rows describe discovered availability; config describes operator selection.
- Status distinguishes configured from effective values and reports source precedence explicitly
  (session snapshot/focus override/config/default/env/auth store/registry) with actionable
  remediation for missing auth, unavailable models, invalid values, or restart requirements.
- Existing `model_registry_v2`/`provider_registry` contracts are reused and completed as needed;
  the legacy registry remains compatibility-readable until a tested gateway cutover is safe.

### Audit and Errors

- Successful changes publish a redacted `config_change` audit event containing revision, changed
  paths, masked before/after values, source surface/session when supplied, and reload semantics.
- Validation errors, unknown keys, lock timeouts, permission failures, malformed stores, and
  revision conflicts use typed machine-readable codes plus a concise safe remediation (UX-01).
- Audit payloads reuse `SECRET_PATTERNS`; secret-bearing auth mutations record metadata only.
- Routine reads are side-effect free. Corrupt config/auth files are preserved for diagnosis and
  fail closed instead of silently resetting operator choices.

### the agent's Discretion

- Exact module split and type names, provided there is one transaction path and one public
  projection.
- Whether config change replay is exposed as a dedicated CLI command in this phase or consumed
  from existing audit records, provided every successful write publishes a structured event.
- Exact provider-health vocabulary mapping, provided configured/effective/source/auth/health/
  fallback/remediation remain distinct and JSON-stable.

</decisions>

  Pydantic validation, `env:` references, provider resolution, masking, and dotted-key access.
- `services/agent-runtime/atlas_runtime/cli/config.py` and Rust gateway `GET /v1/config` are thin
  adapters that can evolve into the shared GET/PATCH contract.
- `packages/atlas-core/atlas_core/schemas/registry_v2.py` and migration
  `infra/migrations/0004_registry_v2.sql` already define provider/model/route records without
  secret material.
- `docs/architecture/AUTH_STORE_DESIGN.md` commits the ATLAS-owned auth layout, OS-handle lock,
  Windows landmine, permissions, corruption preservation, and credential boundary.
- `services/agent-runtime/atlas_runtime/audit_service.py` provides redaction and transactional
  structured audit writes; `mission_service.ensure_operator_run` is the FK-safe run precedent.

### Established Patterns

- Public Python contracts are frozen Pydantic v2 models with JSON-stable `model_dump()`.
- Service functions inject connections/locks and keep CLI/gateway layers thin.
- New runtime dependencies are disallowed; use standard-library file and process primitives.
- Configuration must remain in the Hermes-derived/LLM adapter surface until behavior is stable
  enough for later Rust migration.

### Integration Points

- `NativeAtlasAgent.execute` calls `config_service.resolve_provider`; this is the new-session
  effective-value boundary.
- Phase 10.3 `SurfaceSession` stores model/prompt versions and protects active-session semantics.
- Rust gateway route wiring in `native/atlas-core-rs/crates/atlas-gateway/src/lib.rs` adds PATCH
  dispatch without duplicating config validation.
- Later Phase 10.6/10.7 surfaces consume the masked snapshot and conflict response.

<specifics>
## Specific Ideas

- Keep config GET/PATCH local and network-free with a target under 50 ms p95 on normal local
  files; add no resident watcher process and no dependency.
- Treat `revision` as the public optimistic-concurrency token. Hashes may be internal integrity
  aids but are not a second concurrency API.
- Preserve active-session reproducibility: a model/config change is never retroactively injected
  into a running or resumed session snapshot.

</specifics>
