<user_constraints>
## User Constraints (from CONTEXT.md)

### Locked Decisions

- `config.yaml` uses explicit schema version + monotonic revision with optimistic PATCH conflicts.
- One standard-library OS-handle advisory lock and one durable owner-only write path protect both
  config and auth files.
- The masked GET/PATCH contract is the only surface contract; no watcher daemon or second settings
  implementation.
- Active sessions preserve their frozen Phase 10.2/10.3 model and prompt snapshots.
- Secrets live only in `auth.json` or external `env:` references; masked APIs expose presence and
  remediation, never values.
- External credential stores are detection-only and byte-identical after ATLAS commands.
- Provider/model mutation is a config patch; registry rows describe availability, not operator
  selection.
- Successful writes emit redacted `config_change` audit events; typed errors carry remediation.

### the agent's Discretion

- Exact module/type names and internal status vocabulary mapping.
- Dedicated config-event reader vs consumption through existing audit records.

### Deferred Ideas (OUT OF SCOPE)

- OAuth/device-code flows, OS keychain integration, route-policy enforcement, fallback behavior
  changes, TUI/WebUI forms, and native/Tauri integration.

<research_summary>
## Summary

The repository already has the correct seams but only baseline behavior:
`config_service.py` owns `~/.atlas/config.yaml`, validates `env:` references, and is read by both
the runtime and Rust gateway. Its write path is not yet safe for concurrent surfaces: there is no
schema version/revision, no OS lock, no durability fsync, no owner-only ACL, no conflict detection,
and no audit event. The current comment also overstates Windows `os.replace` atomicity.

The committed auth design is stronger and should be implemented as a shared local primitive rather
than duplicated in config and auth modules. The Hermes reference confirms the correct zero-dependency
shape: OS-handle advisory lock, one-byte/seek-zero on Windows, same-directory O_EXCL temp, fsync,
replace, corruption preservation, and secure permissions. ATLAS must own its implementation and
must not import foundation internals.

The model plane should remain a projection, not a new router. `provider_registry` and
`model_registry_v2` already carry secret-free health/auth metadata; the existing discovery writer
still targets the legacy table. The lean path is dual-write/compatibility-read during this phase,
then expose one `ControlPlaneSnapshot` that reports configured/effective provider/model, source,
auth state, registry health, and remediation.

**Primary recommendation:** build one shared secure-file primitive, then layer revisioned config,
auth storage, status projection, and thin CLI/gateway adapters on it; add no dependency and no
resident watcher.

</research_summary>

<architecture_patterns>
## Architecture Patterns

### System Architecture Diagram

```text
CLI / Rust gateway / future TUI-WebUI
               |
               v
      masked GET / revisioned PATCH
               |
               v
     Config control-plane service
       | validate + compare revision
       | resolve configured/effective/source
       | build remediation
       +--------------------------+
       |                          |
       v                          v
 secure config.yaml store    auth/status service
 (schema + revision)         (auth.json + env + read-only external detection)
       |                          |
       +------------+-------------+
                    v
        provider/model registry projection
                    |
                    v
       frozen ControlPlaneSnapshot response
                    |
                    +--> redacted config_change audit event
```

### Pattern 1: Locked Read-Modify-Write

Acquire the OS-handle lock before loading the current revision. Validate
`expected_revision`, apply dotted changes to a copy, validate the frozen model, write the durable
replacement, then release the lock. Do not read before acquiring the lock and later write from that
stale object.

### Pattern 2: Masked Projection, Private Storage Model

Raw auth records never become public schema output. Public status models contain provider id,
auth type/status, source, timestamps, redacted hint, health, and remediation. The gateway consumes
only serialized public contracts.

### Pattern 3: Config Selection vs Registry Availability

Config answers “what the operator selected.” Registry answers “what is currently discovered and
healthy.” The control-plane projection joins them and may report an unavailable selection without
silently choosing a different model.

### Pattern 4: Snapshot Boundary for Session Stability

New executions call `resolve_provider()` and see the latest revision. Existing sessions use their
stored `SurfaceSession` and `RunContractSnapshot`; a config update never mutates those records.

### Anti-Patterns to Avoid

- Presence-based lock files that remain after a crash.
- Loading config outside the lock and saving later with no revision check.
- Returning `env:` dereferenced values or raw auth records through JSON.
- Treating external Codex/Claude auth as writable or ATLAS-owned.
- Adding a second model-selection API separate from config PATCH.
- Marking the whole config “restart required” instead of field-level metadata.

</architecture_patterns>

<common_pitfalls>
## Common Pitfalls

### Pitfall 1: False Windows Atomicity

**What goes wrong:** concurrent writers or a replace fallback can corrupt/lose state.
**How to avoid:** lock serializes every read/write; fsync temp before replace; preserve corrupt files.
**Warning sign:** code claims `os.replace` alone is the cross-platform guarantee.

### Pitfall 2: Revision Check Outside the Lock

**What goes wrong:** two surfaces both read revision N and both write N+1.
**How to avoid:** load, compare, patch, and replace while holding the same OS lock.
**Warning sign:** service accepts a preloaded `AtlasConfig` object for persistence.

### Pitfall 3: Secret Leakage Through “Helpful” Status

**What goes wrong:** effective API keys or raw external auth payloads reach logs/gateway/tests.
**How to avoid:** status computes presence only; public models have no secret fields; audit metadata
only.
**Warning sign:** `model_dump()` of a public object can contain `api_key`, `access_token`, or
`refresh_token`.

### Pitfall 4: Breaking Active Session Reproducibility

**What goes wrong:** a running/resumed session silently switches model after config change.
**How to avoid:** resolve at new-session/run boundary; preserve stored prompt/model snapshots.
**Warning sign:** live agent loop calls `load_config()` mid-session to replace model identity.

### Pitfall 5: Incomplete Requirement Parsing

**What goes wrong:** wrapped ROADMAP requirement lines are truncated by GSD metadata extraction.
**How to avoid:** keep the Phase 10.4 requirement IDs on one line and verify all 11 IDs manually.
**Warning sign:** `init.plan-phase` reports only through `MOD-01`.

### Pitfall 6: Non-Atomic Audit/File Dual Write

**What goes wrong:** a config file commits but audit emission fails afterward.
**How to avoid:** successful service APIs attempt audit immediately with a stable change id/revision;
tests cover the normal invariant, and failures surface honestly rather than claiming rollback
atomicity across filesystem and SQLite.
**Warning sign:** documentation claims one transaction spans both filesystem and SQLite.

</common_pitfalls>

<open_questions>
## Open Questions (Resolved for Planning)

1. **Should auth secret writes be exposed through the HTTP config PATCH?**
   - Resolution: no. Masked config PATCH accepts references/settings only. ATLAS-owned auth
     mutations use a local CLI/service path with hidden input; gateway exposes status, not raw secret
     transport.

2. **Should hot reload use a watcher?**
   - Resolution: no. Reads at service/new-execution boundaries already provide cross-surface
     visibility without a resident cache or watcher.

3. **Should the v2 registry replace the legacy table immediately?**
   - Resolution: no destructive cutover. Discovery dual-writes or a compatibility reader joins v2
     first and legacy second; gateway status endpoints use the new projection.

4. **What is the Windows identity source for ACL hardening?**
   - Resolution: use `whoami` via explicit argv when available because it returns domain-qualified
     identity; fall back to `USERDOMAIN\USERNAME`/`getpass.getuser()`. Tests mock the subprocess and
     assert `shell=False`.

</open_questions>
