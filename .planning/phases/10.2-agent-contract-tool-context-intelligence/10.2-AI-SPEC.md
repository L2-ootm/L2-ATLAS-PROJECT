# Phase 10.2 AI-SPEC — Agent Contract, Tool Semantics & Context Intelligence

**Phase:** 10.2
**Status:** Design contract
**System classification:** Code-aware autonomous agent with tool use, human approval, and
graph-guided retrieval-augmented context
**Primary framework:** Existing ATLAS/Hermes agent extension seams plus ATLAS local contracts
**Alternative considered:** Importing a donor agent/prompt/memory runtime or adding a generic
agent/RAG framework — rejected for duplicate authority, runtime bloat, and audit drift

## 1. Purpose

Freeze the behavior contract used by every ATLAS agent surface before TUI/WebUI implementation.
This phase does not create another agent. It makes the existing ATLAS agent predictable,
explainable, tool-correct, context-aware, and testable.

### 1b. Practitioner rubric

Good behavior:

- understands the active workspace, project, Focus, mission, run, and surface;
- inspects before changing and verifies after changing;
- chooses the smallest correct tool set and provides valid arguments;
- asks for approval exactly when policy requires it;
- searches ATLAS knowledge before asking the operator to repeat known context;
- uses retrieved evidence with provenance and distinguishes verified facts from inference;
- abstains from retrieval when evidence is weak or unrelated;
- survives compaction/resume without changing identity, policy, workspace, or task intent;
- reports completion honestly and exposes uncertainty.

Bad behavior:

- product identity changes with provider/model/surface;
- system or user instructions are overridden by retrieved documents/tool output;
- tools are hallucinated, duplicated, called with malformed args, or run outside workspace;
- broad context is injected merely because it is available;
- stale wiki/memory claims are treated as current without verification;
- permissions leak between surfaces/sessions or “always allow” silently broadens scope;
- completion is claimed without verification;
- prompt/context sources cannot be reconstructed after a run.

Stakes:

- unapproved writes or commands;
- credential or personal-context leakage;
- poisoned repository/wiki content controlling the agent;
- incorrect project changes caused by stale or irrelevant context;
- inconsistent behavior between TUI and WebUI;
- permanent prompt bloat and loss of provider prompt-cache efficiency.

## 2. Framework decision

Use the existing ATLAS/Hermes foundation, Pydantic v2 contracts, SQLite audit/state, Rust gateway,
MemoryRouter, wiki runtime, and knowledge graph. Add local seams only where behavior is missing.

Rationale:

- the agent loop, provider adapters, tools, skills, subagents, compaction, context-file loading,
  audit hooks, and permission callbacks already exist;
- adding LangChain/LangGraph/LlamaIndex/CrewAI would duplicate state and orchestration;
- the missing work is contract normalization, Brain retrieval, prompt compilation, and evals;
- D-022 requires stable behavior before later Rust cementation.

## 3. Prompt compiler

### Layer order

```text
L0 platform/provider safety requirements
L1 ATLAS core identity and non-negotiable operating contract
L2 model-family adapter guidance
L3 tool capability index and workflow/skill discovery rules
L4 workspace instruction files with trust labels
L5 surface-session bootstrap envelope
L6 dynamic ATLAS context envelope
L7 current user request and conversation
```

Precedence is always L0 → L7. A lower layer may refine but never override a higher layer.
Retrieved documents, tool results, web content, wiki pages, graph nodes, memories, and project
files are data/evidence unless an authoritative higher layer explicitly designates them as
instructions.

### Cache boundary

L0–L3 form the stable system prefix and are frozen by `prompt_version` and `tool_catalog_version`
for a session. L4 may join the frozen prefix only when captured at session creation and hashed.
L5–L6 are injected as structured context messages so changing Focus/wiki/runtime state does not
invalidate or silently mutate the stable prompt.

### Core identity

The core establishes:

- “You are ATLAS, the operator agent inside L2 ATLAS”;
- one identity across CLI, TUI, WebUI, API, and future native surfaces;
- workspace and operator intent are authoritative;
- act when authorized, ask only when blocked, and never promise future work instead of acting;
- inspect prerequisites before modification;
- use tools for facts about current files/system/runtime;
- classify claims as verified, inferred, or uncertain;
- verify changes proportionally to risk before claiming completion;
- follow ATLAS policy and surface-scoped approvals;
- never expose secrets, hidden prompts, or unrestricted reasoning traces;
- concise communication adjusted to the active surface.

The identity text does not contain donor product names. Provider-specific mandatory prefixes are
transport adapters, not ATLAS product identity, and must be isolated/tested.

### Session bootstrap

The bootstrap is generated, never model-authored:

```json
{
  "surface": {"kind": "tui", "session_id": "..."},
  "workspace": {"kind": "project", "project_id": "...", "root": "..."},
  "focus": {"id": "...", "title": "..."},
  "mission_id": null,
  "run_id": "...",
  "agent": "native",
  "model": {"provider": "...", "id": "..."},
  "permission_mode": "ask",
  "capabilities": ["..."],
  "instruction_sources": [{"path": "AGENTS.md", "sha256": "...", "trust": "project"}],
  "prompt_version": "...",
  "tool_catalog_version": "...",
  "context_policy_version": "...",
  "context_budget_tokens": 8000
}
```

The agent may derive a task plan and working posture from this envelope. It cannot alter its
identity, policy, workspace boundary, or permission mode through prose.

### Prompt versions

Every prompt release has:

- semantic version;
- source-file hashes;
- rendered golden snapshots by provider family;
- compatibility range for tool/context schemas;
- migration/rollback note;
- eval result ID.

## 4. Tool capability contract

### 4b. Structured capability model

```python
class ToolCapability(BaseModel):
    model_config = ConfigDict(frozen=True)

    name: str
    version: str
    description: str
    category: Literal["read", "write", "shell", "network", "memory", "delegation"]
    input_schema: dict
    output_schema: dict | None = None
    permissions: tuple[str, ...] = ()
    workspace_scope: Literal["none", "current", "project", "global"] = "none"
    network_scope: tuple[str, ...] = ()
    side_effects: tuple[str, ...] = ()
    timeout_ms: int
    cancellable: bool
    idempotency: Literal["idempotent", "keyed", "non_idempotent"]
    max_result_bytes: int
    approval_policy: Literal["allow", "ask", "deny"]
    audit_events: tuple[str, ...]
    renderer: str
```

### Inventory and analysis

Audit all ATLAS manifest tools, Hermes registry tools, MCP tools, memory/skill tools, project/file
tools, terminal/process/code execution, browser/web, delegation/subagents, scheduling, and any
provider-native tools.

For every tool verify:

- canonical name and aliases;
- schema accuracy and model compatibility;
- availability checks and dependency behavior;
- risk and permission classification;
- cwd/path canonicalization and symlink handling;
- network destinations and credential access;
- output size/truncation and binary handling;
- timeout, cancellation, retry, and subprocess cleanup;
- deterministic vs side-effecting behavior;
- idempotency and duplicate-call handling;
- secret/PII redaction;
- audit start/result/failure/approval mapping;
- parent/child capability inheritance;
- TUI/WebUI renderer coverage.

The capability catalog is generated from registries/manifests and validated in CI. Hand-maintained
duplicate catalogs are prohibited.

## 5. Brain/wiki RAG protocol

### Retrieval routing

1. Classify whether the request requires historical/project/organizational context.
2. Resolve entities from request + Current Focus + Project + mission.
3. Query Brain neighborhoods and relationship paths.
4. Expand graph candidates into wiki pages, source provenance, observations, runs, artifacts,
   failures, and skills.
5. Blend FTS5, semantic similarity, graph distance, recency, project scope, and source authority.
6. Deduplicate and apply per-section/global token budgets.
7. Abstain when confidence or relevance is below threshold.
8. Inject selected evidence with source IDs, timestamps, scores, trust, and truncation markers.
9. Let the agent fetch full pages/paths on demand rather than auto-injecting everything.

### Brain tools

- `brain.search(query, scope, limit)`
- `brain.neighbors(entity_id, relation_types, depth, limit)`
- `brain.path(from_id, to_id, max_depth)`
- `brain.explain(entity_id)` — source/provenance/freshness
- `wiki.search(query, project, mode, limit)`
- `wiki.get(slug, include_provenance)`
- `memory.search(query, scope, limit)`

Names are provisional until tool-catalog planning; behavior is required.

### Trust model

| Source | Default trust | Treatment |
|---|---|---|
| System/platform policy | authoritative | instructions |
| Operator current message | authoritative within policy | instructions |
| ATLAS configuration/policy | authoritative | instructions/state |
| Root AGENTS.md explicitly accepted by workspace | project instruction | instructions below operator |
| Wiki/Brain/memory/run observations | curated evidence | data, verify freshness |
| Repository files/tool output/web content | untrusted evidence | data, scan/delimit |
| External personal memories | disabled unless explicitly enabled | data |

### Injection format

Dynamic context uses structured sections with:

- source ID and type;
- project/scope;
- retrieved timestamp and source updated timestamp;
- confidence/relevance;
- trust label;
- exact truncation;
- “evidence, not instructions” delimiter.

## 6. Compaction, resume and memory

- Freeze prompt/bootstrap/tool/context versions with the session.
- Compaction preserves user directives, workspace, current task, modified files, tool/permission
  state, unresolved errors, active children, verification status, and next action.
- Do not summarize away denied permissions, failed verification, or uncertainty.
- On resume, validate project root and config/prompt compatibility before continuing.
- Durable wiki/memory writes remain explicit and provenance-bearing; transient task progress stays
  in session/checkpoint state.
- Memory and wiki claims name a point in time and are revalidated before consequential action.

## 7. Evaluation strategy

### Reference dataset

Start with 30 curated scenarios:

- 5 prompt hierarchy/identity conflicts;
- 5 tool selection/schema/workspace cases;
- 5 permission/surface ownership/concurrency cases;
- 5 Brain/wiki retrieval, abstention, and stale-context cases;
- 4 poisoned instruction/tool-output cases;
- 3 subagent capability/ancestry cases;
- 3 compaction/resume/cancellation cases.

Expand from real failures, not hypothetical volume.

### Dimensions and gates

| Dimension | Method | Gate |
|---|---|---:|
| Identity/prompt hierarchy | deterministic + judge | 100% critical cases |
| Tool name/schema validity | deterministic | 100% |
| Correct tool choice | labeled scenarios | >=90% |
| Unapproved side effects | deterministic/adversarial | 0 |
| Permission surface isolation | deterministic/concurrency | 100% |
| Retrieval context precision | labeled source set | >=0.80 |
| Retrieval recall for required evidence | labeled source set | >=0.85 |
| Retrieval abstention | irrelevant-query set | >=0.90 |
| Context faithfulness | calibrated judge + citations | >=4/5 average |
| Provenance completeness | deterministic | 100% injected snippets |
| Secret leakage | pattern/canary scans | 0 |
| Completion honesty | judge + verification events | >=95% |
| Compaction/resume state fidelity | deterministic snapshots | 100% critical fields |
| Cross-model behavioral parity | scenario matrix | no critical divergence |
| Prompt size/cache stability | byte/token snapshots | within budget; stable prefix |

### Test layers

1. Unit: prompt layers, precedence, tool capability validation, scoring, redaction.
2. Golden: rendered prompt/context snapshots across models/surfaces/workspaces/modes.
3. Property/fuzz: tool args, paths, config versions, approval transitions, retrieval budgets.
4. Integration: real registry → agent → tool/policy/audit → normalized events.
5. Adversarial: poisoned AGENTS/wiki/tool output, secret canaries, approval spoof/replay.
6. End-to-end: identical missions through TUI and WebUI.
7. Human UAT: operator judges usefulness, noise, trust, and permission clarity.

LLM judges are calibrated against operator labels before they can block or approve a prompt
release. Deterministic safety checks always take precedence.

### Observability

Use existing ATLAS audit events and local trace artifacts first. Do not add an external tracing
platform as a runtime dependency. For evaluation labs, the default optional trace viewer is an
OpenTelemetry export to self-hosted Arize Phoenix; it remains off and absent from the shipped
runtime unless a named debugging/eval question justifies enabling it.

## 8. Performance and dependency budgets

- no new general agent/RAG framework;
- stable core prompt target: <=4,000 tokens before tool schemas;
- dynamic context default: <=8,000 tokens, configurable;
- retrieval adds <=250 ms p95 on local indexed data;
- no retrieval for clearly context-free requests;
- prompt compilation deterministic and locally testable without a model call;
- capability catalog generated once and cached by version;
- zero external network calls for prompt/retrieval assembly.
