# Phase 10.2 Research — Existing Seams and Implementation Strategy

**Researched:** 2026-06-24
**Scope:** local codebase only; official behavior is already fixed by D-023 and the AI-SPEC

## Findings

1. `MemoryRouter` already supplies budget accounting, redaction, ranked retrievers,
   FTS5/semantic wiki fallback, prior failures, observations, runs, and skills.
   Phase 10.2 should evolve its data contract rather than replace it.
2. `context_service` currently emits markdown plus flat source tokens. It needs a
   structured provenance envelope, explicit trust/freshness/confidence, selected
   and rejected sources, and context-policy versioning.
3. The existing `graph_service` is a bounded cockpit visualization graph over
   markdown. It is not a durable/queryable Brain graph and does not connect wiki,
   source, run, artifact, goal, or observation entities. A small additive SQLite
   graph index is justified.
4. ATLAS tool manifests cover five integration tools while the Hermes registry
   exposes the broader execution set. A normalizing adapter can read both without
   editing foundation internals or duplicating handlers.
5. Hermes already builds and persists a stable system prompt once per session and
   injects dynamic context outside that prefix. The ATLAS compiler should produce
   the stable identity/tool/workflow prefix and structured envelopes that later
   session wiring can pass through existing seams.
6. Existing run/audit schemas have session and tool-call identity but no prompt,
   catalog, context-policy, instruction-source, or selected-context snapshots.
   Additive migration tables are preferable to widening every event row.
7. The native runtime signature is still mission/run/prompt based. Phase 10.2
   should freeze and test compilation/replay contracts; Phase 10.3 will bind them
   to persistent surface sessions rather than prematurely changing the runtime API.

## Recommended plan shape

- Wave 1: frozen schemas + deterministic prompt compiler.
- Wave 2 parallel:
  - generated tool capability catalog and conformance;
  - durable Brain index/query tools and structured retrieval envelope.
- Wave 3: prompt/context snapshot persistence, replay, and compaction/resume contract.
- Wave 4: 30-scenario evaluation harness and promotion gate.

## Dependency and performance posture

- Add no Python framework or runtime dependency.
- Reuse Pydantic, SQLite, pytest, and current services.
- Keep graph queries bounded by depth/result caps.
- Cache catalog output by registry generation/version.
- Keep prompt and context assembly network-free.
- Measure prompt size, catalog generation, and retrieval p95 in tests/verification.

## Risks

- Importing the full Hermes registry can trigger optional tool discovery. Catalog
  generation must support an injected/snapshot registry and fail closed on
  unavailable metadata without probing external services during tests.
- Short-lived graph entities can create stale edges. Index writes need source
  version/freshness metadata and deterministic replacement semantics.
- Prompt snapshots can accidentally retain secrets. Redaction/canary tests must
  cover every persistence boundary.
- LLM judging can become a theatrical gate. Deterministic critical checks remain
  authoritative; judges are optional until calibrated against operator labels.
