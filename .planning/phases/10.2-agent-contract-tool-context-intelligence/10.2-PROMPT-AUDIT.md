## Donor strengths worth translating

- coding-specific inspect/edit/verify discipline;
- cautious workspace/git behavior;
- concise terminal communication;
- explicit action blast-radius rules;
- provider-specific prompt variants;
- session checkpoints and structured task progress;
- visible retry behavior and bounded hidden SDK retries;
- instruction-file discovery and display;
- task/subagent return contracts;
- permission dialogs integrated into the session UI;
- prompt/tool compatibility repair paths.

## Donor behavior not to import

- product identity, account/provider coupling, telemetry, updater/share flows, state paths, or
  configuration namespaces;
- a second agent/provider/tool/memory/session backend;
- extreme universal brevity constraints that can suppress necessary audit explanations;
- blanket “no comments” or framework-specific style rules as global ATLAS policy;
- file-memory ownership rules that conflict with ATLAS Wiki/Brain/provenance;
- long autonomous retry horizons without ATLAS run budgets;
- implicit widening into external personal memory stores.

## Hermes strengths worth retaining

- composable identity, model guidance, tools, skills, context files, memory, and platform hints;
- system prompt built once per session and replayed verbatim for prompt-cache stability;
- dynamic recall/plugin context injected outside the stable system prefix;
- context-file threat scanning and bounded loading;
- broad self-registering tool registry and toolset filtering;
- skill discovery and specialized workflow loading;
- subagent capability narrowing and ancestry;
- context compression and session restore;
- interactive approval callbacks and plugin audit hooks;
- provider/model flexibility.

## Hermes behavior to replace or constrain

- Hermes/Nous identity and help/subscription language in operator-facing prompts;
- oversized universal guidance blocks unrelated to the current capability set;
- multiple overlapping memory concepts without one ATLAS ownership contract;
- imperative auto-memory/auto-skill mutation as a universal default;
- project instruction discovery that differs by surface;
- tool descriptions and risk data split across schemas, registries, config, and prompt prose;
- approval behavior coupled to terminal ownership rather than ATLAS surface sessions.

## Current ATLAS findings

- `context_service` and `MemoryRouter` already assemble Focus, goals/tasks, Project, recent runs,
  failures, observations, wiki hits, and skills with redaction, provenance, and budgets.
- Wiki retrieval already supports FTS5 plus semantic fallback.
- The graph service currently builds a useful project/file/planning graph for the cockpit, but it
  is not yet a queryable retrieval spine connected to wiki/source/run entities.
- Tool manifests cover a small ATLAS integration layer, while the Hermes registry exposes a much
  larger capability set. They need one generated catalog.
- Tool approvals are persistent and exactly-once guarded, but do not yet record initiating
  surface/session ownership.
- The WebUI console's `native` path is a stub receipt. Real folder-aware streaming is currently a
  Claude-Code-specific path. Phase 10.3 must route both surfaces into the existing ATLAS agent.
- Current ATLAS context is prepared for mission runs, not yet consistently for free interactive
  surface sessions.

## Resulting ATLAS prompt shape

Do not concatenate donor + Hermes prompts. Compile:

1. immutable ATLAS identity, policy, evidence, verification, and communication contract;
2. model-family adapter;
3. generated catalog for the tools actually available in this session;
4. skill/workflow discovery rules;
5. accepted and hashed project instructions;
6. generated session bootstrap;
7. separately retrieved ATLAS context.

The bootstrap helps the agent orient and form a working plan. It does not let the model invent its
own permanent identity, permissions, workspace, or memory policy.

## Additional requirements captured

- instruction conflict resolution and trust labels;
- prompt/tool/context semantic versioning and rollback;
- context freshness and invalidation;
- retrieval abstention and negative tests;
- graph relationship-path retrieval;
- prompt-injection and malicious tool-output boundaries;
- normalized tool errors and surface renderers;
- cancellation and orphan cleanup;
- compaction/resume fidelity;
- token/cost/latency budgets;
- no persistence of secrets or hidden chain-of-thought;
- cross-model and cross-surface conformance;
- observability sufficient to reconstruct which prompt/context/tool versions shaped a run.
