## Summary

This phase is smaller than its CONTEXT.md framing suggests, because **the setup-wizard and config-service it claims to "pull forward" already shipped** in the six-item operator scope (STATE.md, commit `76df72d`, 2026-06-20) and the migration runner already shipped too (STATE.md "drift correction", `atlas_runtime/db.py`). `atlas setup`, `atlas config get/set/show/export/import`, `~/.atlas/config.yaml` (Pydantic `AtlasConfig`), `GET /v1/config` (masked), and `atlas db init` all exist and work today. The CONTEXT.md decision to "build NOW" is therefore actually "extend what exists" — the planner must not re-create these, only add the missing pieces: a `channels` field on `AtlasConfig` (currently absent — channels config lives in the *foundation's* `~/.hermes/config.yaml`, a separate file, per `atlas_runtime/cli/channels.py:40-48`), `atlas up` (boots gateway + cockpit together — does not exist), `atlas doctor` (does not exist), a deterministic mock provider (does not exist — `config_service.resolve_provider` at `services/agent-runtime/atlas_runtime/config_service.py:101-126` only resolves a *real* provider's API key, returning an empty string on missing creds, which today triggers the Phase A4 honest-failure path in `NativeAtlasAgent.execute`), a `--demo` seed (the only existing seed, `model_registry.seed_default_models`, only seeds 3 model rows — no mission/audit/wiki fixture exists), and the install/build/docs layer (`.env.example`, `docker-compose.yml`, `docs/INSTALL.md`, README quickstart, and scripts that build the gateway binary + cockpit, which `scripts/install-atlas-cli.ps1` currently does NOT do — confirmed by reading the script: it builds the TUI bundle and installs the Python venv/CLI, but never runs `cargo build --release -p atlas-gateway` or `npm run build` for `services/web-ui-react`).

**Primary recommendation:** Extend (do not rebuild) `config_service.py` with a `channels` field and a `provider.mock` sentinel; add `atlas up`/`atlas doctor` as new Typer sub-apps following the existing `gateway_control.py` process-lifecycle pattern; insert mock-provider resolution as a new branch inside `NativeAtlasAgent._resolve_provider` (native.py:116-135) so the Phase A4 honest-failure path is preserved when a provider IS configured but unreachable, while empty/unset credentials now resolve to the mock provider instead of falling through; extend `scripts/install-atlas-cli.ps1`/write its bash twin to also build the gateway binary and the cockpit; write a new demo-seed function modeled on `seed_default_models`'s idempotent `INSERT OR IGNORE` pattern, not a literal reuse of commit `147dd5b` (that commit only seeded model-registry rows, not a mission/audit/wiki fixture).

### Pattern 1: Process-control primitive (boot/health/stop)
**What:** A small module exposing `binary()`/`start()`/`health_ok()`/`stop()`, with a PID file for idempotent stop and a polling loop for "block until healthy."
**When to use:** Any time a new long-running process needs CLI lifecycle control (cockpit dev server, in this phase).
**Example (existing, to mirror exactly):**
```python
# Source: services/agent-runtime/atlas_runtime/gateway_control.py:80-113 (read directly)
def start(poll_seconds: float = 15.0) -> tuple[bool, str]:
    if health_ok():
        return True, "gateway already running"
    binary = gateway_binary()
    if not binary:
        return False, "atlas-gateway binary not found; ..."
    # detached spawn (Windows: DETACHED_PROCESS | CREATE_NEW_PROCESS_GROUP; POSIX: start_new_session)
    proc = subprocess.Popen([binary], ..., env=_child_env(), **kwargs)
    PID_FILE.write_text(str(proc.pid))
    deadline = time.monotonic() + poll_seconds
    while time.monotonic() < deadline:
        if health_ok():
            return True, f"gateway started (pid {proc.pid}) on {GATEWAY_URL}"
        time.sleep(0.5)
    return False, "gateway did not become healthy in time"
```
For `cockpit_control.py`, replace the binary with `["npm", "run", "preview", "--", "--port", str(cockpit_port)]` (or `vite` dev server) run with `cwd=services/web-ui-react`, and the health probe with a plain TCP/HTTP GET to the cockpit port (Vite serves `/` with 200 once ready — no JSON health route needed, a bare connect-and-200 check suffices).

### Pattern 2: Provider resolution with an explicit mock branch
**What:** `config_service.resolve_provider()` returns `{provider, model, base_url, api_key}`; the consumer (`NativeAtlasAgent._resolve_provider`) currently just forwards an empty `api_key` to the foundation, which then honest-fails.
**When to use:** Insert the mock decision at the `NativeAtlasAgent._resolve_provider` boundary (native.py:116-135), NOT inside `config_service.resolve_provider` — keep config resolution provider-agnostic; let the agent runtime decide what an empty key means.
**Example (extension point, current code shown):**
```python
# Source: services/agent-runtime/atlas_runtime/agents/native.py:116-135 (read directly)
def _resolve_provider(self, conn):
    model, provider, base_url, api_key = self._model, self._provider, None, None
    try:
        from atlas_runtime import config_service, focus_service
        focus = focus_service.get_current_focus(conn)
        framework = (focus.framework if focus else "") or ""
        resolved = config_service.resolve_provider(focus_framework=framework)
        model = self._model or resolved["model"]
        provider = self._provider or resolved["provider"]
        base_url = resolved["base_url"] or None
        api_key = resolved["api_key"] or None
    except Exception as exc:
        logger.debug(...)
    return model, provider, base_url, api_key
    # NEW: after this resolution, in execute(), check:
    #   if not api_key and provider configured as non-mock -> use MockAtlasAgent's
    #   factory instead of _default_factory. This preserves the A4 honest-failure
    #   path for a CONFIGURED-but-unreachable provider (api_key present, wrong, 401)
    #   — that case still goes through the real foundation and fails honestly.
```
The mock path must be a distinct branch keyed on "no credential configured" (api_key empty AND provider api_key field is `""` not an `env:` ref that resolves to empty due to a *typo* — CONTEXT.md does not require distinguishing "never configured" from "misconfigured env var"; treat both as the same trigger per the locked decision "auto-engaged when no provider credentials are present").

### Anti-Patterns to Avoid
- **Reusing commit `147dd5b` literally for `--demo`:** That commit added `model_registry.seed_default_models` (3 model rows only) — it contains no mission/audit/wiki fixture. CONTEXT.md's instruction to "reuse the deterministic seed established in commit `147dd5b`" should be read as "reuse the *idempotent-seed pattern* (INSERT OR IGNORE / check-before-write)," not as "find and call existing code" — no such code exists for missions/wiki. Flagged in Assumptions Log.
- **Building a new config file for channels:** `atlas_runtime/cli/channels.py:40-48` shows channel config lives in the *foundation's* `~/.hermes/config.yaml` (`gateway.platforms`), a deliberately separate file from ATLAS's own `~/.atlas/config.yaml`. Adding a `channels` field to `AtlasConfig` per the CONTEXT.md schema should be a thin pointer/summary (e.g. which channels are *known* to ATLAS) rather than duplicating the foundation's live channel state — confirm with the planner whether `channels` in `AtlasConfig` is meant to mirror foundation state (read-only cache) or hold ATLAS-level channel preferences distinct from the foundation file. This is genuinely ambiguous in the drafted schema and should be flagged for discuss-phase or explicit planner decision.
- **Hardcoding gateway/cockpit ports in scripts:** `config_service.GatewayConfig.rust_port` (default 8484) and `CockpitConfig.port` (default 3000, though Vite dev actually defaults to 5173 — note the mismatch in STATE.md's own narrative which says "cockpit `:5173`" while `CockpitConfig.port` defaults to 3000) — `atlas up` must read these from config, not hardcode, and the planner should reconcile the 3000-vs-5173 discrepancy (likely 5173 is the `npm run dev` default and 3000 is an aspirational/preview-mode default that nothing currently enforces — grep confirms `CockpitConfig.port: int = 3000` at config_service.py:70, but `package.json` has no `--port` flag set, so Vite's own default of 5173 is what actually serves today).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Process detached-spawn + health-poll | A bespoke subprocess wrapper from scratch | Mirror `gateway_control.py`'s exact pattern (Popen + PID file + poll loop) for `cockpit_control.py` | Already proven correct on Windows (DETACHED_PROCESS/CREATE_NEW_PROCESS_GROUP) and POSIX (start_new_session); avoids re-debugging platform-specific spawn flags |
| Config atomic write | A new atomic-write helper | `config_service.save_config`'s existing tempfile+os.replace pattern | Already correct (same-dir tempfile avoids cross-filesystem rename issues; os.replace is atomic on both Windows and POSIX) |
| Migration tracking | A new migration runner for the demo seed | `db.apply_migrations`/`schema_migrations` table, called via existing `atlas db init` | The runner already exists and is drift-tolerant; the demo seed is data, not schema — it doesn't need migration tracking at all, just idempotent insert logic |
| Mock LLM response generation | A templating/randomization library | A plain Python dict of canned responses keyed by a simple heuristic (prompt hash or fixed rotation) | CONTEXT.md requires "stable, canned output" — determinism is the explicit requirement, so a complex generator is actively wrong here |

**Key insight:** Almost everything this phase needs a "pattern" for already exists once in this codebase (gateway_control, config_service, db.py, fresh_db_smoke.py). The actual net-new code is small: cockpit_control.py (gateway_control.py's twin), a mock agent class, a doctor aggregator, and a demo-seed function. The risk is NOT technical difficulty — it is mistakenly re-implementing what's already done (config-service, setup wizard, migration runner) because CONTEXT.md was written before the six-item operator scope work landed.

## Common Pitfalls

### Pitfall 1: Re-building the setup wizard / config-service from scratch
**What goes wrong:** A planner reading only CONTEXT.md (not STATE.md) creates duplicate `atlas setup`/`config_service.py`/`AtlasConfig` work, causing a merge conflict or silent duplicate-logic divergence with the already-shipped version.
**Why it happens:** CONTEXT.md was authored before, or without reconciling against, the 2026-06-20 six-item operator scope landing (commit `76df72d`).
**How to avoid:** Planner must read `services/agent-runtime/atlas_runtime/config_service.py` and `cli/config.py` FIRST (both already read in this research) and treat the "build now" language in CONTEXT.md as "extend the existing implementation," scoping new work strictly to: `channels` field, mock-provider wiring, `atlas up`, `atlas doctor`, demo seed, and install/docs.
**Warning signs:** A plan task titled "Create AtlasConfig Pydantic model" or "Implement atlas setup wizard" — these already exist; such a task should instead read "Extend AtlasConfig with `channels`" or similar.

### Pitfall 2: Mock-provider trigger ambiguity with the A4 honest-failure path
**What goes wrong:** If the mock branch is inserted inside `config_service.resolve_provider()` instead of inside the agent's `_resolve_provider`/`execute`, every consumer of `resolve_provider` (including future callers that expect a real-or-empty key, not a mock signal) silently gets mock behavior, and the Phase A4 "empty creds = honest failure, fall back to foundation's own resolution" contract breaks for any caller that isn't the agent runtime.
**Why it happens:** `resolve_provider` is a shared utility (native.py:124, plus any future caller); injecting mock logic there couples an agent-runtime decision into a config utility.
**How to avoid:** Keep `resolve_provider()` exactly as-is (returns the resolved real-provider fields, empty api_key on no creds). Add the mock decision as a new conditional in `NativeAtlasAgent.execute`/`_resolve_provider` (native.py) that checks `if not api_key: use MockAtlasAgent factory` BEFORE falling through to `_default_factory`/the foundation's own AIAgent construction. This is also where the "MOCK MODE" audit/observability signal should be emitted (an audit event or a field on the eventual `RunOutcome`).
**Warning signs:** Tests for `config_service.resolve_provider` start asserting `provider == "mock"` — that's the warning sign the boundary was drawn in the wrong layer.

### Pitfall 3: `atlas up` racing or double-spawning the gateway
**What goes wrong:** `gateway_control.start()` is already idempotent (checks `health_ok()` first), but a new `cockpit_control.start()` written hastily might not check for an already-running Vite process on the configured port, leading to `EADDRINUSE` on re-run.
**Why it happens:** Vite/npm dev servers don't expose a clean `/health` JSON the way the Rust gateway does; a naive `subprocess.Popen(["npm","run","dev"])` with no pre-check will double-spawn.
**How to avoid:** `cockpit_control.health_ok()` should do a plain `urllib.request.urlopen(f"http://127.0.0.1:{port}/", timeout=1)` and treat ANY non-exception response (even non-200, since Vite's dev index may 200 but a built preview could differ) as "already running," mirroring `gateway_control.health_ok()`'s try/except pattern exactly.
**Warning signs:** `atlas up` run twice in a row throws an unhandled `OSError`/`EADDRINUSE` from npm instead of printing "cockpit already running."

### Pitfall 4: Windows console-window flashes during detached spawn
**What goes wrong:** Spawning `npm.cmd` (not `npm`) on Windows via `subprocess.Popen` without the same `creationflags` used for the gateway can flash a visible console window — the exact class of bug fixed in the immediately-prior commit (`fix(gateway): suppress console window flashes on Windows CLI dispatch`, per git log).
**Why it happens:** `npm` on Windows is a `.cmd` shim that needs `shell=True` or explicit `cmd /c`, and any such invocation needs the same `CREATE_NO_WINDOW`/`DETACHED_PROCESS` flags as the gateway spawn, or it will flash a terminal.
**How to avoid:** Reuse `gateway_control._child_env()`-style flag construction; for npm specifically, resolve to `npm.cmd` explicitly and pass the same `creationflags=subprocess.DETACHED_PROCESS | subprocess.CREATE_NEW_PROCESS_GROUP` (Windows) seen in `gateway_control.py:92-97`, and additionally consider `subprocess.CREATE_NO_WINDOW` to suppress the shim's own console.
**Warning signs:** A visible terminal window pops and stays open when running `atlas up` on Windows.

### Pitfall 5: `.env.example` / config.yaml drift (two sources of truth for the same settings)
**What goes wrong:** `.env.example` documents `ATLAS_GATEWAY_BIN`, `ATLAS_HOME`, provider key env-var names, etc., while `~/.atlas/config.yaml` independently stores `gateway.rust_port`, `cockpit.port`, `provider.api_key` (as an `env:VAR` ref). If these two files disagree (e.g. `.env` sets `GATEWAY_PORT=9000` but `config.yaml` still says `8484`), nothing currently reconciles them — `gateway_control.GATEWAY_URL` reads `os.environ.get("ATLAS_GATEWAY_URL", ...)` (an env var, gateway_control.py:29) which is DIFFERENT from `config_service.GatewayConfig.rust_port` (a config.yaml field) — these are two independent settings for "what port is the gateway on" today.
**Why it happens:** The gateway lifecycle primitive predates the config-service; they were never unified.
**How to avoid:** Document this explicitly in `docs/INSTALL.md`: `.env`/`ATLAS_GATEWAY_URL` controls what the CLI *dials* to check health; `config.yaml`'s `gateway.rust_port` is currently informational only (no code reads it to *start* the gateway on that port — the Rust binary's own listen address is presumably a separate constant/arg, not investigated here). The planner should decide whether to wire `config.yaml`'s `gateway.rust_port`/`cockpit.port` into the actual spawn commands (`atlas up`) as the single source of truth, with `.env.example` only documenting the *env-var overrides* (`ATLAS_GATEWAY_URL`, `ATLAS_HOME`, `ATLAS_GATEWAY_BIN`) that exist independently of `config.yaml`. This reconciliation is real work the plan must explicitly scope, not an assumption to gloss over.
**Warning signs:** A user sets `cockpit.port: 4000` via `atlas config set cockpit.port 4000` and `atlas up` still serves on 5173/3000.

## Open Questions

1. **Does `cockpit.port`/`gateway.rust_port` in `AtlasConfig` need to become the actual single source of truth for `atlas up`'s spawn commands, or do `.env`/env-vars stay authoritative?**
   - What we know: `gateway_control.GATEWAY_URL` reads `ATLAS_GATEWAY_URL` (env var); `config_service.GatewayConfig.rust_port` is a separate config.yaml field; nothing currently unifies them.
   - What's unclear: Whether the Rust gateway binary itself reads `config.yaml` for its own listen port, or only ever listens on a hardcoded/env-controlled port (not investigated — would require reading the gateway's `main.rs`/binary entrypoint, not `lib.rs`).
   - Recommendation: Planner should grep the gateway's binary crate (likely `native/atlas-core-rs/src/main.rs` or similar, distinct from `crates/atlas-gateway/src/lib.rs`) for how the listen address is determined, before deciding whether `atlas up` needs to pass `--port` or set an env var to honor `config.yaml`'s port settings.

2. **Should `atlas doctor`'s exit code distinguish "mock mode is fine" from "real provider misconfigured"?**
   - What we know: CONTEXT.md says doctor checks "provider creds-present-or-mock" — implying mock mode should be a PASS, not a warning.
   - What's unclear: Whether a configured-but-broken provider (api_key set, but invalid) should make doctor fail loudly, vs. mock mode (no key at all) being silently fine.
   - Recommendation: Treat "no creds configured -> mock mode" as exit 0 with an informational MOCK MODE line; treat "creds configured but provider unreachable/401" as a distinct doctor warning (exit 0 still, since the app still runs in mock-equivalent fallback per A4, but the message should differ) — this needs explicit task-level acceptance criteria in the plan.

3. **Is the `AtlasConfig.channels` field new schema work, or should it be deferred entirely given channels already have a working CLI (`atlas channels`) over the foundation's own config file?**
   - What we know: `atlas_runtime/cli/channels.py` already provides `status`/`json`/`enable`/`disable`/`gateway start|status|stop` over `~/.hermes/config.yaml`'s `gateway.platforms`.
   - What's unclear: What net-new value an `AtlasConfig.channels` field adds that isn't already served by the existing `atlas channels json` command (which the gateway already dispatches at `GET /v1/channels`, lib.rs:561-567).
   - Recommendation: Default to NOT adding a redundant `channels` field unless the planner identifies a specific consumer need (e.g. the setup wizard wanting to configure channel tokens during `atlas setup` itself) that the separate `atlas channels` CLI doesn't already cover. Flag for discuss-phase if ambiguity persists — this is exactly the kind of schema-completeness-for-its-own-sake risk the CONTEXT.md "adopt the drafted schema" instruction can produce when the draft predates later shipped work.
