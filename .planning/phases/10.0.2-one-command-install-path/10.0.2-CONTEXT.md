# Phase 10.0.2: One-Command Install Path - Context

**Gathered:** 2026-06-22
**Status:** Ready for planning
**Mode:** Smart discuss (autonomous) — 4 grey areas, operator-decided

<domain>
## Phase Boundary

A developer can clone, configure, and run ATLAS without the operator's help.

This phase delivers the **install + first-run experience** for the multi-runtime stack
(Rust gateway + Python agent-runtime + React cockpit, SQLite via `infra/migrations`):

- A **native cross-platform setup path** (primary) that boots gateway + cockpit with one command,
  with **Docker Compose as an optional/secondary path**.
- An **`atlas setup` wizard + `~/.atlas/config.yaml` config-service** (pulled forward from the
  planned 10.0.3 #2 setup-wizard — see Deferred Ideas for the cross-phase reconciliation).
- A **deterministic mock provider** auto-engaged when no LLM credentials are present, so the
  product is demo-able fully offline.
- Seed/demo data and an **`atlas doctor` health-check CLI** wrapping the existing gateway `/health`.
- A quickstart + install-troubleshooting doc, verified on a clean environment (not the operator's
  machine state).

Out of boundary: cockpit redesign/brand (10.0.3), developer integrations (10.0.4), golden-workflow
hardening (10.0.5), public distribution/packaging (10.0.6).
</domain>

<decisions>
## Implementation Decisions

### Install Mechanism
- **Native scripts are the primary one-command path; Docker Compose is provided as an optional
  alternative.** Extend the existing PowerShell pattern (`scripts/install-atlas-cli.ps1`) with a
  cross-platform setup (`scripts/setup.ps1` + `scripts/setup.sh`) that provisions the repo `.venv`
  (the ambient PATH python is the pip-less Hermes venv — known Phase A footgun), builds the gateway
  and cockpit, and exposes a single boot command (`atlas up`) that starts gateway + cockpit.
- **Build-from-source** (cargo build gateway, vite build cockpit, python venv) — no prebuilt binaries.
- **Docker Compose optional:** a root `docker-compose.yml` is provided and documented as a clean-env
  alternative; it is NOT the canonical path. It doubles as the primary clean-environment verification
  surface for success criterion 4.
- Keep the existing native dev scripts working; the new setup wraps/extends them, does not replace.

### Configuration & Secrets (setup-wizard — GROUNDING CORRECTION: already shipped)
- **Operator decision was "pull setup-wizard forward / build now." Research (RESEARCH.md) found it is
  ALREADY SHIPPED** in commit `76df72d` ("feat(config): atlas setup wizard + config-service + gateway
  /v1/config"): `config_service.py` (`AtlasConfig` Pydantic schema, `~/.atlas/config.yaml`, atomic
  write, idempotent), `cli/config.py` (`atlas setup`, `atlas config get/set/show/export/import`),
  gateway `GET /v1/config` (masked), `System.tsx` overview, tests. The setup-wizard phase dir only
  ever held a PHASE.md; the implementation landed under this commit instead.
- **Net effect: the operator's intent is satisfied with no new build.** This phase does NOT rebuild
  config-service. It only (a) verifies it, (b) wires it into `.env.example`/`docs/INSTALL.md` as the
  documented configure step, and (c) fills genuine gaps surfaced during planning (none currently
  required for the phase goal).
- **Open question deferred:** `AtlasConfig.channels` may duplicate the existing `atlas channels` CLI
  over the foundation's `~/.hermes/config.yaml` — do NOT add channels schema work here (Deferred).
- **Secrets are `env:VAR_NAME` references, never inline plaintext** (D-002 posture); import/setup
  rejects inline `sk-`/`Bearer ` values.
- Ship a root **`.env.example`** documenting provider key refs, gateway/cockpit ports, and db path,
  consumed by both the native scripts and the optional compose path. `.env.example` and the wizard
  coexist: the wizard is the interactive path, `.env.example` the manual/CI path.
- No `auth.json` encryption / cross-process lock (that is the paused 10.1 auth store).

### Mock / Demo Mode & Seed
- **Build a new deterministic mock provider**, auto-engaged when no provider credentials are
  configured. Missions/runs/wiki produce stable, canned output so the cockpit shows a working demo
  with zero keys. Surface a clear "MOCK MODE — no live model" banner/state.
- This is distinct from the Phase A4 foundation-fallback (honest failure on empty creds); mock mode
  is an explicit, deterministic demo path. GROUNDING CORRECTION (RESEARCH.md): insert the mock branch
  at **`NativeAtlasAgent._resolve_provider` (native.py:116-135)**, NOT inside
  `config_service.resolve_provider`, so the Phase A4 honest-failure path is preserved. Resolution
  order explicit and documented (configured provider → mock when no creds).
- **Seed/demo data via `atlas db init --demo`** (or equivalent): a sample mission + audit trail +
  one wiki entry. GROUNDING CORRECTION: commit `147dd5b` only seeded 3 model-registry rows — it has
  NO mission/audit/wiki fixture. The demo seed is therefore **genuinely new work**, not a reuse.

### Health Check & Verification
- **New `atlas doctor` CLI** that checks: db reachable + migrated, `~/.atlas/config.yaml` valid,
  gateway/cockpit ports up, and provider creds-present-or-mock. It wraps the existing gateway
  `/health` route (`native/atlas-core-rs/.../atlas-gateway/src/lib.rs`) rather than duplicating it.
- **Install-troubleshooting docs** ship alongside: a `docs/INSTALL.md` (quickstart + troubleshooting)
  plus a README quickstart section.
- **Clean-environment verification (criterion 4):** use the optional Docker path for an inherently
  clean run, and reuse the `scripts/fresh_db_smoke.py` pattern for a fresh-checkout smoke that
  exercises setup → boot → `atlas doctor` → mock-mode mission run.
</decisions>

- `scripts/install-atlas-cli.ps1` — existing installer that targets a dedicated repo `.venv` and
  regenerates the `atlas.cmd` shim (Phase A1 hardening). Extend, don't replace.
- `scripts/fresh_db_smoke.py`, `scripts/foundation_boot_smoke.py` — existing smoke-test patterns for
  clean-environment verification.

- `services/agent-runtime/atlas_runtime/gateway_control.py` + `*_control.py` modules — process
  control for gateway/discord/messaging/cashflow; `atlas up` should follow this pattern.
- Phase A4 provider/model routing (`config_service.resolve_provider`, `NativeAtlasAgent.execute`) —
  the single path all run surfaces flow through; the mock provider plugs in here.
- `infra/migrations/0001_core.sql … 0012_*.sql` — SQLite migration chain; `atlas db init` runs it.
- Deterministic seed from commit `147dd5b` — reuse for `--demo`.
- `packages/atlas-core` — Python core where the Pydantic `AtlasConfig` schema belongs.

### Established Patterns
- Windows-primary dev; PowerShell tooling; repo `.venv` (NOT the ambient pip-less Hermes venv).
- Rust gateway is dispatch-only (D-022); it shells the `atlas` CLI via `ATLAS_CLI` (Phase A1).
- Active cockpit: `services/web-ui-react` (name `atlas-cockpit-react`), Vite SPA — `dev`/`build`/
  `preview` scripts; gateway+cockpit are the two surfaces criterion 1 must boot.
- Secret posture: `env:VAR` refs only, audit-first (D-002), no inline secrets in tracked files.
- No foundation edits (D-001); `~/.atlas/` is ATLAS-owned, independent of `~/.hermes/`.

### Integration Points
- `atlas` CLI (Typer) in `services/agent-runtime/atlas_runtime/cli/` — `setup`, `config`, `doctor`,

- Cockpit System/Overview surface reads `/v1/config` + `/health` (minimal here; full System tabs are
  10.0.3 work).

<specifics>
## Specific Ideas

- One-command boot UX target: `git clone` → (`atlas setup` or `cp .env.example .env`) → `atlas up`
  → gateway `:8080` + cockpit `:5173` live; `atlas doctor` reports READY (MOCK if no keys).
- Mock-mode banner copy: "MOCK MODE — no live model" surfaced in cockpit + `atlas doctor`.
- Adopt the config schema already drafted for the setup-wizard rather than designing fresh.

**Net-new scope (from RESEARCH.md — config-service/setup are already shipped):**
- `cockpit_control.py` (twin of `gateway_control.py` lifecycle pattern) + `atlas up` (boots
  gateway + cockpit together).
- `atlas doctor` aggregator wrapping the existing gateway `/health` + `db.migration_status`.
- Deterministic mock provider branch at `native.py:116-135` (`NativeAtlasAgent._resolve_provider`).
- New demo-seed function (sample mission + audit + wiki) behind `atlas db init --demo`.
- Install/docs layer: `.env.example`, optional `docker-compose.yml`, `docs/INSTALL.md`, README
  quickstart; **extend `scripts/install-atlas-cli.ps1`** which currently does NOT build the gateway
  binary or the cockpit.

**Landmines (from this repo's history):**
- Windows console-window flash: any new `npm`/cockpit subprocess spawn must use the same
  `DETACHED_PROCESS`/`CREATE_NEW_PROCESS_GROUP` flags as `gateway_control.py` (commit `62e9456`).
- Port authority: `gateway_control.GATEWAY_URL` (env `ATLAS_GATEWAY_URL`) vs
  `config_service.GatewayConfig.rust_port` are not unified — `atlas up` must pick one as
  authoritative (default to config.yaml; env overrides). Planner should resolve this explicitly.
- No container engine on the dev machine — Docker path cannot be verified here; use a native
  fresh-checkout smoke (modeled on `scripts/fresh_db_smoke.py`) for criterion 4.
</specifics>
