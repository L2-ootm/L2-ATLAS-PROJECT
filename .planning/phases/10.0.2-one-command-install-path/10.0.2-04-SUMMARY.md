## Decisions Made

- The smoke test deliberately does not subprocess-invoke the real `atlas` CLI for db/gateway/cockpit steps, because `db.connect()`, `gateway_control.PID_FILE`, and `cockpit_control.PID_FILE` all hardcode `pathlib.Path.home() / ".atlas"` and ignore `ATLAS_HOME` — confirmed by reading each module directly. Calling the same underlying service functions against an explicit temp connection/home preserves test isolation while still exercising the identical code paths `atlas up`/`atlas doctor`/`atlas db init --demo` dispatch to.
- `docker-compose.yml` was authored fresh with `dockerfile_inline` blocks rather than separate `Dockerfile` files, since no ATLAS-owned Dockerfiles exist yet (`foundation/atlas-hermes/docker-compose.yml` was checked first as the plan instructed, but its single-image Hermes-specific pattern didn't fit three independently-built ATLAS services).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Fixed `demo_seed.py`'s ATLAS_HOME-ignoring idempotency sentinel**
- **Found during:** Task 2 (writing `tests/test_fresh_install_smoke.py` and running it for the first time — RED-phase discovery, not assumed)
- **Issue:** `SENTINEL_FILE = pathlib.Path.home() / ".atlas" / ".demo_seeded"` was a module-level constant evaluated once at import time using the real home directory, completely ignoring the `ATLAS_HOME` env var that `_wiki_dir()` in the same file already correctly honors. On this dev machine (which had previously run `atlas db init --demo` for real), every call to `seed_demo_data()` — even against a brand-new isolated temp DB/`ATLAS_HOME` — immediately short-circuited with `{"created": False, "reason": "already seeded"}` because it checked the real, pre-existing `~/.atlas/.demo_seeded` file instead of one scoped to the test's isolated environment. This is a pre-existing correctness bug in code outside this plan's `files_modified` list, but it directly blocked Task 2's required behavior test from passing, qualifying as a Rule 3 blocking issue (and Rule 1 bug — the function's own docstring promises ATLAS_HOME-aware sentinel resolution that the constant never delivered).
- **Fix:** Removed the module-level `SENTINEL_FILE` constant; added a lazily-resolved `_sentinel_file()` function that re-reads `ATLAS_HOME` (via the existing `_atlas_home()` helper) on every call, mirroring the file's own `_wiki_dir()` pattern. `seed_demo_data()` now calls `_sentinel_file()` instead of referencing the frozen constant.
- **Files modified:** `services/agent-runtime/atlas_runtime/demo_seed.py`, `services/agent-runtime/tests/test_demo_seed.py` (updated the `_isolated_sentinel` autouse fixture to set `ATLAS_HOME` via `monkeypatch.setenv` instead of monkeypatching a now-removed module attribute)
- **Verification:** Full `test_demo_seed.py` suite (7 tests) passes; full `tests/test_fresh_install_smoke.py` suite (2 tests) passes; full agent-runtime suite (289 tests) passes; confirmed via `stat`/mtime that the real `~/.atlas/.demo_seeded` file's modification time was unchanged before/after the test run, proving real state was never touched.
- **Committed in:** `c90d1fa` (Task 2 commit)

---

No scope creep: the fix is a 2-line surgical change (constant → lazy function) consistent with the file's own existing `_wiki_dir()` convention, plus a matching one-line test-fixture update.
