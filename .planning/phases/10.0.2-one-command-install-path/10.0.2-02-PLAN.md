<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Provider resolution -> mock/real branch | The empty-vs-non-empty `api_key` check is the single gate deciding whether a real external LLM call or a canned local response is used |
| `demo_seed.py` -> filesystem (temp file) -> wiki ingest | A temp file is written, read back by `ingest_source`, then deleted — local-only, but the path/cleanup must be correct to avoid littering or partial state |
| `atlas db init --demo` -> production DB | The `--demo` flag must never run against a database the operator did not explicitly target with `db init` |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10.0.2-05 | Spoofing/Information Disclosure | mock-mode masking a misconfigured real provider | mitigate | Branch gated strictly on `if not api_key:` (empty string/None only) — Task 1 Test 5 is a regression guard asserting a non-empty-but-wrong key routes to `_default_factory`, never to mock, so credential errors surface honestly (preserves Phase A4 contract) |
| T-10.0.2-06 | Tampering | `demo_seed.py` temp file passed to `wiki_service.ingest_source` | mitigate | Path is generated locally via `tempfile.NamedTemporaryFile`, never derived from user/network input; `ingest_source` itself already validates path-is-a-file (T-06-05 path traversal guard, pre-existing) |
| T-10.0.2-07 | Repudiation | Demo seed silently no-ops on re-run | accept | Returns `{"created": False, "reason": "already seeded"}` (visible in CLI echo) rather than failing loudly — acceptable since this is a non-destructive idempotent no-op, not a security-relevant suppression |
| T-10.0.2-08 | Denial of Service / Elevation of Privilege | `--demo` flag writing demo data to an unintended (e.g. production) database | accept | `atlas db init --demo` operates on the exact same `db.connect()` target as a normal `db init` — no new DB-selection surface is introduced; the operator's existing `ATLAS_HOME`/config controls which DB is targeted, identical to today's non-demo `db init` risk profile |
</threat_model>
