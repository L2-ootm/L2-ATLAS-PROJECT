# Phase 10.0.2: Code Review Report

**Reviewed:** 2026-06-21T00:00:00Z
**Depth:** standard
**Files Reviewed:** 20
**Status:** issues_found

## Summary

The core safety-critical logic for this phase is sound: the mock-provider gate in
`native.py` correctly routes only on a falsy `api_key` (`if not api_key:`), and the
honest-failure contract is exercised by a dedicated regression test
(`test_native_execute_routes_to_real_provider_when_key_present_but_wrong`). The
secret-hygiene posture in `doctor.py`/`config_service.py`/`.env.example` is also
correctly implemented — `resolve_provider` never returns the literal secret to a
caller that echoes it carelessly, and `doctor.py` only ever prints
`"configured"`/`"mock"`. `demo_seed.py`'s sentinel is correctly ATLAS_HOME-aware and
lazily resolved (avoiding the import-time-freeze pitfall the docstring calls out).

The most significant defect is that the cockpit-side "Mock Mode" UX promised by
`docs/INSTALL.md` does not exist: `MockModeBanner.tsx` is written but never
imported or rendered anywhere in the cockpit, and no gateway/API surface exposes a
`mock_mode` flag for it to consume. This is a documentation/functionality mismatch
that should block "done" claims for the phase's UX promise, even though the backend
mock-routing itself is correct.

Secondary issues: `cockpit_control.stop()`/`gateway_control.stop()`-style PID-file
kill has no liveness/identity check before `taskkill`/`os.kill`, risking killing an
unrelated process that reused the PID after the cockpit died abnormally; `db_init`
constructs three independent `threading.Lock()` instances across one CLI
invocation, giving no actual cross-call mutual exclusion (lower severity since the
CLI process is single-shot); and a few completeness/robustness gaps in the install
scripts and `cockpit_control._parse_port`.

## Critical Issues

### CR-01: MockModeBanner is dead code — the documented cockpit UX does not exist

**File:** `services/web-ui-react/src/components/MockModeBanner.tsx:1-45`
**Issue:** `MockModeBanner` is never imported or rendered by any other component in
`services/web-ui-react/src`. There is also no gateway/API endpoint or cockpit data
hook that surfaces a `mock_mode` boolean for this component to consume — `mock_mode`
does not appear anywhere else in the cockpit source tree, nor in the Rust gateway
(`native/`). `docs/INSTALL.md` states as a load-bearing claim:

> "every mission run still executes end-to-end through a deterministic, canned
> response and **the cockpit displays a "MOCK MODE — no live model" banner**"

This claim is false as shipped. An operator running in mock mode (the default,
zero-credential install path this entire phase is about) gets no visual indication
in the cockpit that they're seeing canned output — the one UX signal the phase
explicitly promises is absent. This is a correctness/completeness defect for a
phase whose stated purpose is the one-command install + mock-mode demo experience.
**Fix:** Either wire `MockModeBanner` into the cockpit's app shell (driven by a real
`mock_mode` signal — e.g. an `atlas doctor`-equivalent gateway endpoint, or have the
gateway include `mock_mode` in run/mission API responses), or correct
`docs/INSTALL.md`/README to not claim a banner exists until it is actually wired.
Do not ship an unreferenced component as if the feature were complete.

## Warnings

### WR-01: PID-file kill has no liveness/identity check before signaling

**File:** `services/agent-runtime/atlas_runtime/cockpit_control.py:88-104`
**Issue:** `stop()` reads the PID from `PID_FILE` and unconditionally runs
`taskkill /PID <pid> /F` (Windows) or `os.kill(pid, 15)` (POSIX) without verifying
the PID still belongs to the process this primitive spawned (e.g. by checking the
process name/command line, or storing a start-time/cmdline fingerprint alongside
the PID). If the cockpit process crashed or was killed externally and the OS
recycled that PID for an unrelated process before `atlas gateway stop`/`atlas up`'s
teardown runs, this will kill the wrong process. PID reuse is realistic on
long-running dev machines between cockpit restarts.
**Fix:** Store a lightweight fingerprint with the PID (e.g. process start time via
`psutil`, or re-verify the listening port/health before killing), or at minimum
verify the process is still alive and is plausibly `npm`/`node` before signaling:
```python
import psutil
def stop() -> tuple[bool, str]:
    if not PID_FILE.exists():
        return False, "no pid file; cockpit not managed here"
    pid = int(PID_FILE.read_text().strip())
    try:
        proc = psutil.Process(pid)
        if "node" not in proc.name().lower() and "npm" not in proc.name().lower():
            PID_FILE.unlink(missing_ok=True)
            return False, "pid no longer belongs to a cockpit process (removed)"
    except psutil.NoSuchProcess:
        PID_FILE.unlink(missing_ok=True)
        return False, "cockpit process already gone (removed)"
    ...
```

### WR-02: `db_init` creates three unrelated `threading.Lock()` instances within one command

**File:** `services/agent-runtime/atlas_runtime/cli/main.py:803-823`
**Issue:** `db_init()` calls `db.apply_migrations(conn)` (no lock), then
`model_registry.seed_default_models(conn, threading.Lock())`, then (if `--demo`)
`demo_seed.seed_demo_data(conn, threading.Lock())` — each invocation constructs a
brand-new `Lock`, so none of these three writers are mutually exclusive with each
other or with any concurrent `atlas` invocation against the same DB. The module
already has a `_get_lock()` singleton factory (line 112-114) used by every other
command in this file; `db_init` bypasses it entirely. Low real-world risk for a
single-shot CLI process today, but it both defeats the locking convention this file
otherwise follows and would become a real race if `db init --demo` is ever invoked
concurrently with another writer against the same `atlas.db` (e.g. a gateway already
running against the same file).
**Fix:** Use `_get_lock()` consistently:
```python
lock = _get_lock()
seeded = model_registry.seed_default_models(conn, lock)
...
result = demo_seed.seed_demo_data(conn, lock)
```

### WR-03: `cockpit_control._parse_port` silently falls back to 5173 on any unparsable/portless URL, masking misconfiguration

**File:** `services/agent-runtime/atlas_runtime/cockpit_control.py:34-37`
**Issue:** `_parse_port` returns `parsed.port or 5173` for any URL where
`urlparse` doesn't yield a port (malformed `ATLAS_COCKPIT_URL`, a URL with a
non-numeric "port" causing `urlparse` to raise, or a scheme-less value). Combined
with `health_ok()` probing `COCKPIT_URL` (the original string, not the
re-derived port), an operator who sets `ATLAS_COCKPIT_URL=http://127.0.0.1:abc` (a
typo) gets silently routed to spawn `npm run preview -- --port 5173` while
`health_ok()` polls `http://127.0.0.1:abc/` — which will never succeed, and the
resulting failure message ("cockpit did not become healthy in time") gives no hint
that the root cause is a malformed port in `ATLAS_COCKPIT_URL`.
**Fix:** Validate the port at module load / in `start()` and fail fast with an
actionable message rather than silently defaulting:
```python
def _parse_port(url: str) -> int:
    parsed = urllib.parse.urlparse(url)
    try:
        return parsed.port or 5173
    except ValueError as exc:
        raise ValueError(f"ATLAS_COCKPIT_URL has an invalid port: {url!r}") from exc
```

### WR-04: `install-atlas-cli.ps1` / `setup.sh` swallow all build-step failures via `Out-Null`/`--silent`, no exit-code check on `npm run build` / `cargo build`

**File:** `scripts/install-atlas-cli.ps1:52,74`; `scripts/setup.sh:69,89`
**Issue:** Both scripts run `npm install --silent; npm run build` (and the
PowerShell variant wraps it in try/finally with `$ErrorActionPreference = 'Stop'`,
but `setup.sh` uses `set -euo pipefail` which *should* catch a failing `npm run
build`/`cargo build` exit code — however `npm install --silent` suppresses
diagnostic output on failure, making any actual install/build failure on this path
hard to diagnose since stdout is silenced and the operator only sees the bash `set
-e` abort with no further context. More importantly, in the PowerShell script the
`cargo build --release -p atlas-gateway` and `npm run build` calls inside `try {
... } finally { Pop-Location }` are not followed by an explicit
`if ($LASTEXITCODE -ne 0) { throw ... }` check — PowerShell's `$ErrorActionPreference
= 'Stop'` only converts terminating *cmdlet* errors to exceptions; a non-zero exit
code from an external `.exe` (cargo, npm) does NOT automatically throw, so a failed
build is silently treated as success and the script proceeds to print "Done."
**Fix:** Add explicit exit-code checks after each external-process call in the
PowerShell script:
```powershell
cargo build --release -p atlas-gateway
if ($LASTEXITCODE -ne 0) { throw "cargo build failed (exit $LASTEXITCODE)" }
```
and likewise after `npm run build` in both the TUI and cockpit build blocks.

### WR-05: `setup.sh`'s `--claude` flag check is case-sensitive and silently accepts unrecognized args without warning

**File:** `scripts/setup.sh:15-20`
**Issue:** Any argument other than the literal string `--claude` is silently
ignored (no validation, no usage error). A user invoking `./scripts/setup.sh
-claude` or `--Claude` gets the non-Claude install path with zero feedback that
their flag was not recognized. The PowerShell twin (`[switch]$Claude`) at least
fails on unknown named parameters, so the two "kept in sync" scripts diverge in
argument-validation strictness.
**Fix:**
```bash
for arg in "$@"; do
  case "$arg" in
    --claude) CLAUDE=1 ;;
    *) echo "Unknown argument: $arg" >&2; exit 1 ;;
  esac
done
```

## Info

### IN-01: `_default_factory`'s bare `from run_agent import AIAgent` relies on `sys.path` mutation with no cleanup

**File:** `services/agent-runtime/atlas_runtime/agents/native.py:70-77`
**Issue:** `_find_foundation()` inserts the foundation path into `sys.path`
permanently (never removed), and does so on every call to `_default_factory` (the
`if path not in sys.path` guard prevents duplicate inserts but the mutation is
process-global and irreversible). Not a bug in the current single-process-per-run
CLI model, but worth a comment noting the intentional global side effect, since a
future long-lived daemon (`atlas runtime serve`, already present in this same file
set) sharing this code path would accumulate `sys.path` pollution across runs with
different foundation locations if that ever becomes configurable.
**Fix:** Add a one-line comment acknowledging the global, one-time nature of this
mutation, or guard it behind a module-level "already inserted" flag set once at
import time instead of per-call.

### IN-02: `doctor.py`'s `provider: skipped (config invalid)` path never sets `all_ok = False` independently — relies entirely on the earlier config check

**File:** `services/agent-runtime/atlas_runtime/cli/doctor.py:78-79`
**Issue:** When `cfg is None` (config failed to load), the provider check prints
`"provider: skipped (config invalid)"` and does not touch `all_ok`. This is correct
behavior (the docstring states provider/mock is informational-only and the earlier
config check already set `all_ok = False`), but the lack of an explicit comment at
this branch makes it easy for a future editor to assume the skip is itself meant to
fail the run and "fix" it by adding `all_ok = False` here, double-penalizing the
same root cause. Purely a maintainability note.
**Fix:** Add an inline comment: `# all_ok already False from the config check above; this branch is purely informational.`

### IN-03: `fresh_install_smoke.py` mutates the real process `os.environ["ATLAS_HOME"]` for the entire run rather than scoping it

**File:** `scripts/fresh_install_smoke.py:61-84`
**Issue:** The script correctly restores the prior `ATLAS_HOME` value in a
`finally` block, so this is not a correctness bug in isolation. However, because
`ATLAS_HOME` is mutated process-globally rather than passed explicitly to each
ATLAS_HOME-aware call, if any future ATLAS_HOME-aware module reads the env var
lazily on a background thread spawned during `_run_steps` (e.g. a future async
demo-seed worker), a restore in the `finally` block of the main thread would race
with that background read. Currently nothing in `_run_steps` spawns threads that
read `ATLAS_HOME`, so this is purely a latent risk for future contributors, not a
present bug.
**Fix:** No action required now; flag in a comment if/when background-thread work
is added to this script's step sequence.
