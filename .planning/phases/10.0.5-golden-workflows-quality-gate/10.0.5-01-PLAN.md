`AuditEvent.event_type` is a closed `Literal` in atlas_core/schemas/core.py — it
currently has NO "golden_workflow_*" values. This plan must EITHER (a) reuse an
existing literal value (`"artifact"`, `"wiki_update"`, `"tool_completed"` etc.) for

event and `"wiki_update"` for the wiki-recorded event (both already exist and are
semantically correct), and introduce two NEW literal values
`"golden_workflow_started"` and `"golden_workflow_completed"` by editing
`AuditEvent.event_type` in packages/atlas-core/atlas_core/schemas/core.py — this is
a backward-compatible additive change (Literal grows, no migration needed since

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| golden_workflow_service -> artifacts table | first writer ever — must not corrupt FK to runs(id) on a fresh DB |
| golden_workflow_service -> audit_events | every lifecycle event must carry a valid run_id FK (operator run bootstrap) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-1005-01 | Tampering | record_artifact content hashing | mitigate | sha256 computed server-side from actual bytes written, never trusts a caller-supplied hash |
| T-1005-02 | Denial of Service | record_artifact on a fresh DB | mitigate | ensure_golden_run bootstraps the operator run before any artifacts/audit_events INSERT, avoiding FK violation crash loops |
| T-1005-03 | Information Disclosure | AuditEvent.data payload | accept | emit() already redacts via SECRET_PATTERNS at the audit boundary (existing audit_service behavior); golden_workflow_service inherits this for free since it always calls emit() |
</threat_model>
