<decisions>
## Implementation Decisions (operator-confirmed 2026-06-23)

### D-1 — Workflow vehicle: Operations registry
The three golden workflows ship as entries on the premade **Operations** surface
(`operation_service.py` pattern), cockpit-button + CLI triggerable.
- Repo Triage + Research Brief: **internal-risk** (auto-run).
- Self-Review: its writes route through the **10.0.4 approval gate** (tool_service →
  pending ToolApproval; never writes without approval).

### D-2 — Consistency verification: mock-mode + structural assert
A smoke test runs each workflow 3× in deterministic **mock mode** and asserts STRUCTURE,
not byte-identical output:
- artifact produced, audit trail emitted, wiki entry written;
- Self-Review yields a PENDING approval (no write executed).
Plus a **demo-reset path**. Live screenshots deferred to operator UAT.

### Claude's Discretion
Module layout, CLI/gateway surface shape, smoke-test file location, demo-reset mechanism.

<code_context>
## Existing Code Insights (grounding the design)

**CRITICAL — mock provider produces no structural output.**
`agents/mock.py::MockAtlasAgent.run_conversation` returns a single canned text dict
(`api_calls: 0`, no tool calls, no writebacks). So a *pure LLM-instruction* operation
(the current `operation_service` pattern) in mock mode yields only a run record + canned
text — NEVER artifact/wiki/structured audit. **Therefore the golden workflows must be
DETERMINISTIC ORCHESTRATORS**, not instruction templates: they perform real read-scans
via the tool layer, write artifacts + wiki entries, and emit audit events directly —
independent of the LLM. An optional LLM summarization step must degrade gracefully in
mock mode. This is the only way to honor D-2's structural assertions deterministically.

**Reusable surfaces:**
- `operation_service.py` — Operation dataclass + registry + `build_intent`; risk-gating
  (`internal` auto / `outward` needs approval). The golden workflows extend this surface
  but carry real execution logic (a `golden_workflow_service.py` orchestrator), not just
  rendered instructions.
- `tool_service.py` — `invoke(...)` chokepoint (read auto-runs; write/shell → pending
  approval), `list_approvals`, `approve`, `reject`. Self-Review's write goes here.
- Tool adapters (10.0.4): `workspace` (boundary-bounded read), `web_fetch` (SSRF+size-cap
  GET), `github`, `webhook_notify`. Repo Triage uses `workspace` reads; Research Brief
  should prefer **deterministic, offline** sources (wiki/codex search) over `web_fetch`
  so mock-mode smokes don't depend on network.
- `Artifact` model (atlas-core core.py:316): frozen; `run_id`, `path`, `artifact_type`
  (file_write|file_edit|file_delete|unknown), sha256, size_bytes. Artifacts table written
  per-run (mission_service handles the `artifacts` table; see DELETE on run purge).
- `atlas_wiki/wiki_service.py` (services/wiki-runtime) — ingest/update/search with
  provenance (run_id/operator/source/sensitivity). Wiki entries get provenance.
- Run/audit: a workflow needs a run for the audit trail + artifact FK. Reuse the
  `mission_service.ensure_operator_run` precedent (shared operator-run for non-mission
  side effects) — same pattern Phase C / 10.0.4 used for the approval FK.

**Test discipline (carried from 10.0.4 / memory `cli-db-path-not-atlas-home`):**
the `atlas` CLI hits `~/.atlas/atlas.db` regardless of ATLAS_HOME. Smokes MUST drive the
service layer against a temp-DB conftest fixture (apply ALL migrations), never the live CLI.
</code_context>

<specifics>
## Specific Ideas

- **Repo Triage:** workspace-read scan of the repo (key files: README, structure, recent
  changes) → triage artifact (markdown summary file recorded as an Artifact) + wiki page
  "repo-triage-<date>" + audit events via tool_service. Internal/auto.
- **Research Brief:** gather from the codex (searchWiki/listWikiPages) on a topic →
  brief artifact + wiki page. Deterministic + offline. Internal/auto.
- **Self-Review:** review recent run/code state → propose a write (e.g. a workspace_write
  of a review note, or a wiki update) via tool_service.invoke → **pending approval**.
  Assert no write executed until approve().
- **Quality gate:** `scripts/golden_workflows_smoke.py` (or pytest) runs each 3× in mock
  mode, asserts structure; a `--reset`/demo-reset clears generated artifacts/wiki/approvals
  for a clean re-demo.
- **Docs:** `docs/golden-workflows.md` (runbook + sample data) + a KNOWN-FAILURES section
  (or `docs/known-failures.md`).
</specifics>
