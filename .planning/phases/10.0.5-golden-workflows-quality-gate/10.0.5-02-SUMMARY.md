# Phase 10.0.5 Plan 02: Repo Triage + Research Brief Golden Workflows Summary

**Two internal-risk (auto-run) deterministic orchestrators — Repo Triage (real workspace read-scan via the 10.0.4 tool_service chokepoint) and Research Brief (offline FTS5 codex search, zero network surface) — each producing an Artifact + audit trail + wiki page on every invocation**

## Performance

- **Duration:** ~35 min
- **Tasks:** 2 (each RED-then-GREEN)
- **Files created:** 5

## Accomplishments

- `repo_triage.run_repo_triage` lists the workspace root and reads README.md exclusively through `tool_service.invoke(tool_name="workspace", ...)` — never a direct filesystem call — so the audit trail includes real `tool_requested`/`tool_completed` events from the 10.0.4 chokepoint, not just this workflow's own `golden_workflow_started`/`golden_workflow_completed` bookends. A missing README degrades to a "No README found" note instead of raising.
- `research_brief.run_research_brief` queries `atlas_wiki.wiki_service.search_wiki` (FTS5) directly — no `tool_service.invoke`, no `urllib`/`web_fetch` import anywhere in the module — proven network-independent by a test that monkeypatches `urllib.request.urlopen` to raise on any call. An empty search result degrades to an honest "No wiki entries matched '<topic>'" line.
- Both workflows call the wave-1 `golden_workflow_service` core (`ensure_golden_run`, `record_artifact`, `emit_workflow_event`) and `atlas_wiki.wiki_service.update_wiki_page` directly (no new wrapper, per the 10.0.5-01 decision), and both were verified to run 3x against the same input without raising and without losing structural consistency (non-empty `artifact_path` each time).

## Decisions Made
- `wiki_dir.mkdir(parents=True, exist_ok=True)` is called inside both workflow functions (not left to the caller) — the plan's interface block shows production callers (CLI/gateway) already do this, but making the workflow function self-sufficient avoids a `FileNotFoundError` in `wiki_service._update_index` for any caller (including tests) that passes a not-yet-created directory.
- Same-day re-runs of either workflow intentionally upsert the same wiki page (slug includes only the date, not a run-unique suffix) — `update_wiki_page`'s existing upsert-by-slug behavior increments `version` rather than creating duplicate pages, which satisfies D-2's "wiki entry written" assertion without requiring N distinct pages per demo session.

### Auto-fixed Issues

**1. [Rule 1 - Bug] wiki_dir did not exist when update_wiki_page wrote index.md/log.md**
- **Found during:** Task 1 first GREEN test run (`test_run_repo_triage_no_readme_degrades_gracefully` and others failed with `FileNotFoundError` on `wiki/index.md`)
- **Issue:** `wiki_service.update_wiki_page` calls `_update_index(wiki_dir, conn)` which does `(wiki_dir / "index.md").write_text(...)` — this assumes `wiki_dir` already exists; `update_wiki_page` itself only creates `wiki_dir/raw/` (inside `ingest_source`, not `update_wiki_page`). The plan's interface block documents the caller resolving `wiki_dir` with a `mkdir` call, but the workflow `<action>` text did not repeat that step.
- **Fix:** Added `wiki_dir = pathlib.Path(wiki_dir); wiki_dir.mkdir(parents=True, exist_ok=True)` immediately before the `update_wiki_page` call in both `repo_triage.py` and `research_brief.py`.
- **Files modified:** `services/agent-runtime/atlas_runtime/golden_workflows/repo_triage.py`, `services/agent-runtime/atlas_runtime/golden_workflows/research_brief.py`
- **Commit:** included in each task's GREEN commit (not a separate commit)
