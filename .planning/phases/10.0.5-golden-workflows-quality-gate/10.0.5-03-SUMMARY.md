# Phase 10.0.5 Plan 03: Self-Review (Approval-Gated) + Golden Workflow Registry Summary

**Self-Review proves the approval gate end-to-end (propose -> pending -> approve/reject, never an inline write) via a new golden_review_write write-class tool; golden_workflow_registry + `atlas golden list|run` unify all three golden workflows behind one dispatch surface**

## Performance

- **Duration:** ~40 min
- **Tasks:** 2 (each RED-then-GREEN)
- **Files created:** 7, modified: 2

## Accomplishments

- `golden_review_write` ships as a brand-new write-class tool (manifest + adapter + registry binding) — the manifest's `risk_level: write` forces `policy.decide().requires_approval=True` unconditionally, so `tool_service.invoke` always short-circuits to a pending `ToolApproval`. The adapter defensively re-validates the path through `policy.check_workspace_boundary` at execute-time (T-1005-08), independent of the propose-time authorization decision.
- `self_review.run_self_review` surveys the recent audit trail for the shared operator run (no LLM call, deterministic) and proposes a review-note write exclusively through `tool_service.invoke` — there is no code path in the module that calls the adapter directly. Proven by test: 3 calls produce 3 distinct pending approvals, zero files written, until an explicit `tool_service.approve` executes exactly the proposed write (and emits `tool_completed`); `tool_service.reject` leaves zero side effects.
- `golden_workflow_registry.dispatch` is a pure pass-through dispatcher unifying all three golden workflows behind one call shape, verified against `run_repo_triage`'s direct return by key-set comparison (not logic re-derivation). `atlas golden list`/`atlas golden run <id>` CLI (mirroring `cli/tools.py`'s thin-wrapper style) wires the registry into the CLI surface, wired into `cli/main.py` as its own top-level group.

## Decisions Made
- De-dup policy for `run_self_review`: NONE — each call creates a fresh, distinct pending approval. Simplest contract, matches the demo's "run 3x, assess 3x" intent, and avoids ambiguity the plan explicitly called out as needing a single committed answer.
- `self_review`'s own workflow lifecycle (`golden_workflow_completed`) fires once the write is successfully **proposed**, not once it is executed — documented prominently in the module docstring since "completed" is easy to misread as "executed" when the underlying `ToolApproval.status` is still "pending" at that moment.
- `golden_workflow_registry.GoldenWorkflow.risk` uses `"internal"|"approval"` vocabulary, intentionally distinct from `operation_service.Operation.risk`'s `"internal"|"outward"` — avoids conflating "is this workflow goal-bound/external-facing" with "does this workflow's write need approval."
- `golden_app` is a top-level CLI group (not nested under `operation_app`) since golden workflows aren't goal-bound — sized to this phase's actual need rather than overloading the Operations surface (per the plan's objective).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Test for `tool_service.approve` needs explicit `ctx` to resolve the correct workspace_root**
- **Found during:** Task 1, first run of `test_approving_self_review_executes_the_proposed_write`
- **Issue:** `tool_service.approve(db, lock, approval_id=...)` was called without `ctx`, so `golden_review_write.run`'s `_workspace_root(ctx)` helper defaulted to `"."` (the real process CWD), writing a file to the live `services/agent-runtime/golden/` directory on disk instead of `tmp_path`. This is consistent with the adapter's documented behavior (ctx is caller-supplied) but the test omitted it.
- **Fix:** Pass `ctx={"workspace_root": str(tmp_path)}` explicitly to `tool_service.approve` in the test, matching how a real CLI/gateway caller would resolve the workspace at approve-time.
- **Files modified:** `services/agent-runtime/tests/test_golden_workflow_self_review.py`
- **Commit:** included in Task 1's commit (`8e4fe99`)
- **Side effect cleaned up:** the stray `services/agent-runtime/golden/self-review-*.md` file written to the real filesystem during the failing run was deleted before staging; it was never committed.
