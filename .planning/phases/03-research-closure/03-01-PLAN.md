    ## 3. Scoring Criteria
    Define five criteria with weights. For each criterion, state the weight (HIGH/MEDIUM), describe exactly what is being measured, and explain why it matters for the ATLAS cockpit specifically:

    Criterion 1 — Realtime SSE audit stream (Weight: HIGH)
    The cockpit must display a real-time audit event stream (COCKPIT-02, COCKPIT-03). The source is FastAPI's /runs/{id}/events SSE endpoint. Measure: how many footguns exist when the browser's EventSource connects directly to FastAPI (bypassing the frontend framework's server)? See Pitfall 1 in 03-RESEARCH.md for the Next.js buffering issue.

    Criterion 2 — L2 code reuse (Weight: MEDIUM)
    D-006 rationale notes "L2 has existing Next.js muscle/code." Measure: can existing L2 React components, hooks, or API wrappers transfer to the cockpit? Quantify this as developer familiarity, not component portability, since the cockpit is a greenfield operator UI, not a port.

    Criterion 3 — Bundle size / initial load (Weight: HIGH)
    COCKPIT-06 requires < 2s load on localhost. Measure: framework runtime overhead shipped to the browser. Cite the research figures: SvelteKit compiles to vanilla JS (typical < 15 KB JS on initial page); Next.js ships React runtime (~45 KB) plus hydration bundle (~200–300 KB for a simple app). Both meet the < 2s bar on localhost, but SvelteKit has more headroom for future feature growth.

    Criterion 4 — Polish ceiling (Weight: MEDIUM)
    The cockpit must feel like mission control, not generic admin CRUD (from NATIVE_APP_STRATEGY.md). Measure: can an operator-grade dashboard with dense real-time event lists, status badges, and audit viewers be built without a component library? Note React's larger UI component ecosystem (shadcn/ui, Radix) vs. Svelte's smaller ecosystem. Note Svelte 5 rune-based fine-grained reactivity advantage for dense real-time UIs (no virtual DOM reconciliation).

    Criterion 5 — Deployment model (Weight: HIGH)
    D-005 mandates no Electron. Cockpit is served from FastAPI StaticFiles mount (MVP) or Rust sidecar static files (v2). Measure: does the framework produce a static build artifact with no persistent Node.js process required? SvelteKit adapter-static produces pure HTML/CSS/JS. Next.js requires a Node.js server process (output: 'export' has documented constraints on dynamic routes; see Assumption A3 from 03-RESEARCH.md). A third process (Node.js) alongside FastAPI + Rust sidecar adds operational complexity.

    ## 4. Scoring Table
