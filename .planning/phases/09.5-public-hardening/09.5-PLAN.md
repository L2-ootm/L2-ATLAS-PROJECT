# Phase 9.5 — v1.0 Public Hardening, Security Pass & Manual Acceptance Plan

**Type:** hardening / verification / public-readiness gate  
**Position:** after Phase 09, before any public/open-source release or v1.0 milestone archive  
**Scope:** no new product features; only hardening, quarantine, tests, scans, manual UAT, documentation, and release-readiness evidence.

## Goal

Make the v1.0 ATLAS repository safe and credible for a future public/open-source release by hardening the skill surface, vendored foundation, runtime, cockpit, docs, and manual acceptance path.

Phase 09 classified what should ship. Phase 9.5 verifies that the repo can actually be shown, run, tested, and manually accepted without hidden private context, unsafe default skills, stale branding, missing runbooks, or unverified critical flows.

## Non-goals

- Do not build Phase 10 native shell.
- Do not build a skill UI, marketplace, registry loader, or module system.
- Do not merge `feat/integrations-module-catalog`.
- Do not add CRM/Twenty/Pulse/voice/overlay features.
- Do not rewrite vendored Hermes history.
- Do not remove attribution/license references from upstream code.

## Required artifacts

- `.planning/phases/09.5-public-hardening/09.5-PLAN.md`
- `.planning/phases/09.5-public-hardening/09.5-VERIFICATION.md`

- `docs/operations/PUBLIC_RELEASE_HARDENING.md`
- updates to `docs/imports/SKILL_INVENTORY.md`
- updates to `foundation/atlas-hermes/DIVERGENCE_LOG.md` if quarantine/removal changes vendored foundation behavior

## Success criteria

1. Git status is clean at the end.
2. Unsafe/default-shipped skills identified in Phase 09 are quarantined or explicitly excluded from public manifests:
   - `red-teaming/godmode`
   - `inference/obliteratus`
   - `vault-scan`
   - dual-use offensive skills remain opt-in only.
3. Imported SG-3/SG-4 skills have a documented naming plan:
   - no public pack should expose legacy/import-source branding as the user-facing product name;
   - names should describe capability and pack placement.
4. A public-repo scan runs and reports no real secrets, local DBs, runtime sessions, private personal data, or accidental local artifacts.
5. Automated tests pass for the v1.0 surfaces:
   - Python core/runtime/wiki tests;
   - Rust gateway tests;
   - web UI build/check where available;
   - fresh DB bootstrap smoke.
6. Manual UAT guide exists and covers every critical v1.0 use case.
7. Davi can follow `MANUAL_TEST_GUIDE.md` step-by-step without hidden tribal knowledge.
8. `docs/operations/RUNNING.md` explains how to start gateway + web UI from a clean checkout.
9. All public-facing docs use canonical ATLAS/L2/Hermes branding and do not promote imported legacy skill sources as ATLAS product branding.
10. Remaining release blockers are explicit, bounded, and assigned to future phases if not fixed.

## Work plan

### Task 1 — Baseline continuation check

Run:

```bash
git status --short -uall
git branch --show-current
git log --oneline -5
```

Read:

- `AGENTS.md`
- `.planning/STATE.md`
- `.planning/ROADMAP.md`
- `.planning/phases/09-skill-inventory/09-01-SUMMARY.md`
- `.planning/phases/09-skill-inventory/VERIFICATION.md`
- `docs/imports/SKILL_INVENTORY.md`

Expected:

- branch is `main`;
- only intentional Phase 9 / 9.5 docs are dirty;
- scope is hardening only.

### Task 2 — Skill quarantine / exclusion pass

Inspect the current location and packaging status of:

- `foundation/atlas-hermes/skills/red-teaming/godmode`
- `foundation/atlas-hermes/skills/inference/obliteratus`
- any `vault-scan` imported reference
- optional offensive/security skills listed in `SKILL_INVENTORY.md`

Required result:

- either move unsafe default skills to a quarantined/non-default location, or document explicit exclusion from every public pack and manifest;
- update `foundation/atlas-hermes/DIVERGENCE_LOG.md` if vendored foundation content is moved/removed;
- update `docs/imports/SKILL_INVENTORY.md` blocker status.

Do not remove attribution or historical upstream comments. This is distribution hardening, not source laundering.

### Task 3 — Imported skill naming precision plan

Create a naming table in `docs/imports/SKILL_INVENTORY.md` or `docs/operations/PUBLIC_RELEASE_HARDENING.md`.

For SG-3/SG-4 skills that may be reused, define:

- current name;
- source group;
- proposed ATLAS public name or alias;
- reason;
- public-safe status;
- whether rename is required before shipping.

Rule:

Public names describe capability, not import origin.

Examples of naming direction:

- broad design skill → `atlas-interface-design-review` or `operator-ui-design-review` if kept;
- broad review skill → `atlas-extra-marathon-review` only if public-safe and L2-specific parts are removed;
- imported meta/design packs → keep internal until renamed and sanitized.

### Task 5 — Automated test pass

Run the strongest practical v1.0 suite.

Suggested commands, adjust to repo reality:

```bash
# Python
cd packages/atlas-core && pytest -q
cd ../../services/agent-runtime && pytest -q
cd ../wiki-runtime && pytest -q

# Rust
cd ../../native/atlas-core-rs && cargo test -p atlas-gateway

# Web UI
cd ../../services/web-ui
npm run check || true
npm run build
```

If `npm run check` does not exist, record that honestly.

### Task 6 — Fresh DB smoke

Run or create a documented smoke path for:

1. apply migrations 0001–0003 to a fresh SQLite DB;
2. create mission;
3. start run;
4. write wiki page/update;
5. list models;
6. retrieve run events;
7. cancel or complete safe test run;
8. confirm no FK failure and audit rows exist.

If no single script exists, document commands in `docs/operations/RUNNING.md` and mark future script creation as follow-up.

### Task 8 — Edge-case matrix

Add an edge-case matrix to either `MANUAL_TEST_GUIDE.md` or `09.5-VERIFICATION.md`.

Minimum edge cases:

- empty DB;
- long mission title/intent;
- invalid mission id;
- missing gateway;
- gateway starts with wrong `ATLAS_CLI`;
- wrong or missing `ATLAS_WIKI_DIR`;
- refresh on dynamic routes;
- wiki empty search;
- wiki weird slug / spaces / unicode;
- SSE reconnect / finished run;
- JSONL export with many events;
- browser narrow viewport;
- repeated create/cancel clicks;
- stale localStorage sidebar state.

### Task 9 — Idempotency / antifragility pass

Review and document replay/partial-failure surfaces:

- mission create duplicate/retry;
- run start twice;
- cancel twice;
- wiki update retry;
- audit event append;
- SSE reconnect and cursor replay;
- fresh DB bootstrap;
- browser reload during active run.

For each: canonical key, idempotency behavior, current state, required fix if unsafe.

### Task 10 — Public-facing docs polish

Check current public docs for:

- stale phase status;
- imported-source branding leakage;
- private local paths;
- claims like “production-ready” without evidence;
- unclear setup commands;
- missing limitations.

Update only docs needed for public hardening.

## Final report format

2. Files changed.
3. Security scan summary.
4. Test summary.
5. Manual UAT guide path.
6. Edge cases covered.

8. Next recommended phase.

## Stop condition

Stop when Phase 9.5 has either:

- completed and left `main` clean; or
- produced a clear blocker report saying exactly what prevents completion.
