# v1.2 Roadmap Draft — Provider Mesh & Runtime Interoperability

**Status:** Draft; not active and not counted by GSD
**Created:** 2026-06-24
**Activation gate:** v1.1 archived with cross-surface conformance complete
**Identifiers:** milestone-local `PM-*`; canonical phase numbers are assigned at activation

## Milestone outcome

ATLAS can discover, register, route to, and operate across model gateways and
external agent runtimes without surrendering auditability or implying controls
that an opaque runtime cannot actually enforce.

The milestone extends the existing `AgentRuntime` abstraction and D-017 model
router. It does not introduce a competing runtime ABC, import another product's
runtime/config/auth stack, or make self-evolution a release prerequisite.

## Operating model

ATLAS supports three explicit capability tiers:

1. **Model gateway:** ATLAS owns the agent loop, policy, permissions, workspace,
   normalized events, cancellation, and audit trail.
2. **Controlled agent runtime:** an external agent loop may execute through ATLAS
   only when it supports normalized events, cancellation, workspace binding, and
   an enforceable permission bridge.
3. **Opaque external runtime:** ATLAS may detect, inspect, launch read-only or
   advisory workflows, and record observable outcomes. ATLAS must not claim
   permission enforcement or provide write-capable sessions through this tier.

The operator may choose a global default mode/profile, but every project,
surface session, and run records an immutable effective runtime, capability tier,
model profile, and policy mode. Active sessions are never hot-switched.

## Draft phases

### PM-01 — Foundation Boundary Manifest & Upstream Delta

**Goal:** Make the foundation boundary explicit before it is exposed as a mesh backend.

**Scope:**

- classify each foundation module as `tracked`, `owned`, or `diverged-frozen`;
- record upstream path/ref and known scripted transforms;
- produce deterministic, machine-readable upstream-delta reports;
- verify that moving a module from `tracked` to `owned` removes it from the sync surface.

**Success bar:**

- no unclassified foundation modules;
- deterministic partitioning and reproducible output;
- no automatic apply or commit behavior in v1.2;
- foundation-facing product labels use ATLAS-native/foundation terminology;
  Hermes remains provenance, not operator-facing identity.

### PM-02 — Bidirectional WebUI Gap Audit

**Goal:** Protect ATLAS strengths and identify useful foundation UI capabilities before
provider-mesh UI contracts are frozen.

**Scope:**

- evidence-backed ATLAS-better and foundation-has/ATLAS-missing inventories;
- ranked `adopt`, `adapt`, and `skip` backlog;
- provider/profile/control-state recommendations for `PM-04` and `PM-06`;
- research only; no UI rebuild in this phase.

**Success bar:**

- every finding cites routes/files or rendered evidence;
- recommended adoption work has owner phase and rationale;
- no dependency on or mutation of completed v1.1 UI phases.

### PM-03 — Provider Mesh Contract & Dual-Mode Control Plane

**Goal:** Evolve the existing `AgentRuntime` contract and model router into one
capability-aware control plane.

**Scope:**

- version `AgentRuntime` capabilities rather than create `RuntimeAdapter`;
- register model gateways and agent runtimes with explicit capability tiers;
- persist global defaults and bind exact effective values to project/session/run;
- normalize attach, detach, route, fallback, and mode/profile audit events;
- add route trace, hop count, and cycle detection for ATLAS-to-ATLAS routing.

**Success bar:**

- existing `native` and `claude_code` implementations retain behavior;
- active sessions cannot be silently hot-switched;
- unsupported write/permission claims fail closed by capability tier;
- at least one external model gateway passes the versioned contract;
- route loops terminate deterministically and are auditable.

### PM-04 — Role-Keyed Model Rulebook

**Goal:** Resolve provider/model/effort by agent role through one versioned config contract.

**Scope:**

- versioned provider profiles in `~/.atlas/config.yaml`;
- a capability-aware function registry (`routed`, `inherited`, `unavailable`);
- role mappings for main agent, curator, and actual named auxiliary task classes;
- inherited-only visibility for background review until the foundation exposes a seam;
- normalized effort semantics with provider-specific translation;
- atomic CLI/WebUI editing through the existing control plane;
- immutable resolved bindings on each run in a dedicated binding snapshot (the existing
  prompt/tool/context `RunContractSnapshot` does not currently carry provider/model bindings).

**Success bar:**

- partial profiles fall back to explicit documented defaults;
- surfaces cannot offer an override for inherited/unavailable functions;
- secrets remain references and masked;
- concurrent edits use conflict detection;
- runs are replayable from stored profile/version/bindings.

### PM-05 — Capability, Cost & Health Scoring

**Goal:** Fill rulebook gaps and drive fallback using explainable, bounded scoring.

**Scope:**

- capability, cost, health, latency, and recent-failure signals;
- one shared provider/model test substrate: connectivity probe, per-function smoke, and
  10.2 evaluation-gate reuse;
- deterministic scorer inputs and tie-breaking;
- Phase 10.0 error-classification integration;
- cached, budgeted health data.

**Success bar:**

- manual role assignments always override scoring;
- each dynamic choice records inputs and rationale;
- dead providers demote safely;
- startup performs local/read-only discovery only: paid or network probes require
  opt-in, explicit budgets, timeouts, and cache policy;
- direct interactive probes report live/mock/failed, latency, sample output, and cleanup
  their throwaway mission state;
- added background work stays within D-022 runtime budgets.

### PM-06 — Provider Backends & Conformance

**Goal:** Ship useful backends only where their capability tier can be stated and tested honestly.

**Scope:**

- model gateways: OpenAI-compatible endpoints and compatible gateway products;
- controlled agent-runtime adapters selected by verified capabilities;
- optional read-only/advisory integrations for opaque runtimes;
- ATLAS-to-ATLAS routing with cycle guards;
- conformance fixtures for events, cancellation, workspace, permissions, audit,
  routing, and failure behavior.

**Success bar:**

- every shipped backend declares and passes its capability tier;
- write-capable controlled runtimes prove permission-bridge enforcement;
- opaque runtimes cannot enter write-capable ATLAS sessions;
- external MiMo/OpenCode/Codex integrations are adapters only: no donor runtime,
  config, auth, telemetry, or product identity is imported into ATLAS;
- every backend has bounded timeouts, cancellation behavior, and recovery guidance.

### PM-07 — Provider & Model Configuration Surfaces

**Goal:** Make provider wiring, model discovery, function assignment, and testing equally usable
from CLI, TUI, and WebUI without moving business logic into a surface.

**Scope:**

- expanded `atlas setup` flow (mode → credentials → default model → function assignments →
  tests → atomic save);
- scriptable `atlas models`, `atlas profiles`, and shared provider/model test commands;
- Go TUI provider/settings pane over the shared control plane and probe substrate;
- WebUI Models suite: provider mesh board, model catalog, function matrix, test results, and
  effective-resolution summary;
- consistent offline, mock, unavailable, inherited, budget-exhausted, and conflict states.

**Success bar:**

- every surface renders the same masked config, registry, resolution, and test contracts;
- secrets never enter argv, masked APIs, logs, or audit output;
- all mutations remain optimistic/conflict-aware and are audited by Python;
- the Rust gateway remains dispatch-only and Go/React remain render/input clients;
- Windows ASCII safety, keyboard access, responsive layout, and cross-surface conformance pass.

## Quality gates

Until a dedicated L2 extra-marathon skill exists in this workspace, completion uses
available, auditable gates:

- phase plan verification and requirement traceability;
- `gsd-code-review`;
- `gsd-secure-phase` for control-plane and backend phases;
- `gsd-verify-work` conversational UAT;
- `gsd-ui-review` where an operator-facing surface changes;
- measured dependency, startup, memory, probe, and binary/bundle budgets.

The delivery doctrine remains: detect locally before asking, prepare the next safe
action, keep the least invasive path as default, design failure states, and keep
automation reversible and audited.
