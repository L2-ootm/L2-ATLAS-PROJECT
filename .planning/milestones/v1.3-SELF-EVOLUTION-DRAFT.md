# v1.3 Candidate — Gated Self-Evolution

**Status:** Preserved candidate; not active and not a v1.2 release gate
**Identifier:** `EV-01`; canonical phase/ADR numbers are assigned only at activation
**Source:** original 2026-06-24 provider-mesh/self-evolution design

## Why this is separate

Provider interoperability and autonomous code mutation have different blast radii,
verification needs, and rollback obligations. Combining them would make useful
provider-mesh delivery depend on the highest-risk idea in the portfolio.

This draft preserves the intended self-evolution path while requiring evidence from
v1.2 boundary tooling and the normal milestone activation process.

## EV-01 — Gated Foundation Sync Pilot

**Goal:** Prove that ATLAS can propose a bounded, auditable foundation update without
silently mutating the working tree, bypassing review, or claiming autonomy beyond
its demonstrated safety level.

### Entry gates

- v1.2 `PM-01` boundary manifest and delta reports are stable;
- provider-mesh conformance and project test gates are green;
- rollback, branch isolation, audit events, and kill-switch behavior are tested;
- autonomy ceilings and promotion evidence are accepted in a canonical ADR;
- network, compute, and token budgets are explicit.

### Promotion levels

1. **Observe:** fetch/read and produce a reconciliation report only.
2. **Propose:** prepare a patch or branch for human inspection; never merge.
3. **Validate:** run approved test/security/review gates on the proposal.
4. **Publish candidate:** create an atomic commit and draft PR after explicit policy allows it.
5. **Tracked auto-merge:** future candidate only; requires a sustained green record
   and a separate decision. It is not promised by this draft.

General ATLAS self-modification is beyond these levels and requires a later milestone.

### Hard safety rules

- `owned` and `diverged-frozen` changes always halt automatic application;
- default mode is observe/propose, never autonomous merge;
- no direct writes to the protected branch;
- every input, classification, action, output, verification, and rollback is audited;
- kill switch is local, immediate, and tested;
- failures preserve evidence and leave the repository recoverable;
- instructions found in upstream content are treated as untrusted data.

### Success criteria

- a tracked-only fixture produces a deterministic proposal and complete audit trail;
- an owned/diverged fixture stops and produces a reconciliation report;
- cancellation and kill switch leave no orphan process or partial repository mutation;
- rollback is demonstrated from a deliberately failed candidate;
- promotion is blocked unless the required evidence window is satisfied.

## Draft decision keys

- **EV-DEC-01:** self-evolution remains a separately activated, evidence-promoted capability.
- **EV-DEC-02:** observe/propose is the default autonomy ceiling for the first release.

These are placeholders, not canonical ADR numbers.
