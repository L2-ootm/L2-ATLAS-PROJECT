# ATLAS v1.1 — Owned Authentication Architecture Seed

**Status:** seed architecture document.  
**Created:** 2026-06-15.  
**Reference baseline:** Hermes auth system and Codex-style local OAuth UX.  
**Critical constraint:** ATLAS must not mutate Codex auth files.

## 1. Purpose

v1.1 needs an ATLAS-owned authentication system.

The goal is not to copy Codex credentials. The goal is to provide ATLAS with its own local auth flows and credential store, inspired by Hermes/Codex UX patterns, so ATLAS can authenticate providers, refresh tokens, report status, and route models without depending on external CLI state as source of truth.

## 2. Current gap

Current ATLAS v1.0:

- has no `atlas auth` command;
- has no ATLAS-owned auth store;
- does not detect or report provider readiness comprehensively;
- does not detect local Codex status except through manual shell checks;
- does not own OAuth/device-code flows;
- does not manage credential pools;
- does not integrate auth status into model discovery or TUI.

## 3. Required principle

ATLAS may inspect external tools read-only, but ATLAS-owned auth must live under ATLAS-owned state.

Never:

- write to `~/.codex/auth.json`;
- write to `~/.codex/config.toml`;
- copy Codex tokens into repo files;
- print external or ATLAS secrets;
- treat external CLI auth as the only long-term source of truth.

Allowed:

- detect `codex --version`;
- detect existence of `~/.codex/auth.json` and `~/.codex/config.toml`;
- report `Codex: installed/auth-present`;
- recommend user run Codex auth if they want external Codex CLI;
- use ATLAS-owned OAuth/API key credentials for ATLAS runtime.

## 4. Hermes reference patterns to reuse

Hermes auth architecture has these useful patterns:

- provider registry with auth type and base URLs;
- `~/.hermes/auth.json` store;
- cross-process file locking;
- atomic writes;
- OAuth device code / external OAuth handlers;
- access-token refresh skew;
- runtime credential resolver;
- credential redaction/sanitization;
- `hermes auth add/list/remove`;
- `hermes doctor` integration;
- OAuth-over-SSH and manual paste fallback patterns;
- provider-specific status and remediation.

ATLAS should implement equivalent concepts in ATLAS namespace.

## 5. Proposed file layout

### Minimal v1.1 layout

```txt
~/.atlas/
  config.yaml
  auth.json
  logs/
  atlas.db
```

### Profile-ready layout

```txt
~/.atlas/
  active_profile
  profiles/
    default/
      config.yaml
      auth.json
      atlas.db
      logs/
```

Recommendation:

- If profile support is cheap, start profile-ready.
- If not, use minimal layout but wrap paths behind functions so migration is easy.

```json
    },

      "base_url": "http://127.0.0.1:1234/v1",

    "last_auth_status_check": "..."

```

Open question: inline encrypted/obfuscated JSON vs OS keychain.

v1.1 acceptable minimum:

- local file with restricted permissions and redacted output;
- no secrets committed;
- future keychain integration planned.

v1.1 preferred:

- Windows Credential Manager / keyring where feasible;
- auth.json stores references/metadata only.

## 7. Provider registry draft

Each provider config should include:

```python
ProviderConfig(
    id="openrouter",
    display_name="OpenRouter",
    auth_type="api_key",
    default_base_url="https://openrouter.ai/api/v1",
    env_vars=["OPENROUTER_API_KEY"],
    model_list_endpoint="/models",
    api_modes=["chat_completions"],
)
```

Provider attributes:

- id;
- display name;
- auth type;
- default base URL;
- env vars;
- config keys;
- model endpoint;
- supported API modes;
- health check strategy;
- OAuth/device flow support;
- docs/remediation URL;
- public/private risk flag.

## 8. Auth CLI commands

### `atlas auth status`

Purpose: one-screen readiness.

Output shape:

```txt
ATLAS auth store: C:\Users\Davi\.atlas\auth.json
Profile: default

Provider            Auth        Status          Source              Notes
openrouter          api_key     configured      ~/.atlas/auth.json  OK
anthropic           api_key     missing         env/auth store      run atlas auth add anthropic
codex-external      external    detected        ~/.codex            read-only, not ATLAS-owned
freellmapi          local       offline         127.0.0.1:3001      start sidecar
custom-local        noauth      available       127.0.0.1:1234      LM Studio detected
```

### `atlas auth add <provider>`

Add provider credential to ATLAS store.

Modes:

- API key prompt;
- local endpoint registration;
- OAuth device code if supported;
- manual paste fallback;
- no-auth local provider.

### `atlas auth list`

List providers and redacted hints.

### `atlas auth remove <provider>`

Remove credential safely.

### `atlas auth doctor`

Deep diagnostics:

- store exists;
- permissions;
- parse validity;
- token expiry;
- provider reachability;
- model endpoint reachability;
- secrets not in config/docs.

## 9. Codex external detection

Codex should be a status source, not an ATLAS auth store.

Detection:

```txt
codex binary on PATH
codex --version
~/.codex/auth.json exists
~/.codex/config.toml exists
```

Status values:

- `not_installed`
- `installed_no_auth`
- `installed_auth_present`
- `installed_config_present`
- `unknown_error`

Do not parse/print token payload unless a redacted, schema-safe parser is explicitly reviewed.

If parsing is needed later, only extract non-secret metadata:

- account email if non-sensitive and user accepts;
- expiry timestamp;
- plan/entitlement if available through safe API call.

## 10. OAuth clone feasibility

Davi requested a "clone do auth do codex para propria autenticação".

That requires legal/API feasibility review.

Possible meanings:

1. **UX clone:** same local browser/device-code flow style, but for ATLAS providers. Safe and desired.
2. **Protocol clone:** use same OpenAI/Codex OAuth endpoints/client. Must be reviewed; may be disallowed or unstable.
3. **Credential import:** copy Codex tokens into ATLAS. Not acceptable by default.

Recommended stance:

- implement UX clone and ATLAS-owned store;
- do not copy Codex tokens;
- only use Codex endpoints if documented/allowed or already supported by Hermes in a way we can legally inherit inside the vendored foundation;
- document fallback providers.

## 11. Runtime credential resolver

ATLAS needs a resolver that returns:

```python
RuntimeCredentials(
    provider_id="openrouter",
    base_url="https://openrouter.ai/api/v1",
    api_key="...",
    api_mode="chat_completions",
    model="...",
    source="atlas_auth_store",
    expires_at=None,
)
```

Resolver order draft:

1. explicit CLI flags;
2. ATLAS auth store;
3. ATLAS config env references;
4. local sidecar/no-auth providers;
5. read-only external tool status for suggestions only;
6. fail with actionable remediation.

## 12. Security requirements

- Atomic writes.
- Cross-process lock.
- Restricted file permissions where supported.
- No secret values in stdout/stderr/logs.
- Redaction tests.
- No secrets in git diff.
- No auth store in repo.
- No arbitrary OAuth callback host; loopback only.
- Manual paste fallback for SSH/non-browser environments.
- Clear revocation/removal command.

## 13. Tests to require

- auth store read/write roundtrip;
- malformed auth file fails safely;
- concurrent write locking;
- redacted output;
- Codex detection with fake home dir;
- no mutation of fake `.codex` files;
- provider status table formatting;
- expired token status;
- OAuth callback validation if implemented;
- Windows path behavior.

## 14. Integration with models

`atlas models discover` must consume auth status.

Example:

```txt
openrouter/anthropic/claude-sonnet-4 | openrouter | auth=configured | status=available
codex/gpt-5.1-codex                  | codex-external | auth=external-present | status=read-only-detected
lmstudio/local-model                 | lmstudio | auth=noauth | status=available
```

## 15. Integration with TUI/native shell

TUI should show auth readiness before first prompt.

Native shell should show auth/model status in a panel and route remediation to:

```bash
atlas auth add <provider>
atlas auth status
atlas models discover
```

## 16. Definition of done

Auth work is v1.1-complete when:

1. ATLAS owns an auth store.
2. `atlas auth status` is useful and redacted.
3. Codex is detected read-only.
4. At least one provider can be added to ATLAS store.
5. `atlas chat -q` can use ATLAS-owned credentials.
6. Model discovery consumes auth state.
7. Tests prove no Codex mutation and no secret leakage.
