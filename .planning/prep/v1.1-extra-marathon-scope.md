# v1.1 Extra-Marathon Scope — Real ATLAS Harness, Not Just Native Shell

**Created:** 2026-06-15  
**Status:** expanded preparation input for `/gsd-new-milestone`  
**Position:** after v1.0 archive, before formal v1.1 milestone creation

## 0. Executive correction

The prior v1.1 preparation was still too small.

Davi's corrected expectation is not merely:

- add `atlas chat`;
- detect Codex auth;
- list more models;
- wrap the cockpit in Tauri.

The real v1.1 expectation is:

> Build the first credible ATLAS-owned local agent harness, based on Hermes, with its own TUI, its own authentication store/flows, its own provider/model registry, its own chat/agent loop surface, and a native shell that can visually host and operate that harness.

This is more than a CLI patch. It is the transition from **cockpit MVP** to **ATLAS operator runtime**.

## 1. Strategic thesis

v1.0 proved that ATLAS can persist missions, runs, audit events, wiki pages, and show them in a cockpit.

v1.1 must prove that ATLAS can be used as the operator's actual local AI command environment.

That means:

1. **Agent TUI parity:** ATLAS needs a terminal UI equivalent in seriousness to Hermes' TUI, not only Typer help panels — in short, a Hermes-class ATLAS TUI.
2. **ATLAS-owned auth:** ATLAS should not depend on Codex auth files or mutate Codex state. It needs its own auth model/store, inspired by Hermes and compatible with provider APIs, with Codex read-only detection only.
3. **Provider/model truth:** ATLAS must know what providers, models, credentials, limits, and routes exist locally.
4. **Agentic chat and tool loop:** ATLAS must expose a real agentic conversation surface, not only mission database commands.
5. **Native shell as wrapper, not substitute:** the Tauri shell should host a real harness, not paper over a missing one.
6. **Operator trust:** no secrets leaked, no uncontrolled side effects, no fake model availability, no unclear auth state.

## 2. What v1.0 is allowed to remain

v1.0 remains:

> **Operator Cockpit MVP** — mission/run/wiki/audit/cockpit loop, archived.

But v1.0 must not be described as:

- complete ATLAS harness;
- complete ATLAS CLI;
- complete AI operator terminal;
- full Hermes-derived runtime parity;
- native shell product.

## 3. v1.1 proposed title

Recommended milestone name:

> **v1.1 — ATLAS Agent Harness & Native Operator Shell**

Alternative if narrower:

> **v1.1 — ATLAS TUI, Auth, Models & Native Shell**

Avoid:

> Native Cockpit Shell

because that hides the real missing foundation.

## 4. Non-negotiable v1.1 outcomes

v1.1 is not complete until Davi can:

1. run an ATLAS-branded TUI, visually distinct from raw Hermes;
2. authenticate ATLAS-owned providers without touching Codex's local auth state;
3. see auth status for every relevant provider without secrets;
4. discover and list real models from configured providers and local sidecars;
5. run one-shot and interactive agentic chat from ATLAS;
6. run the same chat/agent loop inside the native shell terminal pane;
7. see model/auth readiness in the cockpit/native shell;
8. trust that all actions are audit-safe and no credentials leak.

## 5. Major workstreams

### Workstream A — ATLAS TUI

Build/brand a real ATLAS terminal UI based on Hermes' TUI architecture.

Hermes reference:

- `foundation/atlas-hermes/ui-tui/`
- `foundation/atlas-hermes/tui_gateway/server.py`
- `foundation/atlas-hermes/cli.py`
- `foundation/atlas-hermes/hermes_cli/main.py`

Expected ATLAS behavior:

```bash
atlas
atlas tui
atlas chat
atlas chat -q "summarize current project state"
```

TUI must support:

- transcript;
- composer;
- streaming response;
- tool call activity;
- subagent/task activity;
- model/provider display;
- auth warnings;
- slash commands or ATLAS command equivalents;
- mission context awareness;
- session resume;
- safe exit;
- clear errors when gateway/auth/model unavailable.

### Workstream B — ATLAS-owned auth

Do not rely on or mutate Codex's auth files.

Codex can be detected as an external installed tool, but ATLAS auth must live under ATLAS-owned state.

Proposed state location:

```txt
~/.atlas/auth.json
~/.atlas/config.yaml
~/.atlas/credentials/         # if credential-pool files become necessary
```

or, if profile support is desired immediately:

```txt
~/.atlas/profiles/default/auth.json
~/.atlas/profiles/default/config.yaml
```

Hermes reference concepts:

- `~/.hermes/auth.json`
- provider registry;
- OAuth device code / external OAuth support;
- refresh-token handling;
- cross-process file locking;
- atomic writes;
- redacted status output;
- runtime credential resolution;
- credential pools.

ATLAS auth must provide:

```bash
atlas auth
atlas auth add <provider>
atlas auth list
atlas auth status
atlas auth remove <provider>
atlas auth doctor
```

Minimum providers/status targets:

- ATLAS-owned OpenAI/Codex-compatible OAuth path if legally/API-feasible;
- OpenRouter API key;
- Anthropic API key/OAuth if used;
- Google/Gemini API key/OAuth if used;
- FreeLLMAPI local sidecar key/admin auth if used;
- custom OpenAI-compatible endpoint;
- local no-auth providers such as LM Studio/Ollama if adopted.

### Workstream C — Codex clone/auth inspiration without Codex mutation

Davi specifically asked for a Codex-auth-like clone for ATLAS' own auth.

Interpretation:

- ATLAS should learn from Codex/Hermes auth UX and token storage patterns.
- ATLAS should not modify `~/.codex/auth.json` or `~/.codex/config.toml`.
- ATLAS may detect Codex installation/status for operator awareness.
- ATLAS should store ATLAS credentials in `~/.atlas`, not in Codex.
- ATLAS should never copy Codex secrets into the repo.

Potential UX:

```bash
atlas auth add openai-codex-compatible
atlas auth add openrouter
atlas auth add anthropic
atlas auth add custom --base-url http://127.0.0.1:1234/v1
```

If OpenAI's Codex OAuth flow cannot be legally/technically reused for ATLAS directly, document that and fall back to:

- API-key OpenAI provider;
- OpenRouter;
- Hermes-owned provider bridge;
- explicit external Codex CLI integration as optional, read-only detection only.

### Workstream D — Real model and provider registry

The current v1.0 registry is too shallow.

v1.1 needs separate concepts:

| Concept | Meaning |
|---|---|
| Provider | Auth/runtime backend: OpenRouter, Anthropic, OpenAI, Codex-compatible, FreeLLMAPI, local LM Studio, custom endpoint |
| Credential | Secret/token/API key/OAuth refresh state, owned by ATLAS |
| Runtime | How ATLAS calls provider: OpenAI chat, Responses, Anthropic messages, Codex-compatible, local sidecar |
| Model | Concrete model ID usable for inference |
| Route | Task-class policy mapping to model/provider |
| Health | Online/offline/auth-expired/rate-limited/unknown |

Candidate commands:

```bash
atlas providers list
atlas providers status
atlas models discover
atlas models list --all
atlas models doctor
atlas route show
atlas route set coding <provider>/<model>
```

Model registry must include:

- source;
- provider;
- model_id;
- api_mode;
- auth_status;
- health_status;
- first_seen;
- last_seen;
- deactivated_at if unavailable;
- cost/free-tier notes if known;
- context window if known;
- task suitability tags if known.

### Workstream E — Agent runtime bridge

The ATLAS CLI/TUI must call a real agent runtime.

Options:

1. **Thin wrapper over Hermes AIAgent** — fastest and least duplication.
2. **ATLAS adapter around Hermes AIAgent** — preferred medium-term; isolates ATLAS mission/audit semantics.
3. **New ATLAS agent loop** — avoid for v1.1 unless required; high risk and duplicates Hermes.

Recommendation:

- v1.1 uses a thin ATLAS runtime adapter over Hermes AIAgent.
- Every agentic call emits ATLAS audit events where practical.
- Mission-bound chats can create/run/update mission state.
- Free chats can remain sessions but should still be auditable.

### Workstream F — Native shell / PTY

Native shell remains important, but it should wrap the real harness.

Scope:

- Tauri 2 / Rust, no Electron;
- embed existing cockpit web app;
- PTY terminal pane running `atlas` / `atlas tui`;
- safe IPC allowlist;
- OS keychain integration if auth storage matures enough;
- native approval prompts;
- model/auth status pane;
- threat model before privileged IPC.

### Workstream G — Operator UX and design system

ATLAS TUI/native shell should feel like ATLAS, not stock Hermes or raw Codex.

Must define:

- TUI name/banner;
- colors/skin;
- prompt/composer language;
- model/status bar;
- mission context display;
- approval prompts;
- error/remediation language;
- no imported-source branding leaks.

### Workstream H — Security, privacy, audit

Auth and agent execution make v1.1 higher risk than v1.0.

Must include:

- no credential values in CLI output;
- auth file permissions/locking/atomic writes;
- redaction tests;
- no copying `~/.codex` or `~/.hermes` secrets into repo;
- threat model for OAuth callback server;
- threat model for native IPC;
- audit events for model calls and external actions;
- user approval gates for dangerous tools/commands.

## 6. Expanded acceptance gates

v1.1 should not close unless these pass:

### TUI gates

- `atlas` or `atlas tui` opens ATLAS-branded TUI.
- TUI can send a prompt and stream/display response.
- TUI displays model/provider/auth status.
- TUI supports at least `/help`, `/model` or equivalent command surface.
- TUI exits cleanly and preserves session state.

### Auth gates

- `atlas auth status` shows ATLAS-owned auth store.
- `atlas auth add` works for at least one real provider or a documented local/custom provider path.
- Codex installation is detected read-only.
- ATLAS does not mutate `~/.codex`.
- Auth output passes redaction tests.

### Model gates

- `atlas models discover` does not rely only on seed rows.
- `atlas models list --all` shows source/status/auth columns.
- Offline/unauthenticated sources are shown honestly.
- Cockpit/native model page matches CLI truth.

### Agent gates

- `atlas chat -q "ping"` returns a real response or precise auth remediation.
- Interactive chat works through CLI/TUI.
- At least one agentic tool path is visible/auditable.
- Model calls produce audit metadata.

### Native gates

- Tauri shell launches without Electron.
- Shell embeds cockpit.
- Shell hosts PTY and runs `atlas --help` / `atlas tui`.
- IPC threat model exists.
- No secrets appear in UI/logs/screenshots.
