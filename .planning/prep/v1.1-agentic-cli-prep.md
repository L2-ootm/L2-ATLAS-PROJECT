# v1.1 Preparation — Agentic CLI, Auth, Models, Native Shell

**Date opened:** 2026-06-15  
**Status:** preparation notes — not yet an approved milestone plan  
**Source trigger:** Davi inspected the v1.0 CLI after archive and rejected it as insufficient for a real ATLAS/Hermes-derived harness.

## 1. Executive verdict

The v1.0 archive remains valid as **Operator Cockpit MVP**.

But v1.0 is **not** sufficient as an ATLAS agentic CLI/harness release. The visible CLI is currently a thin operational surface over mission/wiki/model DB commands. It does not yet expose the Hermes-derived agent runtime, local Codex auth, provider discovery, or an interactive/one-shot agentic chat flow.

This must become a hard v1.1 planning input.

## 2. What Davi expected to see

Davi expected the local CLI to behave like the beginning of a real ATLAS harness:

```txt
atlas --help
atlas chat
atlas chat -q "..."
atlas agent
atlas auth status
atlas models discover
atlas models list --all
atlas doctor
```

And he expected it to detect the local AI environment, especially:

- local Codex CLI installation;
- local Codex auth files;
- Hermes/ATLAS provider config;
- real model inventory beyond three seeded records;
- a usable agentic chat entrypoint with the installed/authenticated models.

This means Codex detection is possible and should not require new user setup. The ATLAS CLI simply does not surface it yet.

## 5. Root cause

The system currently has two separate layers:

| Layer | Path | Current role |
|---|---|---|
| ATLAS operational CLI | `services/agent-runtime/atlas_runtime/cli/main.py` | Mission/wiki/models/channels/foundation status wrappers |
| Hermes-derived foundation | `foundation/atlas-hermes/` | Real agent harness, auth resolver, providers, chat/TUI/tools |

The bridge between them is incomplete. `atlas_runtime` can report foundation status, but does not expose the foundation's agentic runtime as ATLAS-branded commands.

## 6. Required v1.1 framing change

The next milestone should not be framed only as "Native Cockpit Shell".

Recommended v1.1 milestone name:

> **v1.1 — Agentic CLI + Native Operator Shell**

Reason: the native shell/PTY work is not credible unless the underlying CLI/harness can actually chat, detect auth, discover models, and invoke the agent runtime.

## 7. Proposed v1.1 pillars

### Pillar A — ATLAS agentic CLI surface

Create real ATLAS-branded CLI commands over the Hermes-derived foundation:

```txt
atlas chat
atlas chat -q "..."
atlas agent
atlas auth status
atlas models discover --include-local-auth
atlas models list --all
atlas doctor
```

Non-negotiable behavior:

- no secret values printed;
- graceful offline/auth-missing states;
- exact provider/model source shown;
- one-shot chat works before native shell work is considered complete;
- interactive chat is available either directly or via a clear handoff into the foundation TUI.

### Pillar B — Codex and provider auth detection

ATLAS should detect local auth like Hermes does:

- `codex` binary on PATH;
- `~/.codex/auth.json` present/not present;
- `~/.codex/config.toml` present/not present;
- token refresh/credential status via safe resolver if available;
- Hermes config/provider availability;
- OpenRouter/custom provider config;
- no credential leakage.

Target command:

```bash
atlas auth status
```

Expected shape:

```txt
Codex: installed, auth present
Hermes profile: default
OpenRouter: configured / not configured
Anthropic: configured / not configured
Google: configured / not configured
Custom providers: N configured
```

### Pillar C — Model registry upgrade

Current `models list` only reads the SQLite registry. v1.1 should merge several model sources:

1. seeded/default registry rows;
2. FreeLLMAPI `/v1/models` if sidecar is live;
3. Hermes/ATLAS configured providers;
4. Codex local authenticated runtime;
5. OpenRouter/custom providers;
6. runtime health/source metadata.

Target commands:

```bash
atlas models discover
atlas models discover --include-local-auth
atlas models list --all
atlas models list --source codex
atlas models doctor
```

Expected row schema:

```txt
model_id | provider | source | status | auth | last_seen
```

Source vocabulary:

```txt
seeded
gateway
freellmapi
codex-auth
hermes-config
openrouter
custom
```

Status vocabulary:

```txt
available
auth-present
needs-login
offline
unknown
```

### Pillar D — Native shell / PTY / cockpit integration

After the CLI/harness is credible, wrap it in the native shell:

- Tauri 2 / Rust shell;
- embedded PTY terminal pane;
- cockpit webview wraps existing SvelteKit app;
- OS keychain integration;
- native approval prompts;
- capability-scoped IPC;
- threat model before enabling privileged commands.

The native shell should call the same ATLAS CLI/runtime paths, not a parallel implementation.

## 8. Proposed v1.1 requirements draft

These are draft requirement IDs for `/gsd-new-milestone` refinement.

### CLI

- **CLI-01:** User can run `atlas --help` and see an ATLAS-branded command tree, not only `python -m ...`.
- **CLI-02:** User can run `atlas chat -q "..."` and receive an agentic response using an available configured provider.
- **CLI-03:** User can start an interactive agentic chat session from the ATLAS CLI.
- **CLI-04:** CLI exits with clear, actionable errors when no provider/auth is available.
- **CLI-05:** CLI command outputs avoid leaking secrets, tokens, local private paths, or raw credentials.

### Auth

- **AUTH-01:** ATLAS detects local Codex installation and reports version/status.
- **AUTH-02:** ATLAS detects Codex auth/config files without printing secret material.
- **AUTH-03:** ATLAS detects Hermes/ATLAS profile provider configuration and reports provider availability.
- **AUTH-04:** `atlas auth status` reports provider readiness in one screen.

### Models

- **MODELS-01:** `atlas models discover` merges seeded, gateway, Hermes config, Codex, OpenRouter/custom, and FreeLLMAPI sources where available.
- **MODELS-02:** `atlas models list --all` shows source/status/auth metadata.
- **MODELS-03:** cockpit Models page shows the same source/status/auth distinctions.
- **MODELS-04:** registry refresh is idempotent and never deletes historical model rows needed for audit.

### Native shell

- **NATIVE-01:** Tauri 2 shell wraps the existing SvelteKit cockpit without duplicating UI logic.
- **NATIVE-02:** Native shell exposes a PTY pane capable of running the ATLAS CLI.
- **NATIVE-03:** IPC surface is capability-scoped and threat-modeled before privileged commands ship.
- **NATIVE-04:** Native shell can show auth/model status and route the user to remediation.
- **NATIVE-05:** Native shell remains local-first and loopback-only unless an explicit integration requires otherwise.

## 9. Suggested phase split

### Phase 10.0 — ATLAS CLI/harness bridge

Goal: expose foundation agent/chat capabilities through ATLAS-branded CLI commands.

Deliverables:

- `atlas` command works directly;
- `atlas chat -q` works;
- `atlas auth status` exists;
- `atlas doctor` exists;
- tests for no-secret output.

### Phase 10.1 — Model/provider discovery

Goal: real provider/model inventory, including Codex auth and Hermes config.

Deliverables:

- Codex detection;
- provider discovery layer;
- merged registry schema or view;
- cockpit Models page upgraded.

### Phase 10.2 — Native shell foundation

Goal: Tauri 2 shell wraps cockpit and can host a PTY pane.

Deliverables:

- Tauri app scaffold;
- web cockpit embedded;
- PTY proof;
- IPC threat model.

### Phase 10.3 — Native operator loop

Goal: CLI + cockpit + PTY + approvals behave as one operator surface.

Deliverables:

- launch mission/run from shell;
- inspect run stream;
- open CLI pane against same repo/runtime;
- auth/model state visible;
- manual UAT guide.

## 10. Acceptance gates

v1.1 should not be considered complete unless all are true:

1. `atlas chat -q "ping"` works or gives a precise auth remediation.
2. `atlas auth status` detects local Codex and Hermes provider state.
3. `atlas models list --all` shows more than seeded rows when local providers exist.
4. Native shell can visually host the cockpit and a terminal/CLI pane.
5. No secrets are printed in CLI, logs, cockpit, JSON exports, or docs.
6. Manual UAT verifies both terminal CLI and visual shell path.

## 11. Open questions for Davi before formal milestone creation

1. Should v1.1 prioritize **agentic CLI first** or **Tauri shell first**?
   - Recommendation: agentic CLI first, because the shell needs a real command surface to host.
2. Should `atlas chat` wrap Hermes foundation directly or call a new ATLAS runtime abstraction?
   - Recommendation: thin wrapper over foundation first, then progressively extract stable ATLAS interfaces.
3. Should Codex be treated as a first-class provider in ATLAS v1.1?
   - Recommendation: yes, detection/status first; full routing after safe resolver review.
4. Should the public v1.0 claim be amended?
   - Recommendation: yes: call it **Operator Cockpit MVP**, not complete ATLAS harness.
