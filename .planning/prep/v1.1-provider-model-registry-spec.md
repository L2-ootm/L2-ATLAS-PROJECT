# ATLAS v1.1 — Provider, Model, Route, and Runtime Registry Specification Seed

**Status:** seed spec for planning.  
**Created:** 2026-06-15.  
**Problem:** v1.0 `models list` showed only three seeded rows and did not reflect real local/provider availability.

## 1. Objective

Build a truthful provider/model/runtime inventory for ATLAS.

The registry must answer:

1. What providers exist?
2. Which are authenticated/configured?
3. Which are online/offline?
4. Which models are actually callable?
5. Which API mode/runtime adapter is needed?
6. Which model should be used for a given task class?
7. What is safe to show in CLI/TUI/cockpit?

## 2. Current limitation

Current v1.0 model registry:

- stores model rows in SQLite;
- can refresh from one OpenAI-compatible `/models` endpoint;
- lists active rows;
- does not represent auth status;
- does not represent provider status;
- does not represent API mode;
- does not merge local auth/config/sidecar/external tool signals;
- does not distinguish seeded defaults from discovered availability.

## 7. Discovery algorithm draft

1. Load ATLAS auth/config.
2. Load ATLAS provider registry.
3. Check local sidecar endpoints with short timeouts.
4. Check configured remote providers only when auth exists.
5. Check external tools read-only (`codex`, possibly `claude`, `gemini`) but do not import/mutate secrets.
6. Query `/models` endpoints when safe.
7. Normalize model IDs and providers.
8. Upsert registry rows.
9. Mark missing models inactive/deactivated per source, not globally.
10. Emit summary with added/retained/deactivated/offline sources.

## 8. SQLite schema direction

Current table may be insufficient.

Candidate new tables:

```sql
provider_registry(
  provider_id TEXT PRIMARY KEY,
  display_name TEXT NOT NULL,
  auth_type TEXT NOT NULL,
  default_base_url TEXT,
  api_modes_json TEXT NOT NULL,
  source TEXT NOT NULL,
  status TEXT NOT NULL,
  last_checked TEXT,
  last_error TEXT
)

model_registry_v2(
  model_id TEXT NOT NULL,
  provider_id TEXT NOT NULL,
  source TEXT NOT NULL,
  api_mode TEXT NOT NULL,
  status TEXT NOT NULL,
  auth_status TEXT NOT NULL,
  context_window INTEGER,
  metadata_json TEXT NOT NULL DEFAULT '{}',
  first_seen TEXT NOT NULL,
  last_seen TEXT NOT NULL,
  deactivated_at TEXT,
  PRIMARY KEY(model_id, provider_id, source)
)

route_policy(
  task_class TEXT PRIMARY KEY,
  provider_id TEXT NOT NULL,
  model_id TEXT NOT NULL,
  fallback_policy_json TEXT NOT NULL DEFAULT '{}',
  updated_at TEXT NOT NULL
)
```

Do not destroy old `model_registry`; migrate or view over it.

## 9. UI requirements

### CLI/TUI

Must show:

- source;
- status;
- auth;
- remediation.

### Cockpit Models page

Should include:

- provider cards;
- model table;
- source/status badges;
- offline/auth-needed states;
- refresh button if safe;
- no mutation of secrets from browser unless native keychain/IPC is ready.

### Native shell

Can add:

- auth/model readiness panel;
- action buttons that open terminal command (`atlas auth add ...`) or native keychain prompt later.

## 10. Safety requirements

- Remote provider discovery only when credential exists and user expects it.
- Short timeouts for local sidecars.
- No model discovery request leaks API keys in logs.
- No secrets in JSONL exports.
- No accidental calls to paid inference endpoints just to list models unless provider's `/models` endpoint is safe and documented.
- External read-only tools remain advisory unless user explicitly imports/links.
