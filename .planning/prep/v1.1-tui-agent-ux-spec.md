# ATLAS v1.1 — TUI and Agent UX Specification Seed

**Status:** seed spec for planning; not final implementation plan.  
**Created:** 2026-06-15.  
**Reference baseline:** Hermes TUI (`foundation/atlas-hermes/ui-tui/` + `tui_gateway/server.py`).

## 1. Why this exists

The v1.0 CLI visual inspection showed a fundamental gap: ATLAS has a browser cockpit and a thin operational CLI, but not a serious terminal-native agent interface.

Hermes already has a real TUI with transcript, composer, gateway, tool progress, slash commands, and session semantics. ATLAS should not ignore that. v1.1 should create an ATLAS TUI based on Hermes' architecture, adapted to ATLAS' mission/audit/model/provider concepts.

## 2. Product principle

ATLAS TUI is not a generic chat clone.

It is the terminal-native cockpit for:

- agentic conversation;
- mission context;
- model/provider visibility;
- tool execution visibility;
- approval and safety;
- audit-aware work.

## 3. Hermes architecture reference

Observed/known Hermes components:

| Component | Path | Role |
|---|---|---|
| Ink TUI frontend | `foundation/atlas-hermes/ui-tui/` | React/Ink terminal UI |
| TUI entry | `ui-tui/src/entry.tsx` | starts terminal app |
| Main app | `ui-tui/src/app.tsx` | composes layout, transcript, composer, status |
| Gateway client | `ui-tui/src/gatewayClient.ts` | JSON-RPC client to Python gateway |
| Python TUI gateway | `foundation/atlas-hermes/tui_gateway/server.py` | JSON-RPC backend, session dispatch, slash command workers |
| Agent loop | `foundation/atlas-hermes/run_agent.py` | AIAgent core loop |
| CLI shell | `foundation/atlas-hermes/cli.py` | interactive CLI orchestration |
| CLI entry | `foundation/atlas-hermes/hermes_cli/main.py` | argument parsing, TUI launch, commands |

Key design lessons:

- TUI should communicate over structured JSON-RPC, not parse terminal output.
- Python stdout must be reserved for protocol; stray prints go to stderr/activity.
- Long-running handlers need worker threads/pool so interrupts/approvals remain responsive.
- TUI and CLI model/provider state must stay synchronized.
- Slash commands should have a single registry of truth.
- Session resume/branch/compress are first-class UX, not afterthoughts.

## 4. ATLAS TUI command targets

Minimum commands:

```bash
atlas
atlas tui
atlas chat
atlas chat -q "..."
atlas --tui
```

Recommended behavior:

| Command | Behavior |
|---|---|
| `atlas` | opens ATLAS TUI by default once v1.1 is ready |
| `atlas tui` | explicit TUI launch |
| `atlas chat` | interactive chat; may alias to TUI or text chat |
| `atlas chat -q` | one-shot non-interactive agent call |
| `atlas --no-tui chat` | fallback plain terminal mode if implemented |

## 5. Required TUI surfaces

### 5.1 Header/status bar

Must show:

- ATLAS brand;
- active profile/store (`~/.atlas` profile);
- provider/model;
- auth status summary;
- mission/session context;
- gateway/local runtime state;
- token/usage summary if available.

Example:

```txt
ATLAS v1.1 | profile: default | model: openrouter/claude-sonnet | auth: OK | mission: none | tools: enabled
```

### 5.2 Transcript

Must show:

- user messages;
- assistant responses;
- tool call summaries;
- agent/subagent activity;
- errors/remediation;
- audit event references when applicable.

### 5.3 Composer

Must support:

- multi-line input;
- submit/cancel;
- slash/command suggestions;
- pasted text;
- clear indicator when message is queued/busy.

### 5.4 Activity panel

Must show:

- model call started/finished;
- tool calls;
- terminal commands;
- file writes;
- web calls;
- approvals pending;
- background process status.

### 5.5 Mission context panel

ATLAS-specific addition beyond Hermes:

- current mission ID/title/status;
- current run ID/status;
- recent audit event count;
- wiki artifact links/slugs;
- option to bind/free current chat to a mission.

Candidate commands:

```txt
/mission
/mission create
/mission bind <id>
/mission run
/mission status
/audit
/wiki
```

### 5.6 Auth/model panel

Must show:

- configured providers;
- auth present/missing/expired;
- selected model;
- source of model availability;
- remediation command.

Candidate command:

```txt
/auth
/models
/model <provider/model>
/doctor
```

## 7. ATLAS TUI architecture options

### Option A — Fork/adapt Hermes TUI into ATLAS namespace

Pros:

- fastest path to visual parity;
- known architecture;
- can brand and add ATLAS panels;
- preserves terminal-native strengths.

Cons:

- divergence from Hermes needs maintenance;
- TypeScript/Ink build complexity moves into ATLAS release.

### Option B — Keep Hermes TUI as vendored component and skin/extend it

Pros:

- less code drift;
- easier upstream updates;
- can start quickly.

Cons:

- may leak Hermes concepts/branding;
- ATLAS-specific mission panels may be hard to add cleanly.

### Option C — Build new Rust TUI

Pros:

- aligns Rust-first strategy;
- potentially simpler native packaging.

Cons:

- repeats Hermes TUI work;
- high risk for v1.1;
- delays auth/model/agent work.

Recommendation:

> Start with Option B or A. Do not build a new Rust TUI for v1.1 unless Hermes TUI reuse proves blocked.

## 8. ATLAS TUI gateway requirements

The gateway between TUI and runtime should support:

- `session.start`
- `session.resume`
- `session.send`
- `session.interrupt`
- `session.status`
- `auth.status`
- `models.list`
- `models.select`
- `mission.status`
- `mission.bind`
- `run.status`
- `audit.tail`
- `tool.approve`
- `tool.deny`

All methods must return structured JSON and stable error codes.

## 9. Integration with native shell

The Tauri shell should be able to host either:

1. a PTY running `atlas tui`; or
2. native panels consuming the same JSON-RPC backend.

v1.1 should prefer PTY first because it is simpler and validates the TUI directly.

Later versions can expose richer native React/Svelte panels around the same backend.

## 10. UX quality bar

TUI is acceptable only if:

- it looks intentionally ATLAS-branded;
- it does not feel like a raw debug CLI;
- errors are human-actionable;
- auth/model state is visible before first prompt;
- tool activity is understandable;
- mission/run/audit context is visible without leaving TUI;
- it handles offline/no-auth states gracefully;
- it does not expose secrets.

## 11. Testing requirements

Add tests for:

- command registry consistency;
- TUI gateway JSON-RPC schema;
- auth/model status payload redaction;
- `atlas chat -q` one-shot;
- TUI launch smoke if feasible;
- no Hermes branding leaks where ATLAS brand is required;
- Windows terminal behavior.

Manual UAT must include screenshots or terminal captures.

## 12. Open design questions

1. Should `atlas` default to TUI or command help in v1.1?
   - Recommendation: default to TUI after stable; `atlas --help` remains help.
2. Should `atlas chat` open TUI or plain chat?
   - Recommendation: `atlas chat` opens interactive TUI/plain session; `atlas chat -q` one-shot.
3. Should Hermes slash commands be inherited wholesale?
   - Recommendation: inherit core session/tool/model commands, add ATLAS mission/audit/wiki commands, hide irrelevant ones.
4. Should TUI be packaged inside ATLAS or referenced from vendored Hermes?
   - Recommendation: start from vendored Hermes, then decide whether to fork namespace after first working prototype.

## 13. Definition of done

Davi opens:

```bash
atlas
```

and sees a real ATLAS agent terminal, not a list of internal service commands.

He can ask a question, switch/inspect models, see auth status, bind to a mission, watch tool/audit activity, and exit without broken state.
